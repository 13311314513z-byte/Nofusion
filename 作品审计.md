# NoFusion / InkOS 作品审计可视化低成本实施报告

## 1. 目标与约束

本方案目标是在现有 NoFusion / InkOS Studio 中增加一个独立的“作品审计”页面，使审计结果不再只是章节按钮弹窗或后台日志，而是可以在前端直观看到章节质量、问题分布、审计进度与修复优先级。

本方案坚持三个原则：

1. **低成本、最小改动**：优先复用现有 `ContinuityAuditor`、`auditIssues`、`reviewNote`、`audit_history.jsonl`、服务商配置与 SSE 事件。
2. **审计与写作解耦**：审计页面、审计 API、审计模型配置应独立于写作流程，避免“同一模型自写自审”造成的幻觉确认。
3. **前端可解释**：审计页面不只显示“通过/失败”，而要展示维度、趋势、问题严重度、章节定位与建议操作。

## 2. 当前架构基础

当前项目已经具备审计基础能力：

- 后端已有 `POST /api/v1/books/:id/audit/:chapter`，可触发章节审计。
- Core 已有 `ContinuityAuditor`，返回 `AuditResult`，包含 `passed`、`issues`、`summary`、`overallScore`。
- 章节索引 `ChapterMeta` 已保存 `auditIssues`、`reviewNote`、`status` 等字段。
- Core 已有 `audit_history.jsonl` 机制，位置为 `books/<bookId>/story/audit_history.jsonl`。
- 前端书籍详情页已有单章“审计”按钮，但目前只是弹窗提示，没有独立可视化页面。
- 左侧主导航已有服务商、文风、导入、市场雷达、环境诊断等独立入口，可以低成本增加“作品审计”入口。
- 服务商管理已支持保存不同服务商的 API Key，后端已有 `.inkos/secrets` 读写能力。

当前主要缺口：

- 审计结果没有独立页面。
- 审计维度没有聚合可视化。
- 审计 API 使用当前项目写作 LLM 配置，无法强制与写作模型/API Key 隔离。
- 审计历史未暴露给前端。
- 批量审计与进度可视化不完整。

## 3. 推荐功能方向

### 3.1 左侧功能栏新增“作品审计”

在左侧功能栏增加独立导航项：

- 中文：`作品审计`
- 英文：`Audit`
- 建议图标：`ShieldCheck` 或 `ScanSearch`
- 路由建议：`#/audit`

页面不强制绑定某一本书，而是在页面顶部选择书籍；如果当前处于书籍写作页，也可以自动默认选择当前书。

低成本实现路径：

- 扩展 `HashRoute`：增加 `{ page: "audit" }`。
- `App.tsx` 增加 `toAudit` 导航函数与 `AuditView` 页面渲染。
- `Sidebar.tsx` 增加一个导航按钮。
- 新增 `pages/AuditView.tsx`。

### 3.2 审计页面 UI 结构

建议采用“操作区 + 总览区 + 章节表 + 问题详情”的紧凑后台式布局，避免做复杂大屏。

页面结构：

1. **顶部控制栏**
   - 书籍选择
   - 审计范围：全部章节 / 未通过章节 / 待审核章节 / 指定章节
   - 审计维度开关：连续性、角色一致性、节奏、伏笔、设定、语言、AI 痕迹、敏感风险
   - 操作按钮：`开始审计`、`停止/取消`、`刷新结果`

2. **审计配置卡**
   - 审计服务商选择
   - 审计模型选择
   - 审计 API Key 输入框
   - 连接测试
   - 与写作 API Key 不同的校验状态

3. **总览指标**
   - 总章节数
   - 已审计章节数
   - 通过率
   - 平均审计分
   - Critical / Warning / Info 数量
   - 最近一次审计时间

4. **维度分布**
   - 按问题类别统计：连续性、OOC、伏笔、节奏、设定冲突、重复、长度、敏感词、AI 痕迹等。
   - 低成本实现可先用横向条形列表，不引入图表库。

5. **章节审计表**
   - 章节号
   - 标题
   - 状态
   - 审计分
   - 问题数量
   - 严重度摘要
   - 最近审计时间
   - 操作：审计本章 / 查看问题 / 打开章节 / 进入修订

6. **问题详情抽屉或右侧面板**
   - 问题严重度
   - 问题类别
   - 描述
   - 建议修复方向
   - 是否已修复/忽略

## 4. 可视化维度建议

审计可视化不应只围绕“是否通过”，建议拆成以下维度。

### 4.1 基础质量维度

- **通过状态**：通过 / 未通过 / 需人工复核 / 未审计
- **综合评分**：`overallScore`
- **问题数量**：critical / warning / info
- **章节状态**：ready-for-review、audit-failed、state-degraded、approved

这些字段可直接来自 `AuditResult`、`ChapterMeta.auditIssues`、`audit_history.jsonl`。

### 4.2 叙事连续性维度

- 时间线冲突
- 地点跳变
- 道具/线索状态冲突
- 前后章节事实不一致
- 当前状态文件与正文不一致

低成本做法：从 `AuditIssue.category` 聚合，若旧数据只有字符串形式的 `[warning] xxx`，先做简单正则归类。

### 4.3 角色维度

- OOC：角色行为不符合角色卡
- 声线偏移：角色说话方式不稳定
- 动机断裂：行为缺少动机铺垫
- 人物关系错误
- 角色状态错误：死亡/离场角色误出现

当前项目已支持角色卡，审计页面可以把角色相关问题单独聚合成“角色风险”。

### 4.4 伏笔与结构维度

- 伏笔未推进
- 伏笔提前泄露
- 回收过快或过慢
- 支线失焦
- 当前章节与卷纲目标不一致

可读取 `pending_hooks.md`、`subplot_board.md`、`chapter_goals.json` 作为审计上下文。

### 4.5 文风与可读性维度

- 句式重复
- 段落过长或过短
- 语气漂移
- 信息密度异常
- 口水化/模板化表达

可复用已有文风分析和 `lengthWarnings`，P0 不需要新增复杂算法。

### 4.6 AI 痕迹与平台风险维度

- AI 痕迹检测分
- 敏感词或平台风险提示
- 重复模板句
- 过度解释性段落

当前已有 `detectionScore`、`detectionProvider`、`detectedAt` 字段，可在审计页面展示，但不建议 P0 强制接入外部检测。

### 4.8 内置维度启用规则

可视化页面应显示“当前书实际启用的维度”，而不是固定展示 37 项。

现有启用逻辑：

- 题材档案 `GenreProfile.auditDimensions` 决定基础维度。
- 书籍规则 `book_rules` 可通过 `additionalAuditDimensions` 增加维度。
- `32 读者期待管理` 与 `33 章节备忘偏离` 为通用强制启用维度。
- 若题材或书籍规则启用年代考据，则启用 `12 年代考据`。
- 若存在 `parent_canon.md` 且非同人模式，则启用 `28-31` 番外审计维度。
- 若存在 `fanfic_canon.md`，则按 fanfic mode 启用 `34-37` 同人审计维度，并可能停用部分番外维度。

前端低成本做法：

- 后端新增 `GET /api/v1/audit/books/:bookId/dimensions`。
- 返回当前书启用维度、维度名称、说明、来源。
- 如果暂不新增 API，P0 可先在前端内置 37 项映射表，并从历史 issue category 中反推“出现过的维度”。

建议返回结构：

```ts
interface AuditDimensionView {
  id: number;
  name: string;
  group: string;
  enabled: boolean;
  source: "genre" | "book-rules" | "universal" | "era" | "spinoff" | "fanfic";
  note?: string;
}
```

### 4.9 审计维度可视化功能需求生成与预期

本节把内置审计维度转化为前端功能需求，目标是让用户一眼看出“哪类问题最多、哪几章最危险、审计是否覆盖了当前书需要的维度”。

#### 需求 A：维度启用面板

功能：

- 展示当前书启用的审计维度。
- 按分组折叠：角色、连续性、设定、伏笔、节奏、文本形态、平台风险、番外/同人。
- 每个维度显示启用来源：题材 / 书籍规则 / 通用 / 年代 / 番外 / 同人。
- 鼠标悬停或点击显示该维度的审计说明。

预期：

- 用户能确认审计不是泛化检查，而是按当前书类型动态生成。
- 用户能知道为什么某些维度出现，例如“年代考据”只在年代约束启用时出现。

P0 实现：

- 先展示 37 项中的“出现过 issue 的维度 + 通用维度 32/33”。
- 不做复杂 tooltip，用展开文本即可。

P1 实现：

- 后端暴露真实 `buildDimensionList` 结果。
- 前端显示完整启用维度和未启用维度。

#### 需求 B：维度风险热力图

功能：

- 行为章节，列为审计维度分组。
- 单元格颜色表示风险：无问题 / info / warning / critical。
- 点击单元格打开该章节该维度的问题列表。

推荐分组：

| 分组 | 包含维度 |
|------|----------|
| 角色 | 1, 13, 14, 16, 25, 34, 36 |
| 连续性 | 2, 3, 9, 18, 19, 28, 29, 30, 37 |
| 数值/规则 | 4, 5, 12, 35 |
| 伏笔/结构 | 6, 11, 15, 24, 32, 33 |
| 节奏 | 7, 17, 26 |
| 文本形态 | 8, 10, 20, 21, 22, 23 |
| 平台风险 | 27 |

预期：

- 用户能看到某类问题是否集中爆发。
- 比如连续 5 章在“节奏”列变黄，说明不是单章问题，而是节奏波形失衡。

P0 实现：

- 用普通 CSS grid 或表格色块，不引入图表库。
- 从 `auditIssues` 字符串和 `audit_history` 聚合。

#### 需求 C：维度问题排行榜

功能：

- 统计所有章节中问题最多的维度。
- 展示每个维度 critical / warning / info 数量。
- 点击维度筛选章节表。

示例：

```text
伏笔检查        critical 1 / warning 5 / info 3
节奏检查        warning 6
OOC检查         critical 2 / warning 1
章节备忘偏离    critical 1
```

预期：

- 用户知道当前作品最大的结构风险是什么。
- 审计页从“单章报错”升级为“全书结构诊断”。

#### 需求 D：章节维度雷达卡

功能：

- 选择某章后，右侧显示该章各维度结果。
- 不建议 P0 做真正雷达图，可用“维度评分条”替代。
- 如果审计结果没有单维度分数，则用 severity 反推风险等级。

风险等级计算：

```text
critical = 100 风险
warning = 60 风险
info = 25 风险
无问题 = 0 风险
```

预期：

- 用户可以快速判断某章是“角色问题”还是“节奏问题”。
- 与章节修订按钮形成闭环。

#### 需求 E：审计覆盖度

功能：

- 展示“已审计章节 / 总章节”。
- 展示“已覆盖维度 / 当前启用维度”。
- 显示最近一次审计使用的审计模型、服务商、时间。

预期：

- 用户知道审计是否过期。
- 用户知道是否存在只审了部分维度或部分章节的情况。

#### 需求 F：维度筛选与修复队列

功能：

- 按维度筛选：只看 OOC / 伏笔 / 节奏 / 敏感词等。
- 按严重度筛选。
- 一键生成“修复队列”：critical 优先，其次 warning。
- 每个问题提供“打开章节”“发送到修订”“忽略/标记已处理”。

预期：

- 审计结果从展示型信息变成行动列表。
- 用户可以按风险优先级处理，而不是逐章盲修。

P0 可只做筛选和打开章节；P1 再做忽略/已处理状态。

## 5. 审计 API 与写作 API 解耦方案

### 5.1 为什么必须解耦

如果写作和审计使用同一个模型、同一个 API Key，容易出现：

- 模型倾向认可自己的输出。
- 审计维度被写作 prompt 的惯性污染。
- 同一上下文策略导致漏检。
- 用户难以判断“审计通过”是否客观。

因此建议审计服务独立配置：

- 写作 API：继续使用当前 `llm` / 服务商默认配置。
- 审计 API：单独保存 `audit.service`、`audit.model`、`audit.apiKey`。
- 前端明确提示：审计 Key 不允许与写作 Key 完全相同。

### 5.2 最小后端配置方案

建议复用现有 secrets 结构，不新增复杂账户系统。

新增 secret key 命名：

```text
audit:<service>
```

例如：

```json
{
  "services": {
    "openai": { "apiKey": "sk-writing-xxx" },
    "audit:openai": { "apiKey": "sk-audit-yyy" }
  }
}
```

新增项目配置字段建议：

```json
{
  "llm": {
    "audit": {
      "service": "openai",
      "model": "gpt-5-mini",
      "baseUrl": "https://api.openai.com/v1",
      "apiFormat": "chat"
    }
  }
}
```

P0 可更简单：只保存在 `.inkos/secrets` 与一个 `audit-config.json`，暂不扩展 `ProjectConfigSchema`。但从长期维护看，扩展 `inkos.json` 的 `llm.audit` 更稳。

### 5.3 API Key 不同校验

前端保存审计 Key 时，应先请求当前写作 Key 的指纹，不返回明文。

低成本后端可新增：

```text
GET /api/v1/audit/config
PUT /api/v1/audit/config
POST /api/v1/audit/test
```

`GET /api/v1/audit/config` 返回：

```json
{
  "service": "openai",
  "model": "gpt-5-mini",
  "connected": true,
  "auditKeyFingerprint": "****abcd1234",
  "writingKeyFingerprint": "****wxyz7890",
  "keySeparated": true
}
```

`PUT /api/v1/audit/config` 接收：

```json
{
  "service": "openai",
  "model": "gpt-5-mini",
  "apiKey": "sk-audit-xxx"
}
```

后端校验：

- `apiKey.trim()` 不为空。
- 与写作服务的 API Key 不完全相同。
- 如果相同，返回 `400`：`Audit API key must be different from writing API key.`

注意：不要在前端保存或比较明文写作 Key，比较应在后端完成。

### 5.4 审计运行 API 独立化

当前已有：

```text
POST /api/v1/books/:id/audit/:chapter
```

建议逐步替换为独立审计命名空间：

```text
POST /api/v1/audit/books/:bookId/chapters/:chapterNumber
POST /api/v1/audit/books/:bookId/run

GET  /api/v1/audit/books/:bookId/issues
```

旧 API 可保留，用于兼容章节表按钮。

### 5.5 单进程问题与“自我审计”边界

“避免单进程自我审计”有两层含义：

1. **模型/API 隔离**：必须 P0 实现，成本低，收益高。
2. **运行进程隔离**：真正单独 worker/子进程运行审计，成本较高，可放 P1。

低成本建议：

- P0：同一 Node 服务内运行，但使用独立 audit config、独立 API Key、独立 prompt、独立 SSE 事件。
- P1：审计任务队列化，支持取消、并发限制。
- P2：审计 worker 子进程，彻底隔离写作运行态。

## 8. 后端实现路径

### 8.1 新增审计配置接口

建议新增：


实现要点：

- 审计 Key 存为 `audit:<service>`。
- 写作 Key 仍存为 `<service>`。

### 8.2 新增审计汇总接口


实现逻辑：

1. `StateManager.loadChapterIndex(bookId)`。
2. `loadAuditHistory(bookDir)`。
3. 聚合章节状态、审计历史、问题数量。
4. 返回 summary + rows。

### 8.3 新增独立审计运行接口


与旧接口区别：

- 不读取 `currentConfig.llm` 作为模型来源。
- 读取 `currentConfig.llm.audit` 或 audit config。
- 使用 `audit:<service>` 的 API Key。
- 广播独立 SSE 事件：

```text
audit:start
audit:progress
audit:complete
audit:error
```

### 8.4 审计结果写回

P0 可先只返回结果并写 `audit_history.jsonl`。

建议 P1 写回：

- `ChapterMeta.auditIssues`
- `ChapterMeta.status`
- `ChapterMeta.reviewNote`
- `ChapterMeta.updatedAt`

状态规则：

- `passed=true`：`ready-for-review`
- 有 critical：`audit-failed`
- 只有 warning/info：`ready-for-review` 但保留 `auditIssues`

## 9. 审计 Prompt 与维度建议

审计 prompt 应与写作 prompt 分离，且要强调“只审计，不续写，不补写”。

当前项目已经内置 37 项审计维度，不建议另起一套前端维度名。低成本方案应把 `ContinuityAuditor` 的 `AuditIssue.category` 直接作为问题分类来源，再用前端映射表把它归入“角色 / 连续性 / 伏笔 / 节奏 / 文本形态 / 平台风险”等可视化组。

审计 prompt 需要保留现有边界：

- 只审完成度与结构，不审文笔细节。
- 文笔、句式、排版类问题只能作为 `info`，不得直接导致不通过。
- 只有存在 `critical` 问题时，`passed=false`。
- `overall_score` 按 0-100 评分，作为页面总览的核心指标。
- 章节 memo 稀疏是合法状态，只检查 memo 已明确提出的要求是否落地。
- 如果题材或书籍规则启用年代考据，应允许审计模型联网检索。

建议前端不要让用户手动勾选全部 37 项。更低成本、更稳定的做法是：

1. 默认使用当前书自动启用的维度。
2. 前端只提供“维度分组筛选”，例如角色、伏笔、节奏、连续性。
3. 高级设置里再允许显示/隐藏单个维度。

输出结构建议：

```json
{
  "passed": false,
  "overallScore": 78,
  "summary": "章节整体可读，但角色动机与伏笔推进存在风险。",
  "issues": [
    {
      "severity": "warning",
      "category": "角色一致性",
      "description": "程时一在未获得新证据的情况下突然信任金井军曹。",
      "suggestion": "补一处证据或心理转折。"
    }
  ]
}
```

为支持维度可视化，建议在后端返回结果时增加一层轻量派生字段，不要求模型额外输出：

```ts
interface AuditIssueView {
  severity: "critical" | "warning" | "info";
  category: string;
  dimensionId?: number;
  dimensionName?: string;
  dimensionGroup?: string;
  description: string;
  suggestion: string;
}
```

派生规则：

- `category` 与内置维度中文/英文名完全匹配时，填充 `dimensionId`。
- 如果无法完全匹配，使用关键词映射到分组，例如包含“伏笔/hook”归入伏笔组。
- 未识别项归入“未分类”，但仍保留原始 category。

## 11. 推荐开发顺序

### 第一步：只做可视化读取

先不触发新审计，只读取：

- `chapters/index.json`
- `auditIssues`
- `audit_history.jsonl`

形成审计页总览和章节表。

收益：最快看到 UI 成果，风险最低。

### 第二步：独立审计配置

增加：

- 审计服务商
- 审计模型
- 审计 API Key
- 与写作 Key 不同校验

收益：解决“自我审计”核心问题。

### 第三步：独立审计运行

增加：

- 单章审计
- 批量审计
- SSE 进度
- 写 audit history

收益：完成审计闭环。

### 第四步：问题详情与筛选

增加：

- 按类别筛选
- 按严重度筛选
- 问题详情面板
- 跳转章节修订

收益：让审计结果可行动。

## 12. 风险与规避

### 风险 1：Key 相同校验泄露明文

规避：

- 前端不读取写作 Key 明文。
- 后端比较明文后只返回 fingerprint。

### 风险 2：审计页面过重

规避：

- P0 不引入图表库。
- 使用普通卡片、表格、进度条。

### 风险 3：批量审计占用过多 token

规避：

- P0 只支持单章审计。
- P1 批量审计默认串行，最多并发 1。
- 加暂停/取消和费用提示。

### 风险 4：审计结果格式不稳定

规避：

- 后端做容错解析。
- 失败时保留原始文本到 `summary`，不阻断页面。

### 风险 5：同一进程仍可能共享状态

规避：

- P0 先隔离模型/API Key/prompt。
- P1 增加任务队列。
- P2 再考虑 worker 子进程。

## 13. 最小版本验收标准

P0 完成后，应满足：

- 左侧栏存在“作品审计”入口。
- 审计页能选择书籍。
- 审计页能展示章节审计状态与问题数量。
- 审计页能展示问题类别分布。
- 审计页有独立审计 API Key 输入框。
- 保存审计 Key 时，如果与写作 Key 相同，会被拒绝。
- 单章审计使用审计 Key，不使用写作 Key。
- 审计开始/完成/失败能在页面上可视化。

## 14. 结论

最合理的低成本路线是：

1. **先做审计结果可视化**，直接复用章节索引与审计历史。
2. **再做审计配置隔离**，新增 `audit:<service>` secret 和 audit config。
3. **最后接独立审计运行 API**，让审计页面成为独立工作台。

这样可以在不重写写作管线的前提下，把审计从“写作流程里的附属按钮”升级为“作品质量控制中心”，同时通过独立 API Key 和模型配置降低自我审计幻觉。
