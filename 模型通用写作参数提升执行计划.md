# 模型通用写作参数提升执行计划

> 基于 `0608LLM写作参数报告.md` 的分析结论，针对 InkOS 当前 `extra` 透传机制已就绪但服务配置层会丢弃 `extra` 的问题，制定最小改动、最大复用、文风稳定优先的执行计划。

---

## 一、目标与约束

### 1.1 核心目标

在**不改动 `provider.ts` API 签名**（`chatCompletion` 仍只暴露 `temperature`/`maxTokens`）的前提下，利用已有的 `extra` 透传通道，将 `top_p`、`presence_penalty`、`frequency_penalty`、`seed`、`repetition_penalty` 五个写作参数纳入用户可控范围，以**提升输出文风的稳定性与可复现性**。

### 1.2 执行约束（来自需求方）

| 约束 | 说明 |
|------|------|
| **少改核心** | 不动 `packages/core/src/llm/provider.ts`，不动 pi-ai 调用链路 |
| **多复用 API** | 复用 `/services/config` PUT/GET 和已有的 `extra` 透传机制 |
| **最小扰动** | 修改文件数 ≤ 5，新增代码量 ≤ 250 行（前后端合计） |
| **文风稳定优先** | 所有新增参数必须服务于"减少重复、控制发散、保持可复现" |
| **前端位置** | `ServiceDetailPage.tsx` 的 `<details>高级参数</details>` 区域内扩展 |

### 1.3 文风稳定性的参数逻辑

| 参数 | 对文风稳定的作用 | 默认值 | 推荐范围 | 适用场景 |
|------|-----------------|--------|---------|---------|
| `top_p` | 截断尾部低概率 token，避免输出"离谱用词" | `1.0` | `0.85–0.95` | 叙事类写作 |
| `presence_penalty` | 抑制已出现过的主题/词汇，避免内容来回绕 | `0` | `0.0–0.3` | 长章节生成 |
| `frequency_penalty` | 按出现频次惩罚高频词，增加词汇多样性 | `0` | `0.0–0.3` | 描写丰富度 |
| `seed` | 固定随机种子，相同 prompt 可复现输出 | `undefined` | 任意整数 | A/B 调试、风格对比 |
| `repetition_penalty` | 惩罚重复 token（兼容 OpenAI 兼容 API） | `1.0` | `1.0–1.15` | 抑制 AI 痕迹 |

> 不选 `top_k/min_p/typical_p/mirostat/dry/xtc` 的原因：OpenAI 兼容层多数不支持，或效果与 `top_p` 重叠，引入会增加用户困惑。

function normalizeServiceEntry(serviceId: string, value: Record<string, unknown>) {
  const extra = value.extra && typeof value.extra === "object" && !Array.isArray(value.extra)
    ? value.extra as Record<string, unknown>
    : undefined;
  return {
    service: serviceId,
    ...(typeof value.temperature === "number" ? { temperature: value.temperature } : {}),

}

## 四、执行必要性与紧急性判断

| Phase | 必要性 | 紧急性 | 理由 |
|-------|--------|--------|------|
| **Phase 1**（后端 extra 透传） | ⭐⭐⭐⭐⭐ | 🔴 P0 | 这是整个计划的**前置条件**。如果不修复，`extra` 中的任何参数都会被丢弃，后续前端工作无效。当前 `INKOS_LLM_EXTRA_*` 环境变量是唯一能用的 workaround，但普通用户不会用。 |
| **Phase 2**（ServiceDetailPage UI） | ⭐⭐⭐⭐⭐ | 🔴 P0 | 文风不稳定是用户最直接感知的问题。`presence_penalty` + `frequency_penalty` 对长文本的重复抑制效果立竿见影。`seed` 对调试体验提升极大。 |
| **Phase 3**（Agent 级覆盖） | ⭐⭐⭐ | 🟡 P2 | Writer Agent 和 Auditor Agent 对参数敏感度不同，Agent 级覆盖能精细化控制。但 Phase 1+2 已覆盖 80% 场景。 |
| **Phase 4**（预设） | ⭐⭐⭐⭐ | 🟠 P1 | 预设能把"参数调试"转化为"场景选择"，大幅降低用户门槛。且实现成本低（纯前端）。 |

**综合判断**：Phase 1 和 Phase 2 应该**合并为同一个 PR**，在同一天内完成。Phase 4 可以作为后续快速跟进。Phase 3 列入 backlog。

## 八、与现有功能的关系

| 现有功能 | 与本次改动的关系 |
|---------|-----------------|
| `analyzeStyleFingerprint`（文风指纹） | **互补**。文风指纹是"事后诊断"，本次参数控制是"事前预防"。两者结合：先用指纹诊断文风偏差，再用参数微调抑制发散。 |
| `StyleManager`（文风管理） | **互补**。StyleManager 管理风格库和改写，本次参数控制影响 LLM 生成时的采样行为。未来可在 StyleManager 的"应用"tab 中一键加载文风对应的参数预设。 |
| `ModelOverridesPanel` | **扩展**。本次 Phase 3 可选地为 ModelOverridesPanel 增加 extra 覆盖，让不同 Agent 使用不同采样策略。 |
| `Pipeline scheduler`（retryTemperatureStep） | **不冲突**。scheduler 的失败重试温度递增是动态自适应，本次是静态配置。两者独立工作。 |
| `AI-Tells` 检测 | **互补**。`frequency_penalty` + `presence_penalty` 能从源头上减少 AI 痕迹（重复、空洞），降低 AI-Tells 触发率。 |
| `QualityGates` | **不冲突**。QualityGates 控制重试次数和暂停策略，与采样参数独立。 |
