# 角色卡定级四级拆分方案

> 生成日期：2026-06-06  
> 目标：将现有 `major / minor` 二级定级扩展为 `core / major / minor / functional` 四级定级  
> 约束：本方案必须与《角色卡标签拆分方案（修订版）》共存，且遵循"不创造 Core 新功能"原则

---

## 1. 改动本质：角色定级 vs 标签拆分

这是两个**正交**的改动，但会冲突在同一组文件上：

| 维度 | 角色定级（本方案） | 标签拆分（修订版方案） |
|---|---|---|
| 改动对象 | **Core + Server + UI** | 仅 UI |
| Core 数据类型 | `RoleTier` 扩展 | 无改动 |
| 存储位置 | frontmatter `roleTier` 字段 | frontmatter `tags[]`（前缀约定） |
| 目录结构 | `roles/核心角色/` `roles/重要角色/` `roles/次要角色/` `roles/功能角色/` | 无影响 |
| 正文呈现 | 元信息区块中 `类型` 行 | 元信息区块中其它标签行 |

**关键冲突**：定级扩展必须改动 Core 类型和目录映射，而标签拆分方案约定"不创造 Core 新功能"。这意味着：
1. 定级方案**允许**改动 Core（`RoleTier` 类型本身就是 Core 的核心抽象）
2. 两个方案在 `BookCharactersSection.tsx` 中会修改同一区域的代码（定级下拉、元信息生成）
3. 建议**先完成定级扩展**，再在其基础上叠加标签拆分

---

## 2. 四级的定义

| 定级 | 标识符 | 目录名 | 含义 |
|---|---|---|---|
| 核心角色 | `core` | `核心角色` | 故事绝对中心，POV 频率最高，不可移除 |
| 重要角色 | `major` | `重要角色` | 主要配角，有完整人物弧光，有 POV 章节 |
| 次要角色 | `minor` | `次要角色` | 功能性配角，少量出场，无 POV |
| 功能角色 | `functional` | `功能角色` | 工具人，仅服务于情节需要，无独立塑造 |

### 与旧二级的映射

```
旧 major → 拆分为 core / major 两级
旧 minor → 拆分为 minor / functional 两级
```

存量数据迁移策略见第 5 节。

## 5. 存量数据迁移

### 目录迁移

旧版角色卡存放在：
```
story/roles/主要角色/    ← old major
story/roles/次要角色/    ← old minor
```

新目录：
```
story/roles/核心角色/    ← new core
story/roles/重要角色/    ← new major
story/roles/次要角色/    ← new minor
story/roles/功能角色/    ← new functional
```

### 迁移策略（首次启动时自动执行）

在 `listRoleCards` 中增加迁移逻辑：

```typescript
// 首次扫描时，如果发现旧目录中的角色，按以下规则迁移：
// 旧 "主要角色" → frontmatter.roleTier 保持 "major"（重要），文件移到 "重要角色" 目录
// 旧 "次要角色" → frontmatter.roleTier 保持 "minor"（次要），文件移到 "次要角色" 目录
// 不自动升降级，保留原定级
```

具体实现：在 `listRoleCards` 中扫描旧目录时，读取角色卡，用 `saveRoleCard` 写入新目录，然后删除旧文件。

**替代方案（不自动迁移）**：保留 `TIER_DIRS` 中的旧目录别名，使新旧目录都可达：

```typescript
const TIER_DIRS: Record<RoleTier, readonly string[]> = {
  core:       ["核心角色", "core"],
  major:      ["重要角色", "major", "主要角色"],  // 保留旧目录别名
  minor:      ["次要角色", "minor"],
  functional: ["功能角色", "functional"],
};
```

这样不需要迁移脚本，旧文件原地可用，但新文件会写入新目录。**推荐此方案**——零迁移成本。

## 6. 与标签拆分方案的协同

### 执行顺序建议

```
1. 完成定级四级扩展（本方案）
   → Core 类型 → Server 类型 → UI 定级下拉 → i18n
2. 在定级完成的基础上叠加标签拆分
   → 利用新的 4 级定级映射到 body 元信息区块
```

### 冲突区域

两个方案都会修改 `BookCharactersSection.tsx` 的以下区域：

| 区域 | 定级方案 | 标签拆分方案 | 解决方式 |
|---|---|---|---|
| `RoleTier` 类型 | 扩展为 4 值 | 不碰 | 定级先改，标签继承 |
| `buildBodyMetaBlock` | 更新 `roleTier` 映射 | 新增其余标签行 | 各自修改不同行，git merge 可处理 |
| 统计计数 | 从 2 个变量变 4 个 | 不碰 | 独立 |
| 下拉菜单 | 从 2 项变 4 项 | 不碰 | 独立 |

**结论**：两个方案可以在同一文件中和平共存，但建议先完成定级方案再叠加标签拆分方案。

## 7. 不做的事项

| 事项 | 原因 |
|---|---|
| 角色卡自动升降级 | 算法不可靠，保留给用户手动调整 |
| 基于定级的 POV 权限控制 | 需要 Pipeline 联动，超出本次范围 |
| UI 上按定级筛选角色列表 | 角色列表尚未做搜索/筛选功能，可后续叠加 |
| 视觉区分（不同定级用不同颜色徽章） | 美观改动，可后续叠加 |
