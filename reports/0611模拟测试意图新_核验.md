## 三、新发现的当前 P0

### P0-1 修辞检测语言参数导致 Studio 无法编译

位置：`packages/studio/src/api/server.ts:5611-5616`

当前请求体将 `language` 声明为普通 `string`：

```ts
const raw = await c.req.json<{ text: string; language?: string }>();
const result = detectDuplicateRhetoric(raw.text, raw.language ?? "zh");
```

`detectDuplicateRhetoric` 只接受 `"zh" | "en" | undefined`，因此产生：

```text
TS2345: Argument of type 'string' is not assignable to parameter of type '"zh" | "en" | undefined'
```

影响范围：

- 根级类型检查失败。
- Studio 服务端构建失败。
- Studio `pack` 和 CLI 发布测试失败。
- 原报告 A-3 的“修复脚本后构建恢复”验收尚未成立。

建议修复：

```ts
const language = raw.language === "en" ? "en" : "zh";
const result = detectDuplicateRhetoric(raw.text, language);
```

并增加中文、英文和非法语言值三组接口测试。

### P0-2 CLI doctor 超时策略与测试预算冲突

CLI 已改用 `Promise.race`，原 `signal` 类型错误已消失。但 localhost 自定义服务测试仍可能连续探测多个模型，每次探测都占用超时预算，最终超过 Vitest 单例 5 秒限制。

应采用“单服务共享总预算”，而不是每个候选模型独立获得完整超时；测试中也应使用可控假客户端，不依赖真实网络等待。

## 六、原报告实施方案中的事实错误

### 6.1 蒸馏端点数量写错

当前 `server.ts:5758-5902` 实际注册 5 个端点：

1. `POST /distillations`
2. `GET /distillations/current`
3. `PATCH /distillations/current`
4. `POST /distillations/current/publish`
5. `GET /distillations/versions`

下列两个原报告标记为“已存在”的端点实际不存在：

- `GET /distillations/versions/:version`
- `DELETE /distillations/current`

Core 虽有读取指定版本的能力，但尚未暴露为 Studio API。

### 6.2 章节版本示例使用了错误的数据结构

原报告示例按以下结构解析章节索引：

```ts
{ chapters: Array<...> }
```

当前真实 `chapters/index.json` 是顶层数组，`StateManager.loadChapterIndexStrict()` 也明确要求数组。直接采用原示例会破坏章节索引读写。

版本功能应复用 `StateManager.loadChapterIndex()` 和 `saveChapterIndex()`，不能在路由中自行假设 JSON shape。

### 6.3 构建问题已变化

`packages/studio/package.json:27` 已删除正确的 `tsconfig.server.tsbuildinfo`，并清理 `dist/api`。原缓存路径问题已不再是当前首要根因。

当前构建失败来自 `server.ts:5615` 的语言类型错误。修复后仍需重新验证：

- `dist/index.html`
- `dist/api/index.js`
- `dist/api/index.d.ts`
- `pnpm pack --dry-run`

### 6.4 封面页面定位错误

独立封面配置页面是 `CoverConfigPage.tsx`，不是 `ServiceListPage.tsx`。两者分别承担独立配置页和服务商列表快捷配置卡。

### 6.5 Error Boundary 尚未实现

Step 4 的契约崩溃已经修复，但原报告 A-2 中提出的面板级 Error Boundary 没有落地。因此“单个面板异常不会清空整页”尚未形成系统保证。
