## 二、P1 级问题 — 按钮防重入保护严重不足

### 2.1 问题概述

全站 324 个按钮中，仅 **41 个（12.7%）** 在异步操作期间设置了 `disabled` 状态。大量关键操作按钮在请求进行中仍可被重复点击，存在严重的重复提交风险。

### 2.2 具体缺陷清单

以下 12 处 `onClick={async () => {...}}` 完全没有 `disabled` 或 `loading` 保护，请求中重复点击将导致重复请求：

| # | 文件 | 行号 | 操作 | 风险 |
|---|------|------|------|------|
| 1 | `App.tsx` | 148 | 导航按钮 | 路由重复跳转 |
| 2 | `App.tsx` | 157 | 导航按钮 | 路由重复跳转 |
| 3 | `NotifyConfigPanel.tsx` | 284 | 通知渠道测试 | 重复测试请求 |
| 4 | `BookChaptersSection.tsx` | 931 | 章节审批 | **重复审批，可能产生脏数据** |
| 5 | `BookChaptersSection.tsx` | 942 | 章节驳回 | **重复驳回，可能产生脏数据** |
| 6 | `BookChaptersSection.tsx` | 1000 | 单章操作 | 重复操作 |
| 7 | `BookDetail.tsx` | 726 | 导出保存 | **重复导出，文件覆盖** |
| 8 | `BookDetail.tsx` | 1030 | 章节审批 | **重复审批** |
| 9 | `BookDetail.tsx` | 1040 | 章节驳回 | **重复驳回** |
| 10 | `BookDetail.tsx` | 1052 | 单章操作 | 重复操作 |
| 11 | `AdjustmentSuggestionsPanel.tsx` | 383 | 接受调整写入章节 | **重复写入，数据不一致** |
| 12 | `StyleManager.tsx` | 1598 | 作家重分析 | **重复触发重分析，浪费计算资源** |

### 2.3 修复方案

**方案 A（推荐）：统一使用 `useApi` 返回的 `loading` 状态**

```tsx
// 改造前（无保护）
<button
  onClick={async () => {
    await postApi(`/books/${bookId}/chapters/${ch.number}/approve`);
    refetch();
  }}
>
  <Check size={14} />
</button>

// 改造后（有保护）
const [approvingChapters, setApprovingChapters] = useState<Set<number>>(new Set());
<button
  disabled={approvingChapters.has(ch.number)}
  onClick={async () => {
    setApprovingChapters(prev => new Set(prev).add(ch.number));
    try {
      await postApi(`/books/${bookId}/chapters/${ch.number}/approve`);
      refetch();
    } finally {
      setApprovingChapters(prev => {
        const next = new Set(prev);
        next.delete(ch.number);
        return next;
      });
    }
  }}
  className="... disabled:opacity-50"
>
  {approvingChapters.has(ch.number) ? <Spinner size={14} /> : <Check size={14} />}
</button>
```

**方案 B（快速修复）：引入 `useAsyncButton` hook**

```tsx
// hooks/use-async-button.ts
export function useAsyncButton<T extends unknown[]>(
  fn: (...args: T) => Promise<void>,
) {
  const [loading, setLoading] = useState(false);
  const execute = async (...args: T) => {
    if (loading) return;
    setLoading(true);
    try { await fn(...args); }
    finally { setLoading(false); }
  };
  return { loading, execute };
}

// 使用
const { loading, execute } = useAsyncButton(async () => {
  await postApi(`/books/${bookId}/chapters/${ch.number}/approve`);
  refetch();
});
<button disabled={loading} onClick={execute}>...</button>
```

**KM 可执行**：是（12 处均可通过脚本批量修复）

## 三、P2 级问题 — 系统级交互缺陷

### 3.1 没有统一的表单校验系统

**现状**：
- 0 处使用 `react-hook-form`、`zod` 或类似表单校验库
- 所有表单校验靠手动 `trim()` + `disabled` 条件实现
- 校验逻辑分散在各组件中，无复用

**影响页面**：
- `BookCreate.tsx`：创建书籍表单
- `StyleManager.tsx`：新建作者、URL 导入
- `GenreManager.tsx`：体裁编辑
- `ServiceDetailPage.tsx`：服务商配置

**修复方案**：
1. 引入 `react-hook-form` + `zod` 组合
2. 定义共享 schema（如 `BookCreateSchema`、`AuthorSchema`）
3. 将手动校验替换为 schema-driven 校验

```tsx
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

const bookCreateSchema = z.object({
  title: z.string().min(1, "书名不能为空").max(100),
  genre: z.string().min(1, "请选择体裁"),
  targetChapters: z.number().min(1).max(1000),
});
```

### 3.2 没有统一的 Toast/通知系统

**现状**：
- 全站仅 **8 处** toast/notification 相关代码
- 错误提示方式混乱：
  - `alert()` — BookDetail.tsx（导出失败）
  - `setActionError()` — BookChaptersSection.tsx
  - `setError()` — StyleManager.tsx
  - 无提示 — 部分操作失败静默

**修复方案**：
1. 引入轻量级 toast 库（如 `sonner`）或自建 toast context
2. 统一错误处理：所有 API 失败调用 `toast.error(message)`
3. 成功提示：关键操作完成后调用 `toast.success(message)`
4. 在 `use-api.ts` 的 `fetchJson` 中统一捕获错误并 toast

```tsx
// hooks/use-api.ts
export async function fetchJson<T>(...) {
  try {
    const res = await fetchImpl(url, { ...init, headers });
    if (!res.ok) {
      const err = await res.text();
      toast.error(err || `请求失败 (${res.status})`);
      throw new Error(err);
    }
    return await res.json();
  } catch (e) {
    toast.error(e instanceof Error ? e.message : "网络请求失败");
    throw e;
  }
}
```

### 3.3 表单未保存离开无提示

**现状**：
- 0 处实现「未保存离开确认」
- `beforeunload` 仅用于 `useAutoSave`（死代码）
- 用户在编辑章节、修改设置、填写表单时，直接关闭页面或切换路由，数据直接丢失

**影响场景**：
- 章节阅读器编辑模式
- 服务详情页参数修改
- StyleManager 文本分析区
- BookCreate 创建流程

**修复方案**：
1. 创建 `useUnsavedChanges` hook：

```tsx
export function useUnsavedChanges(isDirty: boolean) {
  useEffect(() => {
    const onBeforeUnload = (e: BeforeUnloadEvent) => {
      if (!isDirty) return;
      e.preventDefault();
      e.returnValue = "";
    };
    window.addEventListener("beforeunload", onBeforeUnload);
    return () => window.removeEventListener("beforeunload", onBeforeUnload);
  }, [isDirty]);
}
```

2. 在路由切换时增加拦截（若 `isDirty` 为 true，弹出确认对话框）

### 3.4 竞态条件处理不足

**现状**：
- 仅 **3 处**使用 `AbortController` / `signal`
- 大量异步请求（如搜索、分析）没有取消机制
- 用户快速切换筛选条件时，可能收到过期的响应覆盖最新结果

**修复方案**：
1. 在 `useApi` hook 中集成 `AbortController`：

```tsx
export function useApi<T>(path: string | null) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(false);
  const abortRef = useRef<AbortController | null>(null);

  useEffect(() => {
    if (!path) return;
    abortRef.current?.abort();
    const controller = new AbortController();
    abortRef.current = controller;
    setLoading(true);
    fetchJson<T>(path, {}, { fetchImpl: (url, init) => fetch(url, { ...init, signal: controller.signal }) })
      .then(setData)
      .catch((e) => { if (e.name !== "AbortError") setError(e); })
      .finally(() => setLoading(false));
    return () => controller.abort();
  }, [path]);

  return { data, loading, error };
}
```

### 3.5 删除确认不统一

**现状**：
- 大部分删除操作使用 `ConfirmDialog` 组件（8 处）
- `BookDetail.tsx:507` 使用原生 `window.confirm()`，与统一组件不一致
- `ConfirmDialog` 组件支持 Escape 关闭和遮罩点击关闭 ✅

**修复方案**：
1. 将所有 `window.confirm()` 替换为 `ConfirmDialog`
2. 统一危险操作按钮样式（`bg-destructive` + 二次确认）

## 四、P3 级问题 — 体验优化

### 4.1 键盘快捷键覆盖率低

- 全站仅 **22 处**键盘事件处理
- 缺少常用快捷键：
  - `Ctrl/Cmd + Enter` 发送消息
  - `Ctrl/Cmd + S` 保存章节
  - `Esc` 关闭弹窗/抽屉
  - `Ctrl/Cmd + K` 打开搜索/命令面板

### 4.2 骨架屏覆盖不足

- 仅 **16 处**骨架屏/脉冲动画
- 大部分页面加载时只显示「加载中...」文字，无视觉占位
- 建议：为书籍列表、章节列表、会话列表等主要数据区添加骨架屏

### 4.3 乐观更新覆盖不足

- 仅 **8 处**乐观更新
- 大部分操作等待后端响应后才更新 UI，感知延迟明显
- 建议：审批/驳回/删除等即时反馈操作优先使用乐观更新

### 4.4 没有右键菜单

- 0 处右键菜单
- 建议：在书籍列表、章节列表、会话列表增加右键菜单（快速操作：编辑、删除、重命名、复制）

### 4.5 没有拖拽交互

- 0 处拖拽功能
- 建议：章节列表支持拖拽排序、侧边栏会话支持拖拽归档

```bash

  file=$(echo "$line" | cut -d: -f1)
  num=$(echo "$line" | cut -d: -f2)
```
