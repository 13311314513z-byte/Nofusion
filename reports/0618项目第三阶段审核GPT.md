## 一、结论摘要

NoFusion/InkOS 当前已经完成从原型到可用工程产品的主体跃迁：Core 写作引擎、CLI 命令体系、Studio Web 工作台、OpenAPI、CI、性能基准脚本和多 Provider LLM 抽象都已具备。核心写作管线测试稳定，CLI 单独集成链路也可跑通。

**Late Beta / 发布准备前夜，但被 Studio API 回归和 lint 门禁阻断。**

一句话判断：

> Core 与 CLI 已接近 RC 水平；Studio 后端拆分后的 API 契约尚未重新闭合。当前最重要的工作不是继续扩展功能，而是把 Studio route extraction 后的行为一致性、测试门禁和安全边界修回来。

## 四、主要问题清单

### P0-1：Studio API 契约回归，阻断发布

证据：

- `packages/studio/src/api/server.test.ts`：95 个测试中 35 个失败。
- 失败集中在 services、sessions、agent、project files、manual audit、model probe 等核心 API。
- `packages/studio/src/api/phase5-hotfix.test.ts`：11 个测试中 2 个失败。
- `packages/studio/src/api/v13-hotfix-round4.test.ts`：6 个测试中 2 个失败。

判断：

近期拆分后的 route modules 没有完整迁移 `server.ts` 原有行为，出现“旧 server 逻辑还在、新 routes 已注册、两套行为不一致”的状态。当前不能继续堆新功能，应先把 route contract 收敛到单一来源。

建议：

1. 先按失败测试分组修，不要大面积重构：
   - services/model probe：恢复 `selectedModel`、`name`、fallback、401/403、cache by baseUrl、text model filter 等契约。
   - sessions/agent：恢复 unsafe bookId 校验、external edit、write-next 快捷路径和 error envelope。
   - truth browser：修复 `/api/v1/books/:id/truth` 列表端点 404、legacy shim 标记和中英文 roles 目录。
   - notify：修复 private webhook URL 拦截。
2. 每修一组，只跑对应测试文件，再跑全 Studio test。
3. 最后删除或下沉 `server.ts` 中与新 routes 重复的旧块，避免双实现继续漂移。

### P0-2：`routes/truth-browser.ts` 存在编码与行为双重风险

证据：

- 测试中 `/api/v1/books/:id/truth` 多次返回 404，导致 truth browser 相关测试失败。
- 源码中中文 roles 目录名出现 mojibake，如 `roles/涓昏...`，这会导致真实中文角色目录不可被列出。
- `assertBookDirectoryExists` 依赖 `state.loadBookConfig(bookId)`，在部分测试/兼容书籍场景下可能比直接 truth 文件读写更严格，造成 GET/PUT 单文件能过、列表端点失败。

判断：

Truth browser 是 Studio 书籍工作区的基础浏览能力，失败会影响设定文件、角色卡、outline 和 legacy shim 的前端可见性。该问题虽然局部，但对工作台体验影响大。

建议：

- 用 UTF-8 正确字符串恢复中文角色目录：`主要角色`、`次要角色`、`核心角色`、`功能角色`、`重要角色`。
- 列表端点与单文件端点共用同一套 allowlist 与 layout 判断。
- 对 legacy book 和 new-layout book 分别补 API contract 测试。

### P0-3：通知与 Session 安全边界出现回归

证据：

- `notify-test-security.test.ts` 期望 private webhook URL 返回 400，实际返回 200。
- `server.test.ts` 中 unsafe activeBookId、unsafe persisted session bookId、non-string activeBookId、unsafe session bookId 创建等测试未按预期拒绝。
- chat artifact edits against source files 期望 400，实际 200。

判断：

这些不是单纯“测试文案过时”，而是 SSRF、路径越权、会话绑定污染相关的安全边界。发布前必须作为 P0 关闭。

建议：

- 将 `normalizeApiBookId`、路径安全校验、webhook private host 拦截抽成单一 shared guard。
- 在 route module 层调用 guard，不依赖旧 `server.ts` 的内联保护。
- 对安全失败保持 400 + 稳定 error code，不要落入 agent fallback。

### P0-4：Lint 门禁仍失败，CI 不可绿

证据：

- `.\pnpm.cmd lint` 失败：Core 有 33 errors、117 warnings。
- 主要错误类型：`no-useless-escape`、`no-control-regex`、`no-ex-assign`。
- 典型文件：`architect.ts`、`settler-delta-parser.ts`、`style-fingerprint.ts`、`runner.ts`、`book-id.ts`、`document-reader.ts`、`story-markdown.ts`、`text-preprocessor.ts`、`text-relayout.ts`。

判断：

目前 lint 不是“风格建议”，而是 CI 阻断。多数错误是正则写法和异常变量重赋值，修复成本低，不应长期保留。

建议：

- 第一批只修 33 errors，不处理 125 warnings。
- 对控制字符正则使用 `new RegExp(...)` 或局部 eslint 说明，避免误伤意图。
- `runner.ts` 的 `catch (error) { error = retryError }` 改为局部 `let finalError = error`。

### P1-1：服务模型探测逻辑被简化过度

证据：

`routes/services.ts` 中 `probeServiceCapabilities` 当前简化为委托 `listModelsForService`，返回字段明显少于测试期望；失败涉及：

- custom service 模型列表。
- 非文本模型过滤。
- selectedModel 返回。
- 401/403 短路。
- provider bank endpoint check model。
- Ollama API key optional。
- cache 按 baseUrl 隔离。

判断：

服务配置是 Studio 当前最容易被用户感知的部分。这里不稳定会直接导致“模型可选但不能用、连接测试误报、服务切换污染配置”。

建议：

- 不要继续维护 `server.ts` 和 `routes/services.ts` 两份能力。
- 以测试为契约，把旧 `server.ts` 中已经通过的逻辑完整迁入 `routes/services.ts`。
- 迁移完成后删除旧同名 route，防止后续注册顺序制造隐性差异。

### P1-2：CLI 测试存在构建产物顺序风险

证据：

- 第一次跑 `.\pnpm.cmd --filter @actalk/inkos test` 时，部分集成测试报 `Cannot find module ... packages/cli/node_modules/@actalk/inkos-core/dist/index.js`。
- 随后单独跑 `cli-integration.test.ts`，44/44 通过。

判断：

CLI 功能本身可用，但测试之间可能存在构建/pack/清理产物顺序依赖。尤其 `publish-package.test.ts`、`pretest build`、workspace symlink 的组合需要隔离。

建议：

- 将 publish packaging 测试与 CLI runtime integration 分开 job。
- CLI integration 前显式执行 `pnpm --filter @actalk/inkos-core build && pnpm --filter @actalk/inkos build`。
- 检查 `publish-package.test.ts` 是否会清理/恢复影响 workspace symlink 的 `dist`。

### P1-3：CI pnpm 版本使用 `latest` 存在漂移风险

证据：

- 根 `packageManager` 为 `pnpm@11.5.2`。
- `.github/workflows/ci.yml` 中 `pnpm/action-setup` 使用 `version: latest`。

判断：

当前已经有复杂的 workspace build/test 依赖，pnpm latest 可能引入不可控变化。进入 RC 前应固定版本。

建议：

- CI 改为 `version: 11.5.2`，与 `packageManager` 对齐。
- 或使用 Corepack，根据 `packageManager` 自动启用固定版本。

## 八、最终判断

NoFusion/InkOS 第三阶段的真实状态是：

- **能力层面**：核心能力已经成型，Core 与 CLI 可以支撑真实使用。
- **工程层面**：类型系统、OpenAPI、CI、bench 已建立，但 lint 和 Studio test 尚未闭合。
- **产品层面**：Studio 是下一阶段价值核心，也是当前最大风险源。
- **发布层面**：暂不建议打 RC；建议先完成 P0 修复冲刺后再评估。

最终建议：

> 未来 48 小时内暂停新功能，集中修 Studio route contract、truth browser、安全 guard 和 lint errors。修完后若 `typecheck + lint + core/studio/cli test + 一条真实 LLM E2E` 全绿，可进入 RC 候选；否则仍应维持 Late Beta。
