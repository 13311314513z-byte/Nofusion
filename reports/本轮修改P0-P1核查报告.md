### Bug-1：`semantic-duplication.ts` "首语重复"正则在中文场景下失效

**文件**：`packages/core/src/utils/semantic-duplication.ts:217`  
**严重度**：P1（功能失效——中文修辞检测核心场景）

**问题代码**：
```typescript
/^([\u4e00-\u9fff]{2,4}|[a-zA-Z]{2,4})\b([\s\S]*?)\n\n[\s\S]{0,200}\n\1/gm,
```

**根因分析**：
- JavaScript `\b` 是 ASCII 词边界，要求一侧是 `[A-Za-z0-9_]`，另一侧不是
- CJK 字符（`\u4e00-\u9fff`）属于 `\W`，两个 CJK 字符之间不存在 `\b`
- 因此 CJK 分支在中文文本中几乎无法匹配，"首语重复"功能对中文失效
- 英文分支 `[a-zA-Z]{2,4}\b` 仍可正常工作

**影响**：
- 中文章节的首语重复修辞无法被检测
- 依赖该指标的疲劳度/风格诊断结果偏低

**已修复**：拆分为两个独立正则——CJK 分支使用负向前瞻 `(?![\u4e00-\u9fff])` 替代 `\b`，英文分支保留 `\b`。见 `packages/core/src/utils/semantic-duplication.ts:217`。

```typescript
/^([\u4e00-\u9fff]{2,4})(?![\u4e00-\u9fff])([\s\S]*?)\n\n[\s\S]{0,200}\n\1(?![\u4e00-\u9fff])/gm,
/^([a-zA-Z]{2,4})\b([\s\S]*?)\n\n[\s\S]{0,200}\n\1\b/gm,
```

### Bug-2：`bookCreateStatus` TTL 与创建超时严重不匹配

**文件**：`packages/studio/src/api/server.ts:801-802、1848、1859、1905-1925`  
**严重度**：P1（异步 Job 状态丢失）

**问题代码**：
```typescript
const BOOK_CREATE_TIMEOUT_MS = 10 * 60 * 1000; // 10 分钟超时
const BOOK_CREATE_TTL_MS = 60 * 1000;          // 完成后保留 60 秒

// 所有状态都使用 BOOK_CREATE_TTL_MS
bookCreateStatus.set(bookId, { status: "queued", createdAt: Date.now(), ttlMs: BOOK_CREATE_TTL_MS });
bookCreateStatus.set(bookId, { status: "creating", createdAt: Date.now(), ttlMs: BOOK_CREATE_TTL_MS });
```

**根因分析**：
- `bookCreateStatus` 清理定时器每 30 秒扫描一次，删除 `createdAt + ttlMs < now` 的条目
- 进行中（queued/creating）的状态 TTL 只有 60 秒，但创建超时是 10 分钟
- 如果 Architect 生成 foundation 超过 60 秒，状态条目会被清理
- 客户端轮询 `/books/:id/create-status` 将收到 `404 { status: "missing" }`，即使创建仍在进行

**影响**：
- 慢 LLM 或复杂资料导致创建时间 > 60s 时，前端认为创建失败或丢失
- 用户可能重复创建或困惑

**已修复**：新增 `BOOK_CREATE_IN_PROGRESS_TTL_MS = BOOK_CREATE_TIMEOUT_MS + 60_000`，`queued`/`creating` 状态使用该 TTL，终态仍保留 `BOOK_CREATE_TTL_MS`。见 `packages/studio/src/api/server.ts:801-803`。

### Bug-3：Foundation plan 持久化存在启动竞态

**文件**：`packages/studio/src/api/server.ts:1643-1648、6407-6415`  
**严重度**：P1（状态不一致——重启后 plan 短暂不可见）

**问题代码**：
```typescript
const foundationPlansPromise = loadPersistedFoundationPlans(root);
const foundationPlans = new Map<string, FoundationPlanEntry>();
foundationPlansPromise.then((loaded) => {
  for (const [id, entry] of loaded) foundationPlans.set(id, entry);
}).catch(() => {});
```

**根因分析**：
- 启动时异步加载磁盘上的 plan，但 `foundationPlans` 初始为空
- `/books/:id/import/foundation/commit` 直接读取 `foundationPlans.get(planId)`，不等待加载完成
- 如果服务器重启后立刻收到 commit 请求（例如用户保留了 planId 并刷新页面），会返回 409

**影响**：
- 服务器重启后，用户在前端点击"应用"可能失败
- 需要重新 plan，浪费 LLM token

**已修复**：`foundationPlansPromise` 现在被 `commit` 端点 `await`，保证启动加载完成后再读取 plan；加载错误也会记录到控制台而不是静默吞掉。见 `packages/studio/src/api/server.ts:1643-1648` 与 `6417-6420`。

### Bug-4：章节保存端点缺乏输入校验

**文件**：`packages/studio/src/api/server.ts:1993-1998`  
**严重度**：P1（错误处理 / 潜在 DoS）

**问题代码**：
```typescript
app.put("/api/v1/books/:id/chapters/:num", async (c) => {
  const id = c.req.param("id");
  const num = parseInt(c.req.param("num"), 10);
  const { content } = await c.req.json<{ content: string }>();
  // ... 直接写入
});
```

**根因分析**：
- `content` 只是 TypeScript 类型断言，无运行时校验
- `num` 未校验 `Number.isFinite(num)` 或 `num > 0`
- 缺少 `content` 最大长度限制

**影响**：
- 缺失/非字符串 `content` 会导致 500 而非 400
- 超大 payload 可能导致 OOM
- `num = NaN` 时 `String(num).padStart(4, "0")` 得到 `"NaN"`，文件查找失败返回 404，但逻辑混乱

**已修复**：`PUT /books/:id/chapters/:num` 现在校验 `num` 为正整数、`body.content` 为字符串，并对 1000 万字长度返回 413。见 `packages/studio/src/api/server.ts:1993`。

### Bug-5：`doctor.ts` 探测超时产生未捕获的 Promise 拒绝

**文件**：`packages/cli/src/commands/doctor.ts:340-346`  
**严重度**：P1（进程稳定性）

**问题代码**：
```typescript
const probePromise = chatCompletion(client, model, [...], { maxTokens: 16 });
const timeoutPromise = new Promise<never>((_, reject) =>
  setTimeout(() => reject(new Error(`Probe timed out after ${probeTimeout}ms`)), probeTimeout)
);
const response = await Promise.race([probePromise, timeoutPromise]);
```

**根因分析**：
- `Promise.race` 返回后，失败的另一个 Promise 仍会继续执行
- 如果 `probePromise` 获胜，`timeoutPromise` 在定时器到期后仍会 reject
- 该 reject 未被捕获，Node.js 严格模式下会触发 `unhandledRejection`

**影响**：
- 可能导致 CLI 进程在非 0 退出码下终止
- 日志中出现未处理 Promise 拒绝警告

**已修复**：为 `timeoutPromise` 保存 timer，在 `Promise.race` 后 `clearTimeout`；同时给 `probePromise` 补 `.catch(() => {})` 避免其后续 reject 成为未捕获异常。见 `packages/cli/src/commands/doctor.ts:345-356`。

### Bug-6：`doctor.ts` 总预算过紧导致探测不完整 / 测试超时

**文件**：`packages/cli/src/commands/doctor.ts:323-325、329-331`  
**严重度**：P1（可靠性——CLI 唯一失败测试）

**问题代码**：
```typescript
const probeTimeout = Math.max(2000, parseInt(opts.connectivityTimeout ?? "3000", 10) || 3000);
const totalBudget = Math.min(probeTimeout, 4000);
```

**根因分析**：
- 默认 `probeTimeout = 3000ms`，总预算也被限制为 3000ms
- 第一个 probe 超时后，几乎没有时间尝试替代 model/apiFormat
- 测试 `treats localhost OpenAI-compatible endpoints as API-key optional` 仍 5000ms 超时

**影响**：
- `inkos doctor` 在慢网络或本地模拟端点下不可靠
- CLI 测试套件 1/171 失败

**已修复**：保留 `totalBudget = Math.min(probeTimeout * 2, 4500)` 的预算策略，但新增两条约束：
1. 启动新 probe 前检查剩余预算是否足够完成一次完整探测（`Date.now() - startTime + probeTimeout > totalBudget` 则 break）。
2. 为 `chatCompletion` 增加 `signal` 支持，超时后通过 `AbortController` 真正取消底层 fetch，避免进程被挂起的 TCP 连接拖到 10s。

核心改动：`packages/core/src/llm/provider.ts` 为 `chatCompletion` 增加可选 `signal` 并透传至 `chatCompletionViaCustomOpenAICompatible`/`chatCompletionViaCustomAnthropicCompatible` 的 fetch 调用；`packages/cli/src/commands/doctor.ts` 在探测中使用该 signal。

修复后 CLI `treats localhost OpenAI-compatible endpoints as API-key optional` 测试从超时变为 4171ms 通过。

### Issue-3："首语重复"正则存在误报和性能风险

**文件**：`packages/core/src/utils/semantic-duplication.ts:217`  
**严重度**：P2（误报 / 性能）

**问题**：
- 修复 `\b` 后，CJK 分支会匹配任意 2-4 个汉字开头，包括"第1章"、"他说"等常见开头
