# 0615 工作区变更审计与项目发展路线图

> 审计日期：2026-06-15  
> 审计范围：当前工作区、`master` 最新合并、Core 写作管线、Studio 前后端端点、近期新增 E1-E13 能力  
> 验证方式：代码检索、Git 差异核验、全仓类型检查、Core/Studio 自动化测试

---

## 一、结论摘要

项目正在从“自动生成小说的多 Agent 管线”转向“作者可介入、过程可观察、结果可校验的写作工作台”。这一转向是合理的，近期新增的创作访谈、章节意图、章节目标、运行时产物、事件链、场景模板、角色声音画像、风格蒸馏、计划候选等能力，也基本符合高质量创作工具的发展方向。

但当前最主要的问题已经不是缺少 Agent，而是新增能力没有形成完整的产品闭环：

1. 部分功能只有模型或端点，没有前端入口。
2. 部分功能有前端文件，但没有被路由或工作区引用。
3. 部分功能有 API 和 UI，但数据结构不一致。
4. 部分功能完成了存储，却没有进入 Planner、Writer、Auditor 的实际上下文。
5. 测试数量较多，但没有阻止“测试全绿、项目无法通过类型检查”的情况。

因此，下一阶段不应继续横向增加新 Agent，而应把现有能力按“前端入口 -> API 契约 -> 持久化 -> 管线消费 -> 结果验证”逐项闭环。

当前状态判断：

| 维度 | 判断 |
|---|---|
| Core 主写作管线 | 较成熟，测试基础较强 |
| 作者意图介入 | 已形成初步闭环，但交互仍需修正 |
| Studio 基础工作台 | 可用能力较多，但页面和端点开始分叉 |
| 近期 E1-E13 新能力 | 多数处于原型、影子或存储阶段 |
| 前后端契约治理 | 明显不足 |
| 发布状态 | **No-Go：全仓类型检查失败** |

```text

```

最新合并还引入了较完整的书籍运行数据和快照。真实书稿、运行产物和代码仓库长期混合，会带来：

- Git 仓库快速膨胀；
- 测试结果依赖本地历史数据；
- 内容版权和隐私风险；
- 产品数据格式与测试夹具边界不清；
- 合并冲突和无意义 diff 增加。

建议只保留经过脱敏、缩减、可重建的合成夹具，其余书籍数据移出源码仓库。

## 四、前后端与写作管线闭环矩阵

成熟度定义：

- **模型**：只有类型、Schema 或 Agent 类。
- **端点**：后端 API 已存在。
- **可达**：前端有真实路由或工作区入口。
- **消费**：数据实际进入 Planner、Writer、Auditor 或状态更新。
- **闭环**：可编辑、可持久化、可追踪、可验证并有测试。

| 能力 | 模型 | 端点 | 前端可达 | 管线消费 | 当前判断 |
|---|:---:|:---:|:---:|:---:|---|
| 章节目标 | ✅ | ✅ | ✅ | ✅ | 基本闭环 |
| 章节意图/创作访谈 | ✅ | ✅ | ✅ | ✅ | 接近闭环，存在章节选择问题 |
| 书籍资料源 | ✅ | ✅ | 部分 | ✅ | 两套页面分叉，预览端点缺失 |
| 风格诊断 | ✅ | ✅ | ✅ | 部分 | 可生成和保存，历史查看未闭环 |
| 风格蒸馏 | ✅ | ✅ | ❌ | 未确认 | 页面不可达且类型契约错误 |
| 事件链 | ✅ | 失败 | ❌ | ❌ | 原型，当前阻断编译 |
| 场景模板 | ✅ | ✅ | ❌ | ❌ | 仅 JSON 存储 |
| 角色声音画像 | ✅ | 桩实现 | ❌ | ❌ | 返回默认值且不持久化 |
| 计划多候选 | ✅ | 无开关 | ❌ | ❌ | 代码存在但运行路径不会启用 |
| Runtime 产物查看 | ✅ | ✅ | ✅ | 只读 | 原始文件浏览，不是语义化工作台 |
| Beta Reader | ✅ | 管线内部 | ❌ | Shadow | 缺少人工复核、标签与校准入口 |

## 五、未闭环端点与功能问题

### P0-1：事件链端点使用非法包路径

位置：

```text
packages/studio/src/api/server.ts:6941
packages/studio/src/api/server.ts:6965
```

问题：

- 深层导入未被 core 包导出；
- 全仓类型检查失败；
- GET 和 POST 事件链端点均不可可靠发布。

处理：

- 从 `@actalk/inkos-core` 根入口导入已导出的函数，或显式增加受支持的包导出；
- 增加 Studio server build 测试；
- CI 必须依次执行 typecheck、build、test，而不是只跑 Vitest。

### P1-1：风格蒸馏页面不可达且数据契约错误

后端已经提供：

```text
POST  /style/authors/:authorId/distillations
GET   /style/authors/:authorId/distillations/current
PATCH /style/authors/:authorId/distillations/current
POST  /style/authors/:authorId/distillations/current/publish
GET   /style/authors/:authorId/distillations/versions
```

但 `DistillationPage.tsx`：

- 没有进入 `App.tsx` 路由；
- 没有从 StyleManager 或作者列表跳转；
- 只读取 current，不支持生成、修改、发布、查看版本；
- 把后端 `DistillationRule[]` 错误声明为 `string[]`；
- 若页面被接入，直接渲染规则对象可能导致 React 运行时错误；
- 404 的错误字符串判断不可靠，可能无法显示正确空状态；
- 页面还声明了后端并不存在的 `sentencePatterns`、`wordPreferences`、`narrativeHabits` 字段。

建议不再保留独立孤立页面，而是将蒸馏管理并入 StyleManager 的作者工作区。

### P1-2：资料源页面出现双实现和缺失端点

当前真实工作区使用：

```text
BookSourceSection.tsx
```

新增但未接入：

```text
BookSourcesSection.tsx
```

新增页面会请求：

```text
GET /books/:id/sources/:sourceId
```

但后端只有：

```text
GET    /books/:id/sources
POST   /books/:id/sources
DELETE /books/:id/sources/:sourceId
```

结果是新页面未接入，即使接入，资料预览也会 404。

建议保留一个资料源页面，补齐详情读取、编辑、启停、用途标记、被哪些章节引用等能力。

### P1-3：事件链不是实际章节事件提取

当前 POST 端点没有调用已经实现的 `EventChainExtractor` 和 `EventChainInferenceEngine`，而是：

- 读取 `story/sources`；
- 手工解析 frontmatter；
- 为每个来源拼装低置信度事件；
- 不分析章节正文；
- 不进行因果、角色状态或场景结果推理；
- 不进入下一章 Planner/Writer 上下文。

这更接近演示数据生成器，而不是事件链功能。

### P1-4：声音画像端点是桩实现

`POST /books/:id/voice-profiles/analyze` 当前固定返回：

- 平均句长 0；
- 空口头禅；
- 默认复杂度；
- 置信度 0.3；
- 不调用 `VoiceProfileAnalyzer`；
- 不写入 `story/voice_profiles`；
- 前端没有调用；
- Writer 没有消费通用角色声音画像。

该能力目前应标记为 `prototype`，不能标记为完成。

### P1-5：场景模板只有存储，没有创作作用

场景模板只有 GET/PUT JSON 文件端点：

```text
GET /books/:id/scene-templates
PUT /books/:id/scene-templates
```

目前没有：

- 前端模板编辑器；
- Schema 校验；
- 模板版本；
- 从章节意图选择模板；
- Planner 场景拆分引用；
- Writer 场景约束注入；
- 模板效果评估。

建议把场景模板定位为“可复用的场景约束包”，而不是单纯 JSON 设置。

### P1-6：计划多候选不会在真实流程中产生

Planner 支持：

```text
generateAlternatives?: boolean
```

但 `PipelineRunner.resolveGovernedPlan()` 没有传入该选项，Studio 和 CLI 也没有配置入口。因此正常写作不会生成候选计划。

即使生成，目前也缺少：

- 候选持久化规范；
- 前端对比；
- 作者选择；
- 选择原因记录；
- 被选方案和最终正文的关联；
- 候选质量反馈。

### P1-7：创作访谈默认章节可能错误

`BookIntentsSection` 在 API 数据尚未返回时初始化 `chapterNumber`。由于 React 的 `useState` 初始化只执行一次，已有多章书籍进入访谈页后，仍可能停留在第 1 章，而不是下一待写章节。

应在书籍数据加载完成后同步默认章节，同时避免覆盖用户已手动选择的章节。

（本节内容已执行完毕，对应源码和产物已清除，不再构成项目项。）

### P2-1：风格诊断历史只完成后端

后端已有作者诊断的保存、列表、详情端点，前端可以保存诊断，但没有历史列表、详情查看、前后版本对比和回归趋势。

对于写作质量工具，单次分数价值有限，真正有价值的是“修改前后哪些维度改善、哪些维度退化”。

### P2-2：Runtime 页面仍是文件浏览器

当前 Runtime 页面能够按 intent、plan、context、trace、memo、delta 等类型浏览原始产物，这是很好的可观察性基础。

但它尚未完成：

- 计划候选对比；
- 事件链图；
- 角色状态变化；
- 伏笔推进时间线；
- Prompt 输入来源解释；
- 章节质量变化曲线；
- 从异常产物回到修订操作。

它应发展为“章节决策与证据工作台”，而不只是文件查看器。

## 六、项目发展思路调整

### 6.1 产品定位

建议将项目明确定位为：

> 面向长篇、续写、改写和同人创作的证据驱动型写作工作台。作者负责意图、选择与最终判断，Agent 负责材料组织、候选生成、约束检查、状态维护和局部修订。

不建议继续强调“一键自动写完整本书”。项目当前最有差异化价值的部分，是：

- 长篇状态和记忆治理；
- 作者意图进入实际生成；
- 多阶段写作和审校；
- 运行时证据可追踪；
- 对现有书稿的导入、诊断和修订；
- 同人 Canon 与角色一致性；
- 可替换模型和多模型协作。

### 6.2 从 Agent 数量转向三类闭环

#### 闭环 A：作者意图闭环

```text
创作访谈
-> 章节目标/意图
-> 2-3 个计划候选
-> 作者选择和修订
-> 正文生成
-> 意图承诺核验
```

当前目标和意图已经进入 Planner 与校验流程，应优先补齐候选计划选择，而不是继续添加新的提问 Agent。

#### 闭环 B：叙事状态闭环

```text
章节正文
-> 事件链/角色状态/伏笔变化提取
-> 人工确认高风险变化
-> 写入结构化状态
-> 下一章检索和注入
```

事件链必须建立在实际章节正文和既有状态之上，并能更新结构化记忆。单独生成一个 JSON 文件没有产品价值。

#### 闭环 C：质量修订闭环

```text
诊断
-> 问题定位和证据
-> 生成局部修订方案
-> 差异预览
-> 应用修改
-> 重新诊断
-> 记录改善或退化
```

现有风格诊断、AI 痕迹、重复段落、修辞问题和局部改写已经具备较好基础，下一步应增加历史、对比和回归门槛。

### 6.3 统一功能成熟度

建议所有新增功能使用统一状态：

| 状态 | 定义 |
|---|---|
| experiment | 可运行实验，不进入默认流程 |
| shadow | 自动执行和记录，但不影响正文 |
| advisory | 向作者提出建议，由作者决定 |
| actionable | 可以应用修改，但必须可预览、撤销 |
| enforced | 已校准，可作为阻断条件 |

Beta Reader 已经有类似思路，建议推广到事件链、声音画像、计划候选、风格规则和状态更新。

## 七、面向写作需求的进一步建议

### 7.1 建立章节创作驾驶舱

每一章应有一个统一页面，集中呈现：

- 作者本章意图；
- 必须发生和禁止发生的事件；
- 当前伏笔、人物状态、叙事位置；
- 2-3 个计划候选及差异；
- 选中计划；
- 生成正文；
- Beta Reader 和审校反馈；
- 局部修订和最终确认；
- 本章导致的状态变化。

这比把功能拆散在“目标、访谈、Runtime、章节、风格”多个页面中更符合真实写作流程。

### 7.2 区分硬约束、软偏好和诊断信号

建议统一使用：

- Hard：Canon、时间线、角色身份、作者禁止项；
- Soft：风格倾向、节奏偏好、场景模板；
- Diagnostic：AI 痕迹、重复、可读性、Beta Reader 观察。

硬约束可以阻止提交，软偏好只能影响候选排序，诊断信号不能未经校准自动改写正文。

### 7.3 使用候选生成与选择降低单次生成随机性

高质量写作不应只生成一个答案。建议在高价值节点生成少量差异明确的候选：

- 章节计划候选；
- 开场候选；
- 关键对话候选；
- 转折处理候选；
- 局部修订候选。

每个候选必须说明差异维度和风险，避免生成三个内容相近的版本。

### 7.4 以局部修订替代频繁全文重写

全文重写容易破坏已经成立的语气、伏笔和事实。应优先形成：

```text
问题证据 -> 目标段落 -> 修改边界 -> 修订 diff -> 回归检查
```

只有结构性失败时才进入章节级重写。

### 7.5 让角色声音画像成为可验证约束

声音画像应从已确认的角色对话中提取，并保存：

- 句长分布；
- 词汇层级；
- 称呼习惯；
- 打断与停顿；
- 反问、隐喻、口头禅；
- 情绪状态下的变化；
- 正例与反例证据。

Writer 只接收精简后的可执行规则，Auditor 对新增对话做偏移检测。不能只保存一个抽象画像。

### 7.6 多类型管线应共享内核、分离策略

建议形成以下产品管线：

| 管线 | 核心目标 | 特有约束 |
|---|---|---|
| 原创长篇 | 连续创作与长期状态维护 | 世界观、伏笔、人物弧 |
| 导入续写 | 理解现有文本后继续 | 原文事实、语气、未完成线索 |
| 同人创作 | 在 Canon 下创新 | Canon 锁、角色声音、关系边界 |
| 章节改写 | 改善已有章节 | 修改边界、差异预览、事实保持 |
| 短篇创作 | 有限篇幅内完成闭环 | 高密度结构、结尾回收 |
| 风格适配 | 接近目标特征但避免复制 | 规则蒸馏、证据、相似度和引用风险 |

这些管线应共享状态、诊断、修订、模型路由和运行时证据系统，只在规划策略和验收规则上分化。

## 八、建议发展路线图

### R0：恢复可发布状态，0-3 天

目标：让主分支重新满足可编译、可构建、可合并。

任务：

1. 修复事件链非法深层导入。
2. 删除 `package-lock.json`，锁定 pnpm。
3. 清理并停止跟踪 tsbuildinfo。
4. CI 增加 `typecheck -> build -> test -> package export check`。
6. 增加“禁止废弃模块回归”的架构测试。
7. 增加 Studio server 真实构建测试。

验收：

```text
pnpm typecheck
pnpm build
pnpm test
pnpm verify:publish-manifests
```

全部通过，工作区无生成缓存差异。

### R1：完成作者意图垂直闭环，1-2 周

目标：一章从作者意图到最终正文形成完整、可操作工作流。

任务：

1. 修复创作访谈默认章节。
2. 将章节目标和章节意图合并到章节驾驶舱。
3. 正式启用并持久化计划候选。
4. 增加候选对比、选择、修改和确认。
5. 将选择结果与 Prompt Manifest、正文版本关联。
6. 在章节完成后展示意图承诺命中与遗漏。

验收：

- 作者可以在一个页面完成访谈、选计划、生成、修订和确认；
- 每个正文版本能够追溯所用意图和计划；
- 未选择的候选不会污染正式状态。

### R2：完成叙事状态与质量闭环，2-4 周

目标：正文产出后，状态更新和质量修订可被作者理解和控制。

任务：

1. 事件链改为真实正文提取，并调用正式 Agent。
2. 事件链与现有状态做冲突和因果推理。
3. 高风险状态变化进入人工确认。
4. 事件、角色状态、伏笔变化进入下一章检索。
5. Runtime 页面升级为事件链和状态变化可视化。
6. 风格诊断增加历史、前后对比和回归趋势。
7. Beta Reader 增加 Studio 查看、人工标签和反馈入口。

验收：

- 写完一章后可以看到“发生了什么、改变了什么、下一章会使用什么”；
- 修订前后质量差异有证据；
- 状态更新可撤销、可重放。

### R3：闭环场景、声音与风格资产，4-8 周

目标：把新增资产从配置文件变成可复用写作能力。

任务：

1. 场景模板增加 Schema、编辑器、版本和使用记录。
2. 场景模板可被章节意图选择，并进入 Planner/Writer。
3. 声音画像调用真实分析器，持久化正反例证据。
4. 声音画像进入对话生成和偏移审校。
5. 风格蒸馏并入 StyleManager，修复契约。
6. 蒸馏支持生成、人工覆盖、发布、版本和回滚。
7. 增加“目标风格适配”和“过度相似风险”双向检查。

验收：

- 每类资产都能证明自己实际影响了哪一次生成；
- 资产修改后能够检测旧产物是否过期；
- 所有自动规则均可关闭或覆盖。

### R4：多管线产品化与评估，8-12 周

目标：形成真正可比较、可校准的多类型创作工具。

任务：

1. 为六类创作管线定义独立入口、默认阶段和验收标准。
2. 建立统一 Pipeline Profile，而不是复制 Runner。
3. 建立人工偏好数据集和失败案例库。
4. 完成 Beta Reader 与人工判断校准。
5. 建立模型、Prompt、规则、成本、时延、质量的实验记录。
6. 支持同一材料使用不同模型和管线进行 A/B 比较。

验收：

- 每种管线有不少于 10 个端到端基准案例；
- 每次模型或 Prompt 升级可复现对比；
- 自动评分与人工偏好之间有明确相关性报告；
- 未经校准的评价器不得升级为 enforced。

## 九、工程治理建议

### 9.1 建立共享 API 契约

当前服务端有约 155 个路由声明，前端 API 调用点也已经较多，继续手写 URL 和局部接口类型会不断产生漂移。

建议：

- 在共享目录定义 request/response Zod Schema；
- 服务端从 Schema 校验输入并声明输出；
- 前端从同一 Schema 推导类型；
- 集中维护 typed client；
- CI 自动比较前端调用路径与服务端路由；
- 对动态参数进行统一编码和错误状态建模。

### 9.2 为每个功能建立闭环验收模板

新增功能只有满足以下条件才能标记完成：

```text
[ ] 有真实用户入口
[ ] API 契约共享且校验
[ ] 数据可持久化和迁移
[ ] 实际进入目标写作阶段
[ ] 运行产物可追踪
[ ] 错误和空状态可处理
[ ] 可关闭、可回滚或可撤销
[ ] 有端到端测试
[ ] typecheck/build/test 全部通过
```

### 9.3 减少重复页面和重复概念

当前已经出现：

- `BookSourceSection` 与 `BookSourcesSection`；
- 独立 `DistillationPage` 与大型 `StyleManager`；
- Core Interviewer 与 Studio 中重复的规则式访谈构造。

建议确定单一领域实现和单一前端入口，避免服务器、Core、页面各写一套相似逻辑。

### 9.4 用端到端路径衡量进度

后续进度报告不再按“新增多少 Agent、模型、端点”计算，而按以下路径计数：

```text
作者能否完成任务
-> 数据是否影响生成
-> 结果是否可解释
-> 修改是否可撤销
-> 质量是否可比较
```

## 十一、最终判断

项目的核心方向已经比单纯“多 Agent 自动写作”更有价值：它开始具备作者意图、结构化状态、可观察运行时、质量诊断和局部修订能力。

当前最大风险是把“新增文件和端点”误认为“功能完成”。近期代码已经出现非法导入、页面不可达、类型伪契约、桩端点、重复页面和废弃模块回归，这些问题都说明项目进入了必须加强产品闭环和工程治理的阶段。

建议接下来用 2-4 周停止横向扩展，集中完成：

1. 发布基线恢复；
2. 章节创作驾驶舱；
3. 作者意图到正文的可追溯闭环；
4. 正文到叙事状态的确认闭环；
5. 诊断到局部修订的质量闭环。

完成这三个写作闭环后，再扩展更多创作类型和 Agent，项目才会从“功能丰富的原型”进入“真正可持续使用的高品质创作工具”阶段。
