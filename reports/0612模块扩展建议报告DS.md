## 二、Core 模块深度评估

### 2.1 模块结构

```
packages/core/src/
├── agents/          # 22 个 Agent 文件（含 prompts/context/parser）
│   ├── 管线核心 Agent：
│   │   ├── radar.ts          — 雷达扫描（发现可写作方向）
│   │   ├── planner.ts        — 规划师（生成章节意图）
│   │   ├── composer.ts       — 合成师（装配上下文）
│   │   ├── writer.ts         — 写手（产出正文）
│   │   ├── reviser.ts        — 修订师
│   │   ├── consolidator.ts   — 结算师（更新 truth）
│   │   ├── auditor.ts        — 审计师 → chapter-analyzer.ts
│   │   ├── detector.ts       — 检测器
│   │   └── architect.ts      — 架构师（建书/基础设定）
│   ├── 支撑 Agent：
│   │   ├── foundation-reviewer.ts
│   │   ├── state-validator.ts
│   │   ├── style-*.ts（5 个）
│   │   ├── rhetoric-rewriter.ts
│   │   ├── length-normalizer.ts
│   │   └── short-fiction.ts
│   └── 工具模块：rules-reader, ai-tells, continuity
├── pipeline/        # 10 个管线文件
│   ├── runner.ts          — 144KB，管线主调度器
│   ├── scheduler.ts       — 守护进程调度器
│   ├── agent.ts           — Agent 上下文管理
│   └── 支撑模块：chapter-*.ts, detection-runner, short-fiction-runner
├── models/          # 数据模型
├── state/           # 状态管理（MemoryDB）
├── llm/             # LLM 连接层
├── interaction/     # 交互运行时（CLI/Studio 共享）
├── utils/           # 工具函数
├── style-library/   # 文风库
├── notify/          # 通知
└── import/          # 导入模块
```

### 2.4 Core 扩展建议

| # | 建议 | 优先级 | 工作量 | 说明 |
|---|------|:------:|:------:|------|
| C-1 | **runner.ts 拆分** | P1 | 3d | `runner.ts` 144KB 单一文件，含管线调度、写作、审计、修订等全部逻辑。应按 Agent 类型拆分为 `run-write.ts`、`run-audit.ts`、`run-revise.ts` |
| C-2 | **Pipeline 类型收敛** | P2 | 2d | `WriteChapterInput` / `ReviseChapterInput` / `AuditChapterInput` 等输入类型散落在各处，应统一为 `PipelineTask<T>` |
| C-3 | **Agent 错误类型分层** | P2 | 1d | 当前 Agent 错误都用 `Error` 或 `PlannerParseError`，缺少 `AgentError` / `ConfigError` / `LLMError` 分层 |
| C-4 | **StyleAnalysisSession 统一** | P1 | 1d | 各 Step（诊断/调整/复检）不共享文本和结果，应在 Core 层提供统一 Session |
| C-5 | **Hook Core Service** | P1 | 1.5d | 将 Hook CRUD 从 server.ts markdown 操作迁移到 Core，写 JSON + 同步 Markdown |
| C-6 | **正则安全审计** | P2 | 1d | 全仓扫描所有 `RegExp` / `new RegExp`，标记 ReDoS 风险模式 |

### 5.2 数据流断点

```
Studio POST hook CRUD  →  写入 pending_hooks.md（Markdown）
Core 读取 pending_hooks.md  →  parseHooksMarkdown()（正则解析）
```
**问题**：Studio 写 Markdown，Core 读 Markdown 再解析 — 中间无 JSON 结构层，数据一致性依赖 Markdown 格式稳定。

```
Studio PUT chapter-goal  →  写入 chapter_goals.json（JSON）
Core Planner 读取 chapter_goals.json  →  注入 intent/memo（刚完成）
```
**现状**：✅ 刚修复，Planner 已接入 chapter-goal。

## 七、结论

|------|:------:|------|
| Core Agent 管线 | ~85% | 10-Agent 全链路，少数 Agent 功能需闭环 |

### 核心建议
1. **短期（发布门禁）**：修复 CLI 2 项测试失败 → 通过 `pnpm build && pnpm test`
2. **中期（功能闭环）**：版本历史 UI + Error Boundary + AbortController + Hook Core Service
3. **长期（架构优化）**：runner.ts 拆分 + Pipeline 类型收敛 + 正则安全审计
4. **持续（体验提升）**：i18n 覆盖 + CLI JSON 统一 + 任务中心
