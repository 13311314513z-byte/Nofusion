# 文件预处理功能强化方案

> 编制日期：2026-06-06  
> 适用范围：NoFusion / InkOS 1.4.1 当前工作区  
> 文档状态：实施设计稿，仅描述后续工作，不执行代码修改  
> 核心约束：**不为 Core 增加任何新的预处理、检测、去重或排版能力**

---

## 一、目标与结论

本方案继续强化“文件提取 -> 文本清理 -> 文本重排 -> 文风分析”的使用体验，但实现边界调整为：

1. Core 仅作为现有能力提供方，接口和行为保持不变。
2. 新增的产品能力只落在 Studio 页面、Studio API 编排层和 CLI 命令层。
3. 高误伤规则只做检测和提示，不自动删除文本。
4. 所有会改变文本的操作必须可预览、可撤销、可跳过。
5. 文风分析默认优先保证样本保真，而不是追求“清理得越多越好”。

目标调用链：

```text
本地文件 / 粘贴文本 / URL
  -> 现有 document-reader 提取
  -> Studio/API 输入体检（新增，但不进入 Core）
  -> 用户选择预处理预设或自定义选项
  -> 现有 preprocessText()
  -> 可选：现有 relayoutText()
  -> 差异预览与风险确认
  -> 文风分析 / 作家档案 / 导出 / 导入书籍
```

本方案不再建议修改：

```text
packages/core/src/utils/text-preprocessor.ts
packages/core/src/utils/text-relayout.ts
packages/core/src/utils/document-reader.ts
packages/core/src/index.ts
packages/core/src/__tests__/*
```

## 二、当前能力基线

### 2.1 Core 已有能力

`text-preprocessor.ts` 已提供：

| 选项 | 当前作用 |
|---|---|
| `filterCode` | 删除围栏代码块、行内代码 |
| `filterRepeatedPrompts` | 删除包含常见角色/提示词标记的整行 |
| `filterUrls` | 删除 URL 和邮箱 |
| `filterStructuredData` | 删除 XML/HTML 标签及部分 JSON/数组片段 |
| `stripMarkdown` | 清理标题、强调、链接、引用等 Markdown 标记 |
| `minLineLength` | 删除短于阈值的行 |
| `deduplicateParagraphs` | 对规范化后完全相同的段落做精确去重 |
| `filterTimestamps` | 删除 ISO 时间、中文日期和 Unix 时间戳 |
| `filterIds` | 删除 UUID、ObjectId、NanoID 类标记 |
| `filterNoiseMarkers` | 删除 `undefined/null/NaN/true/false` 等噪声 |

返回结果已包含：

```ts
interface PreprocessResult {
  text: string;
  removedChars: number;
  removedLines: number;
  removedParagraphs: number;
  actions: readonly string[];
}
```

`text-relayout.ts` 已提供：

| 选项 | 当前作用 |
|---|---|
| `mergeShortParagraphs` | 合并短段落 |
| `shortParagraphThreshold` | 短段阈值 |
| `formatDialogue` | 尝试拆分对话行 |
| `ensureParagraphSpacing` | 补充段落间空行 |
| `normalizeQuotes` | 规范引号 |
| `compressBlankLines` | 压缩连续空行 |

`document-reader.ts` 已具备：

- `md/txt/jsonl/json/ts/js/html/htm/css` 文本提取。
- JSON/JSONL 字段优先级与元数据过滤。
- 最大 500 万字符提取上限。
- 分片元信息与分片提取入口。
- 截断、原始长度、字符数、哈希和质量警告。

### 2.2 Studio 已有入口

当前 `StyleManager.tsx` 已支持：

- 粘贴文本、读取本地文件、导入 URL。
- 调用 `/style/extract-text` 提取文本。
- 调用 `/style/preprocess` 执行 Core 预处理。
- 调用 `/style/relayout` 执行 Core 重排。
- 展示 Core 返回的 `actions`。
- 将处理结果送入文风分析、导出或书籍文风导入。

### 2.3 当前主要缺口

问题不在于 Core 缺少更多正则，而在于上层缺少安全编排：

1. 所有预处理开关平铺展示，用户难以判断哪些选项会损失正文。
2. 页面没有“原文、提取文本、清理文本、重排文本”的明确阶段选择。
3. 没有删除比例、风险等级和疑似误删提示。
4. 没有针对聊天导出、论坛文本、网页正文等来源的预设。
5. `minLineLength`、`shortParagraphThreshold` 等已有参数没有完整暴露。
6. 高风险操作执行后只能看结果，缺少前后差异摘要。
7. 多文件处理缺少队列、单文件状态和批量一致性配置。
8. CLI 的文风分析仍缺少与 Studio 对齐的预处理流程。

## 三、不可突破的实施边界

### 3.1 Core 冻结规则

本方案实施期间：

- 不新增 `PreprocessOptions` 字段。
- 不新增 `RelayoutOptions` 字段。
- 不新增 Core 正则、相似度算法或语言识别算法。
- 不修改 Core 默认值和执行顺序。

- 不把 Studio 专用预设下沉到 Core。
- 不增加 Core 对浏览器、UI 状态或用户确认的依赖。

如发现 Core 的既有行为存在独立 P0/P1 缺陷，应另建缺陷任务处理，不能借本方案扩大 Core 范围。

### 3.2 允许修改的层级

### 3.3 不做的事情

- 不引入 LLM 做文本清洗或自动分类。
- 不引入向量库、分词服务、语言模型或新 npm 依赖。
- 不自动删除“疑似思维链”自然语言。
- 不自动删除模糊相似段落。
- 不对文风样本默认执行标点转换。
- 不把角色扮演动作、心理描写视为无条件噪声。
- 不为大文本实现第二套文件解析器。
- 不让预处理结果自动覆盖原始文件。

## 四、重新评估原有扩展方向

原文提出的扩展应按“直接复用、检测提示、暂缓自动处理”重新分类。

### 4.1 可通过现有 Core 直接实现

| 扩展方向 | 实现方式 | 结论 |
|---|---|---|
| 对话导出清理 | 组合 `filterRepeatedPrompts/filterStructuredData/filterIds` | 做成预设，不新增 Core |
| 精确段落去重 | 使用 `deduplicateParagraphs` | 保留 |
| Markdown/URL/时间戳/ID 清理 | 使用现有开关 | 保留 |
| 短行处理 | 暴露 `minLineLength` | 默认关闭或设为 0 |
| 短段合并 | 暴露 `shortParagraphThreshold` | 仅排版预设启用 |
| 对话重排 | 使用 `formatDialogue` | 高风险，需预览 |
| 引号与空行整理 | 使用现有重排选项 | 仅用户主动启用 |
| 大文件提取 | 使用现有分片和上限能力 | 不再扩展 Core |

### 4.2 只做检测和提示

以下能力可以在 Studio API 适配层扫描，但第一阶段不自动修改文本：

| 检测项 | 检测结果 | 用户动作 |
|---|---|---|
| 显式思考包装 | 检测 `<think>`、`<analysis>`、对应围栏块 | 提示用户手工检查或使用编辑器删除 |
| 模糊重复段落 | 返回相似段落候选及位置 | 用户逐组保留或删除 |
| 高频重复短语 | 返回重复短语、次数和样例位置 | 仅作为样本质量提示 |
| 混合语言 | 返回中文、英文和其他字符比例 | 用户决定是否拆分样本 |
| Base64/超长编码串 | 返回长度和行号 | 提示该来源可能包含附件数据 |
| ASR 标记 | 返回 `[音乐]`、`[笑声]` 等候选 | 用户确认是否属于风格内容 |
| 翻译对照 | 返回疑似双语成对行 | 用户选择目标语言版本 |
| 引用块 | 返回 Markdown/邮件/论坛引用块数量 | 用户确认引用是否属于作者正文 |
| RP 标记 | 返回 `*动作*`、括号旁白等比例 | 不默认删除，避免损失叙事风格 |

检测输出不能包含“系统已经认定它是垃圾”的表述，应使用“疑似”“候选”“建议检查”。

### 4.3 暂不采用的自动处理

#### 自动 COT 删除

不采用基于“首先、因此、让我想想”等普通语言的删除规则。它会误删侦探推理、人物思考、议论文和第一人称叙述。

只允许检测明确结构包装，例如：

````text
<think>...</think>
<analysis>...</analysis>
```analysis
...
```
````

即使检测到，也默认不自动删除。

#### 模糊去重

不允许在无人确认时根据 Dice、Jaccard 或 Levenshtein 阈值删除段落。小说中的复沓、回环、口头禅和刻意重复都可能是文风特征。

第一阶段只输出候选组：

```ts
interface SimilarParagraphGroup {
  readonly score: number;
  readonly paragraphIndexes: readonly number[];
  readonly previews: readonly string[];
}
```

#### n-gram 自动删句

不采用“某个 n-gram 出现达到阈值就删除整句”的方案。常用词组、人物名和固定叙述视角会产生大量误报。

可将结果作为统计信息，但不得进入自动清理链。

#### 语言一致性自动过滤

不自动删除非目标语言段落。角色外语台词、专有名词、代码混排和翻译体都可能是有效风格信息。

#### 标点自动转换

文风分析直接使用标点节奏指标。自动把中英文标点互换会污染分析结果，因此仅允许在“导出排版”场景手工启用既有重排能力，不进入默认分析预设。

## 五、产品形态：四阶段工作台

### 5.1 阶段模型

页面应明确展示四个不可混淆的文本版本：

```text
Raw        原始输入
Extracted  文档提取结果
Cleaned    preprocessText 结果
Relayouted relayoutText 结果
```

定义统一页面状态：

```ts
type TextStage = "raw" | "extracted" | "cleaned" | "relayouted";

interface TextStageSnapshot {
  readonly stage: TextStage;
  readonly text: string;
  readonly charCount: number;
  readonly lineCount: number;
  readonly paragraphCount: number;
  readonly generatedAt: number;
}
```

页面必须允许用户显式选择“用于分析的版本”，不能继续依赖：

```text
relayoutedText || preprocessedText || extractedText || rawText
```

这种隐式优先级容易让用户误以为分析的是原文，实际却使用了重排文本。

### 5.2 阶段操作

每个阶段提供：

- 查看全文或受控预览。
- 显示字符数、行数、段落数。
- 显示相对上一阶段的变化比例。
- 恢复到上一阶段。
- 清空后续阶段。
- 设为分析输入。
- 导出当前阶段。

当用户重新提取文本时，必须清空旧的 Cleaned 和 Relayouted；当用户重新清理时，必须清空旧的 Relayouted。

### 5.3 不可变原文

Raw 和 Extracted 阶段不得被预处理操作原地覆盖。所有处理都产生新的字符串快照。

本地文件和 URL 原始内容不写回磁盘，导出必须生成新文件。

## 六、预处理预设

### 6.1 预设原则

预设只是在 Studio/CLI 层生成现有 Core options，不改变 Core 默认值。

```ts
interface PreprocessPreset {
  readonly id: string;
  readonly labelKey: string;
  readonly descriptionKey: string;
  readonly preprocess: PreprocessOptions;
  readonly relayout: RelayoutOptions;
  readonly risk: "low" | "medium" | "high";
}
```

### 6.2 推荐预设

#### 保真模式 `fidelity`

用途：小说正文、文风样本、未知来源文本。

```ts
preprocess = {
  filterCode: false,
  filterRepeatedPrompts: false,
  filterUrls: false,
  filterStructuredData: false,
  stripMarkdown: false,
  minLineLength: 0,
  deduplicateParagraphs: false,
  filterTimestamps: false,
  filterIds: false,
  filterNoiseMarkers: false,
}

relayout = {
  mergeShortParagraphs: false,
  formatDialogue: false,
  ensureParagraphSpacing: false,
  normalizeQuotes: false,
  compressBlankLines: false,
}
```

这是文风分析的推荐默认值。

#### 保守清理 `conservative`

用途：网页复制、轻度 Markdown、带少量元数据的文本。

```ts
preprocess = {
  filterCode: true,
  filterRepeatedPrompts: false,
  filterUrls: true,
  filterStructuredData: false,
  stripMarkdown: true,
  minLineLength: 0,
  deduplicateParagraphs: true,
  filterTimestamps: true,
  filterIds: true,
  filterNoiseMarkers: true,
}
```

不启用 `filterStructuredData`，因为当前规则可能把正文中的方括号或花括号内容整体删除。

#### AI 对话导出 `chat-export`

用途：已确认由 user/assistant 多轮对话组成的导出文本。

```ts
preprocess = {
  filterCode: true,
  filterRepeatedPrompts: true,
  filterUrls: true,
  filterStructuredData: true,
  stripMarkdown: true,
  minLineLength: 0,
  deduplicateParagraphs: true,
  filterTimestamps: true,
  filterIds: true,
  filterNoiseMarkers: true,
}
```

启用前必须显示警告：

> “提示词标记过滤会删除命中标记的整行。若角色标签与正文位于同一行，请先查看预览。”

#### 论坛/社区文本 `community`

用途：论坛帖子、评论串、邮件或带引用内容的社区文本。

```ts
preprocess = {
  filterCode: false,
  filterRepeatedPrompts: false,
  filterUrls: true,
  filterStructuredData: false,
  stripMarkdown: true,
  minLineLength: 0,
  deduplicateParagraphs: true,
  filterTimestamps: true,
  filterIds: true,
  filterNoiseMarkers: true,
}
```

引用块仅检测，不自动删除。

#### 仅排版 `layout-only`

用途：不做内容清理，只改善阅读和导出排版。

```ts
preprocess = fidelity.preprocess

relayout = {
  mergeShortParagraphs: false,
  formatDialogue: false,
  ensureParagraphSpacing: true,
  normalizeQuotes: false,
  compressBlankLines: true,
}
```

### 6.3 自定义模式

用户修改任一预设选项后，预设状态切换为 `custom`。

需要补充当前 UI 未完整暴露的已有参数：

- `minLineLength`，范围建议 `0-100`。
- `shortParagraphThreshold`，范围建议 `1-200`。
- `ensureParagraphSpacing`。
- `compressBlankLines`。

输入值只在 Studio/API 层校验，不改变 Core 类型。

## 七、处理风险与差异预览

### 7.1 风险分级

| 操作 | 风险 | 原因 |
|---|---|---|
| 删除 URL、时间戳、ID、噪声标记 | 低 | 通常是元数据，但仍可能属于正文 |
| 清理 Markdown | 低到中 | 会改变格式，不应影响主要语义 |
| 精确段落去重 | 中 | 文学复沓可能是有意表达 |
| 删除代码 | 中 | 科幻、系统文或聊天样本可能包含有效代码 |
| 删除结构化数据 | 高 | 当前正则可能删除正文括号内容 |
| 删除提示词标记行 | 高 | 命中后删除整行，可能连正文一起删除 |
| 短行过滤 | 高 | 对话、诗歌、标题和节奏短句容易被删除 |
| 合并短段、对话重排、引号规范 | 高 | 会直接改变文风统计特征 |

### 7.2 变化率

Studio 根据 Core 返回值和文本长度计算：

```ts
removalRate = removedChars / inputChars
```

建议阈值：

| 变化率 | 提示 |
|---|---|
| `< 5%` | 轻微清理 |
| `5%-15%` | 建议快速检查 |
| `15%-35%` | 高变化，分析前必须确认 |
| `> 35%` | 极高变化，默认阻止自动进入下一阶段 |

阻止只发生在 UI 自动串联，不阻止用户手工确认后继续。

### 7.3 差异展示

考虑到文本可达 500 万字符，不对所有文本无条件执行完整 LCS diff。

建议：

- 20 万字符以内：显示行级差异。
- 超过 20 万字符：显示首部、中部、尾部抽样差异。
- 始终显示 `actions`、删除字符数、删除行数、删除段落数。
- 显示前后字符数和段落数。
- 高风险选项单独列出，不埋在通用 actions 中。

### 7.4 确认规则

以下任一条件成立时，进入分析前弹出确认：

- 变化率超过 15%。
- 开启 `filterStructuredData`。
- 开启 `filterRepeatedPrompts`。
- `minLineLength > 0`。
- 启用任意 relayout 选项。
- 提取结果带有 `truncated: true`。
- 输入体检发现疑似 Base64、大量乱码或混合语言。

## 八、输入体检层

### 8.1 定位

输入体检是 Studio API 的只读检查，不调用 LLM，不修改文本，不进入 Core。

建议文件：

```text
packages/studio/src/api/style-preprocess-adapter.ts
```

建议职责：

- 统计字符、行、段落和空白比例。
- 扫描显式结构标记。
- 生成候选问题列表。
- 对候选提供有限片段，不回传整段敏感内容。
- 不执行删除、重排和自动修复。

### 8.2 数据结构

```ts
type InspectionCode =
  | "explicit-think-block"
  | "similar-paragraphs"
  | "repeated-phrase"
  | "mixed-language"
  | "encoded-data"
  | "asr-marker"
  | "translation-pair"
  | "quote-block"
  | "rp-marker"
  | "high-whitespace"
  | "possible-garbled-text";

interface InspectionFinding {
  readonly code: InspectionCode;
  readonly severity: "info" | "warning";
  readonly count: number;
  readonly lineNumbers?: readonly number[];
  readonly samples: readonly string[];
  readonly messageKey: string;
}

interface InspectionResult {
  readonly charCount: number;
  readonly lineCount: number;
  readonly paragraphCount: number;
  readonly findings: readonly InspectionFinding[];
}
```

### 8.3 安全限制

- 每类 finding 最多返回 5 个样例。
- 每个样例最多 120 字符。
- 不返回完整 `<think>` 或编码数据内容。
- 不对自然语言进行“思维链”语义判断。
- 模糊重复仅在文本低于既定规模时启用；大文本只做哈希和长度分桶候选。
- 检测失败不得阻断现有提取和预处理。

### 8.4 API 路径

新增 Studio API：

```text
POST /api/v1/style/preprocess/inspect
```

请求：

```json
{
  "text": "...",
  "checks": [
    "explicit-think-block",
    "similar-paragraphs",
    "mixed-language",
    "encoded-data"
  ]
}
```

响应：

```json
{
  "charCount": 125000,
  "lineCount": 4380,
  "paragraphCount": 912,
  "findings": [
    {
      "code": "explicit-think-block",
      "severity": "warning",
      "count": 3,
      "lineNumbers": [18, 94, 137],
      "samples": ["<think>..."],
      "messageKey": "style.inspect.explicitThinkBlock"
    }
  ]
}
```

现有 `/style/preprocess` 和 `/style/relayout` 路径保持不变。

## 九、Studio 页面完善

### 9.1 页面结构

文件分析页建议调整为：

```text
1. 来源
   - 本地文件
   - URL
   - 粘贴文本

2. 提取结果
   - 格式、字符数、截断状态、warnings
   - 输入体检

3. 清理设置
   - 预设
   - 基础选项
   - 高风险选项
   - 数值阈值

4. 清理预览
   - actions
   - 变化率
   - 差异摘要
   - 确认/撤销

5. 可选重排
   - 单独开关
   - 明确提示会影响文风统计

6. 输出去向
   - 设为分析输入
   - 加入作家档案
   - 导入书籍
   - 导出
```

### 9.2 页面状态拆分

建议将纯状态逻辑从 `StyleManager.tsx` 提取：

```text
packages/studio/src/pages/style-preprocess-state.ts
```

包含：

- 预设定义。
- 预设到 Core options 的映射。
- 阶段快照选择。
- 变化率计算。
- 风险分级。
- 后续阶段失效规则。
- 提交前确认条件。

这些函数应保持纯函数，便于单元测试。

### 9.3 高风险选项展示

高风险选项默认折叠：

```text
[ ] 删除结构化数据
[ ] 删除提示词标记行
[ ] 删除短行，阈值 [ 0 ]
[ ] 合并短段，阈值 [ 20 ]
[ ] 对话重排
[ ] 引号规范
```

每个选项旁显示准确行为，而不是笼统名称。例如：

> “删除提示词标记行：命中 user:/assistant:/system: 等标记时删除整行。”

### 9.4 大文本展示

- 文本框默认只展示受控长度的预览。
- 显示“查看首段 / 中段 / 尾段”切换。
- 不把多个 50 万字符分片同时挂载到多个 textarea。
- 完整文本仍保留在页面状态中供 API 请求和导出使用。
- 截断时禁止使用“已完整导入”文案。

### 9.5 多文件队列

第二阶段可扩展文件页为批量队列：

```ts
interface PreprocessQueueItem {
  readonly id: string;
  readonly sourceName: string;
  readonly fileType: LocalStyleFileType;
  readonly status: "pending" | "extracting" | "ready" | "processing" | "done" | "error";
  readonly activeStage: TextStage;
  readonly error?: string;
}
```

规则：

- 单次最多 20 个文件。
- 顺序处理，默认并发数 1。
- 每个文件独立失败，不中断整个队列。
- 可统一套用预设，也可单文件覆盖。
- 加入作家档案前逐项显示最终字符数。

## 十、Studio API 完善

### 10.1 请求校验

当前 `/style/preprocess` 和 `/style/relayout` 使用宽泛的：

```ts
Record<string, boolean | number>
```

建议在 Studio API 层增加显式白名单解析，但传给 Core 的结构仍是现有类型：

```ts
interface StudioPreprocessRequest {
  readonly text: string;
  readonly options?: {
    readonly filterCode?: boolean;
    readonly filterRepeatedPrompts?: boolean;
    readonly filterUrls?: boolean;
    readonly filterStructuredData?: boolean;
    readonly stripMarkdown?: boolean;
    readonly minLineLength?: number;
    readonly deduplicateParagraphs?: boolean;
    readonly filterTimestamps?: boolean;
    readonly filterIds?: boolean;
    readonly filterNoiseMarkers?: boolean;
  };
}
```

校验规则：

- 拒绝未知字段。
- `minLineLength` 钳位到 `0-100`。
- `shortParagraphThreshold` 钳位到 `1-200`。
- 文本上限复用现有 `MAX_CHARS`。
- 超限返回 413，不静默截断。
- 非法类型返回 400，不进入 Core。

### 10.2 响应增强

不修改 Core 返回结构；Studio API 可以在外层补充编排元数据：

```ts
interface StudioPreprocessResponse {
  readonly result: PreprocessResult;
  readonly meta: {
    readonly inputChars: number;
    readonly outputChars: number;
    readonly removalRate: number;
    readonly risk: "low" | "medium" | "high";
    readonly highRiskOptions: readonly string[];
  };
}
```

为了兼容现有前端，可分两步实施：

1. 第一阶段保持原响应，仅在前端计算 meta。
2. 第二阶段新增 `/style/preprocess/preview` 返回包装结构。

不要直接破坏现有 `/style/preprocess` 的响应格式。

### 10.3 错误语义

| 场景 | HTTP 状态 |
|---|---:|
| 空文本、非法选项 | 400 |
| 文本超限 | 413 |
| Core 处理异常 | 500 |
| URL 获取失败 | 502 |
| URL 超时 | 504 |

错误响应统一：

```json
{
  "error": {
    "code": "PREPROCESS_TEXT_TOO_LARGE",
    "message": "Text exceeds the 5,000,000 character limit."
  }
}
```

## 十一、CLI 完善

### 11.1 目标

CLI 不新增 Core 能力，只调用现有：

```ts
extractDocument()
preprocessText()
relayoutText()
exportPreprocessed()
```

### 11.2 建议命令

新增：

```powershell
inkos style preprocess ".\sample.jsonl.md" `
  --preset conservative `
  --output ".\sample.cleaned.md"
```

可选参数：

```text
--preset <fidelity|conservative|chat-export|community|layout-only>
--output <path>
--format <txt|md|html>
--relayout
--min-line-length <n>
--short-paragraph-threshold <n>
--inspect
--dry-run
--json
```

### 11.3 CLI 行为

`--dry-run`：

- 提取文件。
- 显示 warnings。
- 显示将启用的 Core options。
- 显示预计风险等级。
- 不写输出文件。

`--inspect`：

- 执行 CLI 本地只读体检。
- 输出候选问题。
- 不自动修改文本。

正常执行：

1. `extractDocument()`。
2. 应用预设映射。
3. `preprocessText()`。
4. 可选 `relayoutText()`。
5. 输出 actions、变化率和最终字符数。
6. 使用 `exportPreprocessed()` 生成内容。
7. 写入用户明确指定的新文件。

若未提供 `--output`，默认只打印摘要，不把数百万字符写到终端。

### 11.4 Studio 与 CLI 一致性

两端至少保证：

- 预设名称一致。
- 预设映射一致。
- 风险分级一致。
- 数值范围一致。
- JSON 输出字段一致。

如果暂时不建立共享包，应通过同一组固定用例做契约测试，避免两端预设逐渐漂移。

## 十四、测试计划

### 14.1 状态与预设测试

- `fidelity` 不启用任何改变内容的选项。
- `conservative` 不启用结构化数据删除和重排。
- `chat-export` 标记为高风险。

- 重新提取后清空 cleaned/relayouted。
- 重新清理后清空 relayouted。
- 分析输入只能来自存在的阶段。

### 14.2 API 测试

- 空文本返回 400。
- 未知 options 字段返回 400。
- 错误类型返回 400。
- `minLineLength` 和短段阈值正确校验。

- 设为分析输入后，分析请求使用用户选择的阶段。

### 14.4 CLI 测试

- `--dry-run` 不创建文件。
- 未指定输出路径时不覆盖源文件。

## 十五、验收标准

### 15.1 功能验收

- 用户能选择预设或自定义现有 Core 选项。
- 用户能明确选择用于分析的文本阶段。
- 清理前后有可读的变化摘要。
- 高风险操作有说明和确认。
- 输入体检只提示，不擅自删除。
- Studio 和 CLI 都不覆盖源文件。

### 15.2 安全验收

- Core 文件无改动。
- 不新增 LLM、数据库或第三方解析依赖。
- 不自动删除模糊重复段落。
- 不自动删除自然语言推理。
- 不自动过滤非目标语言。
- 不因检测失败阻断基本预处理。
- 大文本不会被整段塞入错误消息或日志。

### 15.3 回归验收

- 现有文风分析入口仍可直接分析粘贴文本。
- 现有作家档案创建、追加样本和重新分析不受影响。
- 现有书籍文风导入不受影响。
- 现有 `/style/preprocess`、`/style/relayout` 调用继续可用。
- 不改变章节导入、架构资料导入和运行时状态。

## 十六、风险与缓解

| 风险 | 缓解方式 |
|---|---|
| 用户误把高风险预设用于小说正文 | 默认使用保真或保守清理，显示用途说明 |
| 清理后文风统计失真 | 明确分析阶段，重排默认关闭 |
| 超长文本 diff 卡顿 | 小文本完整 diff，大文本抽样 diff |
| 模糊重复检测复杂度过高 | 限制规模，超限时只做分桶候选或跳过 |
| 检测结果让用户误以为系统已判定垃圾 | 全部使用“疑似/候选/建议检查”措辞 |
| Studio 与 CLI 预设漂移 | 契约夹具和一致性测试 |
| 当前 Core 正则存在误伤 | 上层准确说明行为，高风险默认关闭 |
| 页面状态继续隐式串联 | 使用显式 `activeAnalysisStage` |

## 十七、最终推荐

最短、最安全的实施路线是：

1. 先做四阶段状态和预设，不增加任何新清理算法。
2. 再做变化率、风险提示和明确的分析输入选择。
3. 将 COT、模糊重复、混合语言等需求改为只读体检。
4. 最后补批量队列和 CLI 对齐。

本轮强化的重点不是让系统拥有更多“删除文本”的能力，而是让用户能够看清：

> 原始文件提取出了什么、哪些规则会删除什么、处理后改变了多少、最终分析的到底是哪一版文本。

这条路线可以完整复用现有 Core，同时显著提高预处理的可理解性、可控性和可审计性，并把误删正文、污染文风统计和大文本卡顿的风险控制在 Studio/API/CLI 编排层。
