## 三、代码审查关键问题

### 3.1 阻塞验收的硬性缺陷

#### 3.1.1 `packages/core/src/pipeline/pipeline-memory-sync.ts` 引入 lint error

- **位置**：第 44 行、第 93 行。
- **问题**：`catch (error) { ... error = retryError; }` 违反 `no-ex-assign` 规则。
- **影响**：`pnpm -r lint` 失败，验收门禁未闭合。
- **修复建议**：将 `error = retryError` 改为 `const finalError = retryError`，后续使用 `finalError`。

#### 3.1.2 CLI 集成测试依赖 `dist/index.js` 路径不稳定

- **位置**：`packages/cli/src/__tests__/cli-integration.test.ts`。
- **问题**：11 个测试失败均因 `Cannot find module '.../packages/cli/dist/index.js'` 或 `ERR_MODULE_NOT_FOUND`（`dist/tui/dashboard.js`）。
- **影响**：CLI 测试集无法作为验收门禁稳定通过。
- **根因**：[INFERRED] 测试运行期间 `dist/` 被清理或 `pretest` 构建产物未正确保留；Windows 路径与 ESM 解析存在竞争。
- **修复建议**：
  1. 在 `cli-integration.test.ts` setup 中显式调用 `pnpm run build` 或跳过依赖 dist 的集成测试（标记为 `process.platform === 'win32'` 跳过）。
  2. 将 `run()` helper 改为使用 tsx 直接加载源码，而非依赖预构建产物。

### 3.2 结构性技术债

#### 3.2.1 `runner.ts` 仍 2602 行，目标 < 800

- **当前状态**：已拆出 `pipeline-foundation.ts`、`pipeline-writing.ts`、`pipeline-audit.ts`、`pipeline-revision.ts`、`pipeline-style.ts`、`pipeline-import.ts`、`pipeline-short-fiction.ts`、`pipeline-memory-sync.ts` 等模块，但 `runner.ts` 仍承载调度、状态机、错误恢复、legacy 兼容等 2602 行逻辑。
- **风险**：单文件认知负荷过高，继续扩展会导致回归测试成本指数级上升。

#### 3.2.2 `pipeline-runner.test.ts` 5192 行未拆分

- **修复建议**：按阶段拆分为 `runner-foundation.test.ts`、`runner-writing.test.ts`、`runner-audit.test.ts`、`runner-revision.test.ts`、`runner-error.test.ts`。

#### 3.2.4 显式 `any` 218 处

## 六、风险与建议

### 6.1 高风险

1. **验收基线不可复现**：工作区 72 个未提交文件若不及时处理，验收报告将失去可复现性。
2. **CLI 测试路径依赖 Windows 环境**：`dist/index.js` 缺失问题在 Windows 上持续出现，可能阻塞 CI。
3. **`runner.ts` 拆分可能引入回归**：该文件承载复杂状态机，大拆需配合充分测试。
4. **bundle budget 压线**：Dist 14476.7 / 15500 KB，新增一个中等依赖即可能超限。

### 6.2 中风险

1. **报告虚报倾向**：DS 源多次虚报「已完成」，建议以实测门禁为唯一验收标准。
2. **`any` 收敛影响构建速度**：大量类型窄化工作可能拖慢重构进度。
3. **前端悬空能力过多**：`plan-alternatives`、`write-status`、`state-changelog` 等后端能力长期未消费，验收时可能被质疑「功能完整性」。

### 6.3 建议

1. **建立单一事实源**：在仓库根目录创建 `STATUS.md` 或更新 `CHANGELOG.md`，记录每日实测门禁状态，避免多源报告数据漂移。
2. **收紧 bundle budget 上限**：将 15500 KB 调回 15000 KB 并真正瘦身，或按业务模块动态拆分 chunk（如 `vendor-streamdown` 1616 KB 过大）。
