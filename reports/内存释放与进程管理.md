# 内存释放与进程管理

> 目标：单一后端进程稳定响应多前端操作，每个操作结束后自动释放内存、清理临时资源，
> 确保系统长时间运行不泄漏、不卡顿。

---

## 1. 当前架构与问题

### 1.1 进程模型

```
dev.mjs
 ├── child process 1: API Server (Hono, port 4579)
 │     └── 处理所有 HTTP 请求 + SSE 事件流
 └── child process 2: Vite Dev Server (port 4577)
       └── 前端静态资源 + API 代理到 4579
```

### 1.2 发现的内存泄漏点

| 位置 | 问题 | 影响 |
|------|------|------|
| `PipelineRunner` 实例 | 每次写章/建书都 `new PipelineRunner()`，但**从不调用 `dispose()`** | LLM 客户端缓存在 `agentClients` Map 中，随请求数线性增长 |
| SSE `subscribers` Set | 每个前端标签页占据一个 subscriber，**页面关闭时通过 `onAbort` 清理**（目前正常） | 但如果 `onAbort` 未触发，subscriber 永久残留 |
| `bookCreateStatus` Map | 建书状态完成后保留 60 秒后清理 | ✅ 已修复，但有改进空间 |
| `modelListCache` | 模型列表缓存永不过期 | 模型列表可能过期 |
| 工具执行回调 | `processProjectInteractionRequest` 中的 `collectedToolExecs` 数组 | 长会话中工具执行记录只增不减 |

## 2. 总体设计原则

```
┌─────────────────────────────────────────────────────┐
│                 单一后端进程 (Node.js)                │
│  ┌──────────┐  ┌──────────┐  ┌────────────────────┐ │
│  │ HTTP API │  │ SSE 推送 │  │ Pipeline 执行引擎  │ │
│  │ (Hono)   │  │ (events) │  │ (dispose 及时回收) │ │
│  └──────────┘  └──────────┘  └────────────────────┘ │
│                     ↕ 共享状态                        │
│  ┌──────────────────────────────────────────────────┐│
│  │          ResourceRegistry (资源注册表)            ││
│  │  - 跟踪所有活跃 Pipeline、定时器、SSE 连接       ││
│  │  - 提供 disposeAll / abortAll 接口               ││
│  └──────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────┘
         ↕                          ↕
   ┌──────────┐           ┌──────────────────┐
   │ 前端 Tab 1│  ...      │  前端 Tab N      │
   │ (SSE)    │           │  (SSE + HTTP)    │
   └──────────┘           └──────────────────┘
```

### 2.1 核心抽象：ResourceRegistry

所有需要释放的资源统一注册到一个全局 `ResourceRegistry`，后端进程退出或资源过期时统一清理。

```typescript
// packages/core/src/utils/resource-registry.ts

interface Resource {
  id: string;
  type: "pipeline" | "timer" | "sse" | "task";
  dispose: () => void | Promise<void>;
  createdAt: number;
  ttlMs?: number;     // 可选 TTL，过期自动释放
  meta?: Record<string, unknown>;
}

class ResourceRegistry {
  private resources = new Map<string, Resource>();
  private cleanupTimer: NodeJS.Timeout;

  constructor() {
    // 每 60 秒扫描过期资源
    this.cleanupTimer = setInterval(() => this.scavenge(), 60_000);
    this.cleanupTimer.unref();
  }

  register(resource: Resource): void {
    this.resources.set(resource.id, resource);
  }

  unregister(id: string): void {
    const res = this.resources.get(id);
    if (res) {
      res.dispose();
      this.resources.delete(id);
    }
  }

  /** 释放所有资源（进程退出时调用） */
  disposeAll(): void {
    for (const res of this.resources.values()) {
      res.dispose();
    }
    this.resources.clear();
    clearInterval(this.cleanupTimer);
  }

  /** 清理过期资源 */
  private scavenge(): void {
    const now = Date.now();
    for (const [id, res] of this.resources) {
      if (res.ttlMs && now - res.createdAt > res.ttlMs) {
        res.dispose();
        this.resources.delete(id);
      }
    }
  }
}

export const globalRegistry = new ResourceRegistry();
```

## 3. 内存释放方案

### 3.1 PipelineRunner 生命周期管理

当前问题：`new PipelineRunner(...)` → `.then(...)` → **结束时不调用 `.dispose()`**

修复方案——使用 `withPipeline` 工具函数：

```typescript
// packages/studio/src/api/pipeline-utils.ts

import { PipelineRunner } from "@actalk/inkos-core";
import { globalRegistry } from "@actalk/inkos-core/utils/resource-registry";

/**
 * 创建 PipelineRunner，自动注册到全局资源表，
 * 并在 promise 完成/失败后自动 dispose。
 */
export async function withPipeline<T>(
  label: string,
  fn: (pipeline: PipelineRunner) => Promise<T>,
  ttlMs = 5 * 60_000,  // 默认 5 分钟超时自动清理
): Promise<T> {
  const pipeline = new PipelineRunner(await buildPipelineConfig());
  const id = `pipeline-${label}-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`;

  globalRegistry.register({
    id,
    type: "pipeline",
    dispose: () => pipeline.dispose(),
    createdAt: Date.now(),
    ttlMs,
    meta: { label },
  });

  try {
    const result = await fn(pipeline);
    return result;
  } finally {
    pipeline.dispose();
    globalRegistry.unregister(id);
  }
}
```

**所有 Pipeline 创建点统一改用 `withPipeline`：**

| 位置 | 修改前 | 修改后 |
|------|--------|--------|
| `POST /books/create` | `new PipelineRunner(...)` → 不 dispose | `withPipeline("create-book", ...)` |
| `POST /books/:id/write-next` | 同上 | `withPipeline("write-next", ...)` |
| `POST /books/:id/draft` | 同上 | `withPipeline("draft", ...)` |
| `POST /books/:id/revise` | 同上 | `withPipeline("revise", ...)` |
| `POST /style/adjustments/preview` | 同上 | `withPipeline("style-preview", ...)` |
| ...其余 12 处 | 同上 | `withPipeline("...", ...)` |

#### 3.1.1 `PipelineRunner.dispose()` 内部实现与 `agentClients` 缓存策略

`packages/core/src/pipeline/runner.ts` 中已存在一个相对健壮的缓存清理机制，但被整体不调 `dispose()` 的问题所掩盖。其内部实现如下：

```typescript
export class PipelineRunner {
  private readonly agentClients = new Map<string, LLMClient>();

  dispose(): void {
    for (const client of this.agentClients.values()) {
      client.dispose?.();
    }
    this.agentClients.clear();
  }

  private setAgentClient(cacheKey: string, client: LLMClient): void {
    const MAX_CACHED_CLIENTS = 20;
    if (this.agentClients.size >= MAX_CACHED_CLIENTS && !this.agentClients.has(cacheKey)) {
      const firstKey = this.agentClients.keys().next().value;
      if (firstKey !== undefined) {
        const evicted = this.agentClients.get(firstKey);
        if (evicted) {
          evicted.dispose?.();
        }
        this.agentClients.delete(firstKey);
      }
    }
    this.agentClients.set(cacheKey, client);
  }
}
```

**关键结论：**

| 机制 | 当前实现 | 是否足够 |
|------|---------|---------|
| 单实例缓存上限 | `MAX_CACHED_CLIENTS = 20`（LRU-like FIFO） | 单实例足够 |
| 实例级释放 | `dispose()` 遍历并调用 `client.dispose?.()`，再 `clear()` | 足够 |
| 全局泄漏控制 | **PipelineRunner 实例不被 dispose** → 每个实例最多 20 个 client，实例数线性增长 | **不足** |

**因此核心矛盾不是 `agentClients` 本身无上限，而是 `PipelineRunner` 实例数无上限。** `withPipeline` 修复后，每个实例在其作用域结束时 dispose，20 的上限将自然回落到 0。

**可选增强（P3，非必须）：**

1. 将 `MAX_CACHED_CLIENTS` 从硬编码 20 改为基于环境变量或 `buildPipelineConfig()` 注入，允许大内存机器调高：
   ```typescript
   const MAX_CACHED_CLIENTS = this.config.maxCachedAgentClients ?? 20;
   ```
2. 将 FIFO 改为真正的 LRU（维护 `lastUsedAt`）：
   ```typescript
   private setAgentClient(cacheKey: string, client: LLMClient): void {
     const meta = this.agentClients.get(cacheKey);
     if (meta) {
       meta.lastUsedAt = Date.now();
       return;
     }
     if (this.agentClients.size >= MAX_CACHED_CLIENTS) {
       const oldest = [...this.agentClients.entries()]
         .sort((a, b) => a[1].lastUsedAt - b[1].lastUsedAt)[0];
       if (oldest) {
         oldest[1].client.dispose?.();
         this.agentClients.delete(oldest[0]);
       }
     }
     this.agentClients.set(cacheKey, { client, lastUsedAt: Date.now() });
   }
   ```
3. 在 `resolveOverride()` 调用后更新 `lastUsedAt`，使高频 agent（如 `writer`、`architect`）不会被低频 agent 驱逐。

#### 3.3.1 `bookCreateStatus` 与 `ResourceRegistry` 的集成方式

当前实现（`packages/studio/src/api/server.ts:764-783`）使用独立的 `Map` + `setInterval` 清理：

```typescript
const bookCreateStatus = new Map<string, { ... createdAt: number; ttlMs: number }>();
setInterval(() => {
  const now = Date.now();
  for (const [id, st] of bookCreateStatus) {
    if (now - st.createdAt > st.ttlMs) {
      bookCreateStatus.delete(id);
    }
  }
}, 30_000);
```

这一机制已经工作，但存在以下改进点：

| 问题 | 影响 | 建议 |
|------|------|------|
| 独立清理循环 | 与 ResourceRegistry 不统一，退出时不被 `disposeAll()` 感知 | 将 `setInterval` 包装为 registry 资源 |
| 无取消能力 | 建书过程中用户无法主动中止 | 在 status 中挂载 `AbortController` |
| 仅按时间清理 | 成功/失败状态的生命周期相同 | 错误状态可缩短 TTL，成功状态可延长供前端轮询 |

**推荐集成方案：**

```typescript
// 1. 用 ResourceRegistry 接管 bookCreateStatus 的清理定时器
const bookStatusCleanupId = "book-create-status-cleanup";
globalRegistry.register({
  id: bookStatusCleanupId,
  type: "timer",
  dispose: () => { /* 清理循环随 registry 退出而停止 */ },
  createdAt: Date.now(),
});

// 2. 每个 book 的创建任务注册为独立资源
function trackBookCreation(bookId: string, abortController: AbortController) {
  const resourceId = `book-create-${bookId}`;
  globalRegistry.register({
    id: resourceId,
    type: "task",
    dispose: () => {
      abortController.abort();
      bookCreateStatus.delete(bookId);
    },
    createdAt: Date.now(),
    ttlMs: BOOK_CREATE_TIMEOUT_MS,  // 10 分钟硬超时
    meta: { bookId },
  });
  return resourceId;
}

// 3. 状态完成时更新 TTL，并保留成功状态更久供前端查询
function finalizeBookStatus(bookId: string, status: "completed" | "error", error?: string) {
  const ttl = status === "error" ? 30_000 : BOOK_CREATE_TTL_MS; // 错误 30 秒，成功 60 秒
  bookCreateStatus.set(bookId, { status, error, createdAt: Date.now(), ttlMs: ttl });
  // 完成时从 registry 中移除任务资源（但保留 status 供查询）
  globalRegistry.unregister(`book-create-${bookId}`);
}
```

**交互流程：**

```
用户点击建书
  → 创建 AbortController
  → trackBookCreation(bookId, ac) 注册到 globalRegistry
  → pipeline.runAgentSession("architect", ...)
      ├─ 正常完成 → finalizeBookStatus(bookId, "completed")
      ├─ 出错     → finalizeBookStatus(bookId, "error", err)
      └─ 用户取消 → POST /books/:id/create-cancel → ac.abort() → registry.dispose()
  → 30 秒后 setInterval / registry scavenge 清理过期 bookCreateStatus
```

#### 3.5.1 更细化的 `collectedToolExecs` 清理策略

上述简单上限方案存在一个问题：**若长会话中用户连续触发超过 200 个工具，旧记录被直接丢弃，前端进度条可能出现“跳变”**（已完成的历史条目消失）。需要更精细的策略：

**方案 A：滑动窗口 + 保留关键记录**

```typescript
interface ToolExecRetentionPolicy {
  maxRunning: number;      // 同时追踪的 running 工具数（通常很小）
  maxCompleted: number;    // 保留最近完成的 N 个
  maxCompletedPerAgent: number; // 每个 agent 保留最近完成的 N 个
  alwaysKeep: (exec) => boolean; // 永远保留的关键工具
}

const DEFAULT_RETENTION: ToolExecRetentionPolicy = {
  maxRunning: 50,
  maxCompleted: 150,
  maxCompletedPerAgent: 30,
  alwaysKeep: (exec) => exec.status === "error" || exec.tool === "sub_agent" && ["architect", "writer"].includes(exec.agent),
};

function trimCollectedToolExecs(
  execs: CollectedToolExec[],
  policy = DEFAULT_RETENTION,
): void {
  if (execs.length <= policy.maxRunning + policy.maxCompleted) return;

  const running = execs.filter(e => e.status === "running");
  const completed = execs.filter(e => e.status !== "running");

  // 1. 优先删除 oldest completed，但保留 alwaysKeep
  const protectedIds = new Set(completed.filter(policy.alwaysKeep).map(e => e.id));

  // 2. 按 agent 分组保留最近 N 个
  const keepByAgent = new Map<string, Set<string>>();
  const byAgent = new Map<string, CollectedToolExec[]>();
  for (const e of completed) {
    const key = e.agent ?? "_no_agent";
    if (!byAgent.has(key)) byAgent.set(key, []);
    byAgent.get(key)!.push(e);
  }
  for (const [agent, list] of byAgent) {
    const sorted = list
      .filter(e => !protectedIds.has(e.id))
      .sort((a, b) => (b.endedAt ?? b.startedAt) - (a.endedAt ?? a.startedAt));
    const keep = new Set(sorted.slice(0, policy.maxCompletedPerAgent).map(e => e.id));
    keepByAgent.set(agent, keep);
  }

  // 3. 构建最终保留集合
  const keepIds = new Set<string>([
    ...running.map(e => e.id),
    ...[...protectedIds],
    ...[...keepByAgent.values()].flatMap(s => [...s]),
  ]);

  // 4. 原地过滤，保持顺序
  let writeIdx = 0;
  for (let i = 0; i < execs.length; i++) {
    if (keepIds.has(execs[i].id)) {
      execs[writeIdx++] = execs[i];
    }
  }
  execs.length = writeIdx;
}
```

**方案 B：按会话隔离 + 会话结束时整体丢弃（推荐）**

当前 `collectedToolExecs` 位于 `processProjectInteractionRequest` 的请求级闭包中，理论上请求结束即 GC。但由于它通过 `onEvent` 回调被长时 pipeline 持续 push，若 pipeline 运行时间极长（>30 分钟），仍可能累积。

推荐在以下时机主动清空：

| 时机 | 清理动作 |
|------|---------|
| pipeline 正常结束 | `collectedToolExecs.length = 0` |
| pipeline 出错/取消 | `collectedToolExecs.length = 0` |
| 每 push 50 条后 | `trimCollectedToolExecs(collectedToolExecs)` |
| `withPipeline` 的 `finally` | 确保数组引用释放（设为 `null`） |

```typescript
// 在 withPipeline / runAgentSession 的 finally 中
try {
  // ... 运行 pipeline ...
} finally {
  collectedToolExecs.length = 0;
  // 若使用 let 声明，可进一步赋 null 帮助 GC
  (collectedToolExecs as unknown as null) = null;
}
```

**方案 C：前端分页 + 后端只维护当前窗口**

前端进度条无需展示全部历史，只需当前及最近 N 条。后端仅维护 `windowSize=50` 的滚动窗口，前端加载更多时从持久化日志读取（若有）。这对 InkOS 现阶段较重，标记为 **P3 Future**。

## 4. 进程管理方案

### 4.1 单一后端进程 + 多前端 Tab

当前模型已经满足：**一个 Node.js 进程处理所有请求**。不需要为每个前端操作创建子进程。

重点在于**请求级别的资源隔离**而非进程隔离：

```
后端进程 (PID 固定的 Node.js)
├── HTTP 请求处理 (Hono Router)
│   ├── Request A (用户 1 写章) → Pipeline A → dispose A ✅
│   ├── Request B (用户 2 审计) → Pipeline B → dispose B ✅
│   └── Request C (用户 1 建书) → Pipeline C → dispose C ✅
├── SSE 连接管理
│   ├── SSE Connection 1 (Tab 1) → onAbort → 清理 ✅
│   └── SSE Connection 2 (Tab 2) → onAbort → 清理 ✅
└── 定时任务
    └── createStatus 清理 (每 30 秒) → unref 不阻止进程退出 ✅
```

## 5. 性能基准与监控指标

为确保内存优化真正生效，需要建立可重复的测量基准，而非仅凭肉眼观察。

### 5.1 核心指标定义

| 指标 | 获取方式 | 正常范围 | 告警阈值 |
|------|---------|---------|---------|
| **RSS（常驻内存）** | `process.memoryUsage().rss` | < 512 MB（基线） | > 1 GB 持续 2 分钟 |
| **堆内存 usedHeapSize** | `process.memoryUsage().heapUsed` | < 256 MB | > 512 MB 持续 2 分钟 |
| **堆内存 totalHeapSize** | `process.memoryUsage().heapTotal` | < 384 MB | 增长斜率 > 5 MB/min |
| **外部内存 external** | `process.memoryUsage().external` | < 100 MB | 快速增长提示 Buffer/FFI 泄漏 |
| **PipelineRunner 实例数** | `globalRegistry.resources` 中 `type === "pipeline"` 计数 | 0~5（无活跃请求） | > 10 持续 30 秒 |
| **SSE subscribers 数** | `subscribers.size` | 0~前端 Tab 数 | > 50 |
| **agentClients 总数** | 所有活跃 PipelineRunner 的 `agentClients.size` 之和 | 0~100 | > 200 |
| **bookCreateStatus 条目数** | `bookCreateStatus.size` | 0~并发建书数 | > 20 |
| **GC 耗时（增量）** | `--trace-gc` 解析 | < 50 ms | > 200 ms |

### 5.2 内存测量脚本

```typescript
// packages/studio/src/api/memory-monitor.ts
import { globalRegistry } from "@actalk/inkos-core/utils/resource-registry";

export function startMemoryMonitor(intervalMs = 30_000) {
  const timer = setInterval(() => {
    const mem = process.memoryUsage();
    const pipelineCount = [...globalRegistry.resources.values()].filter(r => r.type === "pipeline").length;
    const sseCount = (global as any).__inkos_sse_subscriber_count ?? 0;

    console.log(JSON.stringify({
      ts: new Date().toISOString(),
      rss_mb: Math.round(mem.rss / 1024 / 1024),
      heap_used_mb: Math.round(mem.heapUsed / 1024 / 1024),
      heap_total_mb: Math.round(mem.heapTotal / 1024 / 1024),
      external_mb: Math.round(mem.external / 1024 / 1024),
      pipeline_count: pipelineCount,
      sse_count: sseCount,
    }));
  }, intervalMs);

  timer.unref();
}
```

在 `server.ts` 启动时调用：

```typescript
import { startMemoryMonitor } from "./memory-monitor";
if (process.env.NODE_ENV !== "test") {
  startMemoryMonitor();
}
```

## 6. Node.js GC 与运行时调优

InkOS 后端主要压力来自：
1. 大文本（章节内容）在 pipeline 各 agent 间传递；
2. 流式响应时大量小 Buffer 分配；
3. 长时间运行后旧生代（old space）中积累的死对象。

### 6.1 推荐启动参数

| 参数 | 作用 | 建议值 |
|------|------|--------|
| `--max-old-space-size` | 限制老生代最大内存，避免系统交换 | 2048（开发） / 4096（生产） |
| `--expose-gc` | 允许手动触发 `global.gc()`（用于测试和压力后回收） | 仅调试开启 |
| `--trace-gc` | 输出每次 GC 耗时和回收量 | 仅性能分析时开启 |
| `--heapsnapshot-near-heap-limit=max_count=3` | 接近内存上限时自动生成堆快照 | 生产调试 |
| `--optimize-for-size` | 牺牲部分执行速度换取更小内存占用 | 低端机器可选 |
| `--no-concurrent-array-buffer-sweeping` | Node 20+ 选项，可能减少大 Buffer 场景的 GC 抖动 | 测试后决定是否使用 |

**开发环境示例：**

```json
// packages/studio/package.json
{
  "scripts": {
    "dev": "node --max-old-space-size=2048 --expose-gc scripts/dev.mjs",
    "dev:profile": "node --max-old-space-size=2048 --expose-gc --trace-gc scripts/dev.mjs"
  }
}
```

### 6.2 手动触发 GC 的端点

```typescript
// server.ts 中仅在开发/调试模式下注册
if (process.env.NODE_ENV === "development" || process.env.INKOS_ENABLE_ADMIN === "1") {
  app.post("/api/v1/admin/gc", (c) => {
    if (typeof global.gc === "function") {
      global.gc();
      return c.json({ ok: true, memory: process.memoryUsage() });
    }
    return c.json({ ok: false, error: "global.gc not available; restart with --expose-gc" }, 400);
  });
}
```

### 6.3 代码层面的 GC 友好实践

1. **避免闭包捕获大对象**：pipeline callback 中不要无意识地捕获 `collectedToolExecs` 全量引用。
2. **流式响应后清空 Buffer 链**：SSE 推送完成后，确保 `ReadableStream` 的 reader 被释放。
3. **大字符串不重复拷贝**：`StateManager` 加载书籍内容时尽量使用引用传递，修改时再做 copy-on-write。
4. **定时器使用 `unref()`**：所有心跳、清理定时器都应 `unref()`，避免空转时阻止进程退出。
5. **避免全局累积的日志缓存**：`logger.child(agent)` 不要无限缓存日志行。

## 7. dev/prod 进程模型差异与部署建议

### 7.1 开发环境（当前）

```
dev.mjs
 ├── API Server (Node.js, 4579)
 └── Vite Dev Server (Node.js, 4577/4578)
```

特点：
- 两个 Node 进程，内存占用翻倍（每个约 200~400 MB）。
- Vite 侧存在模块热更新（HMR）缓存，长时间运行也会增长。
- API 进程泄漏会被 Vite 进程隔离，但前端代理保持连接时，泄漏仍影响 API 进程。

**开发期建议：**
- 每日至少重启一次 `pnpm dev`；
- 在 Vite server 配置中限制 HMR 缓存大小：
  ```typescript
  // vite.config.ts
  server: {
    hmr: { overlay: false },
    warmup: { clientFiles: [] },
  }
  ```

### 7.2 生产环境建议

InkOS 当前主要作为桌面/本地应用运行，但未来若提供服务器部署，需要考虑：

**模式 A：单一 Node 进程（最简单，适合低并发）**

```
Node.js (Hono + 静态文件服务)
 └── 所有请求在同一进程处理
```

- 使用 `serveStatic` 中间件替代独立的 Vite server。
- 静态资源由 CDN / Nginx 前置缓存。
- 适用：个人使用、小团队、并发 < 10。

**模式 B：Cluster 模式（Node.js cluster，适合中等并发）**

```
Master (负载均衡)
 ├── Worker 1 (API)
 ├── Worker 2 (API)
 └── Worker 3 (API)
```

- 每个 worker 独立管理自己的 `ResourceRegistry`。
- SSE 连接需要 sticky session（按客户端 IP 或 cookie 固定到同一 worker）。
- 单 worker 崩溃时 master 重启，不影响其他连接。

**模式 C：Docker / PM2（长期运行的服务器）**

```yaml
# docker-compose.yml 示例
services:
  inkos-api:
    image: inkos:latest
    command: node --max-old-space-size=4096 dist/server.js
    restart: unless-stopped
    deploy:
      resources:
        limits:
          memory: 6G
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:4579/api/v1/health"]
      interval: 30s
      timeout: 5s
      retries: 3
```

PM2 配置示例：

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: "inkos-api",
    script: "./packages/studio/dist/server.js",
    instances: 2,
    exec_mode: "cluster",
    max_memory_restart: "2G",
    env: { NODE_ENV: "production" },
    kill_timeout: 5000,
    listen_timeout: 10000,
    // 优雅关闭
    wait_ready: true,
  }],
};
```

### 7.3 不同模式下的 graceful shutdown 差异

| 场景 | 需要处理的事项 |
|------|---------------|
| dev 双进程 | SIGINT 同时杀 Vite + API；API 进程应独立完成 `globalRegistry.disposeAll()` |
| prod 单一进程 | 收到 SIGTERM 后，停止接受新请求，等待活跃 pipeline 完成（或超时 10 秒），再退出 |
| cluster worker | 收到 disconnect 后，向 master 报告 ready-to-die，避免未完成的 SSE 连接被切断 |
| Docker | 使用 `STOPSIGNAL SIGTERM`，确保 `docker stop` 触发优雅关闭而非直接 SIGKILL |

### 7.4 进程保活与健康检查

