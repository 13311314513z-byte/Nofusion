## 二、发现的 Bug 与修复

### Bug-1：`migrateIntentsIndex` 对无效/空 intent 过于严格

**文件**：`packages/core/src/models/chapter-intent.schema.ts`  
**严重度**：P1（数据加载失败）

**问题**：
- 原实现使用 `ChapterIntentsIndexSchema.partial().parse(raw)`，然后对 `raw.intents` 每个元素调用 `migrateLegacyIntent`；
- 如果 intents 数组中存在 `null` 或缺少 `chapterNumber` 的元素，`parse` 会抛出异常，导致整本书的 intent 加载失败；
- 原有 `loadChapterIntents` 实现会过滤掉无效/null 元素，新实现丢失了这个行为。

**修复**：
- 改用 `safeParse`；
- 逐个 intent 用 `AuthorChapterIntentSchema.safeParse` 验证，跳过 `null`/非对象/验证失败的条目；
- 保留向后兼容的默认值填充。

**验证**：`src/__tests__/chapter-intent.test.ts` 中 "filters out invalid intents on load" 测试从失败变为通过。

---

### Bug-2：`intent_commitments` 表缺少 SQLite 迁移

**文件**：`packages/core/src/state/memory-db.ts`  
**严重度**：P1（现有项目运行时报错）

**问题**：
- 新增 `intent_revision` 列后，`CREATE TABLE IF NOT EXISTS` 不会修改已存在的表；
- 对于已有 SQLite 数据库的项目，`INSERT` 语句会报“table has 6 columns but 7 values were supplied”。

**修复**：
- 在 `initSchema()` 中调用已有的 `ensureColumn()` 辅助方法：
  ```typescript
  this.ensureColumn("intent_commitments", "intent_revision", "INTEGER NOT NULL DEFAULT 0");
  ```
- `ensureColumn` 会静默忽略“列已存在”的错误，兼容新旧数据库。

**验证**：测试创建的是新库，因此不能直接验证迁移；但 `ensureColumn` 模式与现有 `hooks.payoff_timing` 迁移一致，属于已验证的模式。

| 项目 | 评价 |

| 版本追踪字段 | `revision` / `status` / `updatedAt` / `source` 对 intent 的演进管理很有必要 |

| 成对偏好评估 | 相比绝对 1-10 分，成对比较更稳定，符合 Phase 0.5 的实验设计 |

### 3.2 需要关注的问题

#### Issue-1：类型重复定义可能造成维护混乱

`chapter-intent.ts` 和 `chapter-intent.schema.ts` 都定义了 `AuthorChapterIntent` / `AuthorScenePlan` / `AuthorCharacterState` / `ChapterIntentsIndex`。

- 目前 `index.ts` 只导出 `chapter-intent.ts` 的类型，不导出 schema 文件的类型，所以包边界没有冲突；
- 但两个文件中的同名类型结构略有不同（schema 类型数组是 `string[]`，interface 是 `ReadonlyArray<string>`）；
- 长期维护中容易修改一处、忘记另一处。

**建议**：后续重构时让 `chapter-intent.ts` 的类型从 schema 推导，或把 schema 文件中的类型重命名为 `*SchemaType` 避免混淆。

#### Issue-2：`paired-preference.ts` 的 `interReaderAgreement` 命名不够准确

函数实现是“读者两两答案一致率”，但接口字段和注释提到 Fleiss' Kappa。对于当前阶段可接受，但后续如果要做严肃统计应重命名或实现真正的 Kappa。

**建议**：将字段重命名为 `pairwiseAgreement` 或 `rawAgreement`，避免误导。

#### Issue-3：`preference-eval.mjs` 的 CSV 解析器较脆弱

当前按 `split(",")` 解析，如果 freeform 备注中包含逗号或引号会出错。

**建议**：内部工具可先用；若要长期维护，建议改用 `csv-parse` 等库，或至少将 freeform 字段禁止包含换行/逗号。

#### Issue-4：`evaluate-chapter.mjs` 未校验输入文件存在性

直接 `readFile(resolve(versionAFile))`，如果路径错误会抛出未处理异常。

**建议**：添加 `try/catch` 和友好的错误提示。
