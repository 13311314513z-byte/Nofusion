## 一、结论摘要

## 四、P0：发布前必须完成

### P0-1 Studio 构建假成功，发布包缺少服务端入口

**证据**

- `packages/studio/package.json:25-27` 先执行 `vite build`，再执行 `tsc -p tsconfig.server.json`。
- `tsconfig.json:16` 开启 `composite`。
- `packages/studio/tsconfig.server.json:4` 将服务端输出到与 Vite 相同的 `dist`。
- 实测构建退出码为 0，但 `packages/studio/dist/api/index.js` 不存在。

**原因判断**

Vite 清空 `dist` 后，TypeScript 增量构建缓存仍可能认为服务端文件已是最新状态，因此不重新输出文件。

**改进方案**

1. 将客户端与服务端分别输出到临时目录，再合并发布产物，避免 Vite 清理服务端输出目录。
2. 服务端发布构建禁用增量缓存或使用强制重建，例如 `tsc -b --force`，避免仅依赖时间戳缓存。
3. 在 `build` 和 `prepack` 后增加产物断言：`dist/index.html`、`dist/api/index.js`、`dist/api/index.d.ts` 缺一即失败。
4. CI 增加 `npm pack --dry-run` 或 tar 内容检查，禁止“命令成功但包不可运行”。

**验收标准**

- 连续执行两次 Studio build，三项关键产物始终存在。
- 清空 `dist` 但保留 `tsbuildinfo` 后重新构建，服务端入口仍能生成。
- 发布包测试通过。

### P0-2 封面 API Key 可能被静默覆盖为空

**证据**

- 后端 `GET /cover/secret/:service` 返回 `{ hasApiKey, keyPreview }`：`server.ts:2697-2710`。
- 前端仍按 `{ apiKey?: string }` 读取：`ServiceListPage.tsx:106-109`。
- 保存时无条件提交当前输入框内容：`ServiceListPage.tsx:131-134`。
- 后端收到空字符串后仍调用 `setServiceApiKey`：`server.ts:2713-2725`。

用户只修改封面模型并点击保存时，前端输入框为空，已有封面 Key 可能被清空。

**改进方案**

1. 前端增加 `hasStoredKey`、`keyPreview`、`keyDirty` 状态。
2. 未修改 Key 时不发送 `apiKey` 字段。
3. 后端将“保持原值”和“清除 Key”拆成不同语义；清除应使用显式 DELETE 或 `clear: true`。
4. 增加“已有密钥，留空不会覆盖”的提示。

**验收标准**

- 已有 Key 时仅修改模型，Key 保持不变。
- 输入新 Key 时仅替换当前封面服务商 Key。
- 切换封面服务商不会串 Key。
- 清除 Key 必须经过明确操作和确认。

### P0-3 恢复发布门禁

在上述问题修复前，`pnpm test` 和发布包验证均不能作为可信绿灯。下一轮发布必须以以下条件为门禁：

- Core、Studio、CLI 测试全部通过。
- Studio 构建产物断言通过。
- `npm pack` 内容验证通过。
- API Key 保留/替换/清除回归测试通过。

## 五、P1：下一轮主迭代

### P1-1 统一 CLI JSON 契约并处理兼容性

`formatJsonOutput` 已输出 `{ status, data, meta }`，但只有部分命令采用，既有测试仍读取扁平结构。当前状态会同时破坏旧脚本并造成命令间格式不一致。

建议：

1. 明确采用统一包装还是保持旧格式，不要停留在半迁移状态。
2. 若采用包装格式，增加 `schemaVersion`，一次性迁移全部 `--json` 命令和测试。
3. 短期可保留兼容字段，或提供 `--json-version 2`。
4. `--format` 使用 Commander choices 或 Zod 运行时校验，移除类型断言。
5. 为循环引用、BigInt 和未知 Error 增加安全序列化降级。

### P1-2 修复 doctor 对本地兼容端点的长等待

localhost API Key 可选逻辑已经存在，但 doctor 仍会直接探测未启动的本地端点，导致检查结果迟迟不能输出。

建议：

1. 基础配置检查先输出，连通性探测放在最后。
2. 设置明确的 2-3 秒单次超时和总预算。
3. 增加 `--skip-connectivity` 与 `--connectivity-timeout`。
4. 本地端点不可达应报告“Key 可选但服务未启动”，不要混同为鉴权失败。

### P1-3 修正项目根目录边界校验

`server.ts:1714-1719` 使用字符串 `startsWith` 判断路径归属。类似 `Nofusion-main-evil` 的同前缀兄弟目录可能被误判为合法。

建议改为 `path.relative`：

- 相对路径不能以 `..` 开头。
- 不能是绝对路径。
- API 请求原则上不应允许覆盖服务端启动时确定的项目根目录。

同时把非法 `authorId`、非法 URL、越界路径从 500 改为稳定的 400/403。当前多个蒸馏路由在 `try` 外调用 `assertSafeAuthorId`，错误状态不明确。

### P1-4 完成文风“检测到修改”的操作闭环

文风 Step 4 已渲染可读性、重复段落和修辞问题组件，但关键动作仍是日志占位：

- 删除段落：`StyleManager.tsx:1264`
- 合并段落：`StyleManager.tsx:1265`
- 高亮、忽略、AI 改写、标记修复：`StyleManager.tsx:1276`
- 作家对比结果：`StyleManager.tsx:2008`

建议形成统一闭环：

`问题定位 -> 原文高亮 -> 修改预览 -> diff 确认 -> 保存章节/样本 -> 重新检测 -> 记录结果`

`/style/rhetoric/rewrite` 当前只返回 prompt，不是最终改写文本。前端“AI 改写”必须再调用已配置模型，并提供失败回退与人工编辑。

### P1-5 建立真实的文风分析会话状态

`StyleSourceContext`、`StyleAnalysisSession`、`TextRangeIssue` 已定义，但尚未被页面状态实际采用；AI 痕迹检测仍固定 `language="zh"`。

建议：

1. 由统一 session 保存来源类型、bookId、chapterNumber、语言、原文 hash 和各步骤结果。
2. 所有 Step 读取同一 session，避免切换步骤后来源和结果错位。
3. 保存章节前校验原文 hash 或版本号，避免覆盖用户在其他页面的新修改。
4. `TextRangeIssue` 作为诊断、高亮、修辞和可读性问题的统一定位结构。

### P1-6 接入作家搜索与蒸馏工作台

当前后端已有搜索、抓取、样本写入、蒸馏草稿、发布和版本端点，但：

- `AuthorSearchPanel` 仅有定义，没有页面使用。
- `AuthorProfileCard` 仅有定义，没有页面使用。
- 蒸馏端点没有对应前端调用。

建议优先完成最小工作台：

1. 搜索并选择来源。
2. 抓取预览、版权/来源提示、手动确认。
3. 写入样本并生成档案。
4. 生成蒸馏草稿，逐条启用、禁用和编辑规则。
5. 发布版本并保留回滚记录。
6. 应用到文段前展示差异和约束来源。

### P1-7 修复 Pipeline 生命周期管理

`withPipeline` 的 `_ttlMs` 未使用，`dispose()` 又在 `finally` 中直接执行。

风险：

- 名义上有 TTL，实际上没有超时保护。
- dispose 抛错可能覆盖原始业务异常。
- 长任务取消、客户端断连和资源回收没有统一语义。

建议使用 `AbortSignal.timeout` 或统一任务控制器，并在清理异常时保留原始错误。

### P1-8 增加章节正文版本历史与冲突保护

当前 truth state 快照主要用于运行时状态恢复，不等同于用户可见的章节正文版本管理。

建议增加：

- 章节保存版本号或 ETag。
- 保存前冲突检测。
- 正文历史列表与逐行 diff。
- 恢复历史版本时同时触发审计和状态一致性检查。

这也是文风“调整建议保存到章节”安全落地的前置条件。

### P1-9 让“一键检测全部”执行真实连通性测试

`ServiceListPage.tsx:335-343` 的按钮主要刷新服务和模型列表，名称却是“一键检测全部”，容易让用户误以为已完成鉴权、模型和协议验证。

建议：

1. 后端提供批量测试任务，限制并发数。
2. 每个服务返回鉴权、模型、协议、延迟、错误类型。
3. 前端展示进度和逐项重试。
4. CLI 补充 `doctor --all-services`，复用同一测试内核。

## 六、P2：稳定性与体验债务

### P2-1 新组件主题适配不完整

`components/readability` 与 `components/author` 中仍有大量 `bg-white`、`text-gray-*`、`border-gray-*`，与项目主题变量体系不一致，深色模式下可读性存在风险。

应统一使用 `bg-card`、`text-foreground`、`text-muted-foreground`、`border-border` 等语义颜色。

### P2-2 清理死代码和孤立组件

当前确认的孤立内容包括：

- `useAutoSave` 默认调用不存在的 `/api/v1/session/auto-save`。
- `RhetoricHighlightEditor` 未被页面使用。
- 作家搜索和档案卡组件未接入。

处理原则：能在本轮接入的接入；没有明确入口和验收标准的删除或标记为实验模块，避免形成“代码存在即功能完成”的误判。

### P2-3 诊断历史与审计记录

文风诊断已有保存动作，但缺少完整的历史浏览、前后对比和规则版本信息。

建议每次记录：

- 来源、文本 hash、语言。
- 规则版本和作家档案版本。
- 修改前后指标。
- 用户采纳/忽略项。
- 最终保存目标和章节版本。

### P2-4 状态变更日志

`state_changelog.jsonl` 仍未形成统一实现。建议记录人物、地点、伏笔和时间线变更，支持按章节和实体筛选，但不应替代现有 truth snapshot。

### P2-5 规则配置外置

文风诊断、AI 痕迹、敏感词和写后校验仍含较多硬编码阈值。建议先形成有版本的规则配置层，再开放 UI 调整，避免规则散落在多个模块。

### P2-6 前端异步操作重入保护

导入、分析、应用作家、批量检测、抓取来源等操作仍需统一：

- loading 状态。
- 禁止重复提交。
- AbortController。
- 请求序号或版本校验。
- 可恢复错误提示。

### P2-7 导出入口一致性

Core 和 BookWorkspace 已支持 HTML，但旧 `BookDetail`、CLI 帮助文本和部分入口仍只展示 txt/md/epub。应统一格式枚举、帮助文案和下载行为。

### P2-8 前端包体积治理

本轮 Vite 构建主 bundle 约 2.7 MB，且存在多个超过 500 KB 的 chunk。建议对 Mermaid、Shiki、图表和编辑器能力按路由动态加载。

## 九、下一轮验收标准

下一轮不能只以“页面出现了组件”作为完成标准，至少应满足：

- 构建产物可执行、可打包、可安装。
- 全量自动化测试通过。
- 已保存 Key 不会因修改无关配置而丢失。
- 所有新接口有成功、参数错误、越界、安全和失败路径测试。
- 文风问题能够定位到原文并完成预览、确认、保存、复检。
- 作家蒸馏能够在前端完成草稿、编辑、发布、版本查看和应用。
- 章节保存具备版本冲突保护。
- 批量服务检测展示真实探测结果，不以模型列表刷新代替。

## 十、最终判断

当前项目不是“主要功能未实现”，而是已经进入**集成收口和发布治理阶段**。旧文档中大量 P0-P1 已完成，新的主要风险集中在：

1. 构建与发布结果不可信。
2. 掩码密钥改造后存在前后端兼容遗漏。
3. CLI 契约迁移缺乏统一边界。
4. 文风和作家蒸馏后端能力尚未转化为完整用户流程。
5. 章节修改缺少版本化保护。

建议暂停继续铺设大模块，先完成批次 A、B，再以文风闭环作为下一轮核心功能目标。
