## 二、代码结构回顾

InkOS 是一个多智能体小说写作 Agent，采用 pnpm workspace 三件套：

| 包 | 职责 | 关键目录 |
|---|---|---|
| `@actalk/inkos-core` | 引擎：模型、Agent、Pipeline、LLM 接入层、状态管理 | `src/agents/`, `src/pipeline/`, `src/llm/`, `src/state/` |
| `@actalk/inkos` | CLI + TUI（Commander + Ink/React） | `src/commands/`, `src/tui/` |
| `@actalk/inkos-studio` | Vite + React + Hono Web 工作台 | `src/api/routes/`, `src/components/`, `src/pages/` |

此外顶层 `scripts/` 包含大量评估、校准、E2E 验证脚本；`books/` 存放 7 本示例/测试书籍；`reports/` 存放 195+ 份历史汇报。

## 六、诚实标注的限制与风险

1. **Bundle 预算依赖临时上限**：原 15,500 KB 已被突破（超出约 700 KB），当前 16,500 KB 是临时豁免值。
2. **CLI 测试需要全局 pnpm**：在仅安装本地 pnpm 的环境（如本机实测）下，`cli-integration.test.ts` 全部失败。
3. **Studio 测试存在首次抖动**：第一次 `pnpm --filter @actalk/inkos-studio test` 出现 61 个失败，无代码改动重跑后全部通过，疑似与状态/缓存/异步初始化有关。
4. **`inkos eval` 崩溃**：对当前 7 本书中的任意一本运行都会触发，属于可直接复现的 P1 缺陷。
5. **`inkos audit` 覆盖历史 issue**：审计后会丢失该章节之前的 `auditIssues`，仅保留最新一轮结果，对长期追踪不利。
6. **人力校准数据为合成数据**：reports 中 0621 诚实终验已明确，beta-reader / stage5 / human-reader-agreement 等数据为占位/模拟数据，不能作为真实质量验收依据。
7. **部分书籍元数据不完整**：`test-book-0609` 的 genre/platform 显示为 `undefined`；多本通兰县系列书籍仍停留在 `outlining` 状态。

## 七、结论

截至 2026-06-22，`ca39e58` 基线的 Nofusion 项目处于**“自动化门禁基本通过，但存在可复现功能缺陷与测试环境依赖”**的状态：

- **可以交付的程度**：类型安全、构建产物、核心功能测试、Studio 构建预算均已通过；主要历史重构（server 路由拆分、StyleManager 拆分、动态导入静态化）已完成。
- **不能视为完全稳定的点**：`inkos eval` 命令崩溃、`inkos audit` 历史记录丢失、CLI 集成测试强依赖全局 pnpm、Studio 测试偶发失败、runner.ts / writer.ts / server.test.ts 仍未完成拆分、`any` 数量偏高。

建议下一步优先级：

1. **修复 `inkos eval` 的 `auditIssues` 空值防御**（1 行改动）。
2. **评估 `inkos audit` 的 issue 合并策略**，避免覆盖历史审计记录。
3. **改造 `cli-integration.test.ts`**，使其支持项目本地 `node_modules/.bin/pnpm` 或降级为不依赖 pnpm 的子进程调用。
4. **继续拆分 `runner.ts`、`writer.ts`、`server.test.ts`** 并降低 `any` 使用。
5. **为 Bundle 预算制定回退方案**，避免长期依赖临时上限。
