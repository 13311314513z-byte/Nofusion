## 一、🔴 P0：三源共识——立即处理

### P0-1：Fire-and-forget 写作操作缺乏超时与回滚（DS + GPT）

**位置**: `server.ts` — `write-next`(L2298) / `draft`(L2316) / `rewrite`(L3234)

```typescript
// 返回 { status: "writing" } 后完全脱离监控
withPipeline("write-next", ...).catch((e) => broadcast("write:error", ...));
return c.json({ status: "writing", bookId: id });
```

| 风险 | DS | GPT | KM |
|------|:--:|:--:|:--:|
| 客户端无法区分"仍在写"和"已崩溃" | ✅ | ✅ | — |
| 无超时——LLM挂起永久占用资源 | ✅ | — | — |
| 无回滚——步3失败导致步1-2文件变更残留 | ✅ | — | — |
| 前端30s超时误杀长任务，触发重复写作 | — | ✅ | — |

**三源共识修复**: 改为 job 模式（提交→jobId→SSE/轮询→结果）+ 幂等键防重复

---

### P0-2：`new StateManager(root)` 重复创建 12 次（DS）

**位置**: `server.ts` L1717 + L2280 + L3415 + L3430 + L3452 + L3509 + L3584 + L3607 + L3621 + L3637 + L3653 + L3674

每次新建丢失缓存 + 内存分配 + GC 压力 + 并发时缓存不一致风险。

**建议**: 复用顶层 `state` 变量 → `state.bookDir(id)`。预计修复 30min。

---

### P0-3：`fetchJson` 全局 30 秒超时误杀长耗时 AI 端点（GPT 独特发现）

**位置**: `use-api.ts` — 0618 新增的全局 `AbortSignal.timeout(30_000)`

| 受影响端点 | 典型耗时 | 风险 |
|------|:--:|------|
| `POST /write-next` | 60-300s | 前端中断→后台继续→重复写作 |
| `POST /draft` | 60-180s | 初稿误判失败 |
| `POST /revise/:chapter` | 30-120s | 修订链路断裂 |
| `POST /style/authors/:id/distillations` | 60-300s | 蒸馏超时→输出品质链路断裂 |
| `POST /import/foundation/commit` | 60-300s | 长文本导入易超时 |
| `POST /radar/scan` | 60-300s | 市场扫描耗时不可控 |
| `GET /doctor` | 15-60s | 多 provider 探测可能被提前中断 |

**建议**: 分级超时——CRUD 30s / 写作 300s / job 化长任务禁用前端超时

### P0-5：Stream Reader 未释放——连接泄漏风险（KM 独特发现）

**位置**: `core/llm/provider.ts` 多处流式响应处理

```typescript
const reader = response.body?.getReader();
try {
  while (true) { const { value, done } = await reader.read(); ... }
} finally {
  monitor.stop();  // ← 缺少 reader.releaseLock() / reader.cancel()
}
```

异常或提前退出时 HTTP body reader 可能长时间占用连接。

---

### P0-6：API Key 明文存储（KM 独特发现）

**位置**: `core/llm/secrets.ts:51-62` — `saveSecrets` 将 API key 以明文写入 `.inkos/secrets.json`。

**建议**: 使用系统 keychain 或至少加密存储 + 限制文件权限

### P0-7：脆弱 LLM 输出解析——影响输出品质（KM 独特发现）
