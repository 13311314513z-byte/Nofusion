## 2. 最高风险：全局 30 秒前端超时

位置：`packages/studio/src/hooks/use-api.ts`

当前 `fetchJson` 引入了：

- `DEFAULT_FETCH_TIMEOUT_MS = 30_000`
- `AbortSignal.timeout(timeoutMs)`
- 若有外部 signal，则尝试 `AbortSignal.any([deps.signal, timeoutSignal])`

这对普通 CRUD 是正向修复，但对 AI 写作系统非常危险，因为大量业务端点天然超过 30 秒。

### 2.1 直接受影响端点

| 端点 | 位置 | 风险 |
|---|---|---|
| `POST /api/v1/books/:id/write-next` | `server.ts:2291` | 章节写作可能超过 30 秒；前端中断后服务端仍可能继续跑，造成重复写作或 token 浪费 |
| `POST /api/v1/books/:id/draft` | `server.ts:2309` | 初稿生成被误判失败，影响用户对输出质量的判断 |
| `POST /api/v1/books/:id/revise/:chapter` | `server.ts:3004` | 修订链路中断后，前端不知道最终状态 |
| `POST /api/v1/books/:id/audit/:chapter` | `routes/audit.ts:295` | 审校超时会导致质量反馈缺失 |
| `POST /api/v1/style/authors/:authorId/distillations` | `routes/style.ts:1042` | 风格蒸馏可能被前端误杀，输出品质链路断裂 |
| `POST /api/v1/books/:id/import/foundation/commit` | `routes/style.ts:1374` | 长文本导入和 foundation 生成易超时 |
| `POST /api/v1/books/:id/event-chain/extract` | `routes/style.ts:1849` | 事件链抽取可能依赖长文本分析 |
| `POST /api/v1/books/:id/voice-profiles/analyze` | `server.ts:3499` | 角色声音分析容易超过 30 秒 |
| `POST /api/v1/radar/scan` | `server.ts:3763` | 市场扫描依赖网络/LLM，耗时不可控 |
| `GET /api/v1/doctor` | `server.ts:3790` | 多 provider 探测时可能被前端提前中断 |
| `POST /api/v1/services/:service/test` | `routes/services.ts:502` | 弱网络 provider 测试可能误报失败 |
| `GET /api/v1/services/:service/models` | `routes/services.ts:601` | 模型列表刷新受上游延迟影响 |

### 2.2 具体风险

1. 前端 30 秒报错，但服务端任务可能继续运行，用户重试后可能触发重复写入、重复导入或重复消耗 token。
2. 写作、审校、蒸馏、导入这些端点直接影响内容产出；超时策略过短会降低用户感知的输出品质。
3. `AbortSignal.any` 在不支持的运行时会 fallback 到 `deps.signal`，此时 timeout 可能失效；而 `AbortSignal.timeout` 本身也需要浏览器兼容性确认。

### 2.3 建议

短期：

- 为 `fetchJson` 增加 `timeoutMs` 参数，默认 30 秒只用于普通 CRUD。
- 对写作、审校、导入、蒸馏、radar、doctor 设为 120-300 秒，或显式禁用前端短超时。
- 捕获 `AbortError` 时给出“请求超时/后台可能仍在运行”的专门文案。

中期：

- 长任务端点改成 job 模式：提交任务 -> 返回 jobId -> SSE/轮询状态 -> 拉取结果。
- 写作、导入、蒸馏必须通过 `sessionId` 或 idempotency key 防重复提交。

## 3. Studio API 架构债

当前 `server.ts` 仍有 3513 行，已拆出的 route module 有 18 个，但大量高价值端点仍在入口文件 inline：

| 功能域 | 代表端点 | 影响 |
|---|---|---|
| interaction/session | `GET /api/v1/interaction/session`、`/sessions/*` | 会话、agent 上下文核心入口 |
| agent | `POST /api/v1/agent` | 工具调用、thinking、写作协作核心 |
| write/draft | `/write-next`、`/draft` | 章节产出核心链路 |
| revise/export | `/revise/:chapter`、`/export`、`/export-save` | 后处理与交付 |
| runtime/truth | `/runtime`、`/truth/:file` | 状态、设定、truth 文件 |
| voice/roles/templates | `/voice-profiles`、`/roles`、`/scene-templates` | 输出一致性与角色质量 |
| fanfic/radar/doctor | `/fanfic/*`、`/radar/*`、`/doctor` | 辅助生产与诊断 |

这不是当前运行故障，但会带来三类后续风险：

1. 单文件承载过多业务域，局部改动容易误伤其他端点。
2. route module 与 inline route 的边界不清，后续继续拆分时需要额外契约测试保护。
3. API 文档、OpenAPI 生成和前端调用契约更难保持同步。

建议拆分优先级：

1. `sessions-agent.ts`：`/sessions/*`、`/agent`、`/interaction/session`
2. `writing.ts`：`/write-next`、`/draft`、`/write-preview`、`/plan-alternatives`
3. `revision-export.ts`：`/revise`、`/rewrite`、`/resync`、`/export`
4. `runtime-truth.ts`：`/runtime`、`/truth/:file`
5. `book-artifacts.ts`：roles、voice、scene templates、event chain

每次拆分都应先补或搬迁契约测试，避免再次出现“文件存在但未真实生效”的历史问题。

## 4. 输出品质相关债

### 4.1 真实 LLM E2E 尚未成为质量门禁

当前 `e2e-real-llm.test.ts` 已需要 `INKOS_RUN_REAL_LLM_E2E=1` 才执行，这是正确方向。但仍有三个残留问题：

- 测试会读取仓库根 `.env`，不完全符合 CI/本地隔离习惯。
- 日志仍打印 API key 长度。
- 验收只覆盖 create -> write -> audit 的基本路径，尚不能证明长篇一致性、风格稳定性、修订质量。

建议：

1. key 只从环境变量读取；如需 `.env`，明确由测试 runner 加载，不在测试中手动扫描。
2. 移除 key 长度日志。
3. 增加小型真实质量 benchmark：同一 prompt 连续 3 章，检查 hook、角色、状态、审校修订前后差异。

### 4.2 Interaction runtime 仍有未实现意图

`packages/core/src/interaction/runtime.ts` 仍有 book draft、book creation、book export、fallback intent 未实现提示。Studio/CLI 可能通过直接路径绕过，但统一 agent/TUI/interaction 入口会出现能力不一致。

影响：

- 用户在 Chat/TUI 中请求“创建书/导出/构思草稿”时，可能得到未实现响应。
- 同一能力在 Studio 页面存在，在 agent runtime 不存在，削弱产品一致性。

建议将 create/export/draft 迁入 interaction tools，至少提供与 Studio/CLI 等价的薄封装。

### 4.3 `any` 与 schema 边界仍偏宽

高频出现位置包括：

- Core agent tools/session/message 转换
- Studio server LLM response 解析
- Studio service/model 配置
- Chat store runtime

这类债短期不阻塞，但会影响输出品质问题定位：模型响应、tool result、thinking block、audit issue 一旦 shape 漂移，类型系统很难提前拦住。

建议优先收敛：

1. `AgentMessage` / transcript / thinking block schema
2. LLM provider response schema
3. Studio `/agent` request/response schema
4. audit issue 与 revise result schema

## 5. 运行效率相关债

### 5.1 CLI `pretest` 执行 `pnpm install`

位置：`packages/cli/package.json`

当前：

```json
"pretest": "pnpm run build && pnpm install"
```

问题：

- 测试阶段修改依赖环境，可能改变 workspace 状态。
- CI 和本地测试变慢。
- 若网络、registry、lockfile、pnpm store 状态异常，会把“依赖安装问题”混入“代码测试问题”。

建议：

- 移除 `pnpm install`。
- 依赖链接问题交给 CI checkout/setup 阶段处理。
- CLI 测试只做 build/test，不做包管理副作用操作。

### 5.2 Bundle budget 通过但余量薄

本次执行 `pnpm --filter @actalk/inkos-studio build:budget`：

| 指标 | 当前 | 上限 | 判断 |
|---|---:|---:|---|
| JS | 14311.5 KB | 18000 KB | 通过 |
| CSS | 158.4 KB | 300 KB | 通过 |
| Max chunk | 1616.6 KB | 1700 KB | 接近上限 |
| Dist | 14470.7 KB | 15000 KB | 接近上限 |

虽然近期 chunk 优化有效，但 Dist 和最大 chunk 的余量很薄。后续新增 Studio 页面、AI components、markdown/code/math/mermaid 相关依赖时，很容易重新超线。

建议：

- 对 markdown、code highlight、mermaid、style 工具继续保持懒加载。
- 将 `build:budget` 纳入 `test:ci` 或 release gate。
- 增加“新增 chunk 来源”报告，避免只看到总量。

### 5.3 `routes/style.ts` 和 `routes/services.ts` 是热点大文件

建议按业务域拆：

- `style-analysis.ts`

- `book-import.ts`

## 6. SSE / 实时反馈债

当前存在至少两套事件定义：

- 前端：`packages/studio/src/hooks/use-sse.ts`
- 后端类型 helper：`packages/studio/src/api/sse-events.ts`
- 实际 route：`packages/studio/src/api/routes/events.ts`

近期 `thinking:start`、`thinking:end`、`thinking:delta` 已补到前端事件列表，但 `sse-events.ts` 的 canonical list 与实际 route/subscriber 机制仍不是同一个来源。

影响：

- 后端新增事件时，前端可能没有订阅。
- 前端新增事件时，后端类型 helper 不知道。
- thinking / tool / llm progress 这类输出体验核心事件容易出现“服务端发了但 UI 不显示”。

建议：

1. 保留唯一事件 manifest，例如 `shared/sse-events.ts`。
2. 前端 `STUDIO_SSE_EVENTS` 从 manifest 派生。
3. 后端 `broadcast` 也使用同一 union 类型。
4. 加测试：所有 broadcast event 都被前端订阅，所有前端订阅 event 都在后端允许列表内。

## 9. 最终结论

当前项目的主要风险已经从“能不能启动/测试能不能过”转为“长链路是否可靠、长任务是否能被正确观测、真实输出品质是否可复现”。最先应处理的是 `fetchJson` 全局 30 秒超时，因为它直接横跨运行效率和输出品质：既会误杀长任务，也会制造重复请求、重复 token 消耗和用户误判。

只要先把长任务超时/job 化、CLI 测试副作用、真实 E2E 日志与验收补齐，项目就更接近可持续 RC，而不是只在短门禁下看起来健康。
