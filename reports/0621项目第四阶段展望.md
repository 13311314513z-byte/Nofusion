## 二、Phase D（第四阶段）目标定义

**核心口号**: 测试先行、小步提取、稳态收口——先固化已拆模块边界，再渐进式完成剩余拆分。

**策略说明**: B3 已进入"可控拆分阶段"。当前不应冒险硬切 `importCanon`/`importChapters`（含长 prompt 和复杂 I/O，大段文本替换风险高）。正确路径是：先给已拆模块补测试 → 拆测试文件降低耦合 → 把剩余大方法拆成纯 helper → 最后移动主方法。

---

### 阶段一: 固化已拆模块（预计 4h）——"先让边界变硬"

已拆出的模块目前仅通过 runner 集成测试间接覆盖，模块边界不够硬。优先补直接测试。

| 优先 | 任务 | 说明 | 工时 |
|:--:|------|------|:--:|
| **D1** | `pipeline-import.test.ts` | 覆盖 `planFoundationImport` / `commitFoundationImport` 独立行为（mock ctx + deps） | 1.5h |
| **D2** | `pipeline-writing.test.ts` | 从 `pipeline-runner.test.ts` 拆出 writeDraft / writeNextChapter 用例 | 1.5h |
| **D3** | `pipeline-revision.test.ts` | 从 `pipeline-runner.test.ts` 拆出 repair / resync 用例 | 1h |

**验证**: 3 个新测试文件独立通过 + `pipeline-runner.test.ts` 从 5,192 → ~3,500 行

---

### 阶段二: 安全提取大方法（预计 6h）——"先拆 helper，再移主方法"

`importCanon` / `importChapters` 不直接做整段替换。改为先抽纯函数 helper，等 helper 有测试后再移动主方法。

| 优先 | 任务 | 说明 | 工时 |
|:--:|------|------|:--:|
| **D4** | 抽 `importCanon` 的 prompt 构造函数 | 将嵌入的 LLM prompt 模板抽为 `buildCanonImportPrompt()` | 1h |
| **D5** | 抽 `importChapters` 的 split/map helper | `splitChapters`、`mapToImportPlan` 等纯数据转换 | 1h |
| **D6** | 抽 `importChapters` 的 replay seed builder | 导入回放的种子构建逻辑 | 1h |
| **D7** | 安全移动 `importCanon` 主方法 | helper 已有测试后，用委托模式替换（参 pipeline-revision 经验） | 1.5h |
| **D8** | 安全移动 `importChapters` 主方法 | 同上 | 1.5h |

**验证**: 每一步后 `typecheck + test`，避免累积型损坏

## 四、风险与注意事项

| 风险 | 等级 | 缓解 |
|------|:--:|------|
| `importCanon` / `importChapters` 整段替换损坏文件 | 🔴→🟡 | **策略已改**: 不直接大段替换。先抽纯 helper 并测试，再移动主方法 |
| 测试拆分后 `--pool=forks` 可能暴露隐式顺序依赖 | 🟡 | 先在 `--pool=forks` 下验证现有测试，再拆分 |
| 阶段一新增测试文件可能暴露已拆模块的类型/接口缺陷 | 🟡 | 小步迭代，每个测试文件独立运行通过后再继续 |
| Core 拆分影响 Studio/CLI 发布链路 | 🟡 | 每轮收口前跑全门禁（typecheck+build+lint+Core/Studio/CLI tests） |
| `PipelinePool` 测试需 mock LLM client + 并发场景 | 🟡 | 复用 `pipeline-test-helpers.ts` 中已有 mock 基础设施 |
