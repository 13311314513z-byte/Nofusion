## 一、P2-1: server.ts 40 内联路由拆分

### 1.1 问题描述

`packages/studio/src/api/server.ts` 当前约 4,000 行，已提取 19 个路由模块到 `routes/` 目录，但仍有 **~40 个路由** 以 inline 形式存在于主文件中，分布在 8 个业务域：

| 业务域 | 端点示例 | 行数估算 |
|------|------|:--:|
| interaction/sessions | `GET /interaction/session`, `/sessions/*` (5 端点) | ~200 |
| agent | `POST /agent` | ~800 |
| write/draft | `/write-next`, `/draft`, `/write-preview`, `/plan-alternatives`, `/write-status` (5) | ~120 |
| revise/export | `/revise`, `/rewrite`, `/resync`, `/export`, `/export-save` (5) | ~180 |
| runtime/truth | `/runtime`, `/runtime/:file`, `/truth/:file`, `/state-changelog` (4) | ~100 |
| voice/roles/templates | `/roles/*` (5), `/voice-profiles/*` (2), `/scene-templates/*` (2) | ~200 |
| fanfic/radar/doctor | `/fanfic/init`, `/fanfic/import-canon`, `/radar/scan`, `/doctor` (4) | ~120 |
| genres/misc | `/genres/:id`, `/genres/:id/copy`, detect-all, detect-stats, style-score (6) | ~150 |

### 1.2 重构理由

1. 单文件承载过多业务域，局部改动容易误伤其他端点
2. route module 与 inline route 的边界不清，后续维护成本递增
3. API 文档、OpenAPI 生成和前端调用契约难以保持同步
4. 测试覆盖依赖主文件，无法按域独立测试

### 1.3 工序流程

```
Phase 1: 准备（1h）
├── Step 1.1: 梳理 server.ts 中所有 inline 路由的闭包依赖
│   └── 输出: 依赖矩阵（每个路由依赖的 state/buildPipelineConfig/broadcast 等）
├── Step 1.2: 创建 ServerContext 接口扩展
│   └── 将 buildPipelineConfig、loadRawConfig 等纳入 context
└── Step 1.3: 为每个待提取域创建空路由骨架文件

Phase 2: 分批提取（每域 1h，共 5h）
├── Batch A: sessions-agent（interaction + sessions + agent）
├── Batch B: writing（write-next + draft + write-preview + plan-alternatives + write-status）
├── Batch C: revision-export（revise + rewrite + resync + export + export-save）
├── Batch D: runtime-truth（runtime + truth + state-changelog）
└── Batch E: book-artifacts（roles + voice-profiles + scene-templates + fanfic + radar + doctor）

Phase 3: 验证（2h）
├── Step 3.1: pnpm typecheck
├── Step 3.2: pnpm test（确保 1861 tests 不退化）
├── Step 3.3: 手动测试关键端点（write-next / agent / revise）
└── Step 3.4: git bisect 确认每批提交独立可运行
```

## 五、P2-14: `any` / `as any` 242 处收敛

### 5.1 问题描述

242 处 `any` 分布在 3 个核心模块：

| 模块 | 数量 | 典型场景 |
|------|:--:|------|
| `provider.ts` | ~80 | LLM API 响应解析（`response.json() as any`） |
| `agent-tools.ts` | ~60 | 工具调用结果（`result as any`） |
| `agent-session.ts` | ~50 | session 序列化/反序列化 |
| 其余 | ~52 | 分散在各处 |

### 5.2 重构理由

1. LLM 响应 shape 漂移时类型系统无法提前拦截
2. 工具调用结果的字段访问无编译期保护
3. 新贡献者无法从类型推断数据形状

### 5.3 工序流程

```

├── Step 1.2: Tool Result Schema
│   ├── ToolCallResult（status, result, error, details）
│   └── SubAgentResult（agent, kind, bookId, output）
└── Step 1.3: Session Schema
    ├── AgentSession（messages, metadata, execution）
    └── SessionEvent（type, timestamp, payload）

Phase 2: 逐模块迁移（每模块 2h，共 6h）
├── Module 1: provider.ts — LLM response 解析
```

## 七、P2-7: Pipeline 实例池

### 7.1 问题描述

`withPipeline()` 每次创建新的 `PipelineRunner`，其中包含：
- `new StateManager(root)` — 非 trivial 构造（目录结构扫描）
- `new Map()` — agentClients 缓存
- `new PipelineConfig` — LLM 客户端配置

高频写作场景下（daemon mode），重复创建 PipelineRunner 造成不必要的 CPU 和内存开销。

### 7.2 重构理由

1. `StateManager` 构造包含 `books/` 目录扫描，频繁创建浪费 I/O
2. `agentClients` 缓存每次重建，LLM 连接无法复用
3. `PipelineConfig` 在 90% 情况下是相同的（同一项目、同一 LLM 配置）

### 7.3 工序流程

```
Phase 1: 设计池接口（1h）
├── 定义 PipelinePool 类
│   ├── acquire(): Promise<PipelineRunner>
│   ├── release(runner: PipelineRunner): void
│   ├── drain(): void  // 关闭所有实例
│   └── 配置: maxSize, idleTimeoutMs
└── 处理并发安全：同一 bookId 的写操作不能共享 PipelineRunner

Phase 2: 实现（2h）
├── 实现 PipelinePool
├── 修改 withPipeline 使用池
└── 确保 dispose 语义：release 时清理 chapterContentCache 等临时状态

Phase 3: 测试（1h）
├── 单元测试：acquire/release/drain
├── 并发测试：多个 write-next 不共享 runner
└── 集成测试：daemon mode 下 runner 复用
```

### 7.4 代码示例

```typescript
// pipeline-pool.ts
export class PipelinePool {
  private readonly pool: PipelineRunner[] = [];
  private readonly inUse = new Set<PipelineRunner>();
  private readonly maxSize: number;
  private readonly idleTimeoutMs: number;

  constructor(
    private readonly configFactory: () => PipelineConfig,
    options?: { maxSize?: number; idleTimeoutMs?: number },
  ) {
    this.maxSize = options?.maxSize ?? 4;
    this.idleTimeoutMs = options?.idleTimeoutMs ?? 5 * 60_000;
  }

  async acquire(): Promise<PipelineRunner> {
    // 1. 从池中取空闲实例
    const idle = this.pool.find(r => !this.inUse.has(r));
    if (idle) {
      this.inUse.add(idle);
      return idle;
    }
    // 2. 池未满，创建新实例
    if (this.pool.length < this.maxSize) {
      const runner = new PipelineRunner(this.configFactory());
      this.pool.push(runner);
      this.inUse.add(runner);
      return runner;
    }
    // 3. 池满，等待释放
    return new Promise((resolve) => {
      const check = setInterval(() => {
        const freed = this.pool.find(r => !this.inUse.has(r));
        if (freed) { clearInterval(check); this.inUse.add(freed); resolve(freed); }
      }, 100);
    });
  }

  release(runner: PipelineRunner): void {
    // 清理临时状态（章节缓存等）
    if (typeof (runner as any).resetForReuse === "function") {
      (runner as any).resetForReuse();
    }
    this.inUse.delete(runner);
  }

  drain(): void {
    for (const runner of this.pool) {
      if (typeof (runner as any).dispose === "function") {
        (runner as any).dispose();
      }
    }
    this.pool.length = 0;
    this.inUse.clear();
  }
}
```

```typescript
// server.ts — withPipeline 改用池
const pipelinePool = new PipelinePool(() => buildPipelineConfigSync());

async function withPipeline<T>(
  label: string,
  config: PipelineConfig,
  fn: (pipeline: PipelineRunner) => Promise<T>,
  ttlMs = 5 * 60_000,
): Promise<T> {
  // 使用池获取 runner（忽略传入的 config，用池的 factory）
  const pipeline = await pipelinePool.acquire();
  try {
    const result = await withTimeout(fn(pipeline), ttlMs);
    return result;
  } finally {
    pipelinePool.release(pipeline);
  }
}
```

## 八、P1-4: style-score 逐章计算缓存

### 8.1 问题描述

### 8.2 重构理由

2. 缓存失效策略需要与章节内容变更联动

### 8.3 工序流程（1h）

```
Step 1: 在 server.ts 或 routes 中加内存缓存
├── Map<`${bookId}:${chapterNum}`, { score, contentHash, cachedAt }>
└── 每次调用前比对章节内容 hash

Step 2: 缓存失效策略
├── 章节被 revise/resync/write 后删除对应缓存
└── 超时淘汰（1h TTL）

Step 3: 测试
└── 验证同一章节连续 3 次请求只触发 1 次 LLM 调用
```

### 8.4 代码示例

```typescript
// style-score-cache.ts
import { createHash } from "node:crypto";

interface CachedScore {
  readonly score: unknown;
  readonly contentHash: string;
  readonly cachedAt: number;
}

const scoreCache = new Map<string, CachedScore>();
const SCORE_CACHE_TTL_MS = 60 * 60 * 1000; // 1 hour

export function getCachedScore(key: string): CachedScore | undefined {
  const cached = scoreCache.get(key);
  if (cached && Date.now() - cached.cachedAt < SCORE_CACHE_TTL_MS) {
    return cached;
  }
  scoreCache.delete(key);
  return undefined;
}

export function setCachedScore(key: string, contentHash: string, score: unknown): void {
  scoreCache.set(key, { score, contentHash, cachedAt: Date.now() });
}

export function invalidateChapterScore(bookId: string, chapterNum: number): void {
  scoreCache.delete(`${bookId}:${chapterNum}`);
}

export function contentHash(content: string): string {
  return createHash("sha256").update(content).digest("hex").slice(0, 16);
}
```

## 附录 C: 逐项 P0/P1 风险分析

> 每一项重构都可能引入新的 P0（系统不可用）或 P1（功能退化）风险。以下逐项列出可能的风险、触发条件和缓解措施。

### C.1 P2-1: server.ts 40 路由拆分

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **中间件链断裂** — 提取后的路由注册顺序变化，导致 `CORS → bodyLimit → bookId验证 → 路由` 链中某环节被跳过 | 新路由模块的 `app.use()` 或 `app.post()` 注册顺序与原来不一致 |
| 🔴 P0 | **路由丢失** — 提取时遗漏某个端点，生产环境 404 | 人工迁移时漏掉某个 inline 路由 |
| 🟡 P1 | **循环依赖** — 提取的路由模块 import server.ts 的 helper（如 `acquireWriteJob`），server.ts 又 import 路由模块 | 辅助函数未及时提取到 shared 模块 |
| 🟡 P1 | **测试覆盖缺口** — 搬迁过程中部分边界测试丢失 | 手动迁移测试用例时遗漏 |

**缓解措施**:
```
1. 提取前用脚本列出所有 app.get/post/put/delete 路由 URL → 提取后 diff 确认无遗漏
2. 每批提取后立即 git commit，出错时 git revert 单批
3. 辅助函数优先提取到 shared/ 目录，避免 server.ts ↔ routes/ 双向依赖
4. 全量回归测试每批都跑，不等到最后
```

### C.2 P2-2: runner.ts 4,050 行拆分

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **公共 API 破坏** — PipelineRunner 的 public 方法签名改变，导致 server.ts / CLI / 测试三处调用全部崩溃 | 拆分时意外修改了方法参数类型或返回值结构 |
| 🔴 P0 | **Private 方法遗漏** — 某个 private 方法在拆分时被遗漏，运行时 `undefined is not a function` | 方法调用图分析不完整 |
| 🟡 P1 | **PipelineContext 接口漂移** — 提取的 context 对象缺少某个字段，导致子模块行为与原来不同 | context 类型定义不完整 |
| 🟡 P1 | **章节缓存跨模块不一致** — 拆分后的 audit 模块修改了缓存但 write 模块不知道 | chapterContentCache 共享语义未明确 |

**缓解措施**:
```
1. 保留 PipelineRunner 为薄门面，所有 public 方法签名 100% 不变
2. 拆分前用脚本自动提取所有 private 方法名 → 拆分后 diff 确认 0 遗漏
3. PipelineContext 使用 TypeScript interface + 编译期检查确保字段完整
4. chapterContentCache 通过 PipelineContext 共享引用，各模块读写同一 Map
```

### C.3 P2-5: StyleManager 1,900 行拆分

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🟡 P1 | **UI 回归** — 状态提升到 hook 后，Tab 切换时状态丢失或未正确重置 | `useState` 拆分时遗漏某个状态字段 |
| 🟡 P1 | **性能退化** — 新增的 hook + 7 个组件文件导致首屏加载变慢 | 未使用 `React.lazy` 懒加载 Tab 组件 |

**缓解措施**:

```
3. useStyleManagerState 写单元测试覆盖状态转换
```

### C.4 P1-3: agent 800 行内联提取

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **SSE 流式中断** — agent 的 SSE 响应依赖 server.ts 闭包中的 `broadcast`/`streamSSE` 等，提取后流式推送失败 | 闭包变量未正确注入 ServerContext |
| 🟡 P1 | **Chat edit 行为变化** — `tryHandleExternalChatEdit` 提取后错误处理路径改变 | 辅助函数的异常处理逻辑与原内联版本不一致 |

**缓解措施**:

```
1. agent 端点是最后提取的（在 #1 server.ts 拆分完成后），此时 ServerContext 已稳定
2. 辅助函数逐個提取并补单元测试，每个函数独立验证行为不变
3. 手动测试 agent 的 write-next / chapter edit / thinking stream 三个核心流程
```

### C.5 P2-14: any 242 处收敛

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **LLM 调用全断** — Zod schema 过于严格，合法但罕见的 LLM 响应被 `parse()` 抛出异常 | 上游 LLM API 返回了 schema 未定义的字段（如新增的 `service_tier`） |
| 🔴 P0 | **Service probe 失败** — 服务探测依赖宽松的响应解析，严格的 schema 导致探测误报 | 某些 provider 返回非标准格式的 /models 响应 |
| 🟡 P1 | **可选字段丢失** — schema 标记为 optional 但代码依赖该字段存在，Zod 不报错但运行时数据为空 | `usage.total_tokens` 在某些 stream 模式下确实不返回 |

**缓解措施**:
```
1. 全部使用 z.object({...}).passthrough() 而非严格模式 —— 未知字段自动忽略，不抛异常
2. 使用 .safeParse() 替代 .parse()：解析失败时 fallback 到原 any 逻辑 + 日志告警
3. 分 provider（openai/anthropic/google）定义独立 schema，不共用
4. 上线前用真实 API key 跑一遍 service probe + chat + stream 核心链路
```

### C.6 P2-16: legacy 兼容代码清理

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **用户数据丢失** — 存在 Phase 4 格式的书籍数据，删除兼容代码后无法读取 | 未确认所有用户已迁移到 Phase 5 |
| 🟡 P1 | **迁移脚本 bug** — Phase4→Phase5 迁移脚本有缺陷，导致部分文件未迁移或损坏 | 迁移脚本未覆盖所有文件类型（如自定义 role 目录） |

**缓解措施**:
```
1. 迁移前全量扫描所有 books/ 目录，统计 Phase 4 vs Phase 5 格式比例
2. 迁移脚本先 dry-run → 生成报告 → 人工确认 → 再实际执行
3. 迁移前自动备份原 books/ 目录（tar/zip）
4. 提供 v1.5.x 作为过渡版本，标记 deprecated 警告但不删除代码
```

### C.7 P2-7: Pipeline 实例池

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🔴 P0 | **池耗尽** — 并发写操作超过 `maxSize`，新请求在 `acquire()` 中无限等待 | daemon mode 下多个 write-next 同时触发 |
| 🔴 P0 | **状态污染** — 池中复用的 PipelineRunner 残留上一次写作的 state/config，导致章节写入错误书籍 | `release()` 时未正确清理实例状态 |
| 🟡 P1 | **池泄漏** — 异常路径下 `release()` 未被调用，`inUse` 集合持续增长最终耗尽 | `fn(pipeline)` 抛出异常时 finally 块未执行 |
| 🟡 P1 | **并发写共享 runner** — 同一 bookId 的多个 write-next 请求意外分配到同一个 PipelineRunner | 池的分配逻辑未考虑 bookId 隔离 |

**缓解措施**:
```
1. maxSize 默认 4，但可配置；acquire() 增加超时参数（默认 30s），超时抛错
2. PipelineRunner 新增 resetForReuse() 方法：清理 chapterContentCache + agentClients 临时状态
3. release() 必须在 try-finally 中调用，池的 withPipeline 封装确保这一点
4. 按 bookId 分区：同一 bookId 的请求复用同一 runner，不同 bookId 不共享
```

### C.8 P1-4: style-score 缓存

| 级别 | 风险 | 触发条件 |
|:--:|------|------|
| 🟡 P1 | **过期分数展示** — 章节被 revise 后缓存未失效，用户看到旧分数 | `invalidateChapterScore` 未在 revise/resync 路径中调用 |

**缓解措施**:
```
1. 在 revise/resync/write-next 成功回调中显式调用 invalidateChapterScore
2. 缓存 key 包含 contentHash，即使缓存未清除也会因 hash 不匹配而 miss
3. 1h TTL 作为兜底过期策略
```

### C.11 P2-17: 巨型测试文件拆分
