### 2.2 测试基础设施 — 评级：★★★☆☆

| 包 | 框架 | vitest.config.ts | 池隔离 | 测试数 | 通过率 |
|---|:----:|:----------------:|:------:|:------:|:------:|
| Core | vitest | ✅ 已配置 ✅ | `pool: forks, isolate: true` | 1392 | 100% |
| CLI | vitest | ❌ **缺失** | 默认（threads） | 176 | 100% |
| Studio | vitest | ✅ 已配置 ✅ | `pool: forks, isolate: true` | 277 | 100% |

**问题**：
1. **CLI 缺少 `vitest.config.ts`** — 使用 vitest 默认配置（`pool: "threads"`），在多线程环境下可能产生模块状态污染。虽然当前测试通过，但长期稳定性不如 Core/Studio 的 `forks` 隔离方案。
2. **CLI 无 `testTimeout` 显式配置** — 集成测试中的 `execFileSync` 调用依赖系统命令的超时，而非 vitest 的超时控制。

**建议**：
```typescript
// packages/cli/vitest.config.ts
import { defineConfig } from "vitest/config";
export default defineConfig({
  test: {
    include: ["src/**/*.test.ts"],
    pool: "forks",
    poolOptions: { forks: { isolate: true } },
    testTimeout: 60_000, // CLI 集成测试需 58s
  },
});
```

**验证**：

| `.github/` 目录 | ❌ 不存在 | 无 CI 管道定义 |
| PR 检查 | ❌ 不存在 | 无法阻止引入编译错误或测试失败 |
| 自动发布 | ❌ 不存在 | 需手动 npm publish |

**这是最大基础设施缺口**。当前所有验证完全依赖本地手动执行 `pnpm -r typecheck && pnpm -r build && pnpm -r test`。

**最小 CI 配置建议**（`.github/workflows/ci.yml`）：
```yaml
name: CI
on: [push, pull_request]
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "22" }
      - uses: pnpm/action-setup@v2
      - run: pnpm install
      - run: pnpm -r typecheck
      - run: pnpm -r build
      - run: pnpm -r test
```
