## 二、测试宽度扩展建议

### 2.1 建议新增：技术质量维度（Technical Quality）

| 新维度 ID | 维度名 | 测量对象 | 建议工具 |
|:--------:|--------|----------|----------|
| `ai-trace` | AI 痕迹密度 | 单章/多章 | 现有 `detector.ts`（`DetectorAgent`） |
| `readability` | 可读性分数 | 单章 | 现有 `readability-score.ts` |
| `repetition` | 段落/修辞重复 | 单章/跨章 | 现有 `paragraph-dedup.ts` + `semantic-duplication.ts` |
| `paragraph-shape` | 段落形态健康 | 单章 | 现有 `post-write-validator.ts` 段落检测规则 |

> 这些维度有现有工程实现，仅需在测试报告中汇总趋势，无需新增 Reader 判断。

### 2.2 建议新增：叙事完整性维度（Narrative Integrity）

| 新维度 ID | 尺度 | 测量方式 | 说明 |
|:--------:|------|----------|------|
| `continuity-entities` | 章节级 | 自动化 diff + 抽样人工 | 同一人物/地点/道具在相邻章的描述一致 |
| `hook-progress` | 卷级（3-5 章） | 自动化（hook ledger 状态变化） | 伏笔在 N 章内是否推进/回收 |
| `pacing-variance` | 卷级 | 章节字数 + 事件密度曲线 | 节奏是否单调（连续 3 章同一类型事件） |
| `character-graph` | 卷级 | 角色共现矩阵变化 | 角色关系是否有发展而非静态重复 |

### 2.3 建议新增：成本与效率维度（Cost & Efficiency）

| 新维度 ID | 测量方式 | 数据源 |
|:--------:|----------|--------|
| `token-efficiency` | 每千字生成 Token 消耗 | `PipelineRunner` chapter usage |
| `revision-rate` | 需修订章数 / 总章数 | 管线日志 |
| `first-pass-rate` | 审计首次通过率 | audit history |
| `model-cost-per-chapter` | 单章 API 费用估算 | token * 模型定价 |

### 2.4 建议新增：对角色与风格适配维度

| 新维度 ID | 尺度 | 工具 |
|:--------:|------|------|
| `voice-consistency` | 角色级 | `VoiceProfileAnalyzer` 偏移检测 |
| `style-alignment` | 书籍级 | `style-comparator.ts` 目标风格匹配度 |
| `genre-promise` | 书籍级 | `genre-promises.ts` 必需要素兑现 |

### 2.5 全维度覆盖矩阵（建议最终状态）

## 三、测试深度扩展建议

### 3.1 当前深度（手册定义的 8 阶段）

```
阶段 1-2: 设计 + 准备          ← 人工设计
阶段 3-4: 配置 + 执行          ← 半自动
阶段 5-6: 收集 + 统计          ← 全自动（脚本）
阶段 7-8: 校准 + 报告          ← 人工决策
```

### 3.2 建议扩展为分层测试金字塔

```
        ┌──────────────┐
        │  人工双盲评测  │  ← 每周 10 对，顶尖
        │  (Stage 5)    │
        ├──────────────┤
        │ Beta Reader   │  ← 每次 PR 自动 30 对
        │  综合评测      │
        ├──────────────┤
        │ 管线集成测试   │  ← 每次 PR 自动运行
        │  25 tests     │
        ├──────────────┤
        │ 单元测试       │  ← 每次 commit 自动
        │  1845 tests   │
        └──────────────┘
```

### 3.3 各层具体扩展

#### Layer 1：单元测试（已有 1845 tests ✅）

| 扩展建议 | 说明 | 优先级 |
|----------|------|:------:|
| 增加 `VoiceProfileAnalyzer` 单元测试 | 当前无特定测试 | P1 |
| 增加 `scene-template` Schema 校验测试 | 仅 GET/PUT 端点，无 schema 测试 | P1 |
| 增加 `chapter-artifacts` 导入导出测试 | 验证 artifact 读写一致 | P2 |

#### Layer 2：管线集成测试（已有 25 tests，建议扩展到 40+）

| 新增测试 | 覆盖场景 | 优先级 |
|----------|----------|:------:|
| `llm-stub` 模式下全管线端到端 | 用模拟 LLM 替代真实调用，验证管线组装 | P1 |
| 多个体裁 (genre) 的 foundation 生成 | 玄幻/仙侠/都市/恐怖各 1 次 | P1 |
| 续写管线的状态继承 | import → write 3 章 → 验证状态连续性 | P1 |
| 同人管线的 Canon 约束 | fanfic mode → 验证 Canon 锁生效 | P2 |
| 异常恢复路径 | API 失败 → retry → 降级 → 日志 | P2 |

#### Layer 3：Beta Reader 评测（当前 12 对，建议扩展到 50+ 对）

| 扩展 | 说明 | 优先级 |
|------|------|:------:|
| 目标扩展到 50 对 | 满足统计显著性最低要求 | P0 |
| 增加 `tie` 和 `unable` 的强制上限 | 若某 Reader 的 tie 率 > 30%，标记为不可靠 | P1 |
| 引入 `difficulty` 字段 | 标记高难度 pair（两版质量接近），用于培训 Reader | P2 |
| 跨模型一致性矩阵 | 同一 pair 用 2-3 个不同模型 Reader 判断 | P2 |

#### Layer 4：人工评测（当前 0）

| 扩展 | 说明 | 优先级 |
|------|------|:------:|
| 建立固定评测池（≥4 人） | 包含普通读者、写作者、编辑三种角色 | P1 |
| 每轮评测 ≤20 对 | 控制疲劳效应 | P1 |
| 标注指南文档 | 统一标注标准，减少标注员间分歧 | P1 |
| 金标数据集（≥20 对） | 人工已标注的 pair，用于校准所有 Reader | P1 |

### 4.1 建议：质量门控 CI（Quality Gate CI）

```
.simtest/quality-gate.yml

on: [pull_request]
jobs:
  quality-gate:
    steps:
      - 单元测试（1845 tests，必须 100%）
      - 管线集成测试（25+ tests，必须 100%）
```

## 六、风险与约束

### 6.1 真实风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| Beta Reader 自评偏差 | 用 Writer 同一模型做 Reader，结果偏向自己 | 强制异构 Reader |
| 统计显著性不足 | 50 对仍可能无法检测小效应 | 报告 CI，不只看点估计 |
| 人工评测疲劳 | 标注员在 20 对后判断质量下降 | 每轮 ≤20 对，中间休息 |
| 指标漂移 | 长期测试中 Reader prompt 或模型更新导致评分基准漂移 | 金标数据集定期重测 |
| 维度膨胀 | 20 个维度难以同时解读 | 聚焦 3-5 个核心维度，其余为参考 |

### 6.2 预算约束

| 资源 | 预估需求 |
|------|----------|
| LLM API 费用（50 对 × 5 维 × 2 Reader） | 约 500 次判断 × ~500 tokens/次 |
| 人工标注时间 | 每次 20 对 × 每人 30 分钟 = 1h/人 |
| 工程时间（到 Phase B） | 约 2 天 |

## 七、总结

```
当前测试覆盖：30%（仅 5 个阅读体验维度）
建议扩展至：85%（20 个维度 + 4 层金字塔 + 质量门控 CI）

| 维度 | 当前 | 目标 | 路径 |
|------|:----:|:----:|------|
| 测试宽度 | 5 维 | 20 维 | 补充技术质量、叙事完整性、成本效率、风格适配 |
| 测试深度 | 1 层（Beta Reader） | 4 层（单元→集成→Beta→人工） | 建立金字塔 |
| 自动化程度 | 半自动 | 全自动（含 CI） | Phase C 集成质量门控 |
| 校准对数量 | 12 对 | 50 对 + 20 对金标 | Phase A/B 逐步扩展 |
| 人工参与 | 0 | ≥4 人固定池 | Phase B 启动 |
```

**核心原则**：测试宽度决定"能发现多少问题"，测试深度决定"多确信这是问题"。在现有 1845 单元测试的坚实基础上，补齐集成测试和金标数据集是性价比最高的下一步。
