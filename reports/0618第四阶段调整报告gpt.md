## 0. 结论摘要

当前项目已经从 0618 上午的“多处 P0/P1 阻塞”恢复到 **内部 RC 候选 / Early Beta 后段**：标准工程门禁在正确顺序下可以通过，Studio API 主契约、CLI 集成、Core 主体测试均已恢复，前端当前调用未发现明显缺失后端端点。

但项目还不能按“公开 RC / 可长期演进架构”判定。主要原因不是功能不可用，而是仍存在三类高风险：

1. **真实 LLM E2E 未形成可靠门禁**：本机 `.env` 会自动激活 `e2e-real-llm.test.ts`，实跑暴露 `projectRoot` / runner 配置形态问题，且测试输出包含 API key 长度日志。
2. **Studio 后端仍有拆分后遗留的双实现/死模块风险**：`server.ts` 已注册多组 route module，但会话/agent 等关键链路仍保留 inline 实现；`routes/sessions.ts` 中存在“简化桩函数”，当前未注册，但未来一旦重新接入会制造回归。
3. **测试编排存在并发/构建产物污染风险**：Core、CLI、Studio 单独按顺序跑可以通过，但并发跑会因为 workspace core dist 被重建、包自引用走 dist 等原因产生假失败。

因此，本阶段判断为：

| 维度 | 当前判断 |
|---|---|
| 功能可用性 | 主流程可用，Studio/CLI/Core 主干已恢复 |
| 自动化质量 | 顺序执行可通过；真实 LLM E2E 与并发测试仍不稳定 |
| 前后端对齐 | 当前前端主调用与后端端点基本对齐 |
| 架构状态 | 可维护性仍偏紧，Studio API 层需要继续收敛 |
| 发布建议 | 可作为内部 RC 候选；公开 RC 前需先处理真实 E2E 与 sessions 路由债 |

## 4. 前后端对齐复核

### 4.1 Studio API 基础形态

Studio 前端主要通过 `src/hooks/use-api.ts` 和局部 `fetch` 调用 `/api/v1` 下端点。后端入口为 `packages/studio/src/api/server.ts`，并注册了多组 route module：

- `events`
- `cover`
- `project`
- `logs`
- `genres`
- `analytics`
- `health`
- `truth-browser`
- `language`
- `model-overrides`
- `notify`
- `sources`
- `hooks`
- `books`
- `services`
- `chapters`
- `audit`
- `style`
- `daemon`

当前未发现 Studio 前端主页面调用一个完全不存在的后端端点。第三阶段集中修复后的 Studio API 契约测试也覆盖了大量前后端关键约定。

### 4.2 仍未收敛的问题：server.ts inline 路由残留

`server.ts` 仍保留大量 inline API，包括但不限于：

| 功能域 | 典型端点 |
|---|---|
| Interaction | `GET /api/v1/interaction/session` |
| Sessions | `GET/POST /api/v1/sessions`、`GET/PUT/DELETE /api/v1/sessions/:sessionId` |
| Agent | `POST /api/v1/agent` |
| Runtime / truth | `/api/v1/books/:id/runtime`、`/api/v1/books/:id/truth/:file` |
| 写作链路 | `/write-preview`、`/plan-alternatives`、`/write-next`、`/draft` |
| 修订/导出 | `/revise/:chapter`、`/export`、`/export-save` |
| genre 扩展 | `/genres/:id/copy`、`/genres/create` 等 |
| 检测/重写 | `/detect/:chapter`、`/rewrite/:chapter`、`/resync/:chapter` |
| 角色/声音/模板 | `/roles`、`/voice-profiles`、`/scene-templates` |
| fanfic/radar/doctor | `/fanfic/*`、`/radar/*`、`/doctor` |

这说明 Studio 后端仍处于“部分模块化 + 大型入口文件兜底”的过渡状态。当前可用，但不利于长期维护。

### 4.3 sessions route module 是当前最大架构债

`packages/studio/src/api/routes/sessions.ts` 当前没有在 `server.ts` 中导入和注册，但文件存在，并且包含明显的简化桩：

- `tryHandleExternalChatEdit(...)` 直接 `return null`，注释指向 “full logic in original server.ts”
- `validateAgentActionExecution(...)` 直接 `return null`，注释指向 “full logic in original server.ts”

这类文件的风险在于：

1. 当前不会影响运行，因为实际生效的是 `server.ts` inline 实现。
2. 后续维护者很容易以为 `routes/sessions.ts` 是真实实现并在里面修 bug。
3. 如果未来重新注册该模块，agent / sessions 的行为会静默退化。

建议将其列为公开 RC 前的 P0 架构风险处理项：要么删除死模块，要么补齐逻辑并用契约测试证明它完全等价。

## 6. 当前问题分级

### P0：公开 RC 前必须处理

#### P0-1 真实 LLM E2E 必须改成可控、可复现、可通过

现状：

- `packages/core/src/__tests__/e2e-real-llm.test.ts` 会向上查找仓库 `.env` 并自动加载 key。
- 本机存在 `.env` 时，测试会自动进入真实 LLM 路径。
- 实跑暴露 `PipelineRunner` / `StateManager` 配置问题：`projectRoot` 为空导致 path 参数异常。
- 测试日志输出 API key 长度，虽然未打印 key 本体，但仍不应作为常规日志。

建议：

1. 默认跳过真实 E2E，必须显式设置如 `INKOS_RUN_REAL_E2E=1` 才执行。
2. 修正 runner 初始化配置，确保 `projectRoot`、booksDir、runtime/state 路径完整。
3. 不再自动从仓库 `.env` 激活真实测试，或至少把行为写入明确的 opt-in 逻辑。
4. 移除 API key 长度日志。
5. 验收标准：有 key 且 opt-in 时，create -> write chapter -> audit 至少一条真实链路稳定通过；无 key 或未 opt-in 时稳定 skip。

#### P0-2 sessions / agent 路由双实现必须清理

现状：

- 实际生效逻辑在 `server.ts` inline。
- `routes/sessions.ts` 存在但未注册，并含有简化桩函数。

建议二选一：

1. 短期保守方案：删除或改名隔离 `routes/sessions.ts`，明确当前只有 inline 版本生效。
2. 中期正确方案：把 inline sessions / agent 完整迁入 route module，再用现有 95 个 Studio API 契约测试加新增回归测试锁住行为。

验收标准：

- 仓库内不再存在“full logic in original server.ts”这类简化占位。
- `/api/v1/sessions/*` 与 `/api/v1/agent` 只有一个真实实现位置。
- Studio API 契约测试覆盖外部聊天编辑、agent action validation、session CRUD。

### P1：进入稳定 Beta 前应完成

#### P1-1 测试编排固化

当前顺序执行可通过，但并发执行会因构建产物重建和包自引用造成假失败。建议增加显式脚本：

- `test:ci`：build -> core non-e2e -> studio -> cli -> optional real-e2e。
- `test:real-e2e`：只在有 key 且 opt-in 时运行。
- 文档明确禁止把三个 workspace 的测试并发跑作发布依据。

#### P1-2 server.ts 继续降体量，但必须以契约测试驱动

`server.ts` 当前接近大型枢纽文件，后续应继续拆分 runtime、write、export、roles、fanfic、radar、doctor 等域。但拆分顺序应由测试保障，而不是一次性机械搬运。

推荐顺序：

1. sessions / agent
2. runtime / truth
3. write / draft / revise / export
4. roles / voice / scene templates
5. fanfic / radar / doctor

每拆一个域，都要先补契约测试，再迁移，再删除 inline 版本。

#### P1-3 Interaction runtime 与 Studio/CLI 能力补齐

Core interaction runtime 中仍有明确未实现提示：

- book-draft ideation
- book creation
- book export
- fallback intent

这不一定影响 Studio 当前直接路径，但会影响统一交互入口、TUI、未来 OpenClaw/agent 编排的一致性。建议把这些能力列为 Beta 功能补齐，而不是继续依赖各端绕行。

### P2：质量和维护性优化

#### P2-1 前端 bundle 与懒加载

Studio build 已通过，但 Vite 提示 `use-api.ts` 同时被静态和动态导入，动态拆包无法生效；同时主 chunk 体积较大。建议：

- 对 Shiki / Mermaid / Streamdown / language packs 做真正按需加载。
- 检查 `use-api.ts` 是否应拆成轻量 query hooks 与重依赖工具模块。
- 加 bundle budget 到 CI，但先以 warning 运行，避免短期阻塞。

#### P2-2 lint warning 收敛

当前 lint 0 error，但 warning 仍约 379 个。建议按模块逐步清：

1. 先清 production `unused` 与 `any`。
2. 测试里的宽松类型可后置。
3. 把新增 warning 作为 PR 阈值控制，避免继续增长。

#### P2-3 移除生产调试输出

`StyleAuditTab.tsx` 存在前端生产路径 `console.log("[StyleManager] Comparison result:", result)`。这不是阻塞，但应清理。

5. 发布包不包含明显调试日志或误导性测试桩。

## 9. 最终判断

NoFusion 当前已经不是“功能不可用”的阶段，而是进入了“主流程可用、工程门禁基本恢复、发布信心由真实 E2E 和架构收敛决定”的阶段。

第四阶段最关键的调整不是继续堆新功能，而是把三件事做实：真实 LLM E2E、Studio API 唯一实现、测试编排标准化。完成后，项目才适合从内部 RC 候选推进到公开 RC。
