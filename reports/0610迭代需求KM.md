### 3.7 Core 核心架构

#### 3.7.1 现状评估

- `index.ts`：全量导出，含 Node.js 依赖
- `browser-index.ts`：浏览器安全入口，已修复构建问题
- `index.d.ts` + `index.js`：tsc 产物
- `node:sqlite`：实验性 API，需关注 Node.js 版本兼容性

#### 3.7.2 迭代需求

**需求 CORE1-1：统一浏览器/服务端入口（P2）**

- **现状问题**：
  - `browser-index.ts` 手动维护，容易与 `index.ts` 不同步
  - Studio 需要同时知道哪些导入走 `core`、哪些走 `core/browser`
- **方案**：
  1. 在 `index.ts` 中使用条件导出或子路径导出，替代手动维护的 `browser-index.ts`
  2. 或在构建时自动生成 `browser-index.ts`（通过 AST 扫描排除含 `node:` 前缀的模块）
  3. 推荐方案：修改 `package.json` exports，使用 `"./browser"` 子路径，由 `tsup`/`unbuild` 自动生成
- **KM 可执行**：是（构建工具链改造）

**需求 CORE1-2：风格分析结果缓存（P3）**

- **方案**：
  1. 对 `analyze`、`diagnostics`、`ai-tells` 等纯计算函数增加 LRU 缓存
  2. 缓存键：文本内容的 hash + 语言 + 参数版本
  3. 缓存有效期：24 小时
  4. 减少重复计算，提升响应速度
- **KM 可执行**：是

**需求 CORE1-3：错误处理中间件标准化（P2）**

- **现状问题**：`style/rhetoric/rewrite` 500 崩溃暴露出后端缺少统一的错误边界
- **方案**：
  1. 在 Hono app 上注册全局错误处理中间件
  2. 所有 async 路由统一包裹 `try/catch`，返回 `{error: string, code?: string}` 结构
  3. 500 错误记录完整堆栈到日志，前端显示用户友好的错误提示
- **KM 可执行**：是

**需求 CORE1-4：Server.ts 按领域拆分（P0 → P2）**

- **现状问题**：`server.ts` 6353 行 Monolith，维护困难、冲突率高、测试覆盖难
- **方案**：
  1. 按领域拆分为 `packages/studio/src/routers/*.ts`：
     - `books.router.ts` — 书籍 CRUD、章节、导出
     - `sessions.router.ts` — 会话、消息
     - `style.router.ts` — 风格分析、作家档案、蒸馏
     - `services.router.ts` — 服务商配置、测试
     - `audit.router.ts` — 审计、检测
     - `project.router.ts` — 项目设置、语言
     - `daemon.router.ts` — 后台任务
     - `import.router.ts` — 导入、同人
  2. 在 `server.ts` 中使用 `app.route('/api/v1', booksRouter)` 聚合
  3. 每个 router 独立导出，便于单元测试
- **KM 可执行**：是（大规模重构，需分步执行）

**需求 CORE1-5：全局内存缓存治理（P1）**

- **现状问题**：`modelListCache`、`agentCache` 等全局缓存无上限，长期运行可能导致内存泄漏
- **方案**：
  1. 引入 `lru-cache` 或自研 LRU 实现
  2. 为所有缓存设置 maxSize（如 100）和 TTL（如 1 小时）
  3. 在 `/daemon` 状态接口中暴露缓存命中率/大小指标
- **KM 可执行**：是

**需求 CORE1-6：配置持久化加固（P1）**

- **现状问题**：`inkos.json` 直接写入，无文件锁，并发修改可能损坏 JSON
- **方案**：
  1. 短期：使用 `proper-lockfile` 或 `fs.writeFileSync` + 原子写入（先写临时文件再重命名）
  2. 长期：将高频修改的配置项迁移到 SQLite（`config` 表），`inkos.json` 仅保留静态配置
- **KM 可执行**：是

**需求 CORE1-7：MemoryDB 迁移脚本版本化（P2）**

- **现状问题**：`ensureColumn` 等迁移逻辑简陋，无法处理复杂 schema 变更
- **方案**：
  1. 引入 `migrations/` 目录，按版本号命名 SQL 文件（如 `001_init.sql`、`002_add_distillation.sql`）
  2. 启动时读取 `schema_version`，按序执行未应用的迁移
  3. 迁移记录写入 `_migrations` 元数据表
- **KM 可执行**：是

**需求 CORE1-8：日志持久化（P2）**

- **现状问题**：日志仅通过 SSE 实时推送，无持久化，生产环境排障困难
- **方案**：
  1. 引入 `pino` 或按日滚动的文件写入（`logs/inkos-YYYY-MM-DD.log`）
  2. 保留 SSE 实时推送能力，同时写入文件
  3. 日志页面支持读取历史日志文件（最近 7 天）
- **KM 可执行**：是

**需求 API1-1：SSE 事件对齐（P3）**

- **问题**：后端 broadcast 52 种事件，前端只声明 46 种
- **方案**：

  3. 将缺失的 6 种事件添加到前端声明中，或从后端移除未使用的事件
- **KM 可执行**：是

  3. 生成 `openapi.json`，供前端类型生成和第三方集成使用

  2. 对比后端注册端点

  4. 输出报告到 `reports/api-coverage.json`
- **KM 可执行**：是
