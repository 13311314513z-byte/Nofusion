# 文风分析模块完善建议与路线图

> 基于 2026-06-11 代码分析和当前实现状态  
> 目标是让文风分析从「工具集」转变为「诊断→建议→执行」的完整工作流

---

## 零、本轮复核结论

结合当前工作区代码复核，原方案方向正确，但部分状态已经发生变化，后续不应继续按“组件缺失”推进，而应转向“数据闭环和工程化”：

1. **已经进入实现阶段**：书籍首章导入、AI 痕迹自动带入文本、诊断页原文风险区域、调整建议迁移、作家风格对比入口、调整前后 diff、章节风格评分展示均已出现代码实现。
2. **仍属于初步接线**：书籍导入固定读取第 1 章；导入来源没有统一保存 `bookId/chapterNumber/language`；漂移评分也固定请求第 1 章，且当前导入函数没有同步设置 `importBookId`。
3. **风险标注尚未真正定位**：诊断结果已经包含 `examples.start/end`、`occurrences.start/end` 等位置数据，但当前页面只是展示全文和问题摘要，没有完成点击问题后滚动、高亮和上下文定位。
4. **修辞去重组件已经存在**：`RhetoricIssuePanel`、`DuplicateParagraphPanel`、`ReadabilityDashboard` 均已实现，Step 4 的主要工作已从“开发组件”变为“接入统一文本状态、处理修改回写和复检”。
5. **作家蒸馏后端明显领先前端**：草稿生成、当前版本读取、规则覆盖、发布、历史版本接口已经存在，但 Studio 内没有完整工作台，这是下一轮最值得补齐的功能闭环。
6. **语言和规则配置仍不统一**：AI 痕迹检测及部分修辞接口仍固定使用 `zh`；诊断规则、阈值和提示语分散在 Core 与 Studio 中，不利于英文文本及后续扩展。

因此，下一阶段的核心目标应调整为：

> 建立统一分析会话 → 精确定位问题 → 安全生成修改 → diff 审核 → 回写章节 → 自动复检 → 保存趋势与规则版本。

## 二、各步骤完善路线图

### Step 1: 文本导入 — 已创作书籍导入

**当前状态**：当前工作区已经增加书籍下拉框，但只固定导入第 1 章，且没有将书籍、章节和语言保存为统一来源信息。

**进一步完善**：
- 将单一书籍下拉改为“书籍 → 章节 → 分析范围”三级选择，范围至少包含单章、连续章节、全书抽样。
- 不再固定请求 `/chapters/1`，应读取章节索引后使用真实章节号；需要兼容缺少第 1 章、章节不连续和空章节。
- 建立统一 `StyleSourceContext`：

```ts
interface StyleSourceContext {
  sourceType: "manual" | "file" | "url" | "chapter" | "book-sample";
  sourceName: string;
  bookId?: string;
  chapterNumber?: number;
  chapterVersion?: string;
  language: "zh" | "en";
}
```

- 导入章节时同步设置 `bookId` 和 `chapterNumber`，使漂移评分、保存回章节、返回书籍页面都使用同一来源。
- 对全书分析采用抽样或分块聚合，避免直接把整本书送入单次分析。

**工作量**：0.5—1 天  
**依赖**：章节索引接口、统一来源状态

---

### Step 2: 文风诊断 — 增加原文风险标注

**当前状态**：已经增加“原文风险标注”折叠区，但当前实现仍是全文预览加问题摘要，并不是真正的原文定位。

**进一步完善**：
- 直接消费现有诊断结果中的位置字段：
  - `intentRepetitions[].examples[].start/end`
  - `repeatedDescriptions[].occurrences[].start/end`
  - 复杂句和修辞结果中的行号或句子样本
- 使用统一 `TextRangeIssue` 适配层，把不同诊断结果转换成 `{ id, category, severity, start, end, message }`。
- 左侧显示问题列表，右侧使用只读文本编辑器进行区间高亮；点击问题后滚动到对应片段。
- 支持同一区间多个问题叠加，避免后渲染的问题覆盖前一个问题。
- 文本发生变化后比较 `sourceHash`，旧诊断立即标记为“已过期”，禁止继续应用旧位置补丁。
- 增加误报反馈：忽略单项、忽略同类规则、恢复忽略，并把忽略范围绑定到文本 hash 或规则版本。

**工作量**：1—1.5 天  
**依赖**：统一位置适配器、可高亮文本组件

---

### Step 3: AI 痕迹检测 — 自动检测 + 修改建议

**当前状态**：已经增加 `initialText` 并在进入页面时自动检测，但请求逻辑与手动检测重复，语言仍固定为 `zh`。

**进一步完善**：
- 抽取单一 `runDetection(text, language)` 方法，自动和手动检测共用，避免两套请求逻辑漂移。
- 自动检测只在切换到该步骤或来源 hash 变化时触发，并增加 300—500ms 防抖、请求取消和结果缓存。
- 语言从 `StyleSourceContext.language` 传入；手动粘贴时可自动识别并允许用户修正。
- AI 风格风险必须显示“启发式评分”和置信度，不应把结果表述为“确定由 AI 生成”。
- 每个标签关联原文区间、规则说明、修改原则和可选建议；不要直接进行无审查替换。
- 将问题项传给统一调整计划，并通过 diff 预览执行；避免在 `AITellsPanel` 内再建立独立改写流程。
- 增加“修改后复检”，显示风险分、可读性和作家匹配度的前后变化。

**工作量**：0.5—1 天  
**依赖**：统一分析会话、请求取消机制

---

### Step 4: 修辞去重 — 段落 + 修辞双重去重

**当前状态**：页面仍是占位提示，但所需组件和后端检测接口已经存在。

**后端能力**：
| 能力 | 来源 | 当前状态 |
|------|------|---------|
| 段落去重 | `detectDuplicateParagraphs`（browser 入口） | ✅ 纯函数可用 |
| 修辞去重 | `detectDuplicateRhetoric`（browser 入口） | ✅ 纯函数可用 |
| 输入检查发现 | `POST /style/preprocess/inspect` 返回的 `duplicate-*` 码 | ⚠️ 预处理阶段已有结果，未与 Step 4 联动 |
| 预处理去重选项 | `deduplicateParagraphs` checkbox | ✅ 已有 |
| 段落检测 API | `POST /style/paragraph/dedup` | ✅ 后端可用 |
| 修辞改写 API | `POST /style/rhetoric/rewrite` | ⚠️ 目前只返回 prompt，不返回改写文本 |
| 可读性 API | `GET /style/readability/score` | ⚠️ 可用，但长文本 query 设计需要调整 |

**进一步完善**：
- Step 4 进入时并行请求段落去重、修辞检测和可读性评分，结果统一绑定当前 `sourceHash`。
- 接入现有三个组件：
  - `ReadabilityDashboard`：总览评分和维度。
  - `RhetoricIssuePanel`：修辞类别、示例、忽略、跳转和 AI 改写入口。
  - `DuplicateParagraphPanel`：完全重复删除、相似段落合并。
- “删除重复”不能使用行号直接修改原文，应使用稳定的字符区间或段落 ID，并在执行前验证原文仍匹配。
- “合并”先生成 diff，不直接把最长段落当作自动合并结果；最长段落只能作为编辑初稿。
- 将 `/style/rhetoric/rewrite` 改造成真正的服务端改写预览接口，或明确返回 `prompt` 后由统一 LLM 执行器完成调用，避免前端拿到 prompt 后无法继续。
- 将可读性评分改为 `POST /style/readability/score`，避免长文本进入 URL、浏览器历史和代理日志。
- 所有修改完成后自动重新检测，问题计数必须下降或明确提示无改善。

**工作量**：1.5—2 天  
**依赖**：文本区间编辑器、统一 LLM 改写执行器

---

### Step 5: 应用审计 — 调整建议 + 作家对比 + 审计结果

**当前状态**：作家档案应用、调整建议、作家对比和 diff 预览已进入当前工作区，但仍缺少应用前后审计记录。

**进一步完善**：
1. **先比较后应用**：选择作家后先展示样本充分度、置信度、主要偏差和预计影响，再允许写入书籍。
2. **应用作家档案需要预览**：展示将覆盖或创建的 `style_profile.json`、`style_guide.md`、`style_source.json`，并允许取消。
3. **增加版本记录**：`style_source.json` 记录作家档案版本、蒸馏版本和规则版本，不能只记录作家 ID 与时间。
4. **应用后审计**：对当前章节或抽样章节计算应用前后匹配度，保存结果而不是只显示一次。
5. **避免固定第 1 章**：漂移评分必须使用来源上下文中的真实章节号；全书审计应显示章节列表和趋势。
6. **比较结果进入状态**：不要仅 `console.log`，应保存到页面状态，并可作为调整计划的输入。
7. **章节回写安全**：保存修改前检查章节版本或内容 hash，冲突时要求重新加载，不能覆盖并发修改。

**工作量**：1—1.5 天  
**依赖**：来源上下文、章节版本字段、审计记录结构

---

### Step 6: 作家蒸馏工作台 — 补齐后端到前端的最大断点

**当前状态**：后端已经支持生成草稿、读取当前蒸馏、修改规则覆盖、发布版本和查看版本列表，前端尚无完整入口。

**建议做法**：
- 在作家详情中增加“蒸馏规则”页签，展示样本数、总字数、充分度、置信度、档案版本和过期状态。
- 支持生成草稿、逐条启用/禁用规则、编辑规则文本、查看证据片段、发布版本、查看历史版本。
- 档案重分析后，如果 `authorProfileVersion` 变化，明确标记蒸馏草稿过期并禁止直接发布。
- 应用到书籍时优先选择已发布蒸馏版本；未发布草稿只能用于预览，不能成为生产写作规则。
- 书籍侧保存 `authorId + authorProfileVersion + distillationVersion`，保证后续章节生成可追溯。
- 提供规则冲突检查：作家蒸馏规则不得静默覆盖书籍禁令、题材规则和角色声线约束。

**工作量**：1.5—2 天  
**依赖**：现有 distillations API、规则冲突检测

### Step 7: 统一分析会话 — 解决组件之间状态割裂

建议把当前散落在 `StyleManager`、`AITellsPanel`、`AdjustmentSuggestionsPanel` 和 `AuthorStyleComparison` 中的状态整合为一个页面级会话：

```ts
interface StyleAnalysisSession {
  source: StyleSourceContext;
  text: string;
  sourceHash: string;
  profile?: CoreStyleProfile;
  diagnostics?: FullStyleDiagnostics;
  aiTells?: AITellResult;
  readability?: ReadabilityScore;
  comparison?: StyleComparisonResult;
  adjustmentPlan?: AdjustmentPlan;
  preview?: StyleRewritePreview;
  status: "idle" | "analyzing" | "ready" | "stale" | "error";
}
```

实施要求：
- 文本一旦变化，所有依赖旧 `sourceHash` 的结果统一变为 `stale`。
- 各步骤读取同一份文本、语言、来源和目标作家，不再各自维护一份文本框状态。
- 请求层支持 `AbortController`，切换文本或页面时取消旧请求，避免慢请求覆盖新结果。
- 诊断、比较、调整计划和预览都显示规则版本、档案版本和生成时间。
- 页面刷新后仅恢复来源信息和草稿文本，不恢复可能已经过期的分析结果。

## 五、工程质量与验收标准

### 5.1 API 与契约

- 所有文风 POST 请求使用 Zod schema 校验，错误返回稳定错误码，不只返回字符串。
- 长文本不通过 GET query 传输；可读性评分调整为 POST。
- `diagnostics`、`comparison`、`plan`、`preview` 返回 `sourceHash` 和 `ruleVersion`。
- 与作家相关的结果返回 `authorProfileVersion`；与蒸馏相关的结果返回 `distillationVersion`。
- 修辞改写接口应返回可审核的改写结果和变更区间，而不是只返回 prompt。

### 5.2 前端交互

- 用户从任意书籍章节进入文风模块后，页面能正确恢复书籍、章节号、文本和语言。
- 点击任何诊断问题都能定位到原文；文本修改后旧结果明确显示“已过期”。
- 所有 AI 改写必须先展示 diff，用户确认后才更新当前草稿。
- 保存章节前检查内容版本；发生冲突时不覆盖原内容。
- 切换步骤不会重复无意义调用接口，离开页面时旧请求被取消。
- 深色主题和英文界面下不得出现固定白底、中文硬编码和不可读文本。

### 5.3 自动化测试

- Core：为中英文诊断、短样本降置信度、位置区间、规则版本和误报忽略增加单测。
- API：覆盖空文本、错误作家 ID、档案版本冲突、章节不存在、长文本和过期 `sourceHash`。
- Studio：覆盖书籍/章节选择、自动检测只触发一次、文本变化后结果 stale、diff 接受/拒绝和章节冲突。
- 端到端：至少完成“导入章节 → 诊断 → 定位问题 → 生成调整 → 接受 diff → 保存 → 复检”一条完整链路。

### 5.4 可量化验收指标

| 指标 | 验收目标 |
|------|----------|
| 章节来源正确率 | 任意章节进入后 `bookId/chapterNumber` 100% 对应 |
| 问题定位成功率 | 有位置数据的问题 100% 可跳转高亮 |
| 过期结果保护 | 文本 hash 变化后旧补丁 100% 禁止应用 |
| 自动请求控制 | 同一文本首次进入 AI 检测最多发起 1 次请求 |
| 保存安全 | 章节版本冲突时 0 次静默覆盖 |
| 修改闭环 | diff 接受后自动复检并展示前后指标 |
| 蒸馏可追溯性 | 应用规则可追溯到作家档案和蒸馏版本 |

## 七、总结

文风分析模块的基础检测、作家档案、调整计划、diff 预览、修辞去重和作家蒸馏后端已经具备较完整的能力。当前主要问题已经不再是“有没有功能”，而是“同一份文本能否在各步骤中保持来源一致、版本一致并安全回写”。

下一轮应优先解决：

1. **统一来源和分析会话**：消除固定第 1 章、语言硬编码和组件各自维护状态的问题。
2. **诊断到原文的精确关联**：使用已有区间数据实现真正高亮，而不是只展示全文。
3. **修改安全闭环**：所有调整经过 diff、版本校验、用户确认和修改后复检。
4. **修辞去重正式接入**：现有组件可以复用，重点处理稳定区间修改和结果回写。
5. **作家蒸馏前端化**：把已经完成的后端能力转成可编辑、可发布、可追溯的规则工作台。

完成上述工作后，文风模块才能从多个独立检测面板升级为可持续用于章节生产的“文风质量控制系统”。
