# NoFusion 0607 任务实施流程建议

> 基于 `0607项目推进建议.md` 第六节（建议实施顺序）和第七节（建议任务清单）  
> 目标：将 12 项任务转化为可执行的实施流程，标注前置依赖和风险点

---

## 总体原则

1. **先收口，再拓展** —— P0 全部完成后才进入 P1，P1 全部完成后才进入 P2。
2. **每个任务在合并前必须通过 typecheck + 全量测试 + 对应新增测试**。
3. **Core/API/Studio/CLI 四层联动** —— 不做单一层的独立改动。

### 任务 1：修复 Studio 服务端类型边界

**前置**：无  
**依赖**：无  
**风险**：❌ 当前 server typecheck 未通过，所有 server 端改动无法得到类型保障

**实施步骤**：

1. 识别所有 `src/api` 中从前端 `src/pages/` 或 `src/shared/` import 的模块：
   ```bash
   grep -r "from.*\.\./pages/" src/api/ --include="*.ts"
   grep -r "from.*\.\./shared/" src/api/ --include="*.ts"
   ```
2. 将共享类型（contracts、schemas、presets）抽出为独立包或调整 tsconfig 包含路径：
   - 方案 A：`tsconfig.server.json` 增加 `"include": ["src/shared/**/*.ts"]`
   - 方案 B：将共享类型移入 `src/shared/` 并确保两边 tsconfig 都包含
3. 将 `style-preprocess-adapter.ts` 中对前端页面状态的引用改为由参数传入，切断对 `pages/` 的依赖
4. 修复后验证：
   ```bash
   cd packages/studio
   npx tsc -p tsconfig.server.json --noEmit   # 通过
   npx tsc -p tsconfig.json --noEmit           # 前端 typecheck 不受影响
   ```

**验收**：`tsc -p tsconfig.server.json --noEmit` 无错误。

### 任务 2：预处理预设真实闭环

**前置**：任务 1（server typecheck 通过后才能安全修改 server 端）  
**依赖**：Core `preprocessText()` / `relayoutText()` 的选项 schema  
**风险**：⚠️ 预设选项与 Core 实际行为的一致性需要对照验证

**实施步骤**：

#### 2a. 修复前端预设行为

1. 确认初始 `fidelity` 模式对应的实际选项值：
   - `fidelity = true` → 所有清理开关为 `false`（不清理）
   - 当前代码初始显示 `fidelity` 但开关实际为 `true`，修复前端 `preprocess-state.ts` 初始值
2. 预设选择联动修复：
   - 选择预设时同时更新 `preprocessOptions` 和 `relayoutOptions`
   - 当前只更新清理选项，未更新 relayout
   - 在 `usePreprocessState.ts` 或等效位置增加联动逻辑

#### 2b. 接入风险确认流程

1. `computeRemovalStats()` 已有定义但未接入——在用户点击"执行"前调用
2. `requiresConfirmation()` 返回 `true` 时弹出确认弹窗
3. 用户确认后才调用 `/style/preprocess`
4. 预检与执行分离流程：

```
用户点击"执行"
  → computeRemovalStats()
  → requiresConfirmation()?
    → 是 → 弹窗展示删除率及高风险项 → 用户确认 → 调用 API
    → 否 → 直接调用 API
```

#### 2c. 加固 API schema

1. 为 `/style/preprocess` 和 `/style/relayout` 定义 Zod schema，拒绝未知字段：
   ```typescript
   const PreprocessOptionsSchema = z.object({
     removePageNumbers: z.boolean().default(false),
     unifyQuotes: z.boolean().default(false),
     // ... 全部字段枚举
   }).strip() // strip 会删除未知字段，但先 log 警告
   ```
2. 超限校验：输入字符 > 5_000_000 时返回 `413 Payload Too Large`，不要静默截断
3. 在 API adapter 层增加 schema 校验，非 200 响应明确错误结构：
   ```json
   { "error": "PAYLOAD_TOO_LARGE", "message": "...", "maxChars": 5000000 }
   ```

#### 2d. 修复 UI 大文本挂载问题

1. StyleManager 中如果原始文本 > 100_000 字符，只渲染首部/中部/尾部抽样
2. 完整文本只在调用 API 时发送，不挂载到 textarea

#### 2e. 增加测试

1. UI helper 测试：预设选择联动、初始 fidelity、风险确认逻辑
2. API adapter 测试：schema 校验、超限 413、未知字段拒绝
3. 端到端契约测试：同一预设通过 Studio 和 CLI 产生相同的 Core options

**验收**：
- fidelity 初始值为全部关闭
- 预设切换同时更新 preprocess + relayout
- 高风险操作有确认弹窗
- 超限返回 413，不静默截断
- 新增测试覆盖预设选择、风险确认、超限拒绝

### 任务 4：修复分片元数据与前端加载

**前置**：任务 1（server typecheck）  
**依赖**：Core `extractDocumentChunked()`  
**风险**：⚠️ 普通提取与分片提取的 `totalChunks` 口径不一致

**实施步骤**：

1. 统一元数据口径：
   - 普通提取（整体 < 500 万字符）：`totalChunks = 1, chunkSize = totalChars`
   - 分片提取：`totalChunks = ceil(totalChars / chunkSize)`
   - 保证 `totalChunks * chunkSize >= totalChars`
2. StyleManager 读取 `truncated / totalChunks / chunkIndex` 元数据：
   - 第一个分片展示元数据和抽样预览
   - 用户可点击加载后续分片（每次请求指定 `chunkIndex`）
   - 加载失败时允许重试
3. 避免重复解析：
   - 引入短期 `extractionId`（类似 planId 模式），服务端缓存已解析结果
   - 分片请求通过 `extractionId + chunkIndex` 获取，不重新上传原文
4. 前端处理超大文本：
   - 初始展示只渲染首部、中部、尾部抽样预览
   - 完整文本仅用于 API 调用，不放入 textarea

**验收**：
- `totalChunks` 语义统一
- 分片可逐段加载
- 大文本不挂载完整 textarea
- 分片请求不重复解析原文

### 任务 5：修复锁和 Studio 异步测试

**前置**：无  
**依赖**：无  
**风险**：⚠️ daemon start 时序测试和 webhook 安全测试的修复需要了解测试框架

**实施步骤**：

1. **锁问题修复**（P1-1 完全修复）：
   - 方案：使用带 nonce 的锁内容 + 原子 `rename` 方案
   - 临时锁文件以 `.write.lock.{nonce}` 命名，写入后 `rename` 到 `.write.lock`
   - `rename` 在 POSIX 和 Windows 上都是原子的
   - 当前"重读校验 + unlink"方案缩小了窗口但未消除竞争，应升级为此方案

2. **MemoryDB 工厂统一降级**（P1-7 遗留）：
   - 提供 `tryCreateMemoryDB(bookDir): MemoryDB | null` 工厂函数
   - 替换所有直接 `new MemoryDB()` 的调用点
   - 使其调用方不再需要自行 catch

3. **Studio 异步测试修复**：
   - daemon start 超时：检查 `start()` 中 `triggerWriteCycle()` 的首次执行是否设置了合理的超时等待
   - webhook 安全测试超时：检查安全校验逻辑中是否有同步网络请求导致超时
   - 在 `__tests__` 中增加 mock / fake 来隔离网络依赖

**验收**：Core + Studio + CLI 全量测试通过，无偶发超时。

**前置**：任务 1、3  
**依赖**：`story/sources/index.json` 格式

### 任务 8：统一输入工作台

**前置**：任务 2、4、6、7  
**依赖**：BookCreate、ImportManager、StyleManager 三者的文件选择逻辑  
**风险**：⚠️ 涉及三个页面的重构，需做好向后兼容

**实施步骤**：

1. 提取共享组件 `SourceUploader`：
   - 文件拖拽 / 粘贴 / URL 导入
   - Core 文档提取（调用 `/style/extract-text`）
   - 输入体检（调用 `/style/health-check`）
   - 用途分类（auto 或手动选择）
   - 可选预处理（调用 `/style/preprocess`）
   - 影响预览（显示将生成/修改的文件列表）
2. 三个使用入口只配置最终去向：
   - BookCreate → 新书基础架构
   - ImportManager → 存量架构 / 章节 / 文风
   - StyleManager → 风格分析 / 作家档案 / 导出
3. 将共享组件放入 `src/shared/components/` 目录
4. 确保不破坏三个页面现有功能的向后兼容性

**验收**：三个入口复用同一文件选择和提取逻辑，行为一致。

### 任务 9：模型能力标签与路由预设

**前置**：任务 1

**风险**：低

## 风险清单

| 风险 | 影响 | 缓解措施 |

| server typecheck 修复涉及架构决策 | 阻塞所有 server 端 P0 任务 | 优先在半天内决策方案并实施 |
