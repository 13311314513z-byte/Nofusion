### 1.2 已集成的链路

```
输入文本
```

### 1.3 现有数据链路断点

通过代码审查（`style-preprocess-adapter.ts:208-244` 与 `contracts.ts:161-168`），发现以下**结构性断点**：

| 断点 | 位置 | 问题 | 影响 |
|------|------|------|------|
| **A. `InspectionFinding` 缺 `ranges`** | `contracts.ts:161` | `InspectionFinding` 只有 `lineNumbers` + `samples`，无字符级 `ranges` | 编辑器高亮无法直接使用 `inspectText()` 的输出，必须二次调用 `detectDuplicateRhetoric()` 获取位置信息 |
| **B. `severity` 精度丢失** | `style-preprocess-adapter.ts:237` | `finding.severity` 被映射为 `"warning"` 或 `"info"` 两档，丢失了 `low/medium/high` 三级精度 | 问题面板无法按严重度做颜色区分 |
| **C. `id` 未传递** | `style-preprocess-adapter.ts:235` | `DuplicateRhetoricFinding.id` 未映射到 `InspectionFinding` | 问题面板的"忽略/标记"状态无法与具体 finding 关联 |
| **D. `perThousandChars` 未传递** | `style-preprocess-adapter.ts` | 千字频率指标被丢弃 | 阅读性评分仪表盘缺少关键输入 |

**修复建议**：扩展 `InspectionFinding` 接口，新增可选字段，保持向后兼容：

```typescript
// packages/studio/src/shared/contracts.ts
export interface InspectionFinding {
  readonly code: InspectionCode;
  readonly severity: "info" | "warning";
  readonly count: number;
  readonly lineNumbers?: readonly number[];
  readonly samples: readonly string[];
  readonly messageKey: string;
  // === 新增可选字段（修辞检测专用）===
  /** 字符级位置范围，用于编辑器高亮 */
  readonly ranges?: readonly { readonly start: number; readonly end: number }[];
  /** 三级严重度（修辞检测原生精度） */
  readonly rhetoricSeverity?: "low" | "medium" | "high";
  /** 每千字出现次数 */
  readonly perThousandChars?: number;
  /** 置信度 0-1 */
  readonly confidence?: number;
  /** 唯一标识（用于忽略/标记状态持久化） */
  readonly findingId?: string;
}
```

对应的 `style-preprocess-adapter.ts` 映射更新：


### 2.5 英文检测现状与扩展路径

当前 `detectDuplicateRhetoric` 对英文直接返回空数组（`semantic-duplication.ts:251-253`），但 `EN_PATTERNS` 已存在两个 stub 条目（Simile、Rhetorical question）。

**建议：**
1. 短期（P3）：补全 `EN_PATTERNS` 到与 `ZH_PATTERNS` 对等的覆盖度，至少实现 metaphor / repetition / transition / parallelism 四类。
2. 中期：将语言判断从 `isChineseText()` 的 0.1 阈值改为更精确的语言识别，避免中英文混合文本误判。
3. 长期：考虑引入轻量 NLP 库（如 `compromise` 或 `natural`）进行词干提取，提升英文检测准确率。

**高亮策略：**

#### 3.1.3 性能优化：正则预编译与增量检测

当前 `detectDuplicateRhetoric` 每次调用都重新编译正则（`new RegExp(regex.source, ...)`），对于大文本或频繁调用的场景（如用户实时输入时的 debounce 检测）存在性能浪费。

**优化方案：**

```typescript
// packages/core/src/utils/semantic-duplication.ts

// 在模块初始化时预编译所有正则
const COMPILED_PATTERNS = ZH_PATTERNS.map((p) => ({
  ...p,
  compiled: p.patterns.map((regex) =>
    new RegExp(regex.source, regex.flags.includes("g") ? regex.flags : regex.flags + "g")
  ),
}));

// detectDuplicateRhetoric 中直接使用预编译版本
for (const pattern of COMPILED_PATTERNS) {
  for (const regex of pattern.compiled) {
    // 不再 new RegExp(...)
    let m: RegExpExecArray | null;
    while ((m = regex.exec(text)) !== null) {
      // ...
    }
    // 重置 lastIndex 以便下次调用
    regex.lastIndex = 0;
  }
}
```

**增量检测（实时编辑场景）：**

当用户在编辑器中修改文本时，无需全量重新检测，可采用以下策略：

```typescript
interface IncrementalRhetoricState {
  /** 文本内容的哈希 */
  textHash: string;
  /** 各修辞类别的匹配位置缓存 */
  categoryPositions: Map<RhetoricCategory, Array<{ start: number; end: number }>>;
}

function detectDuplicateRhetoricIncremental(
  newText: string,
  oldState: IncrementalRhetoricState | null,
  changedRange: { start: number; end: number; newLength: number },
): DuplicateRhetoricResult {
  // 1. 若文本哈希前缀匹配且修改范围 < 500 字符，仅重新检测修改区域 ±200 字符
  // 2. 否则回退到全量检测
  // 3. 更新位置偏移：修改点之后的 ranges 根据长度差调整 start/end
}
```

> 增量检测标记为 **P3 Future**，当前阶段全量检测已足够（10 万字 < 200ms）。

### 3.3 一键接受/忽略建议

**优先级：P1** | 工作量：1 天

每个 finding 支持三种操作：

| 操作 | 效果 | 实现 |
|------|------|------|
| **标记为已读** | 该 finding 灰显，不再提示 | 持久化到 localStorage |
| **忽略此类** | 同类别所有 finding 隐藏 | 更新 filterCategories |
| **跳转到源码** | 编辑器滚动到对应 `ranges[0].start` | `editor.scrollToRange()` |

#### 3.3.1 持久化策略与多 Tab 同步

当前方案使用 `localStorage` 按 `storageKey` 隔离（如 `"style-rhetoric"`、`"audit-rhetoric"`），存在以下问题：

| 问题 | 场景 | 建议 |
|------|------|------|
| 多 Tab 状态不同步 | 用户在 Tab A 忽略某条，Tab B 仍显示 | 使用 `BroadcastChannel` 同步状态变更 |
| localStorage 大小限制 | 大量 finding 的忽略列表可能超限 | 使用 `IndexedDB` 存储，或只存储哈希指纹 |
| 数据丢失 | 清理浏览器数据后状态丢失 | 将「已修复/已忽略」状态持久化到章节元数据 |

**推荐方案（分层存储）：**

```typescript
interface RhetoricIgnoreState {
  /** 按 findingId 的忽略状态 */
  ignoredIds: string[];
  /** 按类别的全局忽略 */
  ignoredCategories: RhetoricCategory[];
  /** 版本号（用于 schema 迁移） */
  version: number;
}

// Layer 1: 运行时内存（最快）
const runtimeIgnoreState = new Map<string, RhetoricIgnoreState>();

// Layer 2: localStorage（会话级）
function saveIgnoreState(storageKey: string, state: RhetoricIgnoreState) {
  localStorage.setItem(`inkos-rhetoric-ignore-${storageKey}`, JSON.stringify(state));
}

// Layer 3: BroadcastChannel（跨 Tab 同步）
const bc = new BroadcastChannel("inkos-rhetoric-sync");
bc.onmessage = (ev) => {
  if (ev.data.type === "ignore-state-changed") {
    runtimeIgnoreState.set(ev.data.storageKey, ev.data.state);
    // 触发 React re-render
  }
};

// Layer 4: 章节元数据（最持久，审计入口使用）
// 在 saveChapter / audit 结果中写入修辞问题的修复状态
```

#### 3.3.2 撤销/回退设计

一键接受 AI 改写或批量删除重复段落是**破坏性操作**，必须提供撤销能力：

```typescript
interface ReadabilityEditOperation {
  readonly id: string;
  readonly type: "ai-rewrite" | "delete-duplicate" | "merge-paragraph";
  readonly timestamp: number;
  /** 修改前的文本 */
  readonly before: string;
  /** 修改后的文本 */
  readonly after: string;
  /** 受影响的 finding IDs */
  readonly affectedFindingIds: string[];
}

// 操作栈（LIFO，最多保留 20 步）
const operationStack: ReadabilityEditOperation[] = [];

function pushOperation(op: ReadabilityEditOperation) {
  operationStack.push(op);
  if (operationStack.length > 20) operationStack.shift();
}

function undoLastOperation(): ReadabilityEditOperation | null {
  const op = operationStack.pop();
  if (!op) return null;
  // 恢复 before 文本
  return op;
}
```

**UI 体现：**
- 每次批量操作后显示 toast：`「已修改 3 处比喻」↩️ 撤销`
- 工具栏常驻「撤销」按钮（Ctrl+Z / Cmd+Z）
- 章节保存前可查看完整的修改历史时间线

### 3.4 AI 批量改写修辞

**优先级：P1** | 工作量：3 天 | 依赖：LLM 可用

#### 3.4.1 改写策略

对于检测到的过度重复修辞，提供三种改写模式：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **替换** | 用同义表达替换重复修辞 | 比喻/拟人过多 |
| **删除** | 删除冗余修辞，保留语义 | 排比/并列过多 |
| **重组** | 将连续同类修辞分散到全文 | 首语重复/过渡词聚集 |

#### 3.4.2 Prompt 设计

```
你是一个中文小说编辑助手。以下是用户文稿中发现的修辞重复问题：

{findings_json}

请对原文进行最小干预修改，遵循以下规则：
1. 保持原文风格和叙事节奏
2. 每处修改都提供修改理由
3. 修改后修辞重复度应低于阈值
4. 不要引入新的 AI 腔
```

#### 3.4.4 与现有 `style-rewriter.ts` 的具体衔接

当前 `style-rewriter.ts` 的 `rewriteWithAuthorProfile` 接受 `text` + `profile` + `options`，生成风格调整后的文本。修辞去重改写可以作为**前置步骤**或**合并步骤**：

**方案 A：前置去重（推荐）**

```

  → detectDuplicateRhetoric() → 发现问题
  → rewriteByRhetoricDedupe() → 去重后文本
  → rewriteWithAuthorProfile() → 风格统一后的最终文本
```

优点：去重和风格调整分离，职责清晰；去重后的文本更干净，风格调整效果更好。

**方案 B：合并改写（一步完成）**

在 `rewriteWithAuthorProfile` 的 prompt 中注入 `findings_json`，让 LLM 同时处理风格调整和修辞去重：

```
你是{authorName}的编辑。请根据以下风格档案和修辞重复问题，
对文本进行最小干预修改：

【风格档案】
{profile_json}

【修辞重复问题】
{findings_json}

【待修改文本】
{text}
```

优点：一次 LLM 调用完成两件事，减少延迟。
缺点：prompt 更长，可能稀释风格调整的注意力；调试困难。

**推荐采用方案 A**，并通过缓存 `detectDuplicateRhetoric` 结果避免重复计算。

### 3.5 自动删除极端重复段落

**优先级：P2** | 工作量：1 天

#### 3.5.1 完全重复段落检测

当前 `semantic-duplication.ts` 已通过 `repetition` 类别检测同一短语的短距离重复。扩展为检测**完全相同的段落**：

```typescript
export function findDuplicateParagraphs(text: string): {
  paragraphs: ReadonlyArray<{
    hash: string;           // 段落内容的 MD5/快速哈希
    content: string;
    lineNumber: number;
    duplicates: ReadonlyArray<{
      lineNumber: number;
    }>;
  }>;
}> {
  const paras = text.split(/\n\s*\n/);
  const seen = new Map<string, number[]>(); // hash → [lineNumbers]
  
  paras.forEach((para, i) => {
    const normalized = para.trim().replace(/\s+/g, "");
    if (normalized.length < 20) return; // 跳过短行
    const hash = quickHash(normalized);
    const existing = seen.get(hash) ?? [];
    seen.set(hash, [...existing, findLineNumber(text, para)]);
  });

  // 只报告出现 >= 2 次的段落
  return {
    paragraphs: [...seen.entries()]
      .filter(([, lines]) => lines.length >= 2)
      .map(([hash, lines]) => ({
        hash,
        content: extractContent(text, lines[0]),
        lineNumber: lines[0],
        duplicates: lines.slice(1).map((l) => ({ lineNumber: l })),
      })),
  };
}
```

### 3.8 与 Pipeline 的联动：从源头减少修辞重复

当前修辞检测仅在**事后分析**阶段运行（文风分析、导入检查、审计），但最理想的方案是让 **writer agent 在生成本章时即避免过度使用同类修辞**。

**方案：将修辞统计注入 System Prompt**

```typescript
// packages/core/src/agents/rhetoric-rewriter.ts

export function buildRhetoricAwarePrompt(
  basePrompt: string,
  contextText: string,      // 已生成的前文
  maxPerThousandChars: Partial<Record<RhetoricCategory, number>>,
): string {
  const stats = detectDuplicateRhetoric(contextText);
  const warnings: string[] = [];

  for (const finding of stats.findings) {
    const limit = maxPerThousandChars[finding.category];
    if (limit && finding.perThousandChars >= limit) {
      warnings.push(
        `- 已使用 ${finding.label} ${finding.count} 次（${finding.perThousandChars}/千字），` +
        `接近上限 ${limit}/千字，后续段落请减少使用。`
      );
    }
  }

  if (warnings.length === 0) return basePrompt;

  return `${basePrompt}\n\n【修辞使用提示】\n前文已出现的修辞统计如下。` +
    `请在继续写作时自然分散修辞使用，避免同类修辞过度集中：\n${warnings.join("\n")}`;
}
```

**集成到 Pipeline：**

在 `PipelineRunner.writeNextChapter()` 调用 writer agent 前，提取已生成章节的文本，运行 `detectDuplicateRhetoric`，将统计结果追加到 system prompt 中。

```
已生成 3000 字
  → detectDuplicateRhetoric(前 3000 字)
  → 发现比喻 8 次（2.7/千字）超过阈值 2.0
  → 在 writer 的 system prompt 中追加：「前文比喻使用较多，后续请减少」
  → writer 生成下一部分时自然减少比喻
```

**优点：** 从源头解决问题，比事后修改更自然。
**局限：** 增加每次 LLM 调用的 prompt 长度（约 +200 字符）；对 writer 的指令遵循能力有依赖。

**优先级：P2** | 可在 `rhetoric-rewriter.ts` 中先实现工具函数，再在 Pipeline 中逐步试点接入。

### 5.6 接口调整建议：扩展 `InspectionFinding`

如 1.3 断点分析所述，`InspectionFinding` 需要新增可选字段才能完整承载修辞检测信息。调整后的接口：

```typescript
export interface InspectionFinding {
  readonly code: InspectionCode;
  readonly severity: "info" | "warning";
  readonly count: number;
  readonly lineNumbers?: readonly number[];
  readonly samples: readonly string[];
  readonly messageKey: string;
  /** 字符级位置范围，用于编辑器高亮 */
  readonly ranges?: readonly { readonly start: number; readonly end: number }[];
  /** 三级严重度（修辞检测原生精度） */
  readonly rhetoricSeverity?: "low" | "medium" | "high";
  /** 每千字出现次数 */
  readonly perThousandChars?: number;
  /** 置信度 0-1 */
  readonly confidence?: number;
  /** 唯一标识（用于忽略/标记状态持久化） */
  readonly findingId?: string;
}
```

**向后兼容性：** 所有新增字段均为 `readonly?` 可选，不影响现有的 `explicit-think-block`、`encoded-data` 等非修辞检测 finding。

```
  → 用户选择「AI 替换」→ LLM 将 2 处比喻改为直述
  → 预览 diff，确认修改合理 → 接受
  → 比喻使用降至 4 次 (2.0/千字) ✅ 安全
  → 可阅读性评分从 65 → 78 ↑
```
