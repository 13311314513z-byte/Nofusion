# 文风分析调整方案

> 报告日期：2026-06-07  
> 项目状态：Core 文风分析体系已健全（style-analyzer、style-fingerprint、ai-tells、post-write-validator、style-library），Studio StyleManager 已具备基本分析流程  
> 目标：在保留现有分析流程的基础上扩展诊断维度，并在"文本样本"模块下新增"调整建议"，形成"分析→诊断→作家档案对比→预览调整→复检"闭环

### 1.2 缺口分析

| 缺口 | 严重度 | 现有基础 | 说明 |
|------|:------:|----------|------|
| 意图/动机重复 | P1 | `post-write-validator` 部分标记 | 需要区分"动作/表达重复"与真正的"语义意图重复"；前者可规则检测，后者只能给出启发式判断或引入语义模型 |
| 重复描述 | P1 | `detectCrossChapterRepetition()` 可复用 | 同一事物/场景/动作被多次描述，现有 N-gram 能力尚未下沉为通用文本内重复检测 |
| 过多转折 | P1 | `ai-tells` 有转折词密度 | 只检查"然而""不过"等词汇密度，未检测"每段都转折"的节奏问题 |
| 过多从句 | P1 | 标点节奏可复用 | 逗号数只能作为句法复杂度代理指标，不能直接等同从句数 |
| AI风格标签 | P2 | `ai-tells` + marker词 | 已有基础，但未聚合为可解释的"AI 风格启发式风险标签" |
| 无调整建议 | P2 | LLM Provider 已具备 | 当前只诊断；规则可以生成修改计划，但按作家档案改写需要 LLM 预览能力 |
| 无作家档案对比 | P2 | `style-library` | 可导入作者档案，但未做"当前文本 vs 目标风格"的差异分析 |

```
  ├── 预处理/清理/重排

```

### 2.2 Core 层扩展思路

采用两层能力，避免把规则诊断和生成式改写混为一体：

1. **诊断与计划层**：纯函数、统计型、无 LLM 调用，输出证据位置、严重度、可信度和可执行计划。
2. **调整预览层**：仅在用户选择作家档案并主动点击"预览调整"时调用现有 `createLLMClient()` / `chatCompletion()`；只返回预览，不直接写入书籍或作家档案。

新增 Core 代码放在 `packages/core/src/agents/` 下，并复用 `ai-tells.ts`、`post-write-validator.ts` 的已有检测器，避免重复维护规则。

```
packages/core/src/agents/
  ├── style-analyzer.ts        (已有)
  ├── style-fingerprint.ts     (已有)
  ├── ai-tells.ts              (已有)
  ├── post-write-validator.ts  (已有)
  ├── style-diagnostics.ts     [新增] 高级诊断聚合
  ├── style-adjuster.ts        [新增] 规则建议与文本补丁计划
  ├── style-rewriter.ts        [新增] 基于档案的受约束预览改写
  └── style-comparator.ts      [新增] 作家档案对比
```

跨层请求/响应类型不在 `StyleManager.tsx` 内重复声明：Core 暴露领域类型，Studio 在 `packages/studio/src/api/style-schemas.ts` 中增加严格 Zod 请求校验，在共享 contracts 中暴露前端所需响应类型。

## 三、新增分析维度详细设计

### 3.1 意图重复检测 (`style-diagnostics.ts` — `detectIntentRepetition`)

**检测目标**：识别同一叙事目的、信息结论或行为反馈在短篇幅内被反复表达。实现上必须分成两级：

- **一级：表达/动作意群重复**。如反复"转身、叹气、点头、看向"，可由规则稳定检测。
- **二级：语义意图重复**。如连续三段都在说明"他不信任对方"，即使措辞不同也属于重复。首版只根据心理动词、因果词和主题锚点输出低/中可信度提示，不宣称完成了严格语义理解；后续可接入嵌入模型或 LLM 复核。

| 模式 | 示例 | 检测方式 |
|------|------|----------|
| 回头/转身/看向 | "他转过身""她回头""他看向窗外" | 行为动词短语频率 |
| 叹气/深吸一口气 | "叹了口气""深吸一口气" | 习语频率，>3次/千字报警 |
| 点头/摇头 | "点了点头""摇摇头" | 肢体反馈词频率 |
| 目光/视线 | "他的目光""视线交汇" | 名词短语去重统计 |
| 心理活动引导 | "他想""她觉得""她意识到" | 心理动词短语频率 |

```typescript
// style-diagnostics.ts

export interface IntentRepetitionFinding {
  readonly kind: "action-expression" | "semantic-intent";
  readonly pattern: string;
  readonly count: number;
  readonly perThousandChars: number;
  readonly confidence: number; // 0-1；规则命中高，语义代理指标较低
  readonly severity: "high" | "medium" | "low";
  readonly examples: ReadonlyArray<{
    readonly sentence: string;
    readonly start: number;
    readonly end: number;
  }>;
}

  text: string,
  thresholdPerThousand = 3,
): ReadonlyArray<IntentRepetitionFinding> {

  // 2. 按段提取"主体+心理/判断动词+主题锚点"，形成语义意图代理特征
  // 3. 同时计算全局密度与局部窗口密度，输出证据位置和 confidence
}
```

**阈值口径**：中文按每千字，英文按每千词；短于 500 字/词时不输出高严重度结论，只标记"样本不足"。不要承诺固定的 `< 1ms` 性能，实际应通过基准测试确定。

### 3.2 重复描述检测 (`style-diagnostics.ts` — `detectRepeatedDescriptions`)

**检测目标**：同一对象/场景被用相似词汇反复描述。

| 模式 | 示例 | 检测方式 |
|------|------|----------|
| 外貌重复 | "英俊""帅气""棱角分明"多次出现 | 对象锚点 + 描述词重叠 |
| 环境重复 | "漆黑的夜""夜色""黑暗中" | 场景锚点 + 同义词群统计 |
| 属性堆叠 | "冰冷的""冷漠的""冷淡的" | 同义形容词链检测 |
| 近似句重复 | 不同段落出现高度相似的 6 字片段/句式 | N-gram/Jaccard 相似度 |

```typescript
export interface RepeatedDescriptionFinding {
  readonly cluster: string;        // 描述集群名（如"外貌-英俊系"）
  readonly subject?: string;       // 被重复描述的人物/场景/物体
  readonly matchedPhrases: ReadonlyArray<string>;
  readonly occurrences: ReadonlyArray<{ start: number; end: number }>;
  readonly density: number;        // 每千字出现次数
  readonly similarity: number;     // 0-1
  readonly confidence: number;     // 对象绑定失败时降低
  readonly severity: "high" | "medium" | "low";
}
```

**实现约束**：

1. 抽取并复用 `post-write-validator.ts` 中 `detectCrossChapterRepetition()` 的 N-gram 公共逻辑，新增"同一文本内窗口重复"模式，不再另写一套重复算法。
2. 词表只作为补充证据，不能仅因"冷、黑、看"等高频词出现就判定重复。
3. 优先将描述绑定到附近的人物名、代词或场景名；无法绑定对象时降低 `confidence`。
4. 外部 JSON 采用追加/覆盖配置，但必须带 `language` 和 `ruleVersion`。

### 3.3 转折频率分析 (`style-diagnostics.ts` — `detectTransitionClustering`)

**检测目标**：转折词在段落间出现过密，导致"每段都转折"的节奏问题。

```typescript
export interface TransitionClusteringFinding {
  readonly transitionWord: string;
  readonly totalCount: number;
  readonly paragraphsWithTransition: number;
  readonly paragraphRatio: number;
  readonly gapParagraphs: ReadonlyArray<number>;  // 间隔段落数
  readonly minGap: number;
  readonly avgGap: number;
  readonly consecutiveTransitions: number;  // 连续段落含转折词的数量
  readonly severity: "high" | "medium" | "low";
}
```

**检测逻辑**：
1. 扫描所有段落，标记含转折词（然而、不过、但是、却、可、但、不过、尽管如此等）的段落
2. 连续 3+ 段落含转折词 → high severity
3. 段落间隔均值 < 2 → medium severity
4. 计算"含转折段落占比"和局部窗口集中度，避免长文本只看全局均值
5. 复用 `ai-tells.ts` 现有公式化转折词表，并补充段落位置证据

### 3.4 从句复杂度检测 (`style-diagnostics.ts` — `detectClauseComplexity`)

**检测目标**：检测可能存在的过长分句、嵌套定语和多层状语。该模块名称保留，但产品文案应显示为"句法复杂度（代理指标）"，避免把标点数误报为真实从句数。

```typescript
export interface ClauseComplexityFinding {
  readonly sentence: string;
  readonly position: { readonly start: number; readonly end: number };
  readonly sentenceLength: number;
  readonly separatorCount: number;   // 逗号、分号、冒号等分隔符
  readonly connectiveCount: number;  // 因为/虽然/如果/当/而/但是等连接标记
  readonly estimatedClauseCount: number; // 代理估算，不是句法解析结果
  readonly hasNestedClause: boolean; // 是否包含"……的……的……"嵌套定语
  readonly maxAttributeChain: number; // 最长定语链（"的"的连续使用）
  readonly confidence: number;
  readonly severity: "high" | "medium" | "low";
}
```

**检测逻辑**：
1. 综合句长、逗号/分号、连接词、介词结构和"的"链计算复杂度，不以逗号数直接替代从句数
2. 扫描"……的……的……"模式，统计定语链深度
3. 仅当多个指标同时超阈值时给出 high；单一长句只给 low/medium
4. 若未来需要准确从句层级，再单独接入中文句法分析器，不将其伪装成规则函数能力

### 3.5 AI 风格标签总览 (`style-diagnostics.ts` — `summarizeAIStyleTags`)

**聚合现有检测结果**，输出统一评分。

```typescript
export interface AIStyleTagSummary {
  readonly heuristicRiskScore: number;   // 0-100，AI 风格启发式风险，不是作者来源概率
  readonly confidence: number;

  readonly ruleVersion: string;
  readonly paragraphUniformity: number;  // 来自 ai-tells
  readonly hedgeWordDensity: number;
  readonly transitionWordDensity: number;
  readonly markerWordDensity: number;    // 来自 post-write-validator
  readonly metaNarrationCount: number;
  readonly collectiveShockCount: number;
  readonly reportTermCount: number;
  readonly sermonWordCount: number;

```

**说明**：该函数聚合 `ai-tells`、`post-write-validator` 和新增诊断的结果。前端显示"AI 风格启发式风险"，不得显示"AI 生成概率""检测为 AI"等结论性文案。每个标签必须附证据片段、规则来源和可信度；短样本时降低可信度并禁用红色高风险结论。

## 四、调整建议模块设计

### 4.1 Core 层：`style-adjuster.ts`

**核心职责**：基于诊断结果与作家档案差异生成结构化 `AdjustmentPlan`。本模块不调用 LLM，只负责"哪里需要改、为什么改、目标指标是什么"以及少量确定性文本补丁。

```typescript
export interface AdjustmentSuggestion {
  readonly id: string;
  readonly category: "intent-repetition" | "description-repetition" | "transition" | "clause-complexity" | "ai-tell" | "vocabulary" | "rhythm";
  readonly severity: "critical" | "warning" | "info";
  readonly confidence: number;
  readonly description: string;
  readonly originalSnippet: string;     // 原文摘录（~80字）
  readonly instruction: string;         // 修改目标，不伪造具体成稿
  readonly position: { readonly start: number; readonly end: number };
  readonly targetMetric?: string;
  readonly targetRange?: { readonly min: number; readonly max: number };
  readonly patch?: {
    readonly expectedText: string;       // 防止文本变化后错位应用
    readonly replacementText: string;
  };
}

export interface AdjustmentPlan {
  readonly sourceHash: string;
  readonly ruleVersion: string;
  readonly authorProfileId?: string;
  readonly authorProfileVersion?: number;
  readonly suggestions: ReadonlyArray<AdjustmentSuggestion>;
  readonly comparison?: StyleComparisonResult;
  readonly warnings: ReadonlyArray<string>;
}
```

**建议生成逻辑**（按优先级）：

| 优先级 | 检测 | 建议类型 | 自动应用 |
|:------:|------|----------|:--------:|
| P0 | 动作/表达重复过密 | 删除冗余或合并动作 | 仅明确重复且不损失语义时 |
| P0 | 全场震惊、报告术语、显式 marker | 删除固定模板词 | 仅 `expectedText` 精确匹配时 |
| P0 | 转折词连续聚集 | 删除非必要连接词 | 仅删除后语法仍完整时 |
| P1 | 句法复杂度过高 | 建议拆句并指定保留信息 | 不自动，交由预览改写 |
| P1 | 元叙事 | 改为场景内动作/对话展示 | 不自动 |
| P2 | 外貌/环境重复描述 | 合并信息或更换观察角度 | 不自动 |
| P2 | 作家档案偏差 | 按目标区间调整节奏/对话/感官比例 | 不自动 |
| P3 | 句首模式单一 | 调整句首结构 | 不自动 |

```typescript
// style-adjuster.ts

export function generateAdjustmentSuggestions(
  text: string,
  options?: {
    readonly targetAuthorProfile?: AuthorStyleProfile;
    readonly language?: "zh" | "en";
    readonly maxSuggestions?: number;
  },
): AdjustmentPlan {
  // 1. 运行全部诊断
  const diagnostics = runFullDiagnostics(text);

  // 2. 可选：与作家档案做差异对比
  const comparison = options?.targetAuthorProfile
    ? compareWithAuthorProfile(text, options.targetAuthorProfile)
    : undefined;

  // 3. 生成建议、确定性 patch、目标区间和警告
  // 4. 写入 sourceHash 与 authorProfileVersion，供前端判断计划是否过期
  return buildAdjustmentPlan(text, diagnostics, comparison, options);
}
```

`suggestionId` 本身不足以在无状态 API 中恢复替换内容，因此不设计"只传 ID 即应用"。确定性补丁由前端根据 `sourceHash + position + expectedText` 本地执行；任何一项不匹配即标记计划过期并要求重新生成。

建议 ID 必须由 `ruleVersion + category + position + evidenceHash` 稳定生成。预览端点会基于当前文本和档案重新生成计划，再解析 `selectedSuggestionIds`，不信任前端上传的修改指令。

### 4.2 作家档案对比 (`style-comparator.ts`)

```typescript
export interface StyleComparisonResult {
  readonly targetAuthorId: string;
  readonly targetAuthor: string;
  readonly targetProfileVersion: number;
  readonly sampleAdequacy: "insufficient" | "limited" | "sufficient";
  readonly currentProfile: Partial<StyleProfile>;
  readonly targetProfile: Partial<StyleProfile>;
  readonly deviations: ReadonlyArray<{
    readonly metric: string;
    readonly currentValue: number;
    readonly targetValue: number;
    readonly normalizedDeviation: number; // 归一化偏差，处理目标值为 0 的情况
    readonly direction: "above" | "below";
    readonly suggestion: string;
  }>;
  readonly overallMatchScore: number;    // 0-100，仅表示所选统计维度的接近程度
}

export function compareWithAuthorProfile(
  text: string,
  authorProfile: AuthorStyleProfile,
): StyleComparisonResult {
  // 1. 对当前文本运行完整文风分析
  const currentProfile = analyzeStyle(text);
  const currentFingerprint = analyzeStyleFingerprint(text);

  // 2. 与目标档案逐项对比
  const deviations: StyleComparisonResult["deviations"] = [];
  //   - 句长、句长波动、段落长、对话占比、动作/心理/感官密度、标点节奏等
  //   - 使用各指标的容差区间归一化，不直接用 target=0 时失真的百分比

  // 3. 根据 authorProfile.sampleStats 判断档案样本是否足够
  // 4. 计算统计维度综合匹配度
  const overallMatchScore = computeMatchScore(deviations);

  return {
    targetAuthorId: authorProfile.id,
    targetAuthor: authorProfile.name,
    targetProfileVersion: authorProfile.version,
    /* ... */
  };
}
```

**档案使用约束**：

- API 根据 `targetAuthorId` 在服务端调用 `getAuthorProfile(root, authorId)`，不接受前端直接上传整份档案，避免档案被篡改。
- `sampleStats.totalChars < 3000` 或有效来源少于 2 个时，允许查看对比，但将 `sampleAdequacy` 标记为 `insufficient/limited`；默认禁用"按档案预览调整"或要求用户确认。
- 改写提示词只使用聚合统计、标签和高层风格特征，默认不拼接档案原始样本文本。
- 当前文本语言与档案 `language` 不一致时返回 400，或要求用户显式切换语言模式。

### 4.3 基于作家档案的调整预览 (`style-rewriter.ts`)

真正的"根据导入作家档案调整当前文段"不能只靠同义词替换完成。采用受约束 LLM 预览：

```typescript
export interface StyleRewriteRequest {
  readonly text: string;
  readonly authorProfile: AuthorStyleProfile;
  readonly plan: AdjustmentPlan;
  readonly selectedSuggestionIds: ReadonlyArray<string>;
  readonly preserveContent: true;
}

export interface StyleRewritePreview {
  readonly sourceHash: string;
  readonly authorProfileVersion: number;
  readonly adjustedText: string;
  readonly changedRanges: ReadonlyArray<{
    readonly original: string;
    readonly replacement: string;
  }>;
  readonly beforeDiagnostics: FullStyleDiagnostics;
  readonly afterDiagnostics: FullStyleDiagnostics;
  readonly warnings: ReadonlyArray<string>;
  readonly usage: {
    readonly promptTokens: number;
    readonly completionTokens: number;
    readonly totalTokens: number;
  };
}
```

调用约束：

1. 使用现有 `chatCompletion()`，温度建议 `0.2-0.4`；具体模型继续走项目 LLM 配置，不在文风模块另建密钥体系。
2. 系统提示明确要求保留人物、事实、时间顺序、视角、时态和对话含义，不新增剧情事实，只处理选中的风格目标。
3. 模型只返回改写文本或严格 JSON；返回后必须重新运行诊断，生成前后指标和 diff。
4. 预览结果不直接保存。用户点击"接受调整"后只更新 `StyleManager` 当前 `text` 状态，并进入本地撤销栈；保存/导出仍沿用现有流程。
5. 首版限制单次预览文本不超过 20,000 字符；长文本要求用户选段，避免分块改写造成跨段风格不一致和费用失控。
6. LLM 不可用时，规则诊断、档案对比和调整计划仍可使用，前端仅禁用"预览调整"。

### 4.4 API 端点

在现有 `/api/v1/style/analyze`、`/api/v1/style/authors` 基础上新增 4 个端点。全部不修改书籍文件或作家档案：

```typescript
// 1. 高级诊断
POST /api/v1/style/diagnostics
Body: { text: string; language?: "zh" | "en" }
Response: {
  sourceHash: string;
  sampleAdequacy: "insufficient" | "limited" | "sufficient";
  ruleVersion: string;
  intentRepetitions: IntentRepetitionFinding[];
  repeatedDescriptions: RepeatedDescriptionFinding[];
  transitionClustering: TransitionClusteringFinding[];
  clauseComplexity: ClauseComplexityFinding[];
  aiStyleTags: AIStyleTagSummary;
}

// 2. 作家档案对比
POST /api/v1/style/compare
Body: {
  text: string;
  targetAuthorId: string;
  language?: "zh" | "en";
}
Response: StyleComparisonResult;

// 3. 生成调整计划（规则层）
POST /api/v1/style/adjustments/plan
Body: {
  text: string;
  targetAuthorId?: string;
  categories?: AdjustmentSuggestion["category"][];
  maxSuggestions?: number;
}
Response: AdjustmentPlan;

// 4. 生成改写预览（LLM 层）
POST /api/v1/style/adjustments/preview
Body: {
  text: string;
  sourceHash: string;
  targetAuthorId: string;
  authorProfileVersion: number;
  selectedSuggestionIds: string[];
  preserveContent: true;
}
Response: {
  sourceHash: string;
  authorProfileVersion: number;
  adjustedText: string;
  changedRanges: { original: string; replacement: string }[];
  beforeDiagnostics: FullStyleDiagnostics;
  afterDiagnostics: FullStyleDiagnostics;
  warnings: string[];
  usage: { promptTokens: number; completionTokens: number; totalTokens: number };
};
```

**校验与错误约定**：

- 在 `packages/studio/src/api/style-schemas.ts` 增加 `.strict()` Zod Schema。诊断上限沿用现有 5,000,000 字符；LLM 预览单独限制为 20,000 字符。
- 服务端校验 `targetAuthorId` 并通过 `getAuthorProfile()` 加载档案；档案不存在返回 404。
- `sourceHash` 与服务端重算值不一致、档案版本已变化时返回 409，前端提示重新生成计划。
- LLM 未配置返回明确的 422/503，不影响规则端点。
- 记录耗时和 token usage，但日志不得写入完整用户文本。

| 组件/状态文件 | 职责 |
|------|------|
| `pages/style-manager/AdvancedDiagnosticsPanel.tsx` | 高级诊断分类、证据展示 |

`textarea` 无法直接实现多区间富文本高亮。首版采用"证据卡片 + 字符位置 + 独立片段高亮"，不要承诺在原 textarea 内直接高亮；后续若需要可切换到 CodeMirror 等编辑器。

**前端数据流**：

```
用户点击"运行深度诊断"
→ POST /api/v1/style/diagnostics
→ 右栏展示高级诊断
→ 用户在文本样本下方选择目标作家
→ POST /api/v1/style/compare
→ 展示档案可信度和指标偏差
→ POST /api/v1/style/adjustments/plan
→ 展示规则建议；确定性 patch 可本地逐条应用
→ 用户勾选建议并点击"预览调整"
→ POST /api/v1/style/adjustments/preview
→ 展示 diff + 调整前后诊断对比
→ 用户点击"接受调整"
→ 更新本地 text，旧诊断/计划标记 stale，写入撤销栈
→ 以 300-500ms 防抖重新运行诊断，验证改善效果
```

任何文本编辑、重新导入或作家档案版本变化，都会使现有 comparison/plan/preview 失效。前端通过 `sourceHash` 统一判断，禁止把旧位置补丁应用到新文本。

### 最终验收标准

- 同一文本、同一 `ruleVersion` 的规则诊断结果可复现。
- 所有 finding 均有位置、证据、严重度和可信度，不只返回总分。
- "AI 风格"只作为启发式风险，不输出作者来源结论。
- 选定作家档案后能生成目标差异和调整计划；档案版本变化会阻止旧计划继续使用。
- LLM 调整必须先预览，接受后才更新文本，并能一键撤销。
- 调整后自动给出前后指标，但不以降低 AI 风险分数作为唯一优化目标。

## 八、风险与缓解

| 风险 | 阶段 | 缓解措施 |
|:----|:----:|----------|
| 规则误报/漏报 | 一 | 固定正反例集、按语言和样本长度分阈值、所有结论附 confidence 与证据 |
| 把动作重复误称为意图重复 | 一 | 结果分为 `action-expression` 与 `semantic-intent`，后者明确为代理指标 |
| 逗号多被误判为从句多 | 一 | 使用多指标复杂度分数；产品文案标为"句法复杂度（代理指标）" |
| AI 风格评分被当成检测器 | 一/四 | 改名为"启发式风险"，展示规则版本、样本充分度与免责声明 |
| 作家档案样本不足或偏科 | 二 | 利用 `sampleStats` 分级；不足时禁止默认改写，显示来源数和总字符数 |
| 改写造成事实、视角或人物关系变化 | 二 | 保真提示词 + 低温度 + 前后 diff + 人工接受；后续可接连续性审计 |
| 模型调用失败、费用或延迟过高 | 二/三 | 20,000 字符上限、选段改写、token usage 展示、规则功能可独立降级 |
| 旧建议应用到已编辑文本 | 二/四 | `sourceHash + expectedText + authorProfileVersion` 三重校验，冲突返回 409 |
| 前端面板过多导致 `StyleManager` 臃肿 | 四 | 保持顶层标签不变，拆分子组件和 reducer，高级面板默认折叠 |
| 调整应用后文本被破坏 | 四 | 预览优先、本地历史栈最多 20 步、接受后复检、支持立即撤销 |
| 过度追求目标作家导致机械模仿 | 二/四 | 只提供统计目标区间与高层特征，不默认向 LLM发送原始档案样文，不以匹配分数作为唯一目标 |
