## 三、项目可行性四维分析

### 3.1 技术可行性 — ⭐⭐⭐⭐

| 维度 | 判断 |
|------|------|
| **核心架构** | 成熟。47 Agent + 21 模型 + 62 工具模块，分层清晰（Agent → Model → Utils → Pipeline） |
| **类型安全** | TypeScript 严格模式，三包 composite 引用，类型体系整体完善 |
| **测试覆盖** | 1845 测试用例，100% 通过率，Core/Studio 使用 `pool: forks` 隔离 |
| **技术债务** | 可控。P0 是单行修复，P1 多为前端接入问题，无架构性技术债 |
| **外部依赖** | 仅 3 个核心运行时依赖（pi-ai, zod, hono），版本已钉死 |

**关键风险**：无 CI/CD，类型检查不会自动执行，导致 P0 可被合入主分支。

### 3.2 产品可行性 — ⭐⭐⭐⭐⭐

| 维度 | 判断 |
|------|------|
| **定位** | 从"自动写小说"转型为"证据驱动的写作工作台"，方向合理 |
| **差异化** | 长篇状态治理、作者意图注入、可追踪运行时、多模型协作、同人 Canon 一致性 |
| **用户价值** | 覆盖创作全流程：访谈 → 规划 → 写作 → 审计 → 修订 → 导出 |
| **竞争对手** | 明显区别于 Sudowrite（全自动）、NovelCrafter（重世界构建）、个人写作软件（无 AI） |
| **扩展性** | 六类创作管线（长篇/续写/同人/改写/短篇/风格适配）共享同一内核 |

**关键建议**：不应继续增加 Agent 数量，而应聚焦三个闭环（作者意图 → 正文 → 状态 → 诊断）。

### 3.3 流程可行性 — ⭐⭐

| 维度 | 判断 |
|------|------|
| **CI/CD** | ❌ 缺失。无法阻止编译错误或测试失败合入主分支 |
| **代码规范** | ESLint + Prettier 已配置，但无 `lint` 脚本、无 pre-commit hooks |
| **发布流程** | `prepack` + `verify-publish-manifests` 存在，但无自动发布 |
| **分支策略** | Gitee vs GitHub 双远端，历史不相关，需人工协调 |
| **锁文件治理** | pnpm + npm 锁文件共存，存在漂移 |

**关键建议**：建立最小 CI（typecheck → build → test），单日内可完成。

### 3.4 风险可行性 — ⭐⭐⭐

| 风险 | 等级 | 缓解措施 |
|------|:----:|----------|
| 新增功能无闭环检测 | **高** | 建立功能闭环验收模板 |
| 前后端契约漂移 | **中** | 共享 Zod Schema |
| 书籍数据与代码仓库混合 | **中** | 移出源码仓库，保留脱敏夹具 |
| 模型 API Key 依赖 | **中** | doctor 命令可检测 |
| 多远端同步冲突 | **低** | 统一使用单一远程 |

## 五、可行性结论

### 技术可行性：✅ 高

项目已具备坚实的技术基础，P0/P1 修复成本极低（**1-2 天**）。核心管线成熟，测试覆盖充足。最大技术风险不是代码质量，而是缺乏 CI/CD 导致的"已修复的问题被再次合入"。

### 产品可行性：✅ 高

从"自动写作"转向"证据驱动的工作台"是正确方向。三个闭环（意图 → 正文 → 状态 → 诊断）是对用户真实创作流程的合理抽象。当前 85% 的前端完成度足以支撑早期用户验证。

### 流程可行性：⚠️ 需加强

缺少 CI/CD 是最大单一风险。建议在 **1 天内** 完成以下最小流程治理：
1. 修复 P0 事件链导入
2. 删除敏感词、package-lock.json、双实现页面
3. 创建 `.github/workflows/ci.yml` 执行 `typecheck → build → test`
4. 增加"禁止废弃模块回归"的架构测试

### 综合判断：**可推进，但有条件**

```

4. 不继续横向增加 Agent → 至少 2 周

预计 R1（意图闭环）达成时间：1-2 周
```
