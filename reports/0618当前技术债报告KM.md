## 二、运行时效率风险端点

### 2.1 同步阻塞调用（🔴 高优先级）

| 文件 | 行号 | 调用 | 影响 |
|---|---|---|---|
| `core/src/pipeline/runner.ts` | 2225-2226 | `execSync("git rev-parse HEAD")` | 每章 Beta Reader shadow 持久化时阻塞事件循环 3s，长书写作时累积明显 |
| `studio/src/api/index.ts` | 20 | `execSync("npx vite build")` | Studio 启动/打包时完全阻塞 |

### 2.4 流读取器未释放（内存/连接泄漏风险）

`core/src/llm/provider.ts` 多处流式响应处理：

```typescript
const reader = response.body?.getReader();
try {
  while (true) { const { value, done } = await reader.read(); ... }
} finally {
  monitor.stop();
}
```

**缺失**：`reader.releaseLock()` / `reader.cancel()` 未在 `finally` 中调用。异常或提前退出时，HTTP body reader 可能长时间占用。

### 2.5 无上限缓存

`studio/src/api/routes/services.ts:25-28` 的 `modelListCache` 有 10 分钟 TTL 但无容量上限，服务很多时会持续增长。

## 四、安全与资源风险

### 4.1 API Key 明文存储（🔴 高优先级）

`core/src/llm/secrets.ts:51-62`：`saveSecrets` 将 API key 以明文写入 `.inkos/secrets.json`。

**建议**：使用系统 keychain，或至少做 obfuscation 并限制文件权限。

### 4.2 SSRF / 私有 IP 绕过

`studio/src/api/routes/project-config.ts:109-128`：webhook URL 仅校验 scheme 与 hostname 前缀，未做 DNS 解析后二次校验。

**建议**：解析后 `lookup` IP，再校验是否落在私有段（参考 `server.ts` 中 `assertSafeStyleImportTarget`）。

### 4.3 外部 URL 内容拉取

`studio/src/api/server.ts:300-339`：`readStyleImportBody` 拉取外部 URL 时无 content-type 校验，仅限制大小。

**建议**：校验 content-type 为文本类型，限制总时长与重定向次数。

### 4.4 路径遍历风险

`studio/src/api/server.ts:554-575`：`resolveExternalChatEditPath` 虽有路径校验，但 `parseReplacementInstruction` 的文本匹配可能误触敏感文件。

**建议**：替换前再次校验 resolved 路径，并避免基于自然语言的文件路由。

### 4.5 子 Agent 工具越权

`core/src/agent/agent-tools.ts:163-171, 185-198`：`sub_agent` 工具可直接调用 `pipeline.initBook` / `pipeline.reviseFoundation`。

**建议**：在工具层增加用户确认或 scope 校验（仅允许当前 active book）。
