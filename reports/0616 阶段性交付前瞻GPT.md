## 三、关键代码风险

### 3.1 前端调试残留

文件：

```text
packages/studio/src/pages/ChapterReader.tsx
```

位置：

```typescript
setTimeout(() => { if (editing || true) handleStyleAnalysis(); }, 600);
```

问题：

- `editing || true` 永远为真。
- 条件判断失效，替换文本后总会触发风格分析。
- 可能带来不必要的请求、计算、状态刷新和 UI 抖动。

建议：

- 去掉 `|| true`。
- 补一条行为测试，确认未进入编辑态或数据未就绪时不会触发多余分析。

### 3.2 Studio API 超大文件

文件：

```text
packages/studio/src/api/server.ts
```

规模：约 6994 行。

风险：

- API、业务编排、安全校验、服务探测、风格库、session、书籍操作集中在单文件。
- 回归定位成本高。
- 后续功能扩张容易造成耦合扩大。

建议拆分方向：

- `api/routes/books.ts`
- `api/routes/services.ts`
- `api/routes/style.ts`
- `api/routes/agent.ts`
- `api/routes/sessions.ts`
- `api/security.ts`
- `api/service-probe.ts`

### 3.3 Core PipelineRunner 超大类

文件：

```text
packages/core/src/pipeline/runner.ts
```

规模：约 3759 行。

风险：

- 写作、导入、审计、修订、状态持久化、通知等职责集中。
- 任何变更都可能触发大范围测试。
- Agent 管线未来扩展会继续推高复杂度。

建议拆分方向：

- chapter import service
- write chapter service
- revise service
- audit orchestration
- state persistence adapter
- notification adapter

### 3.4 前端包体偏大

构建警告显示：

- 主入口 chunk 约 2.76MB。
- 多个 Mermaid / Shiki / language chunk 偏重。
- `use-api.ts` 同时被静态和动态导入，动态 import 无法有效拆包。

建议：

- Markdown 渲染、Mermaid、代码高亮按页面或组件延迟加载。
- 避免对同一模块既静态导入又动态导入。
- 为 Studio 设定 bundle budget，例如主入口压缩后不超过 1.5MB。

### 3.5 测试门禁存在竞争风险

现象：

- 并行运行 build/test/typecheck 时，`dist` 目录被不同命令清理和重建，导致 Core/CLI 出现模块找不到。
- 顺序复验后相关测试通过。

判断：

- 不是业务功能缺陷。
- 是 CI/本地门禁脚本可靠性问题。

建议：

- CI 中禁止并行运行会清理 `dist` 的命令。
- 将 root `test` 保持为 `pnpm build && pnpm -r --workspace-concurrency=1 test`。
- 单包测试前显式构建依赖包，尤其是 CLI 前先构建 Core。
