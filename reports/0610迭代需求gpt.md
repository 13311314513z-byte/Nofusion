## 二、代码 Review 发现

### P0-1：Studio 质量门禁仍失败

#### 类型错误

`packages/studio/src/pages/style-types.ts` 自行定义了 `CoreStyleProfile`，但缺少 Core `StyleProfile` 的 `paragraphLengthRange`。该类型被传给 `AuthorProfileCard` 时失败。

**根因**

Core 与 Studio 分别维护同一业务结构，字段演进后未同步。

**迭代要求**

- 删除 Studio 内重复的完整 StyleProfile 定义。
- 类型直接复用 Core 的 browser-safe 类型，或在共享契约层定义 API DTO。
- API 响应使用 Zod schema 做运行时校验。
- 增加 `AuthorProfile` API 契约测试。

#### 测试加载错误

两个测试套件无法解析 `@actalk/inkos-core/browser`。当前 `vitest.config.ts` 同时配置了宽泛的 `@actalk/inkos-core` 和 browser 子路径别名，宽泛别名会影响子路径解析。

**迭代要求**

- 改为按精确匹配顺序声明 alias，browser 子路径优先。
- 测试不得依赖 Core 预先构建的 `dist` 才能启动。
- CI 必须独立执行一次干净工作区测试。

**验收**

```text
pnpm --filter @actalk/inkos-studio typecheck
pnpm --filter @actalk/inkos-studio test
pnpm typecheck
pnpm test
```

全部退出码为 0。

### P0-2：封面快捷配置可能清空已保存 API Key

后端 `GET /cover/secret/:service` 已只返回：

```json
{ "hasApiKey": true, "keyPreview": "sk-a...1234" }
```

但 `ServiceListPage.tsx` 仍按 `{ apiKey?: string }` 读取，并把不存在的 `apiKey` 设置为空。点击“保存封面配置”时，页面会把空字符串 PUT 回后端，可能清除已有 Key。

**迭代要求**

- 封面配置页和服务列表快捷卡统一使用 `ApiKeyInput`。
- 新增 `hasStoredKey` 状态，不再回显完整 Key。
- 保存配置时：
  - 未输入新 Key：只保存 service/model，保留旧 Key。
  - 输入新 Key：更新 Key。
  - 清除 Key：必须使用独立“删除密钥”操作并二次确认。
- 后端更新接口区分“字段省略”和“明确清除”，禁止用空字符串隐式删除。

**验收**

1. 已有 Key 时只切换模型，Key 指纹保持不变。
2. 切换服务商不会显示其他服务商 Key。
3. 留空保存不会清除 Key。
4. 只有点击“删除密钥”才能清除。

### P1-1：章节阅读器仍不能通过 F5 恢复

`HashRoute` 和 `HASH_PAGES` 包含 `chapter`，但 `parseHash()` 与 `routeToHash()` 没有章节路由规则。

**迭代要求**

- 章节 URL 统一为 `#/chapter/{bookId}/{chapterNumber}`。
- 增加中文、空格等编码测试。
- 增加点击章节、F5、前进、后退的浏览器测试。

### P1-2：建书等待时间与后端任务时限不一致

- 后端建书超时为 10 分钟。
- 前端 `waitForBookReady` 默认只轮询 `120 × 250ms`，约 30 秒。
- 30 秒后前端可能提示失败，但后端仍在继续创建。
- 后端响应含 `phase` 字段，但创建流程未持续写入真实 phase。
- 前端创建中的三个步骤全部显示完成图标，不能反映真实进度。

**迭代要求**

- 建书接口改为统一长任务协议，返回 `jobId`。
- 使用 SSE 推送真实阶段，而不是高频轮询。
- 支持取消、后台继续、失败重试和刷新恢复。
- 阶段至少包括：校验输入、创建配置、生成基础设定、写入来源、初始化工作台、完成。
- UI 只对已完成阶段显示完成图标。

### P1-3：前端请求没有统一超时、取消和竞态保护

`fetchJson` 直接调用 fetch，没有默认超时；`useApi` 在路由变化时不会中止旧请求。

**风险**

- 上游服务无响应时页面长期 loading。
- 快速切换书籍时，旧请求可能覆盖新页面数据。
- 长任务与普通查询无法采用不同超时策略。

**迭代要求**

- `fetchJson` 支持 `timeoutMs`、外部 `signal` 和错误码。
- `useApi` 每次请求创建 `AbortController`，卸载或参数变化时 abort。
- 默认策略：
  - 普通 GET：15 秒。
  - 本地分析：60 秒。
  - 服务连接探测：配置化 8-30 秒。
  - 超过 60 秒的任务改用 Job + SSE。

### P1-4：SSE 事件定义没有单一事实来源

后端广播包含 `thinking:*`、`tool:*`、`tool:update`，全局 `STUDIO_SSE_EVENTS` 未声明这些事件。会话专用监听器覆盖了部分事件，但全局和会话事件仍由多份字符串列表维护。

**迭代要求**

- 在共享 contracts 中定义 SSE 事件联合类型和 payload 映射。
- 后端 broadcast、全局 useSSE、会话 stream listener 均引用同一事件表。
- 增加自动契约测试，禁止后端新增事件但前端无声明。
- 事件增加 `jobId`、`bookId`、`sessionId`、`timestamp`、`sequence`。

### P1-5：Studio API 单文件过大

`server.ts` 集中 136 个路由和大量辅助函数，已经出现类型、事件和 secret 契约局部修改后其他页面未同步的问题。

**迭代要求**

按领域逐步拆分，不做一次性重写：

```text
api/routes/books.ts
api/routes/chapters.ts
api/routes/services.ts
api/routes/sessions.ts
api/routes/style.ts
api/routes/audit.ts
api/routes/import-export.ts
api/routes/system.ts
api/contracts/
api/services/
```

每次只迁移一个领域，保持现有 URL 不变。

### P2-1：文风分析语言仍有多处硬编码

`StyleManager`、`AITellsPanel` 和作家创建流程仍发送 `language: "zh"`。英文文本会进入中文分析规则。

**迭代要求**

- 默认使用项目语言。
- 增加轻量文本语言检测，允许用户手动覆盖。
- 作家档案语言跟随样本或由用户选择。
- 分析结果明确显示实际使用的语言。

### P2-2：已有组件和后端能力未形成前端闭环

- `RhetoricHighlightEditor` 无页面消费者。
- `useAutoSave` 无消费者。
- 作家蒸馏的生成、编辑、发布、版本列表接口无完整前端入口。

这类“后端已完成、用户不可见”的功能应优先于继续新增新端点。

### P2-3：首屏包体积过大

当前主 JS 约 2.7 MB，gzip 约 758 KB；`App.tsx` 同步导入全部页面，Shiki、Mermaid、Cytoscape 等重组件进入初始依赖图。

**迭代要求**

- 页面使用 `React.lazy` 和 `Suspense` 按路由加载。
- BookWorkspace 各分区按 section 懒加载。
- 代码高亮语言按需加载，不打包全部语言。
- Mermaid、关系图、文风管理等重模块仅在进入页面后加载。
- 增加 bundle size CI 门禁。

目标：主入口 gzip 小于 300 KB，单个异步业务 chunk 尽量小于 500 KB。

## 三、统一前后端功能契约

### 3.1 长任务 Job 契约

建书、写章、全书审计、全文检测、雷达扫描、作家蒸馏、批量导入均应使用统一结构：

```ts
interface JobState {
  jobId: string;
  type: string;
  status: "queued" | "running" | "completed" | "failed" | "cancelled";
  phase: string;
  progress: number | null;
  message?: string;
  bookId?: string;
  sessionId?: string;
  startedAt: string;
  updatedAt: string;
  error?: {
    code: string;
    message: string;
    retryable: boolean;
  };
  resultRef?: string;
}
```

建议端点：

- `POST /api/v1/jobs/{type}`：创建任务，返回 202。
- `GET /api/v1/jobs/:jobId`：查询状态。
- `POST /api/v1/jobs/:jobId/cancel`：取消。
- `POST /api/v1/jobs/:jobId/retry`：重试。
- SSE `job:update`：推送阶段变化。

### 3.2 Secret 更新契约

禁止再用空字符串同时表达“未修改”和“删除”：

```json
{
  "apiKey": "new-key"
}
```

表示更新；请求中不包含 `apiKey` 表示保留；删除使用独立 DELETE：

```text
DELETE /api/v1/services/:service/secret
DELETE /api/v1/cover/secret/:service
```

### 3.3 错误响应契约

统一为：

```json
{
  "error": {
    "code": "BOOK_CREATE_TIMEOUT",
    "message": "书籍创建超时",
    "retryable": true,
    "details": {},
    "traceId": "..."
  }
}
```

前端根据 `code` 做本地化和动作建议，不再解析任意字符串。

### 3.4 并发修改契约

章节、角色、目标、作家蒸馏和配置保存增加 `version` 或 ETag：

- 读取返回版本号。
- 更新携带 `expectedVersion`。
- 冲突返回 409，并提供重新加载或合并入口。

这比仅依赖最后一次写入更适合连续开发和多页面同时编辑。

### Phase 0：恢复质量门禁（1-2 天）

| 任务 | 优先级 | 交付 |
| --- | --- | --- |

### Phase 1：长任务与请求可靠性（2-3 天）

1. 定义共享 JobState 和 SSE 契约。
2. 首先迁移建书任务。
3. `fetchJson/useApi` 增加超时和 abort。
4. 增加全局任务中心，刷新页面后仍能查看正在运行任务。
5. 对写章、审计、检测和雷达保留兼容端点，再逐步迁移。

1. 作家蒸馏工作台。
2. 修辞高亮编辑器接入。

5. 可阅读性、诊断、修辞问题之间建立可点击跳转。

### Phase 3：性能、测试和架构治理（2-3 天）

1. 页面和工作台分区懒加载。
2. 拆分 API routes，先迁移 services/style。
3. 增加 Playwright 或等价浏览器 E2E。
4. 增加 bundle、类型、路由、SSE 和 secret 契约门禁。

建议周期：单人约 9-12 个开发日。

## 五、各模块扩展方案

### 5.1 Core / Pipeline

**当前基础**

多 Agent、审计历史、Runtime 产物、章节状态和锁机制已经存在。

**建议扩展**

- Pipeline 暴露标准阶段进度和取消信号。
- 每次执行生成 `runId`，关联日志、token、模型、审计和产物。
- 支持从最近安全阶段恢复，而不是长任务失败后全部重跑。
- 文件锁增加 owner nonce、释放时所有权校验和跨进程并发压力测试。
- 将 pipeline error 写入持久历史，补齐健康页当前的“无持久错误历史”。

**验收**

- 进程中断后能识别未完成任务。
- 两个进程并发写同一本书时只有一个获得锁。
- 任务取消后不再继续写入章节。

### 5.2 首页与全局导航

**建议扩展**

- 新增全局任务中心：建书、写章、审计、导入、雷达统一展示。
- 书籍卡片显示最近任务、上次失败原因和“继续处理”入口。
- 增加最近访问、最近编辑和待处理审计列表。
- 顶栏增加 SSE 连接状态、API 健康和后台任务数量。
- 增加命令面板，快速进入书籍、章节和模块。

不建议在首页继续堆叠静态统计卡片，应优先展示可执行事项。

### 5.3 新建书籍与导入

**建议扩展**

- 建书真实阶段进度、取消、后台继续和失败恢复。
- 创建前执行名称冲突、目录权限、服务商、模型能力和预计成本预检。
- 资料来源允许逐项选择用途，不再全部固定为 `auto`。
- 导入前生成章节、人物、地点、时间线和伏笔识别预览。
- 导入提交支持事务式回滚，失败时不得留下半初始化书籍。

### 5.4 书籍工作台与章节

**建议扩展**

- 章节版本历史：保存、审计前、重写前自动快照。
- 双栏 diff：原文、修改后、接受部分修改。
- 接入 `useAutoSave`，增加本地草稿和崩溃恢复。
- 批量审批、批量字数目标、批量导出选中章节。
- 章节详情增加关联角色、伏笔、地点、目标和审计问题。
- 对“删除、覆盖、重写”统一使用影响预览和确认。

版本历史应使用增量或保留策略，避免长篇项目无限增长。

### 5.5 聊天与会话

**当前判断**

会话统一端点 `/sessions?bookId=...` 已存在，旧报告中的 `/books/{id}/sessions` 不是当前真实契约。

**建议扩展**

- 消息发送失败支持“原请求重试”，而不是复制文本重发。
- Tool 调用显示输入摘要、影响文件和执行结果。
- 高风险工具执行前要求确认，并支持“仅本次允许”。
- 会话增加上下文范围选择：项目、书籍、章节、选中文本。
- 显示所用模型、token、耗时和降级情况。
- 支持会话搜索、固定、归档和导出。

### 5.6 文风分析与作家档案

这是下一轮最值得扩展的业务模块。

#### 作家蒸馏工作台

后端已有完整端点，应增加前端：

- 生成蒸馏草稿。
- 展示样本充分度和证据来源。
- 逐条启用、禁用和编辑规则。
- 标记档案版本变化导致的 stale。
- 发布不可变版本。
- 查看历史版本并比较差异。
- 将已发布版本应用到指定书籍。

#### 修辞与调整建议

- 接入 `RhetoricHighlightEditor`。
- 点击问题定位到原文位置。
- AI 改写先显示 diff，不直接覆盖。
- 支持接受单项、接受全部、撤销。
- 调整建议记录来源：规则、作家档案、AI 痕迹或可阅读性。

#### 语言与报告

- 自动语言识别和手动覆盖。
- 诊断结果导出 Markdown/JSON。
- 同一文本多次分析形成趋势，不只显示单次分数。

### 5.7 审计与 Analytics

现有趋势图已经可用，下一步应从“展示”转向“治理”：

- 点击趋势柱跳转到章节审计详情。
- 支持按审计类别、严重度、模型和时间筛选。
- 对比“修改前/修改后”分数变化。
- 对重复出现的问题生成治理建议。
- 增加成本视图：章节、Agent、模型的 token 与估算费用。
- 书籍健康页补充持久化 pipeline error 和最近失败任务。

### 5.8 角色、场景、伏笔、真相与目标

**角色**

- 关系图不应只从标签推断，应使用结构化 relationship 模型。
- 节点可跳转角色详情，边可编辑关系类型、强度和生效章节。
- 从章节对话生成“人物声线候选”，由用户确认后写入角色卡。

**场景与时间线**

- 增加结构化时间线：章节、日期、地点、参与角色、事件。
- 检测同一角色同时出现在冲突地点等时间线错误。

**伏笔**

- 当前已有风险和依赖显示，可增加“建议推进章节”和“一键加入章节目标”。
- 伏笔状态修改应记录历史，避免直接覆盖。

**真相文件**

- 增加引用关系：哪些章节、角色和伏笔依赖该事实。
- 修改核心事实前展示受影响范围。

**章节目标**

- 增加目标状态：计划、进行中、已满足、未满足、延期。
- 审计结束后自动建议目标完成度，但由用户确认。

### 5.9 模型配置与服务商

**必须先修**

- 封面 Key 空值覆盖。
- 所有 API Key 输入统一使用同一组件和相同 password manager 属性。

**扩展建议**

- 模型能力标签：文本、视觉、thinking、stream、上下文长度、输出上限。
- 服务商批量检测，并区分认证、额度、模型、网络和协议错误。
- Agent 模型覆盖增加“推荐用途”和冲突检查。
- 内置参数预设支持“另存为用户预设”、编辑、删除和导入导出。
- 增加调用成本预算和超限提醒。
- 保存配置前提供变更摘要，避免误改全局模型。

### 5.10 Runtime、Daemon、日志与诊断

**Runtime**

- 当前文件查看器可继续增强为阶段时间线。
- 展示 intent → plan → context → trace → memo → delta 的关联。
- 支持与章节正文、审计结果互相跳转。

**Daemon**

- 与全局任务中心合并展示。
- 增加下次执行时间、最近成功、连续失败次数和退避状态。
- 产品需明确侧栏是否公开入口，不再保持“路由可访问但入口隐藏”的模糊状态。

**日志**

- 日志级别、任务、书籍和时间过滤。
- 持久化轮转与历史查看。
- 敏感字段统一脱敏。
- 一键导出诊断包，但排除 Key 和正文内容。

**Doctor**

- 从“结果列表”升级为“问题 → 原因 → 修复动作”。
- 对端口占用、代理、证书、Base URL 和模型不可用给出独立诊断。

### 5.11 Radar、同人、导出与自动化

**Radar**

- 扫描结果允许转成选题卡、关键词和平台建议。
- 保存来源时间与可信度，避免历史结论长期复用。

**同人**

- 导入前提供角色/世界观冲突预览。
- 标记哪些原作资料会进入提示词。

**导出**

- 增加导出模板：投稿、连载、审阅、备份。
- 支持章节选择、目录、元数据和审计报告附录。
- 导出前检查从“阻止”与“提醒”两级区分。

**自动化**

- 统一 CLI JSON：`status/data/error/meta`。
- Webhook 使用与 SSE 相同的事件契约。
- 支持任务完成、失败、审计不通过和预算超限通知。

## 六、测试与验收体系

### 6.1 必增自动化测试

| 测试 | 覆盖内容 |
| --- | --- |
| Secret 契约测试 | GET 不返回完整 Key；留空保存不删除；显式删除有效 |
| 路由往返测试 | 每个 HashRoute 都满足 `parse(routeToHash(route))` |
| SSE 契约测试 | 后端事件全部存在于共享事件表 |
| API DTO 测试 | Core schema、API 响应、Studio 类型一致 |
| Job 状态测试 | queued → running → completed/failed/cancelled |
| 并发锁测试 | 多进程竞争和 stale lock |
| Bundle 门禁 | 主入口和重模块 chunk 不超预算 |

### 6.2 浏览器 E2E 最小集合

1. 首页加载无控制台异常。
2. 书籍总览、章节列表、阅读器之间导航和 F5 恢复。
3. 服务商 Key 保存、切换、留空保存和删除。
4. 建书任务刷新后恢复进度。
5. 文风分析、诊断、修辞定位和 diff 接受。
6. 作家蒸馏生成、编辑、发布和应用。
7. 审计趋势点击进入章节。
8. 中文、英文和深色主题基础流程。

### 6.3 发布门禁

正式发布前必须满足：

- Root typecheck、test、build 全绿。
- 当前源码前端浏览器烟雾测试通过。
- 不存在完整 secret 返回。
- 不存在已知的 Key 隐式删除路径。
- 所有动态页面支持可恢复 URL。
- P0/P1 缺陷为 0。

## 八、暂缓事项

本轮不建议投入：

- 多用户、账号、OAuth 和云端协作。
- 移动端完整适配。
- PostgreSQL 或分布式数据库替换。
- 完整插件市场。
- 新增更多独立 Agent。
- 重写现有 hash router。
- 重做全部 Studio UI。

原因：当前主要问题不是生成能力不足，而是已有能力的契约、进度、恢复、前端入口和测试门禁尚未完全闭环。

## 九、最终建议

下一轮目标应定义为：

> 把 NoFusion 从“功能很多但状态可能分离的开发工作台”，推进为“契约一致、任务可恢复、修改可回滚、已有 Core 能力可在前端完整使用的稳定创作工作台”。

优先顺序：

```text
质量门禁与 Key 安全
→ 长任务状态和请求可靠性
→ 作家蒸馏、修辞、章节版本闭环
→ 性能与浏览器 E2E
→ 各业务模块的深度扩展
```

完成 Phase 0-2 后，再评估正式发布；在此之前，不建议用新增大量页面或端点来替代稳定性建设。
