## 2. P0 — 阻塞级问题（必须先修复，否则功能不可用或数据不一致）

### P0-1: `style-fingerprint.ts` 中 `RHETORICAL_PATTERNS` 仍存在，导致 `rhetoricDensity` 与 `rhetoricalFeatures` 数据不一致

**文件位置：** `packages/core/src/agents/style-fingerprint.ts:93-99` + `:260-264`

**问题描述：**

`analyzeStyle` 已经统一调用 `detectDuplicateRhetoric()` 生成 `rhetoricalFeatures`（更完整的 10 类检测），但 `analyzeStyleFingerprint()` 仍然使用独立的 `RHETORICAL_PATTERNS`（只有 5 类粗略正则）计算 `rhetoricDensity`：

```typescript
// style-fingerprint.ts:93-99
const RHETORICAL_PATTERNS = [
  { name: "比喻", regex: /[像如仿佛似](?:是|同|一般|一样)|宛如|好似/g },
  { name: "排比", regex: /[，。；]([^，。；]{2,6})[，。；]\1/g },
  { name: "反问", regex: /难道|怎么可能|岂不是|何尝不|何况|怎么会/g },
  { name: "夸张", regex: /天崩地裂|惊天动地|翻天覆地|震耳欲聋|人山人海|无边无际/g },
  { name: "拟人", regex: /[风雨雪月花树草石山河水](?:在|像|仿佛|好似).*?(?:笑|哭|叹|呻|吟|怒|舞|睡|醒)/g },
];

// style-fingerprint.ts:260-264
let rhetoricCount = 0;
for (const { regex } of RHETORICAL_PATTERNS) {
  rhetoricCount += (text.match(regex) ?? []).length;
}
const rhetoricDensity = Math.min(Math.round((rhetoricCount / sentenceCount) * 100) / 100, 1);
```

**实际影响：**
- 同一文本：
  - `StyleProfile.rhetoricalFeatures`（来自 `detectDuplicateRhetoric`）显示"比喻(6处)、排比(3处)"
  - `StyleFingerprint.rhetoricDensity`（来自独立正则）可能只统计到 3 处比喻 + 1 处排比
- 用户困惑：卡片显示"修辞密度 15%"，但特征列表只有 3 条
- Pipeline 中 writer agent 收到的 `rhetoricDensity` 和 `rhetoricalFeatures` 不匹配

**修复方案：**

```typescript
// style-fingerprint.ts
import { detectDuplicateRhetoric } from "../utils/semantic-duplication.js";

function computeRhetoricDensity(text: string): number {
  const result = detectDuplicateRhetoric(text, "zh");
  const totalRhetoricCount = result.findings.reduce((sum, f) => sum + f.count, 0);
  const sentences = text.split(/[。！？\n]/).filter((s) => s.trim().length > 0);
  return sentences.length > 0
    ? Math.min(Math.round((totalRhetoricCount / sentences.length) * 100) / 100, 1)
    : 0;
}

// 在 analyzeStyleFingerprint 中：
// 移除 let rhetoricCount = 0; for (const { regex } of RHETORICAL_PATTERNS) { ... }
// 替换为：
const rhetoricDensity = computeRhetoricDensity(text);
```

**验证方法：** 同一段文本分别运行 `analyzeStyle` 和 `analyzeStyleFingerprint`，比较 `rhetoricalFeatures` 的总计数与 `rhetoricDensity` × 句子数，两者应一致。

### P0-2: `StyleFingerprint` 模型未扩展为 `ExpandedFingerprint`

**文件位置：** `packages/core/src/models/style-profile.ts:20-39`

**问题描述：**

`models/style-profile.ts` 中的 `StyleFingerprint` 仍是原始定义（9 个字段），而 `style-dimensions.ts` 中已定义了 `ExpandedFingerprint extends StyleFingerprint`（含 4 个新字段）：

```typescript
// models/style-profile.ts — 仍是旧的
export interface StyleFingerprint {
  readonly dialogueRatio: number;
  readonly actionDensity: number;
  // ... 共 9 个字段，没有 sentenceTypeDistribution/paragraphRhythm/rhetoricBreakdown/dialogueFeatures
}

// style-dimensions.ts — 新的
export interface ExpandedFingerprint extends StyleFingerprint {
  readonly sentenceTypeDistribution: SentenceTypeDistribution;
  readonly paragraphRhythm: ParagraphRhythm;
  readonly rhetoricBreakdown: RhetoricBreakdown;
  readonly dialogueFeatures: DialogueFeatures;
}
```

**实际影响：**
- `analyzeStyle` 返回的 `StyleProfile.fingerprint` 类型是 `StyleFingerprint`，不是 `ExpandedFingerprint`
- `mergeStyleProfiles` 处理的是 `StyleProfile`，其中的 `fingerprint` 字段类型决定了只能聚合旧字段
- 即使 `computeExpandedFingerprint` 计算出完整数据，由于类型不匹配，无法写入 `StyleProfile`
- `store.ts` 中创建档案时初始化的 fingerprint 也只能是旧字段

**修复方案（两步）：**

**步骤 A：** 将 `ExpandedFingerprint` 的定义从 `style-dimensions.ts` 提升到 `models/style-profile.ts`，并让 `StyleFingerprint` 继承它（或合并）：

```typescript
// models/style-profile.ts
import type {
  SentenceTypeDistribution,
  ParagraphRhythm,
  RhetoricBreakdown,
  DialogueFeatures,
} from "../utils/style-dimensions.js";

export interface StyleFingerprint {
  // === 现有字段 ===
  readonly dialogueRatio: number;
  readonly actionDensity: number;
  readonly psychologicalRatio: number;
  readonly sensoryDensity: number;
  readonly colloquialismScore: number;
  readonly rhetoricDensity: number;
  readonly punctuationRhythm: PunctuationRhythm;
  readonly aiTellRisk: number;
  readonly sensoryBreakdown: SensoryBreakdown;

  // === 新增字段（可选，保持向后兼容）===
  readonly sentenceTypeDistribution?: SentenceTypeDistribution;
  readonly paragraphRhythm?: ParagraphRhythm;
  readonly rhetoricBreakdown?: RhetoricBreakdown;
  readonly dialogueFeatures?: DialogueFeatures;
}
```

> **注意：** 设为可选字段（`?`）是为了保持向后兼容，否则已有档案（存储在 `style-library/authors/*/profile.json`）在加载时会因为缺少新字段而类型不匹配。在实际使用处用 `??` 提供默认值。

**步骤 B：** `style-dimensions.ts` 中的 `ExpandedFingerprint` 改为从 `models/style-profile.ts` 重新导出（或删除重复定义，统一使用 `StyleFingerprint`）：

```typescript
// style-dimensions.ts
export type ExpandedFingerprint = Required<StyleFingerprint>; // 或删除，统一使用 StyleFingerprint
```

### P0-4: `mergeStyleProfiles` 未聚合 `ExpandedFingerprint` 新字段

**文件位置：** `packages/core/src/style-library/aggregate.ts:75-116` + `:118-156`

**问题描述：**

`mergeStyleProfiles` 中对 fingerprint 的每个字段逐个做加权平均。当前代码只处理了 9 个旧字段（dialogueRatio, actionDensity, ..., sensoryBreakdown.visual/auditory/...）。如果 `StyleFingerprint` 扩展了新字段（P0-2 修复后），这些新字段在聚合时会被忽略，导致多来源档案的新字段值为 `undefined`。

**当前代码（aggregate.ts:129-152）：**

```typescript
fingerprint: {
  dialogueRatio: Math.round(weightedDialogueRatio * 100) / 100,
  // ... 9 个旧字段
  sensoryBreakdown: { ... },
  // ❌ 没有 sentenceTypeDistribution / paragraphRhythm / rhetoricBreakdown / dialogueFeatures
},
```

**修复方案：**

在 `mergeStyleProfiles` 中添加新字段的聚合逻辑。由于这些新字段结构较复杂，建议拆分为独立的辅助函数：

```typescript
function mergeSentenceTypeDistribution(
  profiles: ReadonlyArray<StyleProfile>,
  weights: ReadonlyArray<number>,
): SentenceTypeDistribution {
  const totalWeight = weights.reduce((a, b) => a + b, 0);
  if (totalWeight === 0) return { declarative: 0, interrogative: 0, exclamatory: 0, imperative: 0 };

  let decl = 0, inter = 0, excl = 0, imper = 0;
  for (let i = 0; i < profiles.length; i++) {
    const dist = profiles[i].fingerprint.sentenceTypeDistribution;
    const w = weights[i] / totalWeight;
    decl += (dist?.declarative ?? 0) * w;
    inter += (dist?.interrogative ?? 0) * w;
    excl += (dist?.exclamatory ?? 0) * w;
    imper += (dist?.imperative ?? 0) * w;
  }
  return {
    declarative: Math.round(decl * 100) / 100,
    interrogative: Math.round(inter * 100) / 100,
    exclamatory: Math.round(excl * 100) / 100,
    imperative: Math.round(imper * 100) / 100,
  };
}

// 类似地添加 mergeParagraphRhythm, mergeRhetoricBreakdown, mergeDialogueFeatures
```

然后在 `mergeStyleProfiles` 返回的 `fingerprint` 中包含这些新字段：

```typescript
return {
  // ... 现有字段 ...
  fingerprint: {
    // ... 现有 9 个字段 ...
    sentenceTypeDistribution: mergeSentenceTypeDistribution(profiles, weights),
    paragraphRhythm: mergeParagraphRhythm(profiles, weights),
    rhetoricBreakdown: mergeRhetoricBreakdown(profiles, weights),
    dialogueFeatures: mergeDialogueFeatures(profiles, weights),
  },
};
```

## 3. P1 — 重要问题（影响功能质量，不阻塞主线）

### P1-1: `fetchUrl` HTML 正文提取过于简单

**文件位置：** `packages/core/src/utils/web-search.ts:71-78`

**问题描述：**

```typescript
return text
  .replace(/<script[\s\S]*?<\/script>/gi, "")
  .replace(/<style[\s\S]*?<\/style>/gi, "")
  .replace(/<[^>]*>/g, " ")
  .replace(/\s+/g, " ")
  .trim()
  .slice(0, maxChars);
```

- 对 SPA（React/Vue 渲染）无效，可能抓取到 JS 代码而非正文
- 没有优先提取 `<main>` / `<article>` 等语义标签的内容
- 导航栏、页脚、广告等噪声没有被过滤

**修复方案：**

```typescript
// 方案 A：引入 cheerio（推荐，依赖小，~200KB）
import * as cheerio from "cheerio";

function extractArticleText(html: string): string {
  const $ = cheerio.load(html);
  const selectors = ["article", "main", "[role='main']", ".content", "#content", ".post-content", ".entry-content"];
  for (const sel of selectors) {
    const el = $(sel).first();
    if (el.length && el.text().length > 500) {
      return el.text().replace(/\s+/g, " ").trim();
    }
  }
  return $("body").text().replace(/\s+/g, " ").trim();
}

// 方案 B：不引入新依赖，基于段落密度的启发式算法
function extractByParagraphDensity(html: string): string {
  const withoutTags = html.replace(/<[^>]*>/g, "\n").replace(/\s+/g, " ").trim();
  const paragraphs = withoutTags.split(/\n+/).filter((p) => p.length > 30);
  // 取最长的连续段落块
  let bestBlock = "";
  let currentBlock = "";
  for (const p of paragraphs) {
    currentBlock += p + "\n";
    if (currentBlock.length > bestBlock.length) bestBlock = currentBlock;
  }
  return bestBlock;
}
```

### P2-3: `epistrophe` 正则的准确性待验证

**文件位置：** `packages/core/src/utils/semantic-duplication.ts:229-231`

**问题描述：**

```typescript
patterns: [
  /([\u4e00-\u9fff]{2,4})\s*$\n\n[\s\S]*?\n\n[\s\S]{0,100}\1\s*$/gm,
],
```

该正则试图匹配"连续两段以相同 2-4 字结尾"，但实现较复杂，可能存在以下问题：
- `$` 在 `m` 模式下匹配行尾，但中文段落可能以标点结尾，末尾的标点会影响匹配
- `\s*$` 会跳过尾部空格，但中文段落通常没有尾部空格
- 跨段落匹配的 `\n\n` 假设段落间隔为空行，但实际文本可能使用不同格式

**修复方案：** 需要实际测试该正则的召回率和准确率。建议准备 5-10 个包含尾语重复的测试文本，运行检测并人工验证。

## 8. 风险与注意事项

### 8.1 向后兼容性

`StyleFingerprint` 扩展新字段时**必须设为可选（`?`）**，否则：
- 已有 `style-library/authors/*/profile.json` 文件在加载时会因为缺少新字段而解析失败
- `store.ts` 中 `readJsonSafe` 的 `JSON.parse` 虽然不会报错，但后续代码访问 `fingerprint.sentenceTypeDistribution` 时会返回 `undefined`

### 8.2 `analyzeStyle` 的性能影响

`computeExpandedFingerprint` 内部调用 `detectDuplicateRhetoric`，而 `analyzeStyle` 原本只调用 `analyzeStyleFingerprint`（纯文本统计）。修改后：
- 单次 `analyzeStyle` 调用增加了一次 `detectDuplicateRhetoric`（已有调用）+ 多次正则扫描
- 实际上 `analyzeStyle` 已经在第 66 行调用了 `detectDuplicateRhetoric` 生成 `rhetoricalFeatures`，所以 `computeExpandedFingerprint` 内部的 `detectDuplicateRhetoric` 是**重复调用**

**优化建议：** 将 `detectDuplicateRhetoric` 的结果缓存并复用：

```typescript
export function analyzeStyle(text: string, sourceName?: string): StyleProfile {
  // ...
  const rhetoricResult = detectDuplicateRhetoric(text, "zh");  // 已有
  const rhetoricalFeatures = rhetoricResult.findings
    .filter((f) => f.count >= 2)
    .map((f) => `${f.label}(${f.count}处)`);

  const baseFingerprint = analyzeStyleFingerprint(text);
  // 传入已有的 rhetoricResult 避免重复检测
  const fingerprint = computeExpandedFingerprint(text, baseFingerprint, rhetoricResult);
  // ...
}
```

> 需要修改 `computeExpandedFingerprint` 的签名，接受可选的 `rhetoricResult` 参数。

### 8.3 前端组件与现有 UI 的冲突

`StyleManager` 当前 tabs 较多（text/files/library/apply/detection），新增 "rhetoric" 和 "author-search" 后可能 UI 拥挤。建议：
- 将 "detection" 和 "rhetoric" 合并为一个 Tab（如「检测与去重」）
- 将 "author-search" 作为「library」Tab 下的子功能，而非独立 Tab

### 8.4 `fetchUrl` 引入 cheerio 的依赖影响

cheerio 约 200KB，对启动时间影响较小。若不想引入新依赖，可使用方案 B（启发式段落密度算法），但准确率较低。
