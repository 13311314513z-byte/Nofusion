### P1-5：`isProcessAlive` 在 Windows 上的边缘情况 → ❌ **修复错误**

**文件**：`packages/core/src/state/manager.ts:176-183**

```typescript
private isProcessAlive(pid: number): boolean {
  try {
    process.kill(pid, 0);
    return true;
  } catch (error) {
    const code = (error as NodeJS.ErrnoException | undefined)?.code;
    if (code === "ESRCH" || code === "EACCES") {
      return false;
    }
    return true;
  }
}
```

**验证结果**：
- ⚠️ 原代码：只有 `ESRCH` 返回 false（进程不存在），其他错误（含 `EACCES`）返回 true（视为存活）
- ❌ **当前代码**：将 `EACCES` 也视为"进程不存在"

**问题**：`EACCES` 在 Windows 上表示"进程存在但无权限探测"。当前改动将 `EACCES` 等同于 `ESRCH`，可能导致：
- 实际存活的进程的锁被错误地判定为陈旧
- 锁文件被删除后，存活进程仍然认为自己持有锁
- 两个进程同时写同一本书的竞争条件

**修复建议**：应改为：

```typescript
if (code === "ESRCH") return false;  // 确知进程不存在
if (code === "EACCES") return true;   // 无法探测但可能存活，保守处理
return true;                          // 其他错误，保守处理
```

**严重性**：**P0 级回归** —— 可能在高并发或 Windows 环境下导致数据竞争。
