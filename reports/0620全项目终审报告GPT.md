## 3. 当前发布阻断项

### P0-1：CLI typecheck / build 失败

位置：`packages/cli/src/commands/update.ts`

当前代码：

```ts
import { exec } from "node:child_process/promises";
```

实测错误：

```text
src/commands/update.ts(2,22): error TS2307:
Cannot find module 'node:child_process/promises' or its corresponding type declarations.
```

影响：

- `pnpm typecheck` 失败。
- `pnpm build` 失败。
- 根 `pnpm test` 和 `test:ci` 不能作为发布门禁通过。
- CLI 包当前不可按正常发布路径验收。

判断：这是当前最明确的发布阻断。修复难度不高，但在修复前不能写“全绿”。

建议：

- 改为 `import { exec } from "node:child_process"; import { promisify } from "node:util"; const execAsync = promisify(exec);`
- 注意 `exec("npm install -g ...", { stdio: "inherit" })` 这一用法本身也不适合 `exec`；若要继承 stdio，应考虑 `spawn`。
- 为 `inkos update` 补一条最小测试或至少 typecheck 回归。

### P0-2：Studio bundle budget 失败

实测：

```text
JS:   14696.8 KB / 18000 KB
CSS:  158.4 KB / 300 KB
Max:  1616.6 KB / 1700 KB
Dist: 15282.8 KB / 15000 KB
```

失败项：

```text
Total dist 15282.8KB exceeds limit 15000.0KB
```

主要体积热点：

| 文件 | 大小 |
|---|---:|
| `vendor-streamdown--o991HNQ.js` | 1578.8KB |

Vite 仍提示：

- `use-api.ts` 同时被动态导入和大量静态导入，动态拆包无法生效。
- 有 chunk 超过 1600KB。

判断：此前“bundle CI 已闭合”的说法当前不成立。这个问题不一定影响本地启动，但影响 release gate，也说明代码分割仍未真正稳定。

建议：

- 先把预算失败当成 P0 发布阻断，而不是调高阈值。
- 继续压缩 Streamdown/Shiki/Mermaid/code language packs 的实际打包面。
- 处理 `use-api.ts` 动静态混用；当前拆包收益被抵消。

## 6. 输出品质风险

### 6.1 真实输出质量没有被当前门禁证明

当前自动化测试可以证明许多函数和契约没有明显回归，但不能证明：

- 连续章节写作的情节一致性。
- 长篇 hook 的种植、推进和回收质量。
- 审校问题是否真正改善文本，而不是只生成报告。
- 风格蒸馏是否稳定影响后续写作。
- agent thinking/tool 调用是否提升最终输出，而不是只提供过程感。

这类项目不能只用“测试数 1861”证明输出品质。更合适的质量门禁应包含固定 prompt 的真实样章回归、人工评分和差异记录。

### 6.2 LLM 解析加固有痕迹，但仍需真实样本覆盖

报告中提到 LLM 解析增强、Set 去重、MAX_LOOPS 等修复，本轮能在代码中看到相关痕迹。但这些主要降低崩溃风险，不等价于提升输出质量。

建议把“稳定性修复”和“文学输出品质”分开记账。

## 8. 建议处理顺序

### 第一优先级：恢复发布门禁

1. 修复 `packages/cli/src/commands/update.ts` 的 `node:child_process/promises` 导入问题。
2. 重新跑 `pnpm typecheck`。
3. 重新跑 `pnpm build`。
4. 修复 Studio bundle budget 超线，不建议直接调高阈值。
5. 重新跑 `pnpm --filter @actalk/inkos-studio build:budget`。

### 第二优先级：纠正“有代码痕迹但未接入”的修复

1. 接入或删除 `dedupGetRequest` / `releaseDedupGetRequest`。
2. 补齐长任务 timeout 覆盖，尤其 import、distillation、doctor、provider test/models。
3. 对 `StateManager` 复用重新定义口径：CLI 命令级实例化可接受，Studio server 生命周期内应尽量统一。

### 第三优先级：继续架构还债

1. 拆分 `server.ts` 剩余 46 个 inline API。
2. 拆分 `routes/style.ts`。
3. 拆分 `runner.ts`，但要保留 `PipelineRunner` 作为兼容门面。
4. 将 route manifest / OpenAPI drift check 纳入 CI。

### 第四优先级：建立输出品质门禁

1. 真实 LLM E2E 保持 opt-in，但新增连续多章样本。
2. 固定 3-5 个 prompt，记录章节连贯性、角色状态、hook 推进、审校修订效果。
3. 把“能跑通”和“写得好”拆成两个指标。

## 9. 最终判断

`0618技术债还债重构手册.md` 仍可作为后续重构路线图，但其中一些完成度判断需要按当前代码重新校准。`0620全项目终审报告.md` 的最终结论过于乐观，本轮实测不能支持“全绿、0 虚报、终审通过”。

当前项目更准确的阶段判断是：

> **修复积累较多，但当前 HEAD 未通过发布门禁；仍处于终审前整改阶段。**

如果要重新发一版可信终审，最低条件是：

1. `pnpm typecheck` 通过。
2. `pnpm build` 通过。
3. Studio `build:budget` 通过。
4. `pnpm lint` 仍保持 0 errors，并记录 warning 数。
5. 至少跑一次不含真实 LLM 的全量测试。
6. 对真实 LLM E2E 给出明确执行或未执行说明，不用它替代输出品质评审。

在这些条件满足前，建议停止使用“终审通过”措辞，改为“终审复盘发现阻断项，待整改后复审”。
