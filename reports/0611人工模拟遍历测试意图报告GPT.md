## 一、结论摘要

当前项目已经具备较完整的工作台外形、书籍数据读取能力和多数核心页面，书籍总览与章节正文不一致的旧问题在测试书籍上未再复现。17 个主要入口和书籍工作区 14 个分区均可打开，测试书籍能够正确显示 5 章、正文、审计统计、伏笔及 Runtime 产物。

但当前版本仍不适合标记为“完整可交付”：

1. 根级 `typecheck` 被 CLI 编译错误阻断。
2. Studio 生产构建缺少 `dist/api/index.js`，发布产物校验失败。
3. 文风模块第 4 步实际点击会导致整页空白。
4. 章节“文风诊断”调用不存在的后端路径，错误被静默吞掉。
5. 章节阅读页不支持 URL 直达和刷新恢复。
6. 文风删除、合并处理存在 1 基行号直接当数组下标使用的风险。
7. 作家蒸馏、章节版本、角色声线漂移等功能仍处于后端局部存在或前端入口未闭环状态。

**综合预期完成度：约 69%。**

- 页面与功能入口可见完成度：约 79%。
- 前后端闭环完成度：约 69%。
- 可发布、可回归、可稳定交付完成度：约 52%。

建议当前阶段定义为：**功能型 Alpha 后期，接近 Beta 门槛，但尚未通过发布门禁。**

### 3.1 CLI 编译阻断

`packages/cli/src/commands/doctor.ts:329` 向 `chatCompletion` 选项传入 `signal`，但当前函数类型不接受该字段：

```text
TS2353: 'signal' does not exist in chatCompletion options type
```

影响：

- 根级类型检查失败。
- CLI 的 `pretest`/构建阶段失败。
- 当前不能用自动化测试证明 CLI 修改未回归。

### 3.2 Studio 发布构建阻断

`packages/studio/package.json` 的 `build:server` 删除的是：

```text
dist/api/tsconfig.tsbuildinfo
```

实际增量缓存位于：

```text
packages/studio/tsconfig.server.tsbuildinfo
```

因此服务端 TypeScript 可能判断无需重新输出，最终 `build:verify` 报：

```text
Missing: dist/api/index.js
```

该脚本还使用了 Windows 特定的 `2>nul &`，会降低跨平台可重复性。

### 4.3.1 已复现崩溃

复现步骤：

1. 进入“文风”。
2. 在文本样本中输入重复段落。
3. 点击“分析”。
4. 点击“4. 修辞去重”。
5. 点击“开始检测”。

实际结果：

```text
TypeError: Cannot read properties of undefined (reading 'filter')
at StyleManager
```

页面根节点长度变为 0，界面完全空白。

直接原因：

- 前端请求 `/style/rhetoric/rewrite` 并声明响应为 `{ findings }`。
- 后端该路由的职责是构建改写提示词，实际只返回 `{ prompt }`。
- 前端把 `undefined` 写入 `rhetoricFindings`，渲染时执行 `.filter()` 导致崩溃。

## 五、前后端未对齐问题清单

## P0：必须先修复

### P0-1 CLI 类型错误阻断测试与发布

- 位置：`packages/cli/src/commands/doctor.ts:329`
- 问题：`chatCompletion` 选项类型不接受 `signal`。
- 建议：在核心客户端选项中正式支持 `AbortSignal`，或在 doctor 中使用已支持的超时封装。
- 验收：根级 `pnpm typecheck` 通过，CLI 测试实际执行并全绿。

### P0-2 Studio 服务端构建产物缺失

- 位置：`packages/studio/package.json`
- 问题：清理错误的 tsbuildinfo 路径，导致服务端输出缺失。
- 建议：使用 Node 脚本跨平台删除 `tsconfig.server.tsbuildinfo`，或为生产构建关闭增量缓存。
- 验收：`dist/index.html`、`dist/api/index.js`、`dist/api/index.d.ts` 同时存在。

### P0-3 文风修辞去重导致整页崩溃

- 前端：`StyleManager.tsx` 读取 `rhetoricResult.value.findings`。
- 后端：`POST /style/rhetoric/rewrite` 只返回 `prompt`。
- 建议方案：
  1. 新增 `/style/rhetoric/detect`，固定返回 `{ findings }`。
  2. `/style/rhetoric/rewrite` 只接受 findings 并返回 `{ prompt }`。
  3. 前端对所有数组响应执行 `Array.isArray` 防御。
  4. 增加页面级 Error Boundary，单个面板错误不能清空工作台。
- 验收：重复文本检测有结果或空态，页面不崩溃，控制台无未捕获异常。

## P1：下一轮核心闭环

### P1-1 章节文风诊断请求 404

- 前端请求：`GET /books/:bookId/chapters/:chapterNumber/content`
- 后端实际：`GET /books/:bookId/chapters/:chapterNumber`
- 现象：404 被 catch 忽略，随后仍跳转文风页，用户看到空文本但没有错误提示。
- 建议：统一使用已有章节接口，并在失败时阻止跳转、显示可操作错误。

### P1-2 章节深链接无法恢复

`HashRoute` 声明了 `chapter`，但 `parseHash`、`routeToHash` 和 `HASH_PAGES` 都未完整处理。直接打开或刷新：

```text
#/chapter/test-book-0609/1
```

会显示首页。

同时 `truth` 也存在类型声明与路由序列化不完整的问题；书籍工作区真相层可用，但旧独立路由处于悬空状态。

### P1-3 文风删除/合并可能错改文本

核心检测返回的 `lineNumber` 为 1 基；前端直接使用 `lines[lineNumber]`。至少应转换为 `lineNumber - 1`。此外段落可能跨多行，不能仅按单行数组删除。

建议：

- 检测结果返回 `startOffset/endOffset` 或 `startLine/endLine`。
- 所有修改基于稳定范围，从后向前应用。
- 应用前显示 diff，支持撤销。

### P1-4 封面配置存在两套密钥契约

- 独立封面页仍读取明文式 `{ apiKey }`。
- 服务商列表快捷配置已改为 `{ hasApiKey, keyPreview }`。
- 建议删除重复状态逻辑，复用同一 Hook 和同一组件。

### P1-5 “一键检测全部”不执行连接测试

当前按钮主要刷新服务列表并获取模型列表，没有逐一调用 `/services/:service/test`。建议改名为“刷新全部模型”，或实现有并发上限、单项超时和结果摘要的真实批量连接测试。

### P1-6 章节缺少版本安全网

章节元数据有 `revisionCount`，但正文没有版本快照、比较和恢复接口。重写、重同步、同人刷新等操作一旦结果不佳，用户缺少可靠回退路径。

建议优先建立：

- 保存前自动快照。
- 版本列表。
- 双栏或行级差异。
- 恢复为新版本，而非覆盖历史。

### P1-7 作家蒸馏未形成前端闭环

后端已有：

- 作家 CRUD、来源追加、重新分析。
- 作家网络搜索、抓取、样本写入。
- 蒸馏生成、读取 current、更新、发布、版本列表。

但当前缺少：

- 公开可用的蒸馏工作台。
- 规则逐条启停、证据查看、发布前复核。
- 指定历史版本详情。
- 删除当前草稿或废弃草稿。
- `AuthorSearchPanel`、`AuthorProfileCard` 已存在但未挂载到实际页面。

## P2：质量和体验优化

1. 文风请求中的语言应来自当前语言/样本设置，不应固定为 `zh`。
2. 密钥输入框应统一 `autocomplete`、字段命名和表单包裹策略，降低 Google 密码管理器干扰。
3. 服务商测试应区分 DNS、连接、认证、额度、模型不存在和超时。
4. 大型客户端包应按页面懒加载；当前主包约 2.72 MB。
5. 修复乱码测试数据，并验证文件、JSON、HTTP 全链路 UTF-8。
6. `withPipeline` 的 `_ttlMs` 参数未生效，应实现真正超时或删除误导参数。
7. `dispose()` 失败不应覆盖原始业务异常，应单独捕获并记录。
8. 来源页应补充新增来源、用途编辑、重新导入和引用范围查看。

## 九、新一轮验收清单

下一轮不建议继续以“页面能打开”作为主要验收标准，应按以下任务链验收：

1. 新建书籍 -> 写入章节 -> 保存 -> 刷新 -> 数据仍在。
2. 章节列表 -> 文风诊断 -> 自动带入正文 -> 产生诊断 -> 返回章节。
3. 文风重复检测 -> 高亮 -> 删除/合并 -> diff -> 撤销。
4. 服务商独立保存 Key -> 切换另一服务商 -> 原 Key 状态不变。
5. 批量测试服务商 -> 每项显示连接、认证、模型和额度结果。
6. 重写章节 -> 自动生成版本 -> 比较 -> 恢复。
7. 作家样本 -> 档案 -> 蒸馏草稿 -> 规则复核 -> 发布 -> 应用到文本。
8. `pnpm typecheck` -> 全包测试 -> Studio build -> 产物启动。

只有上述 8 条全部形成自动化或可重复人工证据，项目才适合进入 Beta 验收。

## 十、最终判断

当前项目不是“仅有页面的原型”，核心和书籍工作区已经具备真实数据能力，Core 与 Studio 自动化测试也有较好基础。主要问题已经从“功能完全缺失”转为“跨模块契约、修改安全、异常恢复和发布工程未收口”。

下一轮应停止继续横向增加入口，先集中关闭 P0 和 P1：

1. 让类型检查、CLI 测试和生产构建重新全绿。
2. 修复文风崩溃及章节到文风的断链。
3. 为自动改写和章节正文增加版本、差异和撤销。
4. 统一密钥、路由、语言和作家蒸馏的数据契约。

完成这些工作后，项目预计可从当前 **69%** 提升到约 **80%-84%**，并具备进入系统化 Beta 测试的条件。
