## 八、需架构重构的 12 项——逐项理由

> 以下 12 项被标记为「需架构重构」，每一项均给出**具体理由**说明为何不能作为独立修复处理。

### 1. P1-3: agent 端点内联 ~800 行

**为何需要架构重构**: agent 端点是 server.ts 中最复杂的单端点（~800 行），涉及 LLM 工具调用、thinking block、session 管理、chapter edit 等 7 个子系统。提取为独立路由需：
- 拆分 agent 逻辑与 server.ts 的闭包依赖（`buildPipelineConfig`、`broadcast`、`state` 等）
- 搬迁 5 个辅助函数（`tryHandleExternalChatEdit`、`validateAgentActionExecution` 等）
- 新增 agent 路由模块的契约测试
- **不可独立修复原因**: 提取后的模块必须通过完整的 agent 交互测试，而当前 agent 测试依赖 server.ts 内联状态，单独提取会造成回归风险。

### 2. P1-4: style-score 逐章计算无缓存

**为何需要架构重构**: style-score 是前端驱动的逐章计算，缓存策略需要：
- 决定缓存粒度（按章？按书？按 session？）
- 与 StyleManager 的状态管理对齐
- 处理缓存失效（章节内容修改后需重新计算）
- **不可独立修复原因**: 单纯加 Map 缓存会引入缓存一致性问题——章节被 revise 或外部编辑后缓存不失效，返回过期分数。

### 3. P2-1: server.ts ~40 内联路由未提取

**为何需要架构重构**: 40 个路由分属 8 个业务域（write/draft/revise/export/roles/fanfic/radar/doctor），提取需要：
- 为每个域创建独立路由模块
- 解决 server.ts 闭包依赖（`state`、`buildPipelineConfig`、`broadcast`、`loadRawConfig` 等共享 12+ 个变量）
- 为每个新模块补契约测试
- **不可独立修复原因**: 单次提取一个域（如只提取 write）会导致路由注册顺序变化，可能破坏中间件链。必须整体规划分批提取 + 全量回归测试。

### 4. P2-2: runner.ts 4,050 行单体

**为何需要架构重构**: 24 public + 30 private 方法，覆盖 draft/audit/revise/import/fanfic/export/foundation 7 个业务域。拆分需要：
- 提取共享依赖（state/config/agentCtx/logger）为 context 对象
- 解决 54 个方法间的交叉调用
- 维护向后兼容（PipelineRunner 被 server.ts、CLI、测试 3 方使用）
- **不可独立修复原因**: 拆分一个方法（如 `writeDraft`）会导致其依赖的 8 个 private 方法也需要暴露/搬迁，连锁修改超过 500 行。

### 5. P2-3: `await import()` → 静态导入（20+ 处）

**为何需要架构重构**: 20+ 处动态 import 分布在不同路由处理器中，包括 `node:fs/promises`、`node:path`、`@actalk/inkos-core` 等。改为静态导入需要：
- 确保所有模块在 server 启动时即可用（非 lazy）
- 评估对冷启动时间的影响（当前动态 import 延迟加载未使用的模块）
- **不可独立修复原因**: 部分动态 import 是刻意为之——避免 server.ts 启动时加载全部依赖。统一静态化可能增加冷启动 30%+ 时间。

### 6. P2-4: use-api.ts Vite 双模式导入告警

**为何需要架构重构**: Vite 告警源于同一模块同时被 client 和 server 引用时产生的 chunk 策略冲突。修复需要：
- 分离 client-only 和 shared 的 API 工具函数
- 调整 Vite 的 `manualChunks` 策略
- **不可独立修复原因**: 单纯去掉告警（如添加 `@ts-ignore`）会掩盖真正的 bundle 问题。需要整体审计哪些模块是 shared、哪些是 client-only。

### 7. P2-5: StyleManager ~1,900 行组件

**为何需要架构重构**: StyleManager 是 Studio 中最复杂的页面组件，包含 7 个 Tab（分析/对比/蒸馏/导入/排重/可读性/调整）。拆分需要：
- 提取每个 Tab 为独立组件
- 设计共享状态管理（当前通过父组件 state 传递）
- **不可独立修复原因**: 单拆一个 Tab 会导致父组件接口不稳定，需要整体设计状态提升/context 方案后再拆分。

### 8. P2-7: Pipeline 无实例池

**为何需要架构重构**: 当前每次写作创建新的 PipelineRunner，无复用。实例池需要：
- 设计池的获取/归还/销毁生命周期
- 处理池中实例的状态污染（上一次写作的 state/config 残留）
- 与 `withPipeline` 的 dispose 语义兼容
- **不可独立修复原因**: 实例池的并发安全性需要整体设计——多个 write-next 请求共享同一个 PipelineRunner 会导致 state/config 竞态。

### 9. P2-14: `any` / `as any` 约 242 处

**为何需要架构重构**: 242 处分布在 3 个核心模块（provider.ts、agent-tools.ts、agent-session.ts），涉及 LLM 响应解析、工具调用结果、session 序列化等。收敛需要：
- 为每个 any 定义精确的 Zod/TypeScript 类型
- 确保 LLM provider 的响应格式与类型定义一致
- 逐模块迁移，每步都需要测试覆盖
- **不可独立修复原因**: 随意改一个 `any` 可能导致 LLM 响应解析失败——上游 API 返回的字段可能不完全匹配类型定义。

### 10. P2-16: 遗留兼容代码堆积

**为何需要架构重构**: architect.ts、audit-issue.ts、book-rules.ts 中的 legacy 回退逻辑用于兼容 Phase 4（flat file）和 Phase 5（outline 目录）两种布局。清理需要：
- 确定迁移窗口期（所有用户已升级到 Phase 5）
- 删除 flat file 读写路径
- 更新所有引用到 outline 路径
- **不可独立修复原因**: 存在 Phase 4 用户数据时删除兼容代码会导致数据丢失。必须先确保无遗留数据。

### 11. P2-17: 巨型测试文件（pipeline-runner.test.ts 5,220 行 / server.test.ts 3,810 行）

**为何需要架构重构**: 拆分测试文件需要：
- 按测试域重新组织（audit 测试、write 测试、import 测试等）
- 确保测试间的隔离性（setup/teardown 不交叉影响）
- 维护测试覆盖率不下降
- **不可独立修复原因**: 巨型测试文件中存在隐式的测试顺序依赖，简单切分会导致部分测试因 setup 缺失而失败。

### 12. P2-18: 条件跳过测试

**为何需要架构重构**: memory-retrieval、pipeline-runner、e2e 依赖环境变量跳过。修复需要：
- 为每个跳过测试提供 mock/fixture 替代真实依赖
- e2e 测试需要提供 sandbox LLM endpoint
- **不可独立修复原因**: 强制启用这些测试会导致 CI 中因缺少 API key 或环境而失败。需要搭建 mock infrastructure。
