### 0.0.3 约束编译：红 / 黄 / 蓝

|------|------|------|

| 🔴 红级 | [COMMON][HIGH] 不允许泄露真实 API Key、通知 Webhook、私有文件内容。 | 证据中密钥必须脱敏，只保留前后 4 位或 `hasApiKey/keyPreview`。 |
| 🔴 红级 | [COMPUTED][HIGH] 不允许破坏真实工作区数据。 | 删除、批量创建、压力测试必须在临时项目执行；真实项目只做只读或备份后执行。 |

| 🔵 蓝级 | [FRAME][HIGH] 测试记录应短、硬、可复验。 | 每条证据优先写命令、状态码、产物路径、截图名，不写泛泛感受。 |

| 状态 | 使用条件 |
|------|------|
| ✅ PASS-1 | [COMPUTED][HIGH] 用例一次执行通过，证据完整，未触发补测。 |
| ✅ PASS-2 | [COMPUTED][HIGH] 首次失败或信息不足，经补测、修正环境或更换数据后通过；必须记录校准摘要。 |

## 测试执行规范
## ═══════════════════════════════════════

### 0.1 执行原则

| 规范项 | 要求 |
|------|------|
| **诚实记录** | 不允许用“看起来可用”“理论可行”替代实测。每个用例必须记录 Pass/Fail/Blocked，并附至少 1 条证据。 |
| **代码优先** | 若手册中的接口、页面名、命令与当前代码不一致，以当前代码为准，并在备注中记录“手册待修正”。本版已按 `packages/studio/src/api/routes/*`、`packages/cli/src/program.ts` 做第一轮校准。 |
| **隔离执行** | P0/P1 用例建议在新建测试项目目录执行，避免污染真实书籍与长期配置。需要写入真实 `inkos.json`、`.env`、`books/` 时必须先备份。 |
| **可复现** | 每个 Fail 必须能复现：记录命令、URL、请求体、响应码、关键日志、浏览器 Console 截图或终端输出摘要。 |
| **LLM 不确定性** | 内容质量类用例不以单次文学偏好判定，通过结构完整性、状态流转、审计字段、错误恢复和可人工复核产物判定。 |
| **安全边界** | 涉及密钥、通知、文件服务、导入 URL、路径参数的用例必须额外核验脱敏、目录穿越、非法字符和错误提示。 |

### 0.2 Pass / Fail / Blocked 判定

| 状态 | 判定标准 | 记录要求 |
|------|------|------|
| **Pass** | [COMPUTED][HIGH] 操作步骤全部完成，实际结果满足 Pass 标准，无 Console error、未捕获异常或数据丢失。 | 记录响应码/截图/生成文件路径/日志片段。 |
| **Fail** | [COMPUTED][HIGH] 功能可执行但结果不满足预期，或出现错误、错路由、数据损坏、状态卡死、密钥泄露、白屏。 | 记录最小复现步骤、实际结果、错误栈或响应体。 |
| **Blocked** | [COMPUTED][HIGH] 外部条件缺失导致不能执行，如无可用 LLM Key、网络不可达、依赖未安装、端口被占用。 | 记录阻塞条件和已确认的前置检查结果。 |
| **N/A** | [COMPUTED][HIGH] 当前版本未暴露该入口或功能已被移除。 | 必须指向代码位置或页面入口缺失证据。 |

### 0.3 基准环境准备

| 步骤 | 操作 | 通过标准 |
|------|------|------|
| 1 | 执行 `node -v`、`pnpm -v`、`git rev-parse --short HEAD` | Node ≥20，pnpm ≥9，Git HEAD 与本手册记录一致或在备注中更新。 |
| 2 | 在临时目录执行 `inkos init`，或复制仓库根目录作为测试副本 | `inkos.json` 可解析，`books/` 等目录存在。 |
| 3 | 执行 `inkos doctor --skip-connectivity` | 环境与项目结构检查可完成，LLM 连通性可单独执行。 |
| 4 | 启动 Studio：`inkos studio`，浏览器访问 `http://localhost:4577` | 首屏 ≤5s，无白屏，Console 无红色错误。 |
| 5 | 配置至少 1 个 LLM 服务，并执行服务连接测试 | `POST /api/v1/services/:service/test` 返回 `ok: true`，或记录 Blocked。 |

### 0.4.1 事实标签与置信度

| 标签 | 允许使用场景 | 置信度上限 |
|------|------|------|
| `[KNOWN]` | 来自稳定项目文档、代码中明确命名的事实，且未发生时间敏感变化。 | HIGH |
| `[COMPUTED]` | 本次实际执行、命令输出、API 响应、文件检查、截图证据。 | HIGH |
| `[INFERRED]` | 从代码结构、日志、状态流转推导，但未完整跑通端到端。 | MED |
| `[COMMON]` | 通用工程规范，例如状态码、密钥脱敏、路径安全原则。 | HIGH |
| `[FRAME]` | ABE、终局假定、测试哲学、分类框架。 | HIGH；不得当成真实执行结果 |
| `[GUESS]` | 无充分依据的临时猜测。 | LOW；不得作为 Pass 依据 |

### 0.4.2 证据等级

| 等级 | 标准 | 是否可支撑 Pass |
|------|------|:--:|
| **E0** | 无证据，仅口头判断。 | 否 |
| **E1** | 有截图或 UI 观察，但无 API/日志/文件佐证。 | P2-P4 可临时支撑，P0/P1 不可单独支撑 |
| **E2** | 有 API 状态码、响应体摘要、CLI 退出码或 Console 记录。 | 可支撑多数 Pass |
| **E3** | E2 + 产物文件/持久化数据/重启后复验。 | 可支撑 P0/P1 Pass |
| **E4** | E3 + 反向边界验证，如错误输入、重复请求、并发、重启恢复。 | 可支撑最终验收 |

### 0.4.3 退出标准

| 级别 | 退出条件 |
|------|------|
| P0 | [COMPUTED][HIGH] 必须 100% Pass 或有明确 Blocked 外部原因；任何 Fail 阻断交付验收。 |
| P1 | [INFERRED][MED] ≥95% Pass，剩余项必须不影响核心写作链路且有缺陷单。 |
| P2-P3 | [INFERRED][MED] 可接受 DEGRADED，但必须记录影响范围和复测入口。 |
| P4 | [COMMON][HIGH] 不允许出现白屏、密钥泄露、不可退出弹窗、移动端关键操作不可达。 |
