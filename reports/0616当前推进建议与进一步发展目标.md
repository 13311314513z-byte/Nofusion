### 2.2 当前核心矛盾

```
代码能力  >  产品闭环能力  >  质量证据能力
   95%          73%              42%
```

**矛盾 1：后端能力 > 前端/CLI 入口**
- 事件链、场景模板、声音画像、蒸馏等后端已就绪，但用户触达不到。
- `DistillationPage.tsx` 已完成但**未挂载到路由**。
- 多个 helper（修辞 aware prompt、段落去重、可读性评分）已封装但组件未调用。

**矛盾 2：Agent 数量 > 产品闭环比例**
- Agent 从 35 增至 47，但新增 12 个中仅约 1/3 形成有效闭环。
- 事件链、声音画像端点存在**桩实现**，返回假数据但标记为"完成"。

**矛盾 3：自动化评测 > 人工校准**
- Beta Reader 可规模化运行，但缺乏人类读者校准，胜率数字不可解释。
- 校准集已达 50 对，但全部来自单一 Reader 合成标注。

## 三、近期推进建议（未来 1–2 周）

### 3.1 第一优先级：修复发布阻断与工程债务

#### 任务 1：Core 包导出治理（半天）

**目标**：消除 Studio 服务端对 Core 未导出深层路径的依赖。

**执行步骤**：
1. 扫描 `packages/studio/src/api/server.ts` 中所有 `import("@actalk/inkos-core/...")` 动态导入。
2. 将实际需要的工具函数显式导出到 `@actalk/inkos-core` 根入口或 `package.json#exports`。
3. 统一使用根入口导入：
   ```typescript
   import { readArtifactIndex, readLatestArtifact, saveArtifactAutoVersion } from "@actalk/inkos-core";
   ```
4. CI 增加构建步骤：
   ```bash
   pnpm -r typecheck
   pnpm -r build
   pnpm -r test
   ```

**验收标准**：
- 删除 Core 包 `exports` 中任意路径后，CI 能立即失败。
- 不允许新的深层导入合入。

#### 任务 2：构建缓存与锁文件治理（半天）

**目标**：避免工作区被类型检查缓存污染，统一包管理。

**执行步骤**：
1. 将以下文件加入 `.gitignore`：
   ```gitignore
   packages/*/tsconfig.tsbuildinfo
   package-lock.json
   ```
2. 删除已跟踪的 `tsconfig.tsbuildinfo` 和 `package-lock.json`。
3. 将真实书籍运行数据（`books/` 中除合成夹具外）移出源码仓库，或加入 `.gitignore`。

**验收标准**：
- 执行 `pnpm -r typecheck` 后，`git status` 无业务无关变更。

#### 任务 3：CLI → Studio 依赖方向修正（半天）

**目标**：清理包依赖方向。

**执行步骤**：
1. 将 `packages/cli/package.json` 中的 `"@actalk/inkos-studio": "workspace:*"` 从 `dependencies` 移到 `optionalDependencies` 或完全移除。
2. 如果 CLI 通过子进程启动 Studio，改为运行时动态 resolve 路径：
   ```typescript
   const studioPath = require.resolve("@actalk/inkos-studio/bin");
   spawn("node", [studioPath]);
   ```
3. 更新 `CONTRIBUTING.md` 说明包依赖方向。

### 3.2 第二优先级：补齐 5 个高价值产品闭环

#### 任务 4：事件链 — 决定真实落地或暂时下线（1 天）

**当前问题**：
- `GET/POST /api/v1/books/:id/event-chain` 端点存在但用假数据构造。
- 未调用 `EventChainExtractor` / `EventChainInferenceEngine`。
- 未注入 Writer prompt，未进入管线消费。

**二选一方案**：

**方案 A：快速真实落地（推荐，若 1 周内可完成）**
1. 端点调用真实 `EventChainExtractor` 提取事件。
2. 调用 `EventChainInferenceEngine` 建立因果链。
3. 用 `renderEventChainAsNarrativeControl()` 渲染后注入 Writer system prompt。
4. 前端 `BookWorkspace` 增加 "事件链" Tab，展示事件节点图。

**方案 B：暂时下线**
1. 端点返回 `501 Not Implemented` 或从路由中移除。
2. 保留 Core Agent 代码，待资源充足后再启用。

**验收标准**：
- 若选 A：端到端生成一章，事件链内容出现在 Writer prompt 中，前端可查看。
- 若选 B：CI 全绿，无 unreachable 端点维护负担。

#### 任务 5：声音画像 — 端点接桩 + 持久化 + 前端入口（1.5 天）

**执行步骤**：
1. 修改 `POST /api/v1/books/:id/voice-profiles/analyze`：
   - 加载角色卡 `loadRoleCard(bookDir, characterId)`。
   - 从最近 5 章提取该角色对话/行动样本 `collectCharacterDialogue()`。
   - 实例化 `VoiceProfileAnalyzer` 调用 `analyze()`。
   - 持久化到 `story/voice_profiles/<characterId>.json` 和 `index.json`。
2. 在 `BookCharactersSection` 增加 "分析声线" 按钮，展示声线画像。
3. 将声音画像规则注入 Writer prompt（可选，R2 阶段）。

**验收标准**：
- 端点返回真实 `VoiceProfile`，非空 `signaturePhrases` / `avgSentenceLength`。
- 分析结果持久化到磁盘。
- 前端角色卡页面可查看。

#### 任务 6：场景模板 — 从 JSON 存储到 Planner 消费（1 天）

**执行步骤**：
1. `PUT /scene-templates` 增加 `SceneTemplateIndexSchema` Zod 校验。
2. 在 Planner 中读取 `scene_templates.json`，按当前章节 `location/type` 匹配模板。
3. 将匹配到的模板约束注入 memo prompt（如：地点、氛围、道具、关联场景）。
4. 前端 `BookWorkspace` 增加 "场景" Tab 管理模板。

**验收标准**：
- 保存场景模板后，Planner prompt 中出现对应模板约束。
- 模板变更可追踪。

#### 任务 7：计划变体 — 暴露开关 + 前端选择（1 天）

**执行步骤**：
1. `write-preview` 端点支持 `?alternatives=true`，返回主 memo + 2 个变体。
2. `write-next` / `draft` 请求体支持 `{ generateAlternatives?: boolean; selectedVariantId?: string }`。
3. 前端在生成前展示 3 个 memo 候选，作者可切换选择。
4. `PlannerAgent` 根据 `selectedVariantId` 使用对应 memo。

**验收标准**：
- 用户能在写作前看到 3 个不同侧重点的 memo 方案。
- 选择后 Writer 按选中方案执行。

#### 任务 8：风格蒸馏 — 并入 StyleManager 并修复契约（1.5 天）

**执行步骤**：
1. 修复 `DistillationPage.tsx` 类型：
   - `rules` 从 `string[]` 改为 `DistillationRule[]`。
   - 移除后端不存在的 `sentencePatterns`/`wordPreferences`/`narrativeHabits` 假设。
2. 将蒸馏页面作为 `StyleManager` 的新 Tab（如 "蒸馏规则"）。
3. 增加操作按钮：生成蒸馏、编辑覆盖规则、发布版本、查看历史版本。
4. 在 `apply-author` 或写作管线中加载蒸馏规则。

**验收标准**：
- 从 StyleManager 可进入蒸馏管理。
- 能完整走通：生成 → 编辑 → 发布 → 应用到书籍。

### 3.3 第三优先级：启动 Stage 5 数据证据

#### 任务 9：修订收益实验（半天）

**目的**：验证自动修订是否真改善文本质量。

**执行步骤**：
1. 从 `药` / `通兰民间故事集` / `test-book-0609` 选取 10 章（覆盖 opening/transition/conflict/climax/resolution）。

3. 用 Beta Reader 做成对比较（5 维度）：
   ```bash
   node scripts/preference-eval.mjs revision-results.csv --output reports/修订收益报告.md
   ```
4. 统计修订版胜率与置信区间。

**成功标准**：
- 修订版在 `clarity` / `emotion` 维度胜率 > 50%（CI 下限 > 45% 即有参考价值）。

#### 任务 10：人工双盲标注启动（持续）

**执行步骤**：
1. 从 `calibration-results.csv` 50 对中抽取 20 对作为第一轮标注。
2. 准备标注指南 `reports/baseline-data/annotation-guide.md`：
   - 5 个维度定义与示例
   - A/B/tie 判断标准
   - 去标识化要求
3. 邀请 2 位标注员（普通读者 1 + 编辑/写作者 1）。
4. 使用 `scripts/human-reader-agreement.mjs` 计算：
   - 人工间一致性（Kappa）
   - 人工 vs Beta Reader 一致性

**成功标准**：
- 人工间 Kappa ≥ 0.4（可接受）/ ≥ 0.6（良好）。
- 人机一致性 ≥ 70%。

## 六、明确"现在不该做什么"

以下方向应明确暂缓，避免分散有限资源：

| 不做 | 原因 |
|------|------|
| 继续大量新增 Agent | E1-E13 还有 60%+ 未闭环，新增只会加剧"模型/端点存在但不可用" |
| 完整插件系统 | 类型、安全、生命周期、测试成本过高 |
| PostgreSQL/多机数据库 | 当前文件系统 + SQLite 已足够 |
| 多用户 SaaS | 权限、隔离、同步、部署复杂度过高 |
| 前端路由框架迁移 | 当前 hash route 够用，迁移收益不明显 |
| 全量 Pipeline 重写 | 风险高，应先做局部拆分与可观测 |
| 过早平台化/商业化 | 质量证据不足，无法证明产品价值 |
