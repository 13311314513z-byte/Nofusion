## writer.ts 业务逻辑拆分执行流程规划（仅规划，不改源码）

> **本节目的**: 根据当前 `packages/core/src/agents/writer.ts` 的真实结构，补充后续拆分执行流程、阶段顺序、边界契约、验收标准与回滚策略。  
> **执行约束**: 拆分实施时必须保持 `WriterAgent` 对外 API 不变；本次仅补充规划文档，不修改 `writer.ts`。

### 1. 当前可拆分边界复核

`writer.ts` 当前是“写作编排器 + 多阶段 helper + I/O”的混合文件。拆分前应先锁定以下边界：

| 当前成员 | 业务职责 | 建议归属 |
|---|---|---|
| `writeChapter()` | 主写作链路编排：读取上下文、生成正文、结算状态、后写校验、组装输出 | 保留在 `writer.ts` facade，逐步下沉子阶段 |
| `settleChapterState()` | 独立状态结算入口，用于已存在正文的 truth/state 重算 | 保留 facade 入口，内部复用 `writer-settlement.ts` |
| `settle()` | Observer + Settler 双阶段 LLM 调用、delta/legacy 解析、governed merge | `writer-settlement.ts` 或 `writer-observer-settler.ts` |
| `saveChapter()` | 章节正文、current_state、pending_hooks、runtime snapshot 落盘 | `writer-io.ts` |
| `saveNewTruthFiles()` | chapter_summaries、subplot、emotional_arcs、character_matrix 落盘 | `writer-io.ts` |
| `buildUserPrompt()` / `buildGovernedUserPrompt()` | legacy/governed 正文生成 prompt 构建 | `writer-creative.ts` |
| `buildChapterContextBlock()` / `joinGovernedEvidenceBlocks()` / `buildSettlerGovernedControlBlock()` | governed 上下文和控制块拼装 | `writer-context.ts`；settler 控制块可留在 settlement 模块 |
| `loadRecentChapters()` / `readFileOrDefault()` / `findRelevantSummaries()` / `loadVoiceProfiles()` | 上下文加载、裁剪、筛选 | `writer-context.ts` |
| `buildRuntimeStateArtifactsIfPresent()` / `resolveRuntimeStateArtifactsForOutput()` / `normalizeRuntimeStateDeltaChapter()` | runtime structured state 适配与规范化 | `writer-runtime-state.ts` 或 `writer-io.ts` |
| `validatePostWrite()` 调用链、`analyzeAITells()`、hook health 汇总 | 后写质量门禁 | `writer-post-write.ts` |
| `buildStyleFingerprint()` / `extractDialogueFingerprints()` | 风格和声音画像辅助信息 | `writer-context.ts` 或 `writer-style-context.ts` |

拆分原则：`writer.ts` 最终只保留 `WriterAgent` 类、对外方法签名、阶段调用顺序、输出汇总逻辑；具体阶段实现下沉到纯函数或小型 service 函数。

### 2. 目标模块结构

建议最终形成以下结构：

```text
packages/core/src/agents/
├─ writer.ts                         # facade：保留 WriterAgent 对外 API 与主流程编排
├─ writer-types.ts                   # 已存在：继续承载 WriteChapterInput/Output 等公共类型
├─ writer-creative.ts                # Phase 1：正文创作 prompt、manifest、LLM call、parseCreativeOutput
├─ writer-settlement.ts              # Phase 2/2a/2b：Observer + Settler、settlement parsing、governed merge
├─ writer-context.ts                 # truth/context/recent chapters/voice/style/relevant summaries 加载与裁剪
├─ writer-post-write.ts              # 后写校验、AI tell、重复/段落漂移、hook health 汇总
├─ writer-runtime-state.ts           # RuntimeStateDelta 规范化、runtime artifact 构建与输出解析
```

其中 `writer-runtime-state.ts` 可以与 `writer-io.ts` 合并为第一阶段临时模块；若后续 runtime state 逻辑继续增长，再单独拆出。

### 4. 推荐执行顺序

#### 阶段 1：先拆 I/O，风险最低

目标：把落盘逻辑与文本生成逻辑分离。

执行：
1. 新建 `writer-io.ts`。
2. 移入 `saveChapter()` 主体、`saveNewTruthFiles()` 主体、`appendChapterSummary()`、`sanitizeFilename()`。
3. `WriterAgent.saveChapter()` 改为薄包装，调用 `saveWriterChapter()`。
4. `WriterAgent.saveNewTruthFiles()` 改为薄包装，调用 `saveWriterTruthFiles()`。

验收：
- 章节文件名、标题格式、`current_state.md`、`pending_hooks.md`、`particle_ledger.md` 写入路径完全一致。
- runtime snapshot 保存行为不变。
- 旧测试不需要改调用方。

#### 阶段 2：拆 runtime state 适配

目标：隔离 structured runtime state 兼容逻辑。

执行：
1. 新建 `writer-runtime-state.ts`。
2. 移入 `normalizeRuntimeStateDeltaChapter()`、`buildRuntimeStateArtifactsIfPresent()`、`resolveRuntimeStateArtifactsForOutput()`。
3. `renderDeltaSummaryRow()` 可一起迁移，或作为 output assembly helper 保留在 facade。
4. 对外提供纯函数：`normalizeWriterRuntimeDelta()`、`buildWriterRuntimeArtifacts()`、`resolveWriterRuntimeArtifactsForOutput()`。

验收：
- `runtimeStateDelta.chapter` 纠偏行为保持一致。
- hook `startChapter` / `lastAdvancedChapter` 不超过当前章节的保护逻辑保持一致。
- `settleChapterState(... allowReapply)` 行为保持一致。

#### 阶段 3：拆上下文加载与裁剪

目标：减少 `writeChapter()` 前半段的上下文准备噪声。

执行：
1. 新建 `writer-context.ts`。
2. 先提取无 LLM 调用的纯上下文函数：`readWriterTruthContext()`、`loadRecentChapters()`、`buildWriterStyleContext()`、`buildLegacyFilteredContext()`、`buildGovernedEvidenceContext()`、`findRelevantSummaries()`。
3. 让 `writeChapter()` 调用 `prepareWriterContext(input)`，返回结构化上下文对象。

验收：
- legacy 路径下 POV 过滤、hooks/summaries/subplots/arcs/matrix 过滤不变。
- governed 路径下 `contextPackage`、`ruleStack`、selected evidence 拼接不变。
- `parent_canon.md`、`fanfic_canon.md` 存在与否的判定不变。

#### 阶段 4：拆 post-write 校验

目标：把零 LLM 成本的规则校验变成可单测模块。

执行：
1. 新建 `writer-post-write.ts`。
2. 提供 `runPostWriteChecks(params)`。
3. 输入包含 creative content、genreProfile、bookRules、language、fingerprintChapters、hook health 所需 snapshot/delta/book target chapters/priorHookIds。
4. 返回 `surfaceNormalizedContent`、`surfaceNormalizedWordCount`、`postWriteErrors`、`postWriteWarnings`、`aiTellIssues`、`hookHealthIssues`。

验收：
- `normalizePostWriteSurface()` 仍在所有后写校验之前执行。
- `detectCrossChapterRepetition()` 和 `detectParagraphLengthDrift()` 仍使用更长窗口的 `fingerprintChapters`。
- error/warning 分级不变。

#### 阶段 5：拆 creative 阶段

目标：隔离 Phase 1 正文生成。

执行：
1. 新建 `writer-creative.ts`。
2. 提供 `runCreativeWritingPhase(params)`。
3. 该函数内部负责 `buildWriterSystemPrompt()`、voice profile 注入、legacy/governed user prompt 构建、prompt manifest、`chat()`、`parseCreativeOutput()`、`writerPromptHash` 返回。
4. `WriterAgent.writeChapter()` 只接收 creative 阶段结果。

验收：
- `writerPromptHash` 与拆分前同输入下保持一致，除非 manifest fragment id 有意变更；默认不应变更。
- `temperatureOverride ?? 0.7` 行为不变。
- `stopReason === "length"` 的 warn 行为不丢失。
- `PRE_WRITE_CHECK` 与 memo 对齐的软校验仍执行。

#### 阶段 6：拆 settlement 阶段

目标：让 `writeChapter()` 与 `settleChapterState()` 复用同一个 settlement 模块。

执行：
1. 新建 `writer-settlement.ts`。
2. 把当前 `settle()` 改造成 `runSettlementPhase(params)`。
3. 保留两段式逻辑：Observer 使用 temperature 0.5；Settler 使用 temperature 0.3。
4. 保留 delta 优先解析：先 `parseSettlerDeltaOutput()`，失败再 `parseSettlementOutput()`。
5. governed control 下继续 merge hooks/subplots/emotionalArcs/characterMatrix。

验收：
- `writeChapter()` 和 `settleChapterState()` 都调用同一个 settlement 实现。
- `validationFeedback` 仅在独立 settlement/修复路径传入，普通写作路径仍为 `undefined`。
- governed merge 的 key 列保持不变：hooks `[0]`，emotional arcs `[0, 1]`。
- token usage 仍正确返回并由 facade 汇总。

### 5. 拆分后的 facade 目标形态

`writer.ts` 不应变成空壳，但应变成清晰的 orchestration：

```ts
export class WriterAgent extends BaseAgent {
  async writeChapter(input: WriteChapterInput): Promise<WriteChapterOutput> {
    const context = await prepareWriterContext(...);
    const creative = await runCreativeWritingPhase(...);
    const settlement = await runSettlementPhase(...);
    const runtime = await buildWriterRuntimeArtifacts(...);
    const postWrite = runPostWriteChecks(...);
    return assembleWriteChapterOutput(...);
  }

  async settleChapterState(input: SettleChapterStateInput): Promise<WriteChapterOutput> {
    const context = await prepareSettlementContext(...);
    const settlement = await runSettlementPhase(...);
    const runtime = await buildWriterRuntimeArtifacts(...);
    return assembleSettlementOutput(...);
  }

  async saveChapter(...) {
    return saveWriterChapter(...);
  }

  async saveNewTruthFiles(...) {
    return saveWriterTruthFiles(...);
  }
}
```

目标行数：
- `writer.ts`: 由约 1,580 行降至 350-600 行。
- 单个新模块：控制在 150-350 行；`writer-settlement.ts` 可接受 400 行以内。
- 不新增跨层依赖，不让 `pipeline/runner.ts` 直接依赖 writer 子模块。

### 7. 风险门禁与停止条件

拆分过程必须设置以下停止条件：

1. **出现 prompt hash 非预期变化**：停止继续拆分，先定位 fragment id、content 或排序是否改变。
2. **token usage 合计变化**：停止继续拆分，确认 creative/observer/settler usage 是否重复或漏加。
3. **runtime snapshot 与 markdown truth 不一致**：停止继续拆分，优先回滚 runtime-state 或 I/O 阶段。
4. **governed 路径通过但 legacy 路径失败**：禁止继续拆 creative/settlement，先补 legacy golden case。
5. **`WriterAgent` public method 签名变化**：直接回滚该阶段，除非同步完成全仓调用方迁移并有明确升级理由。

### 8. 推荐实际提交顺序

为了降低审查和回滚成本，建议拆成 6 个独立提交：

1. `test(writer): add golden coverage before writer split`
2. `refactor(writer): extract writer io helpers`
3. `refactor(writer): extract runtime state artifact helpers`
4. `refactor(writer): extract context preparation helpers`
5. `refactor(writer): extract post-write validation`
6. `refactor(writer): extract creative and settlement phases`

每个提交都应满足：
- 类型检查通过；
- 当前阶段对应单测通过；
- `WriterAgent` 对外调用不变；
- 不混入业务行为修复，除非测试证明原行为有 bug 且单独记录。

拆分完成后，以以下标准判定成功：

- 无新增循环依赖，无新增 `any` 扩散。

### 1. 当前可拆分边界复核

### 5. 拆分后的 facade 目标形态

`runner.ts` 拆分完成后应类似：

```ts
export class PipelineRunner {
  constructor(config: PipelineConfig, ctx?: PipelineContext) {
    this.ctx = ctx ?? new PipelineContext(config);
  }

  async initBook(book: BookConfig, options: InitBookOptions = {}): Promise<void> {
    return initBookPipeline({ ctx: this.ctx, book, options });
  }

  async writeDraft(bookId: string, context?: string, wordCount?: number): Promise<DraftResult> {
    return writeDraftPipeline({ ctx: this.ctx, bookId, context, wordCount });
  }

  async writeNextChapter(bookId: string, wordCount?: number, temperatureOverride?: number): Promise<ChapterPipelineResult> {
    return writeNextChapterPipeline({ ctx: this.ctx, bookId, wordCount, temperatureOverride });
  }

  async auditDraft(bookId: string, chapterNumber?: number) {
    return auditDraftPipeline({ ctx: this.ctx, bookId, chapterNumber });
  }
}
```

目标不是删除 `PipelineRunner`，而是让它成为稳定入口，业务复杂度留在领域模块中。

### 7. 风险门禁与停止条件

拆分 runner 时，以下情况必须停止继续推进：

1. **public API 或导出类型变化**：除非全仓同步迁移并单独记录，否则回滚该阶段。
2. **write-next 锁语义变化**：任何重复写章、锁未释放、并发测试变慢或卡死，都先回滚 write-next 拆分。
3. **truth/runtime/memory 三者不一致**：章节正文落盘成功但 truth 或 memory index 缺失时，停止拆分并定位 support/memory 边界。
4. **审计问题丢失**：auditIssues、actionable issue、sequence-level issue 合并结果变化时，停止 revision 拆分。
5. **foundation import 输出漂移**：相同资料包生成的 source bundle 摘录策略变化时，停止 import-foundation 拆分。
6. **循环依赖出现**：任何子模块 import `runner.ts` 都视为结构性失败，先重构依赖方向。

### 8. 推荐实际提交顺序

建议拆成 7 个独立提交，每个提交只处理一个业务域：

1. `test(runner): add golden coverage before pipeline split`
2. `refactor(runner): extract memory index helpers`
3. `refactor(runner): extract foundation and import planning`
4. `refactor(runner): extract canon and fanfic flows`
5. `refactor(runner): extract chapter writing support`
6. `refactor(runner): extract audit and revision flows`
7. `refactor(runner): extract write-next coordinator and trim facade`

每个提交都应满足：
- `PipelineRunner` public API 不变；
- 对应领域单测通过；
- `packages/core` typecheck 通过；
- 不混入行为修复；确需修 bug 时单独提交并同步测试手册。

### 9. 最终验收标准

拆分完成后，以以下标准判定成功：

- `runner.ts` 行数低于 900 行，理想目标 600-800 行。
- `PipelineRunner` 仍是 CLI/Studio 访问 pipeline 的唯一 facade。
- foundation、chapter、revision、write-next、import、memory index 均有独立模块与核心测试。
- `writeNextChapter` 并发锁、状态修复、resync 行为不变。
- 新书初始化、资料导入、同人 canon、章节导入、审计修订、memory index 重建六条链路均通过 golden case。
- `pnpm --filter @actalk/inkos-core test`、CLI integration、Studio server 相关测试通过。
- 无新增循环依赖，无跨模块复制大型 helper，无新增 `any` 扩散。

## 二、`runner.ts` 拆分分析

### 2.1 当前结构

`PipelineRunner` 已把部分逻辑外移到：
- `chapter-persistence.ts`
- `chapter-review-cycle.ts`
- `audit-helpers.ts` 
- `chapter-truth-validation.ts`
- `pipeline-audit.ts`
- `pipeline-book-status.ts`
- `persisted-governed-plan.ts`

但类中仍保留以下大类方法（按出现顺序）：

| 方法 | 业务域 | 行数占比估算 |
|---|---|---|
| 导入相关：foundation import / fanfic canon / import chapters / import canon | 导入/母本 | ~400 行 |
| 书籍初始化：`initBook` / `initFanficBook` / `reviseFoundation` | 初始化 | ~250 行 |
| 单章操作：`writeDraft` / `planChapter` / `composeChapter` / `auditDraft` / `reviseDraft` | 单章 Pipeline | ~600 行 |
| 连贯链路：`writeNextChapter` / `repairChapterState` / `resyncChapterArtifacts` | 主链路 | ~200 行 |
| 状态/记忆索引：`syncCurrentStateFactHistory` / `syncNarrativeMemoryIndex` / `rebuildCurrentStateFactHistory` / `rebuildNarrativeMemoryIndex` / `withMemoryIndexRetry` | 记忆索引 | ~350 行 |
| 工具/内部：`buildLengthWarnings` / `buildLengthTelemetry` / `persistAuditDriftGuidance` / `evaluateMergedAudit` / `createGovernedArtifacts` | 工具方法 | ~400 行 |

### 2.2 拆分方案

推荐按“业务域 coordinator + 工具类”拆分，而非简单按方法数量切割：

```
pipeline/
├── runner.ts                    # 保留 facade，只负责委派
├── runner-foundation.ts         # initBook / reviseFoundation / import foundation / fanfic canon
├── runner-import.ts             # importChapters / importCanon
├── runner-chapter.ts            # writeDraft / planChapter / composeChapter / auditDraft / reviseDraft
├── runner-write-next.ts         # writeNextChapter / repairChapterState / resyncChapterArtifacts
└── runner-memory-index.ts       # MemoryDB 索引同步与重建逻辑
```

### 2.3 前后端对应风险

- **Studio 端依赖的 API 不变**：`/api/v1/books/:id/write-next` 等路由只调用 `runner.writeNextChapter()`，拆分到 `runner-write-next.ts` 后，只要 `PipelineRunner` 仍暴露同名公共方法，HTTP 契约无影响。
- **CLI 端依赖的公共方法不变**：`inkos write next`、`inkos audit`、`inkos revise`、`inkos plan`、`inkos compose` 都通过 `PipelineRunner` 调用，应保持 `PipelineRunner` 的公开 API 签名不变。
- **PipelineContext 是天然隔离层**：`runner.ts` 已通过 `this.ctx` 访问 state / config / agentClients，拆分时各 coordinator 接收 `ctx: PipelineContext` 即可，不需要新增全局状态。

### 2.4 功能漂移风险

| 风险点 | 说明 | 防控措施 |
|---|---|---|
| 锁与并发 | `writeNextChapter` 依赖 `_writeNextChapterLocked` 的写锁语义 | 拆分时把锁管理保留在 `runner-write-next.ts` 内，不要跨文件持有锁 |
| 事件/日志 | `logStage`、`emitWebhook`、`dispatchNotification` 分散在多个方法中 | 提取 `PipelineLogger` 或 `PipelineEventBus`，避免各 coordinator 直接调用 notify/webhook |
| 记忆索引重试 | `withMemoryIndexRetry` 在多处被调用 | 应整体迁移到 `runner-memory-index.ts`，不要在多个 coordinator 中重复实现重试 |
| 货币/字数规格 | `buildLengthSpec`、`buildLengthTelemetry` 被多个方法使用 | 已可放到 `length-governance` 工具模块，不应随业务域重复 |

### 2.5 逻辑缺失风险

- **导入母本（importCanon）与同人（fanfic）的共享逻辑**：当前 `importCanon` 和 `initFanficBook` 都读取 parent_canon / fanfic_canon，拆分时需识别并保留共用 helper，避免两边实现不一致。
- **Foundation import 的 source bundle 校验**：`assertValidArchitectOutput`、`buildFoundationReviewFeedback` 是 foundation 导入的核心校验，必须随 `runner-foundation.ts` 完整迁移，不可遗漏。
- **`writeNextChapter` 的错误恢复路径**：`repairChapterState` / `resyncChapterArtifacts` 与 `writeNextChapter` 共用 `_writeNextChapterLocked` 的底层逻辑，拆分时需明确哪些 private 方法需要提升到 `runner-write-next.ts` 的模块级 helper。

### 2.6 推荐优先级

**P1**：先拆分 `runner-memory-index.ts`（最独立、最容易单元测试）和 `runner-foundation.ts`（与主写作链路耦合度低）。  
**P2**：再拆分 `runner-chapter.ts` 和 `runner-write-next.ts`。  
**P3**：最后把 `runner.ts` 缩减为纯 facade。

## 三、`writer.ts` 拆分分析

### 3.1 当前结构

`WriterAgent` 目前只有 4 个公开/半公开方法：
- `writeChapter()` — 主写作流程（~350 行）
- `settleChapterState()` — 独立结算入口（~300 行）
- `saveChapter()` — 落盘（~420 行）
- `saveNewTruthFiles()` — 初始化真值文件（~60 行）

但内部有清晰的阶段：
- Phase 1: Creative writing（创作正文）
- Phase 2: State settlement（状态结算）
- Phase 3: Observer（可选观察者校验）
- Phase 4: Settler（最终定稿）
- Post-write validation（后写校验）
- I/O: `saveChapter` / `saveNewTruthFiles`

### 3.2 拆分方案

推荐按“写作阶段 + I/O”拆分，保持 `WriterAgent` 作为 facade：

```
agents/
├── writer.ts                    # facade：writeChapter / settleChapterState 入口
├── writer-creative.ts           # Phase 1 creative prompt + LLM call + parseCreativeOutput
├── writer-settlement.ts         # Phase 2 settlement prompt + delta parsing
├── writer-observer-settler.ts   # Phase 3/4 observer + settler
├── writer-post-write.ts         # 后写校验、AI tells、hook health
├── writer-io.ts                 # saveChapter / saveNewTruthFiles / runtime state artifacts
└── writer-context.ts            # 上下文加载、POV 过滤、relevant summaries 等 helper
```

### 3.3 前后端对应风险

- **CLI/Studio 均通过 `PipelineRunner` 间接调用 `WriterAgent`**：`PipelineRunner.writeDraft` / `writeNextChapter` 调用 `new WriterAgent(agentCtx).writeChapter(input)`。拆分后只要 `WriterAgent.writeChapter` 签名不变，上层无感知。
- **`WriteChapterInput` / `WriteChapterOutput` 是核心契约**：这两个类型已被 `writer-types.ts` 独立定义，拆分时必须保持其字段稳定，否则 `pipeline/chapter-persistence.ts` 等消费者会断裂。
- **`saveChapter` 落盘路径影响前后端状态一致性**：`saveChapter` 写入 `chapters/`、`story/current_state.md`、`story/pending_hooks.md` 等。若拆分到 `writer-io.ts`，需确保 `writer.ts` 仍按原顺序调用，避免前端读取到半写入状态。

### 3.4 功能漂移风险

| 风险点 | 说明 | 防控措施 |
|---|---|---|
| Prompt hash 一致性 | `writeChapter` 返回 `writerPromptHash`，来自 creative manifest | 拆分到 `writer-creative.ts` 后，必须原样返回给 `writer.ts` 再组装 `WriteChapterOutput` |
| Token 汇总 | `tokenUsage` 汇总 creative + settle 的 usage | 各阶段模块应返回 usage，由 `writer.ts` 汇总，避免重复或遗漏 |
| Post-write errors/warnings | 当前在 `writeChapter` 末尾过滤并加入 output | 拆分后 `writer-post-write.ts` 应返回结构化 violations，`writer.ts` 负责注入 output |
| Runtime state artifacts | `buildRuntimeStateArtifactsIfPresent` 影响 `updatedState` / `updatedHooks` / `chapterSummary` | 这是跨阶段数据流，需在 `writer.ts` 中保留 orchestration，不要完全下放到阶段模块 |

### 3.5 逻辑缺失风险

- **`settleChapterState` 与 `writeChapter` 的 settlement 路径共享 `this.settle()`**：`settle()` 是私有方法（~150 行）。若把 settlement 拆到 `writer-settlement.ts`，需把 `settle()` 提升为模块级函数，并确保 `settleChapterState` 和 `writeChapter` 都调用同一实现。
- **Governed vs Legacy 双路径**：`writeChapter` 中 `input.chapterMemo ? buildGovernedUserPrompt : buildUserPrompt` 的选择逻辑必须保留完整，拆分时不能只做“提取 governed 部分”而遗漏 legacy 路径。
- **多语言分支**：大量 `if (language === "en")` 散落在 prompt 构建、日志、校验中。拆分后需确保每阶段模块都正确处理两种语言。

### 3.6 推荐优先级

**P1**：先拆 `writer-io.ts`（落盘逻辑最独立，且已有 `runtime-state-store.ts` 可复用）。  
**P2**：再拆 `writer-post-write.ts` 和 `writer-context.ts`。  
**P3**：最后拆 creative / settlement / observer-settler，保留 `WriterAgent` 为 orchestrator。

### 4.3 前后端对应风险

- **测试拆分不改变前后端契约**：只是测试代码组织方式变化。
- **mock 共享需要关注**：当前 `server.test.ts` 顶部有一大段 `vi.mock("@actalk/inkos-core", ...)`，mock 了约 40 个 core 函数。拆分时需要：
  1. 把通用 mock 提取到 `src/api/__tests__/mocks/core.mock.ts`；
  2. 各测试文件按需引入并覆盖特定 mock；
  3. 避免每个测试文件重复定义 `endpointMocks`、`SERVICE_PRESETS_MOCK` 等大数据结构。

### 4.4 功能漂移风险

| 风险点 | 说明 | 防控措施 |
|---|---|---|
| mock 行为不一致 | 拆分后若各文件对同一 core 函数 mock 返回值不同，会导致测试通过但产品行为未覆盖 | 建立统一的 `mockFactories`，对 `createLLMClient`、`loadProjectConfig` 等高频依赖提供默认实现 |
| 测试夹具（fixture）漂移 | 当前测试内联创建临时目录、写 `inkos.json`、`.env` | 提取 `createTestProject()` helper，确保所有测试使用同一项目结构 |
| 生命周期钩子重复 | `beforeEach` / `afterEach` 创建/删除临时目录的逻辑应统一 | 提取到 `vitest.setup.ts` 或测试工具函数 |

### 4.5 逻辑缺失风险

- **拆分过程中遗漏 it()**：从 3,874 行文件迁移 95 个测试时，容易漏掉边界用例。建议：
  1. 先复制原文件到新测试文件；
  2. 删除不属于该域的 `it()`；
  3. 运行该文件确认通过；
  4. 重复直到原文件只剩 daemon 生命周期。
- **agent 测试依赖 session 测试**：部分 agent 用例会先 `POST /api/v1/sessions` 创建会话。拆分后 `server-agent.test.ts` 应保留最小会话创建逻辑，或从 `server-sessions.test.ts` 导入 helper。

### 4.6 推荐优先级

**P0（应最先做）**：`server.test.ts` 拆分是最安全、收益最高的技术债清理。  
**拆分顺序**：
1. 提取公共 mock 和 fixture helper；
2. 拆分 `server-services.test.ts`（最大一块，约 25 个测试）；
3. 拆分 `server-agent.test.ts`；
4. 拆分其余小域；
5. 原 `server.test.ts` 保留 daemon + server 启动测试，目标 <300 行。

## 五、跨文件共同风险

### 5.1 类型契约漂移

- `runner.ts` 和 `writer.ts` 都有大量内部接口（如 `InitBookOptions`、`ImportChaptersInput`、`WriteChapterInput`）。拆分时应优先把类型定义迁移到独立 `*.types.ts` 文件，而不是留在原文件中通过 `export` 间接暴露。
- 当前 `writer.ts` 已把 `WriteChapterInput` 等放到 `writer-types.ts`，这是好的实践，应延续。

### 5.2 导入循环

- `runner.ts` 已导入 `chapter-persistence.ts`、`chapter-review-cycle.ts` 等。若进一步把 `writeNextChapter` 拆到 `runner-write-next.ts`，需避免 `runner-write-next.ts` 回头 import `runner.ts` 中的 helper。
- 建议：被多个模块使用的 private helper 先提升到 `pipeline/utils/*.ts` 或 `pipeline/shared/*.ts`。

### 5.3 测试覆盖下降

- 大文件拆分后，若不在每个新模块上补充单元测试，可能出现“总测试数不变但单模块无独立测试”的情况。
- 建议：每拆分出一个模块，至少新增该模块的核心路径单元测试，避免把验证责任全部压在 `server.test.ts` 或 `pipeline-runner.test.ts` 上。

## 六、实施建议与验收标准

### 6.1 推荐实施路径

| 阶段 | 任务 | 预计改动文件数 | 预期收益 |
|---|---|---|---|
| 1 | 拆分 `server.test.ts` | +8 / -0（新增测试文件） | 单测试文件 <800 行；CI 调试效率提升 |
| 2 | 提取 `runner-memory-index.ts` | +1 | 记忆索引逻辑独立可测 |
| 3 | 提取 `runner-foundation.ts` | +1 | 初始化/导入逻辑独立可测 |
| 4 | 提取 `writer-io.ts` | +1 | 落盘逻辑可独立测试 |
| 5 | 拆分 `runner-chapter.ts` / `runner-write-next.ts` | +2 | `runner.ts` 降至 <1,000 行 |
| 6 | 拆分 `writer-creative.ts` / `writer-settlement.ts` | +2 | `writer.ts` 降至 <800 行 |

### 6.2 验收标准

- 每个被拆分出的模块有独立的单元测试覆盖核心路径。
- `runner.ts` < 1,000 行，`writer.ts` < 800 行，`server.test.ts` < 300 行。
- 所有现有 CLI / Studio API 调用无需修改即可继续工作。
- `pnpm typecheck`、`pnpm build`、`pnpm test:ci` 全部通过。
- 不引入新的前后端行为差异（通过现有 E2E / 集成测试保证）。

## 七、最终结论

`runner.ts` 与 `server.test.ts` 是当前最应优先拆分的大文件：`server.test.ts` 风险最低、收益最直接；`runner.ts` 拆分可显著降低主写作链路的认知负担。`writer.ts` 虽然行数也超标，但其阶段边界已较清晰，拆分收益略低于前两者，可作为第二梯队。

拆分的核心原则是：**按业务域而非按行数切割，保持 `PipelineRunner` / `WriterAgent` 作为 facade，先提取工具/独立域，再拆核心 orchestration，最后补充模块级单元测试**。
