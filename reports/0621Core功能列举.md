# 0621 Core 功能列举

> 日期：2026-06-21  
> 范围：`packages/core/src` 当前代码与 `packages/core/src/index.ts` 导出面  
> 目的：列举 Core 当前已经实现的核心功能，不等同于发布验收报告。  
> 当前注意：工作区存在未提交修改；最近一次全项目 `typecheck` 已知被 `runner.ts` 类型 re-export 写法阻断，需单独修复后再认定门禁状态。

---

## 一、总体结论

Core 已经不是单一“写作流水线”模块，而是一个较完整的小说创作内核。它当前覆盖：

- 项目、书籍、章节、设定、运行时状态的数据模型与校验；
- LLM provider、服务预设、密钥、模型解析、流式响应与工具调用；
- 长篇写作 pipeline、短篇创作 pipeline、fanfic 导入、章节导入、基础设定导入；
- 多 agent 写作、规划、审稿、修订、状态结算、文风分析和角色声音分析；
- 交互式会话、自然语言路由、工具调用、转录恢复、编辑事务；
- 文风库、作家档案、文风蒸馏、文本预处理、重复检测和可读性评分；
- 运行时状态、记忆索引、hook 治理、状态变更日志、资源锁和一致性检查；
- 通知、结构化日志、错误上报、导出、分析统计等支撑能力。

当前 Core 的主要问题不是“功能缺失”，而是功能面已经很大，部分实现仍集中在大文件里，尤其是 `pipeline/runner.ts` 和部分测试文件。

## 十七、当前不应过度声明的部分

以下能力已有实现或雏形，但不宜描述为完全闭合：

1. **Pipeline 拆分完成**  
   已拆出多个 `pipeline-*` 文件，但 `runner.ts` 仍然很大。

2. **PipelinePool 已充分验证**  
   pool 已实现并导出，但专项测试不足。

3. **真实 LLM E2E 已稳定覆盖**  
   真实 LLM 测试是 opt-in，默认测试不能证明真实服务质量。

4. **所有输出品质问题已解决**  
   Core 已有大量质量治理工具，但输出品质仍依赖模型、prompt、书籍状态和运行时反馈闭环。

5. **当前门禁全绿**  
   最近一次全项目 typecheck 已知存在 `runner.ts` 类型 re-export 阻断，需要修复后重新验证。

## 十九、后续建议

短期优先级：

1. 修复 `runner.ts` 类型 re-export 的 `export type` 问题，恢复全项目 typecheck。
2. 继续拆 `runner.ts`，优先抽 `pipeline-import.ts`、`pipeline-memory-sync.ts`、`pipeline-style.ts`。
3. 给 `PipelinePool` 补专项测试。
4. 拆分 `pipeline-runner.test.ts`，避免 Core 功能越多、单测试文件越不可维护。
5. 将真实 LLM E2E 与 mock-based quality baseline 分层，不要把 opt-in 真实服务测试当作默认质量门禁。

中期目标：

- Core 保持“功能内核”定位；
- Studio / CLI 只作为调用与展示层；
- PipelineRunner 最终退到编排 facade，业务逻辑沉入可独立测试的模块。
