# 作家蒸馏意见与建议

> 基于当前文风分析模块的扩展能力，讨论如何构建「作家蒸馏」体系——
> 将作家创作风格提炼为可复用、可注入、可对比的知识资产。
> 参考 SKILL.md 的结构化思路，为每位作家建立独立风格定义文件。

---

## 0. 基于当前项目的执行结论

本方案与 NoFusion 当前架构总体兼容，但原方案对“已有能力”和“最小改动范围”的判断偏乐观。当前项目已经具备作家档案、文本诊断、作家对比、调整建议、LLM 改写预览和作家档案应用到书籍等主链路，因此“作家蒸馏”不应再建立一套平行系统，而应作为现有 `AuthorStyleProfile` 之上的**可审核规则层**。

建议采用以下定位：

```
原始样本
  → StyleSourceDocument / analyzeStyle()
  → AuthorStyleProfile（统计事实层）
  → AuthorDistillation（人工可审核的规则层）
  → 应用到书籍时生成不可变快照
  → story/style_profile.json + story/style_guide.md + story/style_source.json
  → Writer / Reviser / Continuity 继续复用现有读取链路
```

### 0.1 当前项目真实能力对照

| 能力 | 当前状态 | 对蒸馏方案的影响 |
|------|----------|------------------|
| 作家档案创建、追加来源、重新分析、删除 | 已实现 | 蒸馏直接消费现有档案，不重复建设 |
| 多来源加权聚合 | 已实现 | 数值指纹以 `aggregateProfile` 为事实来源 |
| 文本与作家档案比较 | 已实现，当前比较 8 个核心指标 | 蒸馏后的验证应扩展现有 comparator，而不是新建重复函数 |
| 调整建议计划 | 已实现 | 蒸馏规则可转化为现有 suggestion，不另建调整系统 |
| LLM 改写预览、前后诊断、版本校验 | 已实现 | “按作家调整”已具备闭环，蒸馏主要增强提示约束 |
| 作家档案应用到书籍 | 已实现 | 应在此处固化蒸馏快照和来源版本 |
| Writer 读取 `style_guide.md` / `style_profile.json` | 已实现 | 第一阶段无需让 Writer 直接读取全局风格库 |
| `buildRhetoricAwarePrompt` 独立接口 | 已实现，但未接入章节写作主链路 | 不能据此认定 Pipeline 集成为“零代码修改” |
| 原始文本永久保存在 `sources/*.json` | 未实现 | 本地导入来源只保存分析结果和 hash，后续无法从中重新抽取原文选段 |
| 蒸馏状态、人工标定、版本发布 | 未实现 | 是本方案需要新增的核心能力 |

### 0.2 必须修正的原方案假设

1. `getAuthorProfile()` 是当前公开读取接口，文档中的 `loadAuthorProfile()` 是 `store.ts` 内部函数，不能直接从 core 包调用。
2. `sources/*.json` 保存的是 `StyleSourceDocument.profile`，不保存完整原文；代表选段必须在导入当次生成并保存，或者从 `samples/*.md` 读取互联网抓取样本。
3. `samples/` 与 `sources/` 是两个不同用途的目录：前者保存互联网抓取正文，后者保存结构化分析结果，不能合并理解。
4. Writer 已经读取书籍目录下的 `story/style_guide.md` 和 `story/style_profile.json`。更稳妥的集成方式是“应用作家档案时写入书籍快照”，不是每次写作跨目录读取 `style-library`。
5. Markdown 适合人工审核，不适合作为唯一机器配置源。阈值、状态和版本应以 JSON 为事实源，Markdown 作为可读投影。
6. 当前作家比较只有 8 个指标，尚未覆盖扩展句式、段落节奏、修辞细分和对话标签。文档中的全维度偏差检测属于新增开发，不是现成功能。

## 1. 什么是作家蒸馏

**作家蒸馏（Author Distillation）** 是指从一位作者的作品样本中，通过自动分析 + 人工标定，提取出其独特的创作风格特征，形成结构化、机器可读的风格定义文件。这些定义文件可以：

- 注入到 Pipeline 的 writer agent prompt 中，指导生成风格的模仿
- 用于多作者风格对比，辅助用户理解差异
- 作为「风格快照」版本化管理，追踪作家随时间的风格演变
- 作为社区共享资产，形成风格库

### 1.1 与现有 SKILL.md 的类比

| 维度 | SKILL.md | 作家蒸馏文件 |
|------|----------|-------------|
| 对象 | 编程语言/框架能力 | 作家创作风格 |
| 用途 | 指导 AI 生成符合规范的代码 | 指导 AI 生成符合作家风格的文本 |
| 结构 | YAML frontmatter + Markdown 指令 | YAML frontmatter + Markdown 风格指令 |
| 生成方式 | 人工编写 | 自动分析 + 人工标定 |
| 存储位置 | 项目级 `.github/skills/` | `style-library/authors/<id>/` |

### 1.2 蒸馏粒度

```
作家级（宏观）   → 余华.md, 王小波.md, 莫言.md
   ├── 时期级（中观）→ 余华_早期.md, 余华_转型期.md, 余华_成熟期.md
   │   ├── 作品级（微观）→ 活着.md, 许三观卖血记.md, 兄弟.md
   │   │   └── 章节级（片段）→ 活着_第3章.md
```

当前系统主要支持**作家级**和**作品级**分析，篇章级需要更细粒度的样本提取。

## 2. 当前已具备的蒸馏能力

### 2.1 自动分析管道

```
采集样本（互联网搜索 / 本地导入）
  → preprocessText()  清洗
  → analyzeStyle()    基础统计 + 扩展指纹
      ├── avgSentenceLength / stdDev
      ├── vocabularyDiversity
      ├── topPatterns
      ├── rhetoricalFeatures
      └── fingerprint
          ├── dialogueRatio / actionDensity / psychologicalRatio
          ├── sentenceTypeDistribution  ← 新增
          ├── paragraphRhythm           ← 新增
          ├── rhetoricBreakdown         ← 新增
          └── dialogueFeatures          ← 新增
  → detectDuplicateRhetoric()  修辞习惯检测
  → extractDimensionSamples()  代表性选段
  → mergeStyleProfiles()       多来源聚合
  → buildAuthorProfile()       档案构建
  → generateDistillation()     从统计事实生成待审核规则
  → reviewDistillation()       人工确认、禁用或修正规则
  → publishDistillation()      发布不可变版本
```

> **数据边界说明：** 当前 `addStyleSource()` 在完成分析后只持久化文本 hash、字数和分析结果，不持久化完整原文。因此 `extractDimensionSamples()` 应在文本导入当次执行，并将短选段及来源定位保存为独立 evidence 数据；不能假设以后可从 `sources/*.json` 恢复全文。

### 2.2 可蒸馏的特征维度（当前）

| 维度 | 数据来源 | 类型 | 蒸馏价值 |
|------|---------|------|:--------:|
| 句长分布 | `avgSentenceLength` + `stdDev` | 数值 | 判断作家偏好长句还是短句 |
| 词汇多样性 | `vocabularyDiversity` | 数值 | 词汇丰富度指标 |
| 对话占比 | `dialogueRatio` | 比率 | 对话驱动 vs 叙事驱动 |
| 动作密度 | `actionDensity` | 比率 | 动作描写密集程度 |
| 心理占比 | `psychologicalRatio` | 比率 | 内心独白倾向 |
| 口语化程度 | `colloquialismScore` | 数值 | 书面语 vs 口语化 |
| 修辞密度 | `rhetoricDensity` | 数值 | 修辞使用频率 |
| AI 腔风险 | `aiTellRisk` | 数值 | 自然度指标 |
| 句式分布 | `sentenceTypeDistribution` | 分布 | 陈述/疑问/感叹/祈使比例 |
| 段落节奏 | `paragraphRhythm` | 分布 | 短中长段落比例 |
| 修辞细分 | `rhetoricBreakdown` | 分布 | 各类修辞使用频率 |
| 对话细节 | `dialogueFeatures` | 复合 | 对话长度/频次/标签偏好 |
| 主要模式 | `topPatterns` | 文本 | 高频句式开头 |
| 修辞特征 | `rhetoricalFeatures` | 文本 | 修辞使用特征描述 |
| 示例选段 | `extractDimensionSamples` | 文本 | 各维度代表性段落 |

### 2.3 缺失但可蒸馏的特征（建议）

| 维度 | 提取方式 | 优先级 | 说明 |
|------|---------|:------:|------|
| **标点偏好** | 统计逗号/句号/引号/破折号使用频率 | P1 | 部分已存在 `punctuationRhythm`，可细化 |
| **叙事视角** | 统计第一/二/三人称代词频率 | P1 | 区分全知视角/限知视角 |
| **时间表达** | 统计时间词（曾经/正在/将要等）频率 | P1 | 区分过去时叙事/现在时叙事 |
| **情感基调** | 情感词典匹配积极/消极词汇比率 | P2 | 判断作家整体情感倾向 |
| **地域特征** | 方言/地域词汇识别 | P2 | 识别作家方言背景 |
| **意象偏好** | 高频意象词聚类（如余华偏爱「血」「雨」「夜」） | P2 | 核心风格标识 |
| **修辞手法搭配** | 比喻+拟人的共现频率 | P3 | 判断修辞使用习惯 |

## 3. 作家蒸馏文件的格式设计

### 3.1 文件路径

```
style-library/
└── authors/
    └── <author-id>/
        ├── profile.json              ← 已有：自动聚合的数值档案
        ├── sources/                  ← 已有：结构化来源分析，不含完整原文
        │   └── <source-id>.json
        ├── diagnostics/              ← 已有：诊断结果
        │   └── <diagnostics-id>.json
        ├── samples/                  ← 已有但仅覆盖互联网抓取正文
        │   ├── sources.json
        │   └── 20260609_<title>.md
        ├── index.md                  ← 已有：互联网样本索引
        └── distillation/             ← 新增：可审核蒸馏资产
            ├── current.json          ← 机器事实源，指向当前草稿或发布版本
            ├── current.md            ← 面向人工审核的 Markdown 投影
            ├── evidence.json         ← 短选段、来源 ID、位置和维度
            ├── overrides.json        ← 人工覆盖、禁用规则、备注
            └── versions/
                ├── 1.json
                ├── 1.md
                └── ...
```

不建议第一阶段拆成 `voice.md`、`vocabulary.md`、`syntax.md`、`devices.md` 四个文件。当前数据量不足以保证每个子文件都有稳定内容，多文件还会增加版本一致性问题。先采用 `current.json + current.md + evidence.json`，当单份文件超过约 500 行或出现独立编辑需求时再拆分。

### 3.2 蒸馏主文件格式（参考 SKILL.md）

```markdown
---
name: 余华风格定义
version: 1.0.0
type: author-distillation
language: zh
sourceCount: 3
totalChars: 12400
analyzedAt: 2026-06-09T12:00:00Z
tags: [中国当代文学, 现实主义, 先锋文学]
thresholds:
  metaphorRate: 2.0
  parallelismRate: 1.0
  dialogueRatio: 0.35
---

# 余华 — 文风蒸馏

> 基于《活着》《许三观卖血记》《在细雨中呼喊》三部作品的样本分析。
> 自动生成于 2026-06-09 | 人工标定建议见下方「待标定」章节。

## 1. 核心风格标识

### 1.1 数值指纹

| 维度 | 值 | 与基准对比 |
|------|:--:|:---------:|
| 平均句长 | 12.5 字 | 📏 短句为主，低于平均值 15.2 |
| 词汇多样性 | 42% | 🔼 高于平均水平 38% |
| 对话占比 | 35% | 💬 对话驱动 |
| 动作密度 | 22% | 🏃 中度动作描写 |
| 修辞密度 | 8% | 🎨 简洁克制 |

### 1.2 句式分布

- 陈述句: 72% — 叙事主体
- 疑问句: 15% — 高于平均，常用于角色内心独白
- 感叹句: 10%
- 祈使句: 3%

### 1.3 修辞习惯

- 比喻: 2.1 次/千字（低于平均 3.5）— 克制使用，多为生活化比喻
- 排比: 0.5 次/千字（显著低于平均）— 极少使用
- 口语化程度: 38% — 叙事口语化强，对话占比高

## 2. 风格注入指令

> 以下内容直接用于 writer agent 的 system prompt 注入。

### 2.1 叙事风格

- 使用简洁、冷静的叙述语调，避免过度修饰
- 句子以短句（8-15 字）为主，平均句长控制在 12-14 字
- 段落长度控制在 50-120 字，避免超过 200 字的长段落
- 对话占比保持在 30-40%，对话简洁自然，少用对话标签

### 2.2 修辞限制

- 比喻使用频率 < 2.5 次/千字
- 排比使用频率 < 1.0 次/千字
- 拟人使用频率 < 1.0 次/千字
- 避免使用夸张修辞
- 反问句式 < 1.0 次/千字

### 2.3 词汇倾向

- 多用口语化词汇，少用书面成语
- 避免西方化的长修饰语结构
- 使用具象词汇（涉及具体物象）而非抽象词汇
```

### 3.4 机器数据契约（新增，建议作为事实源）

Markdown frontmatter 适合阅读和导出，但 Pipeline、API 和版本校验不应依赖 Markdown 标题或自然语言解析。建议新增以下结构化契约：

```typescript
export interface AuthorDistillation {
  readonly authorId: string;
  readonly authorProfileVersion: number;
  readonly version: number;
  readonly status: "draft" | "reviewed" | "published" | "archived";
  readonly generatedAt: string;
  readonly reviewedAt?: string;
  readonly publishedAt?: string;
  readonly sampleAdequacy: "insufficient" | "limited" | "sufficient";
  readonly confidence: number;
  readonly rules: ReadonlyArray<DistillationRule>;
  readonly evidenceRefs: ReadonlyArray<string>;
  readonly warnings: ReadonlyArray<string>;
}

export interface DistillationRule {
  readonly id: string;
  readonly dimension:
    | "sentence-length"
    | "paragraph-rhythm"
    | "dialogue"
    | "action"
    | "psychological"
    | "sensory"
    | "vocabulary"
    | "rhetoric"
    | "punctuation"
    | "ai-tell";
  readonly mode: "target-range" | "prefer" | "avoid" | "instruction";
  readonly instruction: string;
  readonly targetRange?: { readonly min: number; readonly max: number };
  readonly confidence: number;
  readonly source: "automatic" | "manual";
  readonly enabled: boolean;
}
```

设计要求：

- `authorProfileVersion` 必须与生成蒸馏时的档案版本绑定。
- 新增或删除样本导致档案版本变化后，旧蒸馏应标记为 `stale` 或要求重新生成，不能静默继续使用。
- 自动生成规则不得覆盖人工规则；重新蒸馏时合并 `overrides.json`。
- `published` 版本不可原地修改，只能创建新版本。
- 规则 ID 必须稳定，使前端能够保留用户禁用、修改和审核结果。

### 3.5 Evidence 数据设计

代表性选段的作用是解释规则来源，不应直接把长原文嵌入 prompt。建议每条 evidence 控制在 40-120 字，并保存可追溯信息：

```typescript
export interface DistillationEvidence {
  readonly id: string;
  readonly authorId: string;
  readonly sourceId: string;
  readonly sourceName: string;
  readonly dimension: string;
  readonly excerpt: string;
  readonly start?: number;
  readonly end?: number;
  readonly lineNumber?: number;
  readonly textHash: string;
  readonly approved: boolean;
}
```

本地文件导入时，应在请求仍携带原文的阶段同步执行 `extractDimensionSamples()` 并保存 evidence。互联网抓取样本可从 `samples/*.md` 提取，但仍应校验来源 URL、文本 hash 。

## 4. 蒸馏流程设计

### 4.1 核心循环

```
[采集] 搜索/导入样本
  │
  ▼
[分析] analyzeStyle() → 数值指纹
  │
  ▼
[蒸馏] 自动生成蒸馏文件
  │  ├── <author-id>.md    主要风格定义
  │  ├── vocabulary.md     词汇偏好
  │  └── voice.md          语气特征
  │
  ▼
[标定] 用户人工审核
  │  ├── 确认数值是否合理
  │  ├── 补充人工观察
  │  ├── 标记代表选段
  │  └── 填写禁忌列表
  │
  ▼
[注入] writer prompt 集成
  │  ├── 发布版本 → 复制到书籍 story 目录
  │  ├── 数值范围 → style_profile / adjustment plan
  │  ├── 风格指令 → style_guide.md 的作家规则区块
  │  └── 仅使用已审核的短选段，且默认不直接注入原文
  │
  ▼
[反馈] 生成文本后再次检测
     ├── 计算生成文本的 fingerprint
     ├── 与目标作家特征对比
     └── 偏差反馈 → 调整注入强度
```

### 4.1a 推荐状态转换规则

| 当前状态 | 允许操作 | 转换条件 |
|----------|----------|----------|
| `draft` | 自动重建、编辑、禁用规则 | 档案存在且至少有一个 ready source |
| `reviewed` | 预览、试写、退回草稿 | 人工确认规则和 evidence，解决阻断警告 |
| `published` | 应用到书籍、导出、创建新版本 | 样本充分度至少为 limited，版本与档案一致 |
| `archived` | 查看、对比 | 被新版本替代或人工归档 |

当 `authorProfile.version !== distillation.authorProfileVersion` 时：

1. API 返回 `stale: true`。
2. 前端禁止直接发布旧草稿。
3. 已经应用到书籍的快照不自动变化，避免后续写作结果漂移。
4. 用户明确点击“更新书籍文风”后，才写入新快照。

### 4.2 自动蒸馏函数

```typescript
// packages/core/src/style-library/distillation.ts

export interface DistillationInput {
  readonly profile: AuthorStyleProfile;
  readonly sources: ReadonlyArray<StyleSourceDocument>;
  readonly evidence: ReadonlyArray<DistillationEvidence>;
  readonly previous?: AuthorDistillation;
}

export interface DistillationOutput {
  readonly distillation: AuthorDistillation;
  readonly markdown: string;
}

/**
 * 从已有作家档案生成结构化蒸馏草稿。
 * 生成器是纯函数，不直接写入文件，也不覆盖已发布版本。
 */
export function generateDistillation(
  input: DistillationInput,
): DistillationOutput {
  const adequacy = evaluateAuthorSampleAdequacy(
    input.profile.sampleStats,
    input.sources,
  );
  const automaticRules = buildAutomaticRules(
    input.profile.aggregateProfile,
    adequacy,
  );
  const distillation = buildDraftDistillation({
    profile: input.profile,
    rules: mergeManualOverrides(
      automaticRules,
      input.previous?.rules ?? [],
    ),
    evidence: input.evidence,
    adequacy,
    previous: input.previous,
  });

  return {
    distillation,
    markdown: renderDistillationMarkdown(
      distillation,
      input.profile,
      input.evidence,
    ),
  };
}
```

上述函数只负责生成草稿，不负责直接覆盖已发布版本。生成步骤还应包含：

1. 计算样本充分度和整体置信度。
2. 将每项指标转换为“范围”，避免把单点均值当作硬指标。
3. 对极端值进行保护，例如平均段落长度异常大时给出数据清洗警告。
4. 合并上一个版本的人工覆盖项。
5. 生成 JSON 事实源，再由同一份 JSON 渲染 Markdown，防止两者内容漂移。

### 4.3 蒸馏文件的生命周期管理

```
Absent（尚未生成）
  │  └── 用户点击生成，或在来源分析完成后明确触发
  ▼
Draft（待标定）
  │  ├── 自动生成的初稿
  │  ├── 含「待标定」标记
  │  └── 主要数值置信度 < 0.7 的部分标记为 ⚠️
  ▼
Reviewed（已标定）
  │  ├── 用户人工审核通过
  │  ├── 补充了人工观察和代表选段
  │  └── 可用于 writer prompt 注入
  ▼
Published（已发布）
  ├── 版本号递增
  ├── 可作为社区共享资产
  └── 可在多个项目中复用

任意状态下档案版本变化
  └── 标记 Stale（已过期），保留原版本，不自动覆盖人工内容
```

### 5.2 偏差检测

```typescript
export function checkStyleDeviation(
  generatedFingerprint: StyleFingerprint,
  targetFingerprint: StyleFingerprint,
): {
  deviations: ReadonlyArray<{
    dimension: string;
    expected: number;
    actual: number;
    deviation: number;  // 百分比偏差
    severity: "low" | "medium" | "high";
  }>;
  overallScore: number;  // 0-100, 越高越接近
} {
  const dimensions: Array<{ key: keyof StyleFingerprint; label: string; weight: number }> = [
    { key: "dialogueRatio", label: "对话占比", weight: 0.15 },
    { key: "actionDensity", label: "动作密度", weight: 0.10 },
    { key: "rhetoricDensity", label: "修辞密度", weight: 0.20 },
    // ... 更多维度
  ];

  let totalScore = 0;
  let totalWeight = 0;
  const deviations: Array<{
    dimension: string;
    expected: number;
    actual: number;
    deviation: number;
    severity: "low" | "medium" | "high";
  }> = [];

  for (const dim of dimensions) {
    const expected = targetFingerprint[dim.key] as number;
    const actual = generatedFingerprint[dim.key] as number;
    if (typeof expected !== "number" || typeof actual !== "number") continue;

    const deviation = expected > 0 ? Math.abs(actual - expected) / expected : Math.abs(actual);
    const dimScore = Math.max(0, 100 - deviation * 100);

    totalScore += dimScore * dim.weight;
    totalWeight += dim.weight;

    if (deviation > 0.3) {
      deviations.push({
        dimension: dim.label,
        expected,
        actual,
        deviation: Math.round(deviation * 100),
        severity: deviation > 0.5 ? "high" : "medium",
      });
    }
  }

  return {
    deviations,
    overallScore: totalWeight > 0 ? Math.round(totalScore / totalWeight) : 100,
  };
}
```

> **复用建议：** 当前已有 `compareWithAuthorProfile()`，不要新增第二个同职责的 `checkStyleDeviation()`。应扩展现有 comparator 的指标定义，使其支持扩展 fingerprint，并把不同指标的容差、权重集中到一份版本化配置中。

### 5.3 不建议立即实施的自动行为

- 不要因为一次章节偏差就自动覆盖正文。
- 不要把所有数值都作为硬阈值，否则会产生机械化文本。
- 不要默认注入大段作家原文作为 few-shot。
- 不要让全局作家档案更新后自动改变已经应用到书籍的文风。
- 不要在 Writer、Reviser、Style Rewriter 中分别拼接不同版本的作家提示词。

建议统一由一个 `renderDistillationInstructions()` 输出规则区块，再写入书籍 `style_guide.md`，避免三个 Agent 对同一档案产生不同解释。

## 6. 建议实施路线

### 总体原则


### Phase 1：蒸馏草稿生成与持久化（P0，1-2 天）

**设计原则：**
- 不新增任何检测逻辑（全部复用 `style-dimensions` / `semantic-duplication` / `store`）

```typescript
import type {
  AuthorStyleProfile,
  StyleSourceDocument,
} from "./models.js";

export interface DistillationInput {
  readonly profile: AuthorStyleProfile;
  readonly sources: ReadonlyArray<StyleSourceDocument>;
  readonly evidence: ReadonlyArray<DistillationEvidence>;
  readonly previous?: AuthorDistillation;
}

export interface DistillationOutput {
  readonly distillation: AuthorDistillation;
  readonly markdown: string;
}

/**
 * 生成作家蒸馏 Markdown 文件内容。
 * 全部数据来自已有分析结果，不做新计算。
 */
export function generateDistillation(input: DistillationInput): DistillationOutput {
  const adequacy = evaluateAuthorSampleAdequacy(
    input.profile.sampleStats,
    input.sources,
  );
  const automaticRules = buildAutomaticRules(
    input.profile.aggregateProfile,
    adequacy,
  );
  const rules = mergeManualOverrides(
    automaticRules,
    input.previous?.rules ?? [],
  );

  const distillation: AuthorDistillation = {
    authorId: input.profile.id,
    authorProfileVersion: input.profile.version,
    version: input.previous?.status === "published"
      ? input.previous.version + 1
      : input.previous?.version ?? 1,
    status: "draft",
    generatedAt: new Date().toISOString(),
    sampleAdequacy: adequacy.level,
    confidence: adequacy.confidence,
    rules,
    evidenceRefs: input.evidence
      .filter((item) => item.approved)
      .map((item) => item.id),
    warnings: adequacy.warnings,
  };

  return {
    distillation,
    markdown: renderDistillationMarkdown(
      distillation,
      input.profile,
      input.evidence,
    ),
  };
}
```

这里的 `evaluateAuthorSampleAdequacy()`、`buildAutomaticRules()`、`mergeManualOverrides()` 和 `renderDistillationMarkdown()` 应保持纯函数，分别测试。文件读写、版本落盘和状态转换放在 `distillation-store.ts`，不要混入生成器。

```typescript
export interface PerspectiveDistribution {
  readonly firstPerson: number;   // 第一人称代词占比
  readonly secondPerson: number;  // 第二人称代词占比
  readonly thirdPerson: number;   // 第三人称代词占比
}

const FIRST_PERSON = ["我", "我们", "咱", "咱们"];
const SECOND_PERSON = ["你", "您", "你们"];
const THIRD_PERSON = ["他", "她", "它", "他们", "她们", "它们"];

export function computePerspectiveDistribution(text: string): PerspectiveDistribution {
  let first = 0, second = 0, third = 0;
  // ... 使用 Set 和 includes 统计
  const total = first + second + third;
  if (total === 0) return { firstPerson: 0, secondPerson: 0, thirdPerson: 0 };
  return {
```

## 7. 风险与注意事项

### 7.1 样本数量与质量

- **系统当前 comparator 判定：** 0 来源或少于 1,000 字为 insufficient；少于 2 来源或少于 3,000 字为 limited；其余为 sufficient。
- **蒸馏发布建议门槛：** 至少 3 个相互独立的样本、总字数 15,000 以上；高可信发布建议 5 个样本、总字数 20,000 以上。
- **样本品类：** 应覆盖作者不同创作时期的作品，避免单一作品偏差
- **抓取质量：** `fetchUrl` 提取的正文可能存在噪声（广告、评论等），建议在蒸馏前进行人工审查
- **异常值保护：** 当平均段落长度、词汇多样性或转折密度明显异常时，应先怀疑导入了 JSON、日志、提示词或重复文本，而不是直接把异常值定义成作家风格。

### 7.2 风格演变

- 作家风格会随时间和阅历演变（如余华的先锋期→成熟期）
- 建议同一作家按时期维护多份蒸馏文件（`余华_1990s.md`、`余华_2000s.md`）
- Pipeline 注入时需明确指定目标时期

### 7.3 过度拟合风险

- 蒸馏文件中的数值限制如果过于严格，会导致生成文本失去自然度和创造性
- 建议将数值约束作为「参考基线」而非「硬限制」
- 修辞阈值应保留 20% 的弹性空间

### 7.4 版权与伦理

- 蒸馏文件可以包含大段的受版权保护的原文
- 示例选段应控制在 60-100 字的合理引用范围内
- 社区共享的蒸馏文件应仅包含风格特征数据，而非原文内容
- 默认提供“完全复制某位在世作家”的产品承诺，应描述为特征参考、节奏约束和文本诊断。
- 互联网抓取内容必须保留 URL、抓取时间和 hash。
- 删除作者档案时，应明确是否连带删除蒸馏版本、evidence 和已抓取样本。

### 7.5 数据一致性与迁移风险

- 旧 `profile.json` 可能缺少四个扩展 fingerprint 字段，生成器必须提供默认值或要求重新分析。
- `buildAuthorProfile()` 每次重建都会递增版本；仅因无实质变化而重新分析时，也可能导致蒸馏过期，需要后续考虑内容 hash。
- 当前 `sampleStats.sourceCount` 包含 failed source，而 `totalChars` 只累计 ready source，UI 和发布判断应明确采用哪一种。
- `samples/` 只覆盖互联网抓取路径，本地上传与粘贴文本没有原文归档，evidence 生成时机必须前移。
- `style_guide.md` 还包含通用写作方法论，应用作家蒸馏时不能整文件覆盖，应更新受控区块。

建议在 `style_guide.md` 使用稳定边界：

```markdown
<!-- author-distillation:start version=2 -->
## 作家蒸馏规则
...
<!-- author-distillation:end -->
```

再次应用时只替换该区块，保留通用写作方法论和用户手工内容。
