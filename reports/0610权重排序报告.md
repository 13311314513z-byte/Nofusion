## 详细分析

### P0 — 立即执行（2 项，~0.8 天）

#### 1️⃣ BookConfig 编辑表单 🔥 最高优先级

| 维度 | 评估 |
|------|------|
| **当前状态** | `BookConfigSchema` 已含 `volumeCount`/`currentVolume`/`keywords`/`targetAudience`/`serializationStatus` ✅；后端 API `buildStudioBookConfig` 和 `server.ts:1922 update` 均已处理这些字段 ✅；`BookDetail.tsx` 已渲染只读显示（line 613-635） ✅ |
| **缺失部分** | `BookCreate.tsx` 的 `BookCreateFormState` **缺少这 5 个字段**（line 14 只有 title/genre/platform/targetChapters/chapterWordCount/brief）；`BookDetail.tsx` settings 编辑区 **缺少对应输入控件**（line 743+ 只有 wordCount/targetChapters/status） |
| **用户价值** | 作者无法在创建时或创建后填写卷数、关键词、目标读者、连载状态，每次都要手动编辑 `book.json` |
| **用户反馈** | 0605 报告将其列为"ROI 最高的剩余统一项" |
| **技术成本** | 纯前端：BookCreate 表单加 5 个字段（~2h）+ BookDetail 编辑区加 4 个控件（~1h） |
| **依赖** | 无。后端已完全就绪，不需要改任何 API |
| **风险** | 极低——只是前端表单控件的增删 |
| **操作路径** | `BookCreate.tsx` → `defaultBookCreateForm()` + `BookCreateFormState` + 渲染；`BookDetail.tsx` → settings 区域加字段 |

---

#### 2️⃣ 章节风格漂移评分

| 维度 | 评估 |
|------|------|
| **当前状态** | `style-fingerprint.ts` 分析引擎已可用并用于风格指纹面板；但**无专用端点**将每章指纹与书籍绑定的 style profile 对比产出评分 |
| **用户价值** | 作者无法知道当前章节是否偏离了既定的文风——这是文风控制闭环中**最后一块拼图** |
| **技术方案** | 后端新增 `POST /books/:id/chapters/:num/style-score` → 调用 `analyzeStyle()` 分析本章 → 与 `style_profile.json` 对比 → 返回相似度评分 + 偏离维度；前端在章节详情或 Audit 分卡展示评分条 |
| **成本** | 后端 API ~0.3d + 前端展示 ~0.2d |
| **依赖** | 无。`style-fingerprint.ts` 已独立可用 |
| **风险** | 低——纯函数计算，无 IO 操作，不修改现有数据 |
| **复用性** | 完成后可用于：Audit 面板 → 章节详情 → Writer hook 自动检测 |

### P1 — 本周执行（4 项，~2 天）

#### 3️⃣ 模型能力标签

| 维度 | 评估 |
|------|------|
| **当前状态** | `ModelInfo` 类型已有 `maxOutput`/`contextWindow`；后端 `GET /services` 返回模型列表；Studio 服务列表页直接渲染模型 ID，**无能力标签**（如"适合写作"、"适合审计"、"性价比高"） |
| **用户价值** | 用户面对 10+ 模型不知如何选择，能力标签能显著降低选择成本 |
| **成本** | 纯前端：在模型选择器/服务列表增加基于 `maxOutput` + `contextWindow` 的自动标签渲染（~0.3d） |
| **风险** | 极低 |

#### 4️⃣ 批量连通性测试

| 维度 | 评估 |
|------|------|
| **当前状态** | `cli/src/commands/doctor.ts` 已存在，但只检查 Node 版本 + 运行时环境；`ServiceListPage.tsx` 可单服务测试但**无一键检测全部** |
| **用户价值** | 调试 API Key / Base URL 问题时用户需逐一点击每个服务，费时且易漏 |
| **成本** | CLI `--all-services` 模式 ~0.3d + Studio 按钮 ~0.2d |
| **风险** | 低 |

#### 5️⃣ CLI JSON 标准化

| 维度 | 评估 |
|------|------|
| **当前状态** | 所有命令支持 `--json`，但输出格式不统一：有的直接输出对象、有的包装在结果中、有的只输出 error。**无统一 `{status, error, data, meta}` 合约** |
| **用户价值** | 关注脚本化/自动化的用户无法可靠解析输出——需为每个命令写不同的解析逻辑 |
| **成本** | 封装统一的 `formatJsonOutput(result, error?)` 函数 + 逐个命令迁移（~0.5d） |
| **风险** | 中——需要修改所有 `--json` 分支，可能破坏已有脚本 |
| **建议** | 加 `outputVersion` 字段实现渐进迁移，同时在 CHANGELOG 标注 breaking change |

#### 6️⃣ HTML / MD TOC 导出

| 维度 | 评估 |
|------|------|
| **当前状态** | `ExportFormat` 仅 `"txt" | "md" | "epub"`；`writeExportArtifact` 无 HTML 分支 |
| **用户价值** | 作者需要 HTML 格式发布到网络平台，MD 无目录页不便于长篇导航 |
| **成本** | `writeExportArtifact` 增加 `html` 格式分支 + TOC 生成（~0.5d） |
| **风险** | 低——纯新增输出格式，不影响现有格式 |
