## 二、项目当前特征画像

### 2.1 产品形态

- **定位**：AI 辅助长篇小说创作工具（中文小说为主，支持英文）。
- **架构**：pnpm monorepo，三端分离：
  - `@actalk/inkos-core`：公共模型、Agent、PipelineRunner
  - `@actalk/inkos`（CLI）：命令行入口，适合自动化/脚本化
  - `@actalk/inkos-studio`：React SPA，可视化创作工作台
- **交互模式**：同步 REST + 异步 SSE 双通道；复杂写作任务统一走 `fire-and-forget + 事件流`。
- **质量工程**：Stage 0-4 已落地（意图合同化、Prompt Manifest、确定性增强、Beta Reader shadow、Issue Normalizer、patch 边界、跨章 issue 计数）。

### 2.2 能力成熟度

| 维度 | 成熟度 | 说明 |
|------|:------:|------|
| 工程组件 | 82% | 后端 API、CLI、Studio 核心链路完整 |
| Stage 0-4 代码 | 85% | 功能已实现并通过集成测试 |
| 质量验证闭环 | 42% | 缺少人工评测与足够校准数据 |
| 前后端对齐 | 90% | 110+ 端点中仅约 10 个未对齐或废弃 |
| 测试稳定性 | ✅ | 修复后三包包测试连续稳定 |

### 2.3 核心矛盾

1. **代码能力 > 证据能力**：Stage 0-4 工程代码已闭环，但无法证明这些改进真的提升了写作质量。
2. **后端能力 > 前端/CLI 入口**：蒸馏、诊断历史、修辞 aware prompt、段落去重、可读性评分等能力已后端实现，但用户触达不到。
3. **自动化评测 > 人工评测**：Beta Reader shadow 可规模化运行，但缺乏人类读者校准，胜率数字不可解释。

## 三、发展思路建议

### 3.1 总体思路：从“功能交付”转向“质量证明”

项目已从“堆功能”阶段进入“验证功能有效性”阶段。下一阶段不应再大量新增 Agent 或端点，而应：

1. **补数据**：把校准集从 12 对堆到 ≥30 对，引入人工双盲标签。
2. **补入口**：把已后端实现但用户触达不到的能力（蒸馏、诊断历史、可读性等）至少暴露一个最小可用入口。
3. **补闭环**：让每一次写作/审计/修订的结果都能被测量、被比较、被追溯。

### 3.2 三条发展主线

| 主线 | 目标 | 关键动作 |
|------|------|----------|
| **主线 A：质量证据化** | 证明 Stage 0-4 改进有效 | 补校准集、人工评测、Beta Reader 校准报告、修订收益实验 |
| **主线 B：能力入口化** | 让后端已有能力被用户用到 | 蒸馏工作台、诊断历史、修辞/去重/可读性小工具、CLI 命令化 |
| **主线 C：写作工作流化** | 把零散能力串成创作者日常 | 从“单章写作”扩展到“卷/书级规划→章节写作→审核→修订→发布”完整链路 |

### 3.3 阶段性判断

- **Stage 5 不应启动新 Agent**，直到质量证据补齐。
- **P2 前端半成品不应大规模投入**，但可以选择 1–2 个对写作流程有直接增益的能力做最小闭环。
- **CLI 与 Studio 能力差距需要收敛**，否则自动化脚本与可视化工具会长期割裂。

## 六、结合写作需求的进一步意见

### 6.1 对创作者最有价值的能力排序

基于“减少创作阻力”和“提升成品质量”两个维度：

1. **意图管理（Stage 0）**：`BookGoalsSection` 已支持，但意图与正文的对齐检查仍可强化——例如高亮“未兑现意图”的章节。
2. **风格蒸馏与迁移**：后端已完整，前端无入口。对长篇续作、同人创作、仿写训练非常有价值。
3. **诊断历史与可追溯性**：创作者需要看到“这本书/这个作家风格出了什么问题、之前怎么修的”。
4. **章节级可读性与重复检测**：直接影响阅读体验，且计算成本低，适合实时反馈。
5. **Beta Reader shadow 结果可视化**：目前结果只躺在 `story/beta-reader-shadow/` 里，应有一个面板展示“哪段被批评、为什么”。

### 6.2 对写作工作流的建议

当前工作流以“单章”为中心，建议向“书级”延伸：

```
书级规划（卷/章目标、伏笔网、角色弧）
   ↓
章节写作（意图 → 生成 → 人工审阅）
   ↓
质量审计（ continuity / AI tells / long-span fatigue / Beta Reader ）
   ↓
局部修订（patch 边界、Issue Normalizer、跨章计数）
   ↓
风格一致性检查（与目标作家/本书风格对比）
   ↓
导出/发布
```

**建议优先补齐的链路环节**：
- 书级伏笔追踪与回收率（当前有 hooks，但缺少“伏笔回收状态”视图）。
- 章节意图与正文的对齐评分（`post-write-validator` 已有，但未在前端显式展示）。
- 跨章 issue 计数的前端展示（目前只影响 severity，创作者看不到趋势）。

### 6.3 对数据/评测的建议

1. **不要把胜率当成唯一指标**：53.3% / 58.2% 只是方向性信号，需要配合人工标签才能解释。
2. **建立“坏案例”集**：收集 Beta Reader 与人工判断不一致的样本，用于校准和 prompt 迭代。
3. **固定评测协议**：明确每对 A/B 的生成模型、温度、prompt 版本、Git commit，确保可复现。
4. **控制变量**：在校准阶段，一次只变一个因素（模型 / 温度 / prompt），否则无法归因。

## 七、关键决策建议

| 决策 | 建议 | 理由 |
|------|------|------|
| 是否立即启动 Stage 5 新 Agent | **否** | 缺少人工评测与足够校准数据，新 Agent 无法验证 |
| 是否立即补齐蒸馏/诊断历史 UI | **是（最小闭环）** | 后端已完整，投入小、对创作者价值大 |
| 是否移除废弃端点 | **下一版本移除** | 当前已标记 `@deprecated`，保留一个版本过渡 |
| 是否扩展 CLI 高级命令 | **中期做** | 自动化脚本需要，但优先级低于数据闭环 |
| 是否把 Core 测试 forks 隔离纳入长期配置 | **是** | 已验证有效，避免未来再次出现模块污染 |

## 八、一句话总结
