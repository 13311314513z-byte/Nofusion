## 一、执行结论

NoFusion 当前已经具备完整产品轮廓，595 个 TypeScript/TSX 源文件、40 个 Studio 页面文件、约 142 个 Studio API 路由和 180 个测试文件共同覆盖了 Core、CLI、Studio、书籍工作台、文风分析、服务商、导入、审阅、导出和自动化等模块。

项目当前的主要矛盾已经不是“缺少页面”，而是以下四类闭环问题：

1. **发布闭环未成立**：类型检查通过，但测试、构建、打包和 lint 仍未全绿。
2. **数据事实源不统一**：Studio 部分编辑功能写 Markdown，Core 实际以结构化 JSON 为准。
3. **前端存在可配置但不生效的功能**：章节目标、伏笔编辑、部分文风调整没有完整进入 Core 决策。
4. **安全和可靠性边界不足**：导出格式、静态文件、网络暴露、路径和并发保存仍需收口。

因此 0612 迭代不应继续以增加独立页面和新 Agent 为主，应定义为：

> **发布门禁恢复 + 安全边界收口 + Core/Studio 唯一事实源统一 + 已有半成品功能闭环。**

当前综合功能完成度估算为 **74%**，内部试用准备度约 **68%**，公开发布准备度约 **55% 至 60%**。

完成本报告的“0612 必做范围”后，预计：

- 综合功能完成度：**82% 至 85%**
- 内部试用准备度：**88%**
- 公开发布准备度：**75% 至 80%**

### 3.2 Studio 五项真实失败

失败全部集中在异步建书流程：

1. 异步失败状态不能立即反映为 `error`。
2. LLM 配置错误没有在请求阶段返回 400。
3. 测试无法确认请求已进入统一 interaction runtime。
4. 空 blurb 的请求内容在响应时尚不可观测。
5. Ollama 无 Key 创建路径在响应时尚未执行到模型初始化。

这些失败不是五个独立业务 Bug，而是同一个契约问题：

> 接口从同步执行改为 fire-and-forget 后，响应、状态、错误、测试和调用方没有共同迁移到明确的 Job 模型。

### 3.3 CLI 两项失败

1. Doctor 本地 OpenAI 兼容端点测试超过 5 秒。
2. Studio 发布包缺少 `dist/api/index.js`。

### 3.4 构建性能现状

Studio 客户端构建可完成，但存在：

- 主入口约 `2746.57 kB`
- gzip 后约 `764.33 kB`
- 多个 500 kB 以上 chunk
- `use-api.ts` 同时被静态和动态导入，动态导入不能拆包
- 大量 Shiki 语言和主题进入构建产物

当前包体问题不阻断内部开发，但应在公开发布前治理。

## 五、0612 必须先处理的阻断项

### P0-01 修复 Studio 服务端构建

**问题**

`packages/studio/tsconfig.server.json` 继承根 `noEmit: true`，导致 TypeScript 校验成功但不输出文件。

**执行**

1. 覆盖 `noEmit: false`。
2. 明确 `declaration` 和输出目录。
3. 验证 `dist/api/index.js` 与 `index.d.ts`。
4. 执行真实 `npm pack` 检查包内容。

**预期工作量：0.25 至 0.5 人日。**

### P0-02 修复导出格式路径边界

**问题**

`export-save` 将用户传入的 `format` 拼入文件路径，只使用 TypeScript 类型断言，没有运行时白名单。

**执行**

- 只允许 `txt/md/html/epub`。
- 非法格式返回 400 和稳定错误码。
- 输出路径使用 Core 安全路径函数复核。
- 增加路径穿越测试。

**预期工作量：0.25 人日。**

### P0-03 修复静态资源路径边界

**问题**

静态资源路径直接 `join(staticDir, c.req.path)`，缺少 resolved-prefix 校验。

**执行**

- URL 解码后使用 `resolve/relative` 校验。
- 只允许读取 `staticDir` 子文件。
- 拒绝 `..`、编码穿越、绝对路径和符号链接逃逸。

**预期工作量：0.25 至 0.5 人日。**

### P0-04 收紧 Studio 网络暴露

**问题**

当前 `cors()` 全开放，服务端启动未明确绑定 `127.0.0.1`。在局域网或公网环境中，未认证 API 可触发写文件、删除和 LLM 调用。

**执行**

- 默认绑定 `127.0.0.1`。
- CORS 仅允许当前 Studio origin。
- 若需要远程访问，增加显式启动参数和认证 token。

**预期工作量：0.5 至 1 人日。**

### P0-05 修复异步建书契约

**执行**

1. 入队前同步验证请求和 LLM 配置。
2. 返回 `202 + jobId`。
3. 状态为 `queued/running/completed/failed/cancelled`。
4. 清理任务超时 timer。
5. 状态存储从模块全局移到 server 实例。
6. 修复 5 项 Studio 测试。

**预期工作量：1 至 1.5 人日。**

### P0-06 伏笔编辑改用 Core 结构化状态

**执行**

- Core 提供 Hook CRUD service。
- 写 `hooks.json` 后由 Core 生成 `pending_hooks.md`。
- 保留 `promoted`、依赖数组和诊断字段。
- 添加“Studio 修改后下一章可见”的链路测试。

**预期工作量：1 至 1.5 人日。**

## 十二、暂缓事项

0612 不建议投入：

- 实时协同编辑

- 全面重写 Hash Router
- 全量重做 Studio UI

原因是当前最有价值的工作仍是让已存在功能形成可信闭环。

## 十三、最终建议

0612 应以以下顺序推进：

1. **修复安全边界、Studio 构建和测试门禁。**
2. **修复异步建书，建立最小 Job 契约。**
3. **统一 Hook、Goal 和其他工作台编辑的 Core 事实源。**
4. **完成章节版本、冲突保护、AI 改写和撤销。**
5. **完成文风分析会话与作家蒸馏前端。**
6. **最后执行任务中心、E2E 和包体治理。**

当前项目不需要重新设计核心架构。最合理的路径是保留已验证的 Core Pipeline，在 Core 边界补充统一服务，让 Studio 从“直接改文件”转为“调用 Core 领域操作”，再用发布门禁和浏览器 E2E 固化这些契约。

如果严格按此顺序执行，项目可以在下一完整迭代结束时从“功能丰富的开发测试版本”推进到“状态可信、可恢复、可打包的内部 Beta”，并为后续公开测试版本建立稳定基础。
