## 七、当前 P1/P2 缺口

### P1-1：Scene Templates 路径与模型错位

**现象：**

- Studio API：`GET/PUT /books/:id/scene-templates` 读写 `story/sources/scene_templates.json`。
- Core Planner：`loadSceneTemplates()` 读取 `story/state/scene_templates.json`。
- Core Schema：`SceneTemplateSchema` 是 `type/location/atmosphere/props/routines/defaultCharacters/...`。
- Planner Loader：期望 `description/beats/mood/pacing/tags`。

**影响：** 用户在前端保存的场景模板不会被 Planner 读取；即使路径修正，字段也无法完整转成 Planner prompt 需要的模板描述。

**建议修复：**

1. 统一持久化路径，优先采用 schema 文档已有的 `story/sources/scene_templates.json`，修改 `scene-template-loader.ts` 读取该路径。
2. `buildSceneTemplatesBlock()` 改为使用当前 `SceneTemplateSchema` 字段：`location`、`atmosphere`、`props`、`routines`、`defaultCharacters`、`linkedEvents`、`notes`。
3. 增加一条集成测试：前端格式 JSON → `loadSceneTemplates()` → Planner prompt 包含地点/气氛/道具/惯例。

### P1-2：Voice Profiles 前端列表与分析结果不同步

**现象：**

- GET `/voice-profiles` 读取 `story/voice_profiles/index.json`。
- POST `/voice-profiles/analyze` 写入 `story/voice_profiles/<characterId>.json`。
- Writer 直接遍历目录读取 JSON 文件，所以写作注入有效。
- 前端 `BookVoiceProfilesSection` 调用 POST 后 `refetch()`，但 GET 不读取单文件，可能仍返回空列表。

**建议修复：**

1. 简单方案：GET 端点改为扫描 `story/voice_profiles/*.json`，排除 `index.json` 后聚合 profiles。
2. 或者 POST 写单文件后同步更新 `index.json`。
3. 增加 API 测试：POST analyze 后立即 GET，返回包含该 `characterId` 的 profile。

### P1-3：Plan Alternatives 没有用户选择闭环

**现象：**

- Core Planner 已能生成 alternatives。
- Runner 已传 `generateAlternatives: true` 并记录 `alternativesCount`。
- Studio `write-preview` 只返回 contextSummary/warnings，没有返回 alternatives。
- `WriteConfirmPanel` 只能“直接写章”，不能选择方案。

**建议修复：**

先不要让 `write-preview` 临时调用 LLM 生成 plan，避免预览页面变慢。更稳妥的路径是：

1. 新增 `POST /books/:id/plans/preview?chapter=N`，显式生成并持久化候选 plan。
2. `WriteConfirmPanel` 展示 primary + alternatives。
3. 用户选择后，将 selected plan id 传入 `write-next` 或写入 pending plan。

### P2-1：State Changelog 写入存在，但产品化不足

**现象：**

- `state-logger.ts` 写 `story/state/state_changelog.jsonl`。
- `state-changelog.ts` 另有根级 `state_changelog.jsonl` 设计。
- Runner 已记录 plan/chapter/audit。
- Studio 无 `/state-changelog` 读取端点，也无 Runtime 页面呈现。

**建议修复：**

1. 保留一个路径：推荐 `story/state/state_changelog.jsonl`。
2. 新增 `GET /api/v1/books/:id/state-changelog?limit=100`。
3. BookRuntimeSection 增加“状态变更日志”Tab。

### P2-2：BookConfig 扩展字段部分 UI-only

**现象：**

- `volumeCount/currentVolume/keywords/targetAudience/serializationStatus` 已在 Core schema、BookCreate、BookDetail、server 中接入。
- `genreTags/contentWarnings` 在 BookDetail UI 中出现，但 Core `BookConfigSchema` 和 server 保存逻辑未接。

**建议修复：**

要么删掉 UI 字段，要么补 schema + server 保存 + 创建流程。不要保留“看似可编辑但不持久化”的字段。

## 十、结论

当前项目已经不是“缺前端、缺端点”的阶段，而是进入“端点、前端、管线三者必须精确同源”的阶段。  

最核心的闭环标准应改成：

```text
前端可触达
→ API 契约与 Core schema 一致
→ 持久化路径唯一
→ Planner/Writer/Auditor 至少一个主链路消费
→ UI 能看到结果或失败原因
→ 有测试覆盖这个完整路径
```

按这个标准，当前最优先的真实缺口是 Scene Templates 和 Voice Profiles，不是继续新增 Agent。
