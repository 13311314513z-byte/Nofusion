# NoFusion 0607 项目推进建议

> 评估日期：2026-06-07  
> 评估范围：`0605阶段发展报告.md`、`0606nufusion项目推进评估.md`、`0607版本待修复问题.md`、`NoFusion最终发展方向报告.md`、`书籍模块强化.md`、`文本导入方案修改.md`、`文件预处理功能强化.md`，以及当前 `packages/core`、`packages/studio`、`packages/cli` 实现。  
> 评估重点：连续开发进度、前端模块与 Core 预设能力的连接程度、下一阶段功能发展顺序。

## 一、总体判断

NoFusion 已经完成从“基础写作 Pipeline”向“可观察、可治理、可导入”的增强阶段迁移，当前不再缺少主要功能框架。项目已经具备：

- 完整的多 Agent 写作、审计、修订和状态沉淀链路。
- Runtime Trace、审计历史、伏笔治理、章节元数据、风格指纹、Token 统计等可观察能力。
- 新建书籍资料包、存量书籍架构资料 plan/commit、安全角色合并和来源持久化。
- 500 万字符文本提取上限及 Core/API 分片基础。
- Studio 预处理预设、输入体检、阶段选择和风险计算基础。
- Studio、CLI、TUI 共用 Core 能力的整体架构方向。

当前阶段的主要矛盾已经从“缺功能”转为：

1. **Core、API、Studio、CLI 的功能完成度不一致。**
2. **前端存在重复定义 Core 契约和预设的情况，容易发生语义漂移。**
3. **部分功能已出现界面，但确认、校验、分片和错误语义尚未闭环。**
4. **0607 稳定性问题虽已修复一部分，但文件锁、静默降级和 Studio 工程验证仍有风险。**

因此，下一阶段不建议继续横向增加新页面或新 Agent。应先完成“契约收口、能力联动、稳定性闭环”，再进入时间线、人物声线和状态变更日志等长篇治理功能。

### 当前成熟度估计

| 层面 | 成熟度 | 判断 |
|---|---:|---|
| Core 写作与治理能力 | 85% | 主链成熟，测试覆盖较强，仍有锁和异常治理债务 |
| Studio 用户功能 | 75% | 页面覆盖广，但部分功能只完成展示或第一段调用 |
| Core/API/前端契约一致性 | 60% | 资料包较好，预处理和分片存在明显漂移 |
| CLI 与 Studio 对齐 | 65% | 资料包已对齐，预处理、统一 JSON、批量诊断未对齐 |
| 运行稳定性与可维护性 | 70% | Core 测试健康，Studio 类型边界和异步测试未完全通过 |

### 2.2 书籍资料包导入已形成跨层闭环

当前已实现：

- Core `FoundationSourceBundle`、用途分类、哈希去重、容量限制、上下文组装。
- `PipelineRunner.planFoundationImport()` 与 `commitFoundationImport()`。
- `supplement/rebuild` 模式及角色 merge/replace 行为。
- Studio 新建书籍附加多文件资料。
- ImportManager 存量书籍架构资料预览与提交。
- 服务端 plan 绑定、30 分钟失效和架构 revision 校验。
- CLI `book create --source` 与 `import foundation`。
- `story/sources` 来源持久化。

该功能线完成度约为 **85%**，剩余问题主要在产品闭环：

- BookCreate 中所有文件固定为 `auto`，未开放逐文件用途选择。
- ImportManager 未开放 Core 已支持的 `chapter/style` 分流选项。
- 缺少 `story/sources/index.json` 的前端来源管理视图。
- `rebuild` 虽展示可能移除角色，但缺少明确的二次危险确认。
- 缺少“已导入、重复、内容变化、重新导入”的来源级状态。

### 2.3 500 万字符与分片能力只完成到 Core/API

已实现：

- `MAX_CHARS = 5_000_000`。
- `ExtractedDocument` 的截断和分片元数据。
- `extractDocumentChunked()` 及 JSON/JSONL 分片入口。
- `/style/extract-text` 的 `maxChars/chunk` 参数。
- URL 导入 50 MB 限制和 60 秒超时。

未完成或存在风险：

- StyleManager 未读取 `truncated/totalChunks/chunkIndex`，没有加载剩余分片。
- 普通提取按 500 万字符计算 `totalChunks`，分片提取按 50 万字符计算，元数据口径不一致。
- JSON/JSONL “分片”当前仍先完整提取文本，再切片，不是真正的流式或增量解析。
- 每次请求指定分片都会重新上传和重新解析完整原文。
- 前端仍可能把超大文本直接放入 textarea。

该功能线完成度约为 **60%-65%**。在前端闭环前，不应宣称“超大文件分片导入已完整可用”。

### 2.4 文件预处理工作台处于半完成状态

已实现：

- 五个预设及纯函数状态模块。
- 原文、提取、清理、重排四阶段数据模型基础。
- 输入体检端点和多类 finding。
- 清理选项、阶段选择、删除率展示和分析输入切换。
- Core `preprocessText()`、`relayoutText()` 的复用。

当前关键缺口：

- 页面初始显示 `fidelity`，但清理开关初始值实际多数为 `true`，界面标签与执行行为不一致。
- 选择预设只更新清理选项，未完整更新 relayout 选项。
- `computeRemovalStats()`、`requiresConfirmation()` 和风险确认状态已定义，但没有接入实际执行流程。
- `/style/preprocess` 和 `/style/relayout` 仍接收宽泛 `Record<string, boolean | number>`，未使用白名单和数值边界。
- 未实现超限 413、未知字段 400 和统一错误结构。
- CLI 尚无 `style preprocess`。
- 多文件队列尚未实现。
- API adapter 从前端页面模块导入类型，导致 Studio 服务端 TypeScript 工程边界失败。
- 设计文档要求的 adapter/API 专项测试尚未形成完整覆盖。

该功能线完成度约为 **50%-55%**，应作为下一阶段第一个产品收口任务。

## 三、0607 稳定性问题复核

### 已修复或基本修复

- Scheduler 已由 `Promise.all` 改为 `Promise.allSettled`。
- `processBook()` 中提前退出已由 `return` 改为 `break`。
- 写章异常前已先取得下一章节号，不再固定传入 `0`。
- 空 `sourceBundle.contextBlock` 已增加日志。
- PipelineRunner 已在缓存淘汰和 `dispose()` 时调用 client 清理钩子。
- Scheduler 停止时会清理 PipelineRunner。

### 仍需继续处理

1. **文件锁仍不是原子陈旧锁接管。**  
   当前“重新读取后 unlink”只缩小 TOCTOU 窗口，没有消除竞争。建议使用带 nonce 的锁内容和原子 rename/compare 方案。

2. **Windows `EACCES` 判断存在反向风险。**  
   当前将 `EACCES` 视为进程不存在，可能删除一个仍存活但无权限探测的进程所持有的锁。应将“确知不存在”和“无法确认”分开处理。

3. **MemoryDB 仍在构造器中抛错。**  
   Pipeline 部分调用点已有 Markdown 回退并通过测试，但降级责任仍分散。建议提供统一 `tryCreateMemoryDB()` 或工厂。

4. **静默 catch 尚未系统治理。**  
   文件不存在等预期分支可安静回退；解析失败、写入失败、网络失败应进入结构化日志和运行记录。

5. **Studio 验证未全绿。**  
   当前 Studio 全量测试有 daemon start 响应时序失败和 webhook 私网安全测试超时；服务端 typecheck 因跨目录 import 未通过。

稳定性问题应与新功能并列为 P0，而不是继续作为“以后优化”。

## 五、下一阶段架构原则

### 5.1 Core 输出能力契约，前端不手写镜像

当前 StyleManager 自行镜像 `PreprocessOptions`，Hook 页面自行计算风险，服务页面自行决定展示字段。这会导致 Core 已更新而前端仍使用旧语义。

建议形成数据型契约：

- Core 输出可用选项、类型、默认值、范围、风险级别和版本号。
- API 负责权限、请求校验、错误语义和任务编排。
- Studio 根据契约渲染，不复制 Core 常量。
- CLI 使用同一契约生成参数校验和 JSON 输出。

预处理方案仍可保持“不增加 Core 清理算法”的边界；只需共享契约和能力描述，不必向 Core 增加新的文本规则。

### 5.2 所有高风险写入统一使用 plan/commit

资料包导入已经证明 plan/commit 模式有效。后续应复用到：

- 新书资料创建预览。
- 批量章节元数据修改。
- 真值文件结构调整。
- 风格档案应用。
- 时间线导入和状态修复。

plan 必须返回：

- 输入摘要和 hash。
- 将修改的权威文件。
- 不会修改的文件。
- 冲突、删除和高风险项。
- Core 能力版本和预设版本。

### 5.3 每次处理生成可审计回执

建议为导入、预处理、风格应用和批量评分保存统一回执：

```json
{
  "operation": "style-preprocess",
  "presetId": "conservative",
  "presetVersion": 1,
  "inputHash": "...",
  "outputHash": "...",
  "options": {},
  "warnings": [],
  "startedAt": "...",
  "completedAt": "..."
}
```

这能把“前端点过什么”和“Core 实际执行什么”连接起来，也为问题复现和撤销提供依据。

## 六、建议实施顺序

### P0：契约与稳定性收口，预计 2-3 天

#### 1. 完成预处理闭环

- 初始 `fidelity` 必须真实对应全部关闭。
- 预设选择同时应用 preprocess 和 relayout 配置。
- 正式接入删除率、风险等级和确认弹窗。
- API 使用明确 schema，拒绝未知字段和非法范围。
- 超过 500 万字符返回 413，不静默截断。
- 将共享类型移出 `pages`，修复 Studio server typecheck。
- 增加 UI、adapter、API 契约测试。

#### 2. 完成分片契约

- 统一普通提取和分片提取的 `totalChunks/chunkSize/truncated` 语义。
- StyleManager 支持加载指定分片、失败重试和受控预览。
- 大文本只渲染首部、中部、尾部抽样，不挂载完整 textarea。
- 避免每个分片重复上传并完整解析原文；可复用现有 planId/run-store 模式建立短期 extractionId。

#### 3. 修复稳定性剩余项

- 修正 Windows 锁存活判断。
- 完成原子陈旧锁接管方案。
- 统一 MemoryDB 工厂降级。
- 修复 Studio daemon start 和 webhook security 超时测试。
- 让 Core、Studio、CLI typecheck 与测试全部通过。

### P1：前端与 Core 预设深度联动，预计 3-5 天

#### 1. 建立统一“输入工作台”

将 BookCreate、ImportManager、StyleManager 中重复的文件选择和文本处理抽成共享能力：

```text
来源选择
  -> Core 文档提取
  -> 输入体检
  -> 用途分类
  -> 可选预处理
  -> 影响预览
  -> plan/commit 或送入分析
```

不同页面只配置最终去向：

- BookCreate：新书基础架构。
- ImportManager：存量架构、章节或文风。
- StyleManager：风格分析、作家档案或导出。

#### 2. 增加书籍资料来源管理

在书籍工作区新增“资料来源”子模块，读取 `story/sources/index.json`：

- 显示名称、用途、字数、hash、导入时间和模式。
- 标记重复、内容已变化和导入失败。
- 支持重新生成预览。
- 删除来源记录时明确“不回滚权威架构”。
- 可跳转到受影响的 outline、roles 和 hooks 文件。

#### 3. 模型能力标签与路由预设

Core 已有 service preset、model bank、context window、max output 和 temperature hint。Studio 应直接展示：

- 上下文窗口。
- 最大输出。
- 是否适合长篇写作、审计、快速分类、本地运行。
- 推荐 Agent。

再增加 `fast/creative/audit/local` 路由预设，映射到 `modelOverrides`，而不是要求用户逐 Agent 配置。

#### 4. 统一书籍健康视图

将以下能力聚合到书籍首页：

- 审计分数与问题趋势。
- Token 趋势。
- 风格漂移。
- 伏笔风险。
- 最近导入和预处理回执。
- 最近失败的 Pipeline 阶段。

Analytics 保留深度统计，BookOverview 提供行动入口，避免两个页面重复绘制相同数据。

### P2：长篇治理增强，预计 1-2 周

#### 1. Planner 伏笔建议

Core 根据 hook 的状态、半衰期、依赖、最近推进章节和当前卷，输出：

- 本章建议推进。
- 本章建议回收。
- 暂缓原因。
- 前置依赖未满足提示。

Studio 在章节规划和 Hook 页面展示建议，但首阶段不自动改写 Planner 结果。

#### 2. 时间线与状态变更日志

- 新增结构化 `story/timeline.md` 或 JSON 投影。
- Settler 每章写入 `state_changelog.jsonl`。
- Studio 提供按章节查看的状态 delta。
- 章节修订页显示正文 diff 与状态 diff，避免“正文修了但真值未同步”。

#### 3. 人物声线最小版本

不新增独立 Agent，先基于现有章节和角色卡生成：

- 对话句长。
- 常用词和称谓。
- 语气词比例。
- 标点和停顿习惯。
- 与最近章节的漂移提示。

输出 `voice_profiles.json`，在角色页和审计页只做提示，不作为强制质量门。

#### 4. 导出与自动化补齐

- Core 增加 HTML 和带目录 Markdown artifact。
- CLI JSON 统一为 `{status,error,data,meta}`。
- Doctor 支持所有已配置服务批量检测。
- 写章完成 Webhook 增加字数、审计摘要、风格漂移和 Token 信息。

## 八、验收标准

下一阶段完成时，应满足：

1. Core、Studio、CLI 类型检查全部通过。
2. Core、Studio、CLI 全量测试通过，异步测试无偶发超时。
3. 前端不再复制 Core 的默认值、范围和能力字段。
4. 同一预设在 Studio 与 CLI 产生相同 Core options。
5. 500 万字符文本不会创建多个完整 textarea，也不会出现分片总数不一致。
6. 高风险预处理必须预览和确认，原文永不被静默覆盖。
7. 资料导入可追溯到来源 hash、用途、预设和影响文件。
8. Foundation plan 取消或失效时不修改书籍权威文件。
9. 单个书籍或单个文件失败不影响同批其他任务。
10. 所有降级行为都有用户可读提示和结构化日志。

## 十、最终结论

NoFusion 当前已经具备较完整的产品能力，下一阶段不应继续以“增加功能数量”为目标，而应以“让已有能力真正贯通”为目标。

最优推进路线是：

1. 先修复 Studio 工程边界、预处理预设、分片契约和文件锁。
2. 再把 BookCreate、ImportManager、StyleManager 合并为共享输入能力。
3. 随后建设资料来源管理、模型路由预设和书籍健康总览。
4. 最后进入 Planner 伏笔建议、时间线、状态 delta 和人物声线。

完成上述收口后，NoFusion 才会从“功能丰富的写作工具”稳定进入“前端可操作、Core 可治理、过程可追溯、长期可持续”的长篇创作系统阶段。
