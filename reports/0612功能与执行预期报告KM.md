## 一、项目总览

NoFusion（InkOS）是一个面向中文网络小说的 AI 辅助创作系统，采用 pnpm monorepo 架构：

| 包 | 职责 | 规模 | 健康度 |
|----|------|------|--------|
| `packages/cli` | 命令行工具（30+ 命令） | ~4,000 LOC | 🟡 中等 |
| `packages/core` | 业务核心（Agent、Pipeline、State、LLM） | ~25,000 LOC | 🟡 中等 |
| `packages/studio` | 前端 React SPA + Hono 后端 API | ~20,000 LOC | 🟡 中等 |

**当前构建状态**：Core `tsc` ✅ | Studio client ✅ | Studio server ❌（`noEmit` 继承导致零输出）  
**当前测试状态**：CLI 5/171 失败 | Core 121/1294 通过 | Studio 67 失败  
**未提交改动**：15 个文件有工作区修改  
**已知 P0 安全漏洞**：2 处路径遍历 + 1 处任意文件写入 + 硬编码 API 密钥

## 三、关键阻塞项（执行前必须解决）

### Blocker-1：构建系统静默失败
- **问题**：`tsconfig.server.json` 继承根 `noEmit: true`，`build:server` 成功但零输出
- **影响**：无法分发 Studio 服务端；CLI publish 测试级联失败
- **修复**：`tsconfig.server.json` 添加 `"noEmit": false`
- **工作量**：5 分钟

### Blocker-2：P0 安全漏洞（3 处）
- **S-01** `export-save` 任意文件写入 — 白名单校验 `format`
- **S-02** 静态资源路径遍历 — 校验解析后路径在 `staticDir` 内
- **S-03** 硬编码 API 密钥 — 删除 `.env` 和 `.inkos/secrets.json`，轮换密钥
- **工作量**：各 10-30 分钟

### Blocker-3：测试失败（72 处）
- CLI 5 处：超时 + publish 因构建失败而失败
- Studio 67 处：server 断言错误 + 超时
- **修复顺序**：先修 Blocker-1 → 重新跑测试 → 定位剩余失败根因
- **工作量**：1-2 天

### Blocker-4：非原子文件操作
- **问题**：`writeFoundationFiles`、`saveChapter`、`memory-db` 批量替换均无事务/回滚
- **影响**：部分失败导致书籍目录/SQLite 处于腐败状态
- **修复**：staging + 原子重命名；SQLite `BEGIN/COMMIT`
- **工作量**：1-2 天

## 六、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| Phase 0 测试修复耗时超预期 | 中 | 阻塞后续所有阶段 | 先修 Blocker-1（构建），其余失败可并行定位 |
| `mammoth.js` DOCX 提取能力不足 | 高 | 功能一体验差 | 同步启动 docx-indexer 微服务方案，mammoth 作为 fallback |
| ChapterGoalCard 接入 Planner 改变生成风格 | 中 | 老用户不满 | 增加功能开关，默认关闭，用户可选启用 |
| 前端 i18n 提取工作量巨大 | 高 | Phase 3 延期 | 仅提取 Top 20 文件（覆盖 80% 用户界面），其余随日常开发顺手替换 |
| indexer 微服务增加部署复杂度 | 中 | 从纯 Node.js 变为 Node+Python | 提供 Docker Compose 一键启动；CLI 自动检测 Python 环境 |
| LLM 上下文长度超限（提示卡注入后） | 中 | Writer 生成失败或截断 | 提示卡内容经过 `context-filter` 预算控制，与现有 truth 共享配额 |
