## 一、🔴 P0：影响运行正确性 / 输出品质的债务

### P0-1：Fire-and-forget 写作/改写操作缺乏超时保护与回滚机制

**位置**: `server.ts` L2298 / L2316 / L3234 — `write-next`、`draft`、`rewrite`

```typescript
// write-next: 返回 { status: "writing" } 后完全脱离监控
withPipeline("write-next", await buildPipelineConfig(), async (pipeline) => {
  const result = await pipeline.writeNextChapter(id, body.wordCount);
  broadcast("write:complete", { ... });
}).catch((e) => {
  broadcast("write:error", { ... });
});
return c.json({ status: "writing", bookId: id }); // ← 客户端收不到最终结果
```

**风险**:
1. Pipeline 内部崩溃时，客户端只知道 `status: "writing"`，无法区分"仍在写"和"已崩溃"
2. 无超时机制——如果 LLM 调用挂起，该任务永久占用资源
3. 无回滚——如果第 3 步失败，前 2 步的文件变更不会回滚

**影响范围**: 3 个写作端点，是产品最核心的交互路径

**建议修复**:
- 为 fire-and-forget 任务添加 10 分钟 TTL + 状态轮询端点
- Pipeline 关键步骤增加 try/finally 清理逻辑
- 客户端通过 SSE 事件 + 状态轮询双重确认

### P0-2：`new StateManager(root)` 重复创建 12 次

**位置**: `server.ts` L1717(1) + L2280 + L3415 + L3430 + L3452 + L3509 + L3584 + L3607 + L3621 + L3637 + L3653 + L3674

**问题**:
```typescript
// 第 1 次（正确）
const state = new StateManager(root);

// 第 2-12 次（应复用上面的 state 变量）
const bookDir = new StateManager(root).bookDir(id);  // ← 每次都 new
const state2 = new StateManager(root);               // ← 又 new
```

`StateManager` 内部维护书籍缓存、索引缓存等，每次重新创建会：
1. 丢失已有的缓存——后续操作需重新读取文件
2. 增加不必要的内存分配和 GC 压力
3. 在并发场景下因多个 StateManager 实例可能产生缓存不一致

**影响**: 12 处冗余创建，每处浪费 ~500μs + 缓存失效

**建议**: 全部替换为复用顶层 `state` 变量：`state.bookDir(id)`

---

### P0-3：20+ 处 `await import()` 动态导入可静态导入的模块

**位置**: `server.ts` 约 20 处

```typescript
// 每次请求都执行异步 import（开销 ~1-5ms）
const { readFile } = await import("node:fs/promises");
const { join } = await import("node:path");
const { analyzeAITells } = await import("@actalk/inkos-core");
const { readGenreProfile } = await import("@actalk/inkos-core");
const { isNewLayoutBook } = await import("@actalk/inkos-core");
```

**问题**:
1. 这些模块完全可以静态导入（已在文件顶部有 `import { readFile } from "node:fs/promises"`）
2. 动态 `import()` 会触发模块解析、加载、初始化——每次请求额外 1-10ms
3. 部分调用在热路径上（如 `write-preview` 每章写作前触发）
4. 还阻碍了 Node.js 的 ESM 静态分析和 tree-shaking

**影响**: 每个请求额外 5-50ms 延迟（取决于模块大小）

**建议**: 全部改为静态 `import` 到文件顶部

## 二、🟡 P1：影响运行效率 / 体验的债务

### P1-1：Pipeline Runner `agentClients` LRU 缓存无 TTL 过期

**位置**: `runner.ts` L399-406

```typescript
const MAX_AGENT_CLIENTS = 20;
if (this.agentClients.size >= MAX_AGENT_CLIENTS) {
  const firstKey = this.agentClients.keys().next().value;
  const evicted = this.agentClients.get(firstKey);
  // dispose evicted...
}
this.agentClients.set(cacheKey, client);
```

**问题**: 仅按数量淘汰（LRU），无基于时间的 TTL。如果短时间内创建 20 个不同参数组合的 Agent，最早的可能已失效但不会被淘汰（只要不超过 20 个）。

**建议**: 添加 `Map<cacheKey, {client, createdAt}>` + 定时清理超过 TTL 的条目

---

### P1-2：server.ts 中 `loadRawConfig()` 每次请求读取磁盘

**位置**: `server.ts` 约 10 处调用

```typescript
async function loadRawConfig(root: string): Promise<Record<string, unknown>> {
  const raw = await readFile(join(root, "inkos.json"), "utf-8");
  return JSON.parse(raw);
}
```

**问题**: 配置文件在进程生命周期中极少变化，但每个配置相关请求都重新读取+解析 JSON

**影响**: 不必要的 I/O 开销，在多次并发请求时会明显

**建议**: 添加内存缓存（读取时检查文件 mtime，变化时刷新）

---

### P1-3：`/api/v1/agent` 端点内联 ~800 行逻辑

**位置**: `server.ts` L2405-2970

**问题**: 整个 Agent 聊天处理逻辑（模型解析、多服务尝试、fallback、book-create、session migrate）全部内联在一个 handler 中。这是 server.ts 中**最复杂的单个端点**。

**影响**:
- 无法独立测试 Agent 逻辑
- 修改任何环节都需要理解全部 800 行
- 如果此端点出问题，是整个 Studio 最核心的交互路径崩溃

**建议**: 提取为独立模块（类似 routes/style.ts 的做法）

---

### P1-4：`/api/v1/books/:id/chapters/:num/style-score` 逐章计算风格分

**位置**: `server.ts` L2113-2170

**问题**: 每次请求都重新读取章节文件 + style_profile.json + 计算 9 维指纹。如果用户连续点击"计算全部文风分"按钮，会触发 N 次独立计算。

**建议**: 
- 添加章节级指纹缓存（内容 hash 作为 key）
- "计算全部"时批量 Promise.all 而非串行

---

### P1-5：`runner.ts` `loadChapterContent()` 每次读取磁盘

**位置**: `runner.ts` L780-890

**问题**: 写作流程中多次调用 `readFile(readdir(...))` 读取章节内容，同一文件可能在不同阶段重复读取。

**建议**: 在 PipelineRunner 实例中维护章节内容缓存（LRU + mtime check）

---

### P1-6：`/api/v1/books/:id/detect-all` 串行检测所有章节

**位置**: `server.ts` L3276-3302

```typescript
const results = await Promise.all(
  mdFiles.map(async (f) => {
    const content = await readFile(...);
    return analyzeAITells(content);
  }),
);
```

**问题**: `Promise.all` 并行虽好，但如果书籍有 100 章，同时触发 100 个 `analyzeAITells` 可能耗尽内存。

**建议**: 使用 `p-limit` 或手动限流（如最多 5 个并发）

---

### P1-7：`/api/v1/project` GET 端点缺少响应缓存

**位置**: `routes/project.ts`

**问题**: project 信息变化频率极低，但每次 Dashboard 加载都请求。可添加 `ETag` / `Cache-Control` 头。

**建议**: 基于 `inkos.json` 的 mtime 生成 ETag，返回 304 而非重新序列化

---

### P1-8：Studio 前端无请求去重/防抖

**位置**: `use-api.ts` `useApi` hook

**问题**: 组件快速挂载/卸载（如 Sidebar 切换书籍）时可能触发多次相同请求。虽然有 AbortController，但请求已经发出。

**建议**: 添加请求级去重 Map（相同 URL + params 的 in-flight 请求共享 Promise）

## 三、🟢 P2：影响可维护性的债务

### P2-1：server.ts 仍有 ~40 个内联路由未提取

19 个路由模块已提取，但 ~40 个端点仍滞留 `createStudioServer()` 闭包内。包括核心的 write/draft/revise/export/roles/fanfic/radar 等。

**建议**: 按域分批提取（优先级: agent > write > export > roles > fanfic > radar > doctor）

### P2-2：`runner.ts` 4050 行——单体 Agent 工厂

包含 24 个 public 方法 + 30 个 private 方法 + 10 个 Agent 创建逻辑。虽然"高内聚"辩护成立，但 4050 行仍然影响 IDE 响应速度和代码审查效率。

**建议**: 继续提取 stage 实现（已启动 Phase 1，继续 Phase 2-3）

### P2-3：多个 `await import("@actalk/inkos-core")` 动态导入核心模块

除了 server.ts（P0-3），routes 模块中也有动态 import 模式。核心模块应完全静态导入。

### P2-4：`use-api.ts` 同时静态和动态导入

Vite 构建告警：被 50+ 文件静态导入，同时被 `ChapterReader.tsx` 动态导入。动态导入不会产生新的 chunk（模块已在主 bundle 中）。

**建议**: 将 `rewriteRhetoric` 等仅少数页面使用的函数独立文件

### P2-5：StyleManager.tsx ~1900 行

虽然已通过 Tab 组件拆分，主文件仍包含大量状态管理 + Tab 切换 + 共享逻辑。

**建议**: 将状态管理提取到独立 hook（`useStyleManagerState`）

### P2-6：缺少请求体大小限制

`server.ts` 中几乎所有 POST/PUT 端点都接受 `await c.req.json()` 无大小限制。恶意或意外的大 JSON body 可导致内存耗尽。

**建议**: 添加全局 body 大小中间件（如限制 10MB）+ 按端点细化（文本导入可放宽）

### P2-7：`write-next` 和 `draft` 端点共用一个 `buildPipelineConfig`

每次调用都创建新的 `PipelineRunner` + `LLMClient` + `Logger`。如果用户在短时间内连续触发写作，会创建多个并发的 Pipeline。

**建议**: 为每本书维护一个 Pipeline 实例池，避免重复创建/销毁

### P2-8：`/api/v1/sessions` 端点无分页

`listBookSessions` 返回全部 session。如果用户积累了大量 session，响应可能过大。

**建议**: 添加 `?limit=&offset=` 参数

### P2-9：缺少结构化日志格式

所有 `console.error/log` 都是非结构化字符串。在生产环境中难以解析和告警。

**建议**: 使用现有的 `createJsonLineSink` + Logger 输出 JSON 行

### P2-10：缺少 API 响应时间监控

无任何端点级别的响应时间记录。性能退化无法及时发现。

**建议**: 添加 Hono 中间件记录每个请求的 `{method, path, status, duration}`
