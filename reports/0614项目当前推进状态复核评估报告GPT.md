## 三、分 Stage 严格评估

### Stage 0：意图合同化，92%

已完成 Schema、Core/Studio 共用合同、revision/status、章节元数据绑定和生成后精确 revision 确认。

剩余问题：

- 缺少 Runner 并发场景集成测试：生成过程中创建新 revision 时，只能确认生成起点 revision。
- `load -> map -> save` 仍是整文件覆盖，极端并发写入下缺少原子更新或乐观锁。

结论：功能主体可用，不应因缺少关键并发测试直接计为 100%。

### Stage 1：评测与运行追踪，42%

已完成：

- 成对偏好指标与报告脚本；
- 探索集清单 15 项，覆盖五种章节职能；
- 保留集清单 12 项；
- 自动生成基线报告的脚本与示例输出。

未完成：

- 当前 CSV 只有 **12 个唯一 A/B 对**，低于要求的 30 对；
- 当前只有 **1 个自动模拟 reader**，未达到至少 6 人、3 类角色；
- 探索集和保留集当前主要是 manifest，未形成可复查的完整输入、A/B 正文和运行元数据包；
- 基线报告未记录完整 Git commit、provider、模型版本、temperature、Token、延迟、重试和人工修改；
- 报告中的 Stage 5 建议仍为占位符；
- `写作质量基线报告.md` 与 `baseline-report-detailed.md` 的总体胜率分别为 53.3% 和 58.2%，统计口径未统一。

结论：工具完成度高于数据完成度，当前结果只能作为管线演示，不能作为质量决策基线。

### Stage 2：Prompt Manifest Lite，78%

已完成：

- Manifest/Fragment 数据结构、Token 预算和 prompt hash；
- Planner、Continuity、Writer 的实际装配接入；
- dropped fragment 日志和组件接入文档。

主要缺口：

- Writer 将整个 creative/observer/settler user prompt 标记为 `optional: true`。超预算时可能只向模型发送 system prompt，丢失章节任务、正文或状态更新输入。
- Fragment 粒度仍主要是“整个 system + 整个 user”，尚不能解释规则、意图、角色声音、记忆等内部来源分别占用多少 Token。
- 缺少接入前后 prompt 等价快照和“必需输入绝不丢弃”的 Agent 级测试。
- Manifest 仅写日志，尚未进入可关联章节、模型和运行 ID 的 trace 资产。

结论：已从“日志包装”升级为真实装配，但目前的超预算降级策略存在 P1 风险。

### Stage 3：确定性质量增强，82%

已完成：

- `AuditIssue`、`IssueNormalizer`、fixScope 路由；
- 手动和自动修订边界拒绝；
- 跨章计数持久化及阈值执行顺序修复；
- 段落范围和 evidence 文本锚定；
- Genre Promise 数据结构和独立检查函数。

主要缺口：

- evidence 在全文未找到时只输出 warning，原错误 location 仍保留在 issue 中，后续 Reviser 仍可能修改错误段落。
- 跨章签名使用 `source + category + description 前 40 字`，且在 category 标准化前计数；LLM 轻微改写描述就会中断“连续出现”链。
- 自动修订边界、跨章持久化和 evidence 未命中的运行级场景缺少专门回归测试。
- Genre Promise 检查已从生产管线撤下，只有模型和独立测试，不能计作完整生产验收。

结论：确定性防护已明显增强，但跨章识别和错误证据降级仍不够可靠。

### Stage 4：Beta Reader 影子评测，55%

已完成：

- 证据型 Observation 和 `BetaReaderOutput.modelInfo`；
- `off/shadow/advisory/actionable` 模式；
- 默认 `off`；
- Beta Reader 失败不阻断写作；
- shadow 文件持久化；
- 校准报告生成脚本。

主要缺口：

- `betaReaderModelFamily` 虽在 Schema 中定义，但 CLI `createRunnerConfig()` 和 Studio Runner 均未传入，因此运行时检查通常不可达。
- 即使传入，相同模型家族也只记录 warning，不会强制切换、跳过或失败，不满足“Writer 与 Reader 使用不同模型家族”。
- 比较依据是主模型 `config.model` 和字符串前缀，不是 Writer/Beta Reader 实际 resolved override，可能判断错误。
- shadow 未记录 Git commit、Writer prompt hash、输入版本和完整生成参数。
- run ID 为“毫秒时间戳 + 正文前 256 字 hash”；同毫秒、同正文会生成同名文件，理论上仍可覆盖。
- 当前工作区未发现真实 `story/beta-reader-shadow/` 样本，也没有实际 Beta Reader 校准报告。
- 校准脚本主要统计 judgment 分布和一致性，尚未输出执行方案要求的 A/B 选择准确率、Pearson、Spearman 及其置信区间。
- 当前 12 对单 reader 自动模拟数据不满足 30 对和人类对照要求。

结论：Agent 和脚本骨架可用，但还没有形成可信的影子评测闭环。

### Stage 5-6：0%，保持暂缓

Stage 5 需要从真实基线和 Beta Reader 校准结果中只选择一个专项，并用保留集证明收益。当前数据无法支持可靠归因，不应提前启动多项高成本创作架构。

## 四、当前 P1 问题

### P1-1：Writer 超预算时可丢弃完整 user prompt

位置：`packages/core/src/agents/writer.ts:298-305`、`641-648`、`724-731`

影响：

- Creative 可能不知道本章具体写作任务；
- Observer 可能拿不到待分析正文；
- Settler 可能拿不到观察结果和待更新状态；
- 调用仍可能返回看似合法内容，属于静默质量退化。

建议：

- 完整任务正文必须 `optional: false`；
- 将可裁剪的背景材料拆为独立 optional fragments；
- 超预算且必需 fragments 已超过预算时明确失败或先压缩上下文，不能静默删除任务。

### P1-2：Beta Reader 异构模型约束未真正生效

位置：`project.ts:99-107`、`cli/src/utils.ts:157-174`、`studio/src/api/server.ts:1748-1768`、`runner.ts:2168-2183`

影响：

- CLI/Studio 未透传 `betaReaderModelFamily`；
- 相同模型家族仍会继续评测；
- 自我偏好会污染校准数据。

建议：

- 基于 `resolveAgentLLMConfig("writer")` 和 `resolveAgentLLMConfig("beta-reader")` 的实际 provider/model 判断；
- 在 shadow/advisory/actionable 启动前执行强校验；
- 不同家族无法满足时跳过并记录结构化原因，而不是只 warning；
- 补 CLI、Studio 和 Runner 三层测试。

### P1-3：Shadow 尚未达到严格 append-only 与可复现实验要求

位置：`runner.ts:2190-2210`

影响：

- 极端情况下同名覆盖；
- 缺少 Git commit、Writer prompt hash 和输入版本，无法重建某次评测；
- 校准报告无法判断样本来自哪次代码和 Prompt。

建议：

- 使用 `randomUUID()` 或单调 ULID；
- 以 exclusive create 写文件，冲突时重试；
- 持久化 commit、Writer/Reader resolved model、两个 prompt hash、temperature、输入版本和章节内容 hash。

### P1-4：evidence 未找到时错误 location 仍进入修订

位置：`runner.ts:2396-2423`

影响：

- `anchorEvidenceText()` 已报告 notFound，但 issue 未移除 location；
- Reviser 仍可能将不存在的证据映射到错误段落，削弱 patch boundary 的保护价值。

建议：

- evidence 全部未找到时移除 location，并将 issue 降为 advisory；
- blocking issue 应转入人工复核，不能自动修订；
- 使用 issue ID 映射 relocation，避免仅按相同 evidence 字符串更新错 issue。

### P1-5：跨章重复问题签名对 LLM 文案变化过度敏感

位置：`issue-persistence.ts:75-103`、`issue-normalizer.ts:168-180`

影响：

- 同类问题只要描述前 40 字变化，就被视为新问题；
- category 同义词在计数前尚未标准化；
- 实际运行中连续问题可能长期不升级。

建议：

- 先标准化 source/category，再生成稳定 `issueFingerprint`；
- 优先使用规则 ID、维度 ID、角色/线程 ID 等结构化字段；
- 无稳定 ID 时再使用规范化关键词或 embedding/相似度辅助；
- 增加“不同描述、同一语义连续三章”的测试。

## 六、建议执行顺序

### 第一批：关闭 P1，预计 1-2 天

1. 修复 Writer 必需 fragment 被丢弃。
2. 打通并强制 Beta Reader 异构模型配置。
3. 修复 evidence notFound 降级行为。
4. 改造稳定 issue fingerprint。
5. 完善 shadow run ID 和实验元数据。

### 第二批：补运行级回归，预计 1 天

1. 生成期间新增 intent revision 的并发确认测试。
2. 自动修订越界后正文、快照和审计状态不变测试。

1. 建立至少 30 个有效 A/B 对，建议 50 个。

## 七、最终判断

当前项目不是“Stage 0-4 全部完成”，而是：

> **Stage 0 基本收口，Stage 2/3 进入可靠性修复，Stage 1/4 等待真实数据闭环。**

工程骨架已经足以支持下一阶段，但真实质量证据仍弱。应先完成 P1 修复与运行级测试，再投入真实双盲基线和 Beta Reader 校准。完成这些工作后，项目才具备从“多 Agent 功能集合”进入“可证明质量收益的创作系统”的条件。
