## 二、稳定性问题逐项对照

对照 `0607项目推进建议.md` 第三节"仍需继续处理的稳定性问题"：

### 2.1 文件锁 → ✅ 已升级为原子操作

**原问题**："当前'重新读取后 unlink'只缩小 TOCTOU 窗口，没有消除竞争。"

**当前实现**（`packages/core/src/state/manager.ts:95-160`）：已重构为完整的原子锁方案：
1. 创建临时锁文件 `.write.lock.{nonce}`（含 pid + timestamp + nonce）
2. 以 `open(lockPath, "wx")` 独占创建目标锁文件
3. 成功后 `rename(tmpLockPath, lockPath)` 原子替换
4. `EEXIST`/`EPERM` 时读取锁数据判断是否陈旧
5. 陈旧时先二次校验（TOCTOU 缩减）再 unlink 重试

**结论**：✅ 已修复，方案合理。`rename` 在 POSIX 和 Windows 上均为原子操作。

---

### 2.2 Windows EACCES 判断 → ⚠️ 刚修复，需验证未回滚

**原问题**："当前将 `EACCES` 视为进程不存在。"

**当前实现**（`packages/core/src/state/manager.ts:188-201`）：复检发现 **代码曾被回滚**（用户或其他工具修改过此文件），已重新应用修复：

```typescript
if (code === "ESRCH") return false;  // 确知不存在
// EACCES 不再返回 false，而是继续到 return true（视为存活）
return true;
```

**结论**：⚠️ 已重新修复。需监控以免后续合并时再次回滚。建议将此方法加入不受格式化工具影响的"受保护代码"列表。

---
