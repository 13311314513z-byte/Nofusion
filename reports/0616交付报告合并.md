### 分歧 2：G8 genreTags/contentWarnings 闭合 vs 断裂

| 报告 | 原判 | 依据 |
|------|:--:|------|
| DS | ✅ 闭合 | `book.ts:71-72` Schema 有字段，`server.ts:2215-2216` PUT 处理 |
| KM | ❌ 断裂 | curl 实测 PUT 后字段被静默丢弃 |
| GPT | — | 未专项核验 |

**代码验证**（逐行审查 `cleanStringArray` + PUT handler + `saveBookConfig`）：

```typescript
// server.ts:2191-2198 — cleanStringArray 逻辑
const cleanStringArray = (value: unknown): string[] | undefined => {
  if (Array.isArray(value)) {
    const arr = value.filter((v): v is string => typeof v === "string")
      .map((v) => v.trim()).filter(Boolean);
    return arr.length > 0 ? arr : undefined;  // ← 空数组→undefined
  }
  // ...string parsing...
  return undefined;
};

// server.ts:2215-2216 — PUT 合并
...(cleanStringArray(body.genreTags) !== undefined
  ? { genreTags: cleanStringArray(body.genreTags) } : {}),
```

**关键发现**：

| 场景 | 输入 | cleanStringArray 返回 | 合并结果 |
|------|------|:---:|------|
| 正常发送标签 | `["标签1", "标签2"]` | `["标签1", "标签2"]` | ✅ 正常保存 |
| 发送空数组 | `[]` | `undefined` | ❌ 字段不合并（旧值保留） |
| 不发送字段 | `undefined` | `undefined` | ✅ 字段不变（正确） |
| 发送含空字符串 | `[""]` | `undefined` | ❌ 字段不合并 |

**深坑**：用户发送 `[]` 意图清空标签时，`cleanStringArray` 返回 `undefined`，扩展运算符产出 `{}`，导致 `genreTags` 完全不被合并到 `updated` 对象中——旧值被保留，前端看到"已保存"但实际未变。KM 发现的"静默丢弃"现象确认为此边界 case。

**同时验证**：`state.saveBookConfig(id, updated)` 使用 `writeFile` 直接覆写 `book.json`，`updated` 对象中的字段会正确持久化（经 `BookConfigSchema.parse()` 校验）。`genreTags`/`contentWarnings` 在 `BookConfigSchema` 中均为 `.optional()`，允许缺失不报错。

**✅ 最终裁决：代码已修复（主线路径正常），但存在「空数组语义歧义」边界 bug。DS 代码存在性判断正确，KM 运行时发现有效。需修复 `cleanStringArray` 对空数组的处理。**

**严重度：P1（主线可用，边界 case 需修复）。** 修复方案：空数组应被允许清除字段 → 区分"未发送"和"发送空数组"。

### 分歧 3：CLI 测试失败数量

| 报告 | 原判 | 依据 |
|------|:--:|------|
| DS | 13 fail | `cli-integration.test.ts` — `ERR_MODULE_NOT_FOUND` |
| KM | 1 flaky fail | `publish-package.test.ts` — workspace 协议 |
| GPT | 冷启动全 fail | CLI 集成测试找不到 `core/dist/index.js` |

**代码验证**（`cli-integration.test.ts` 行 35-40）：

```typescript
// 子进程通过 node 直接启动 CLI 入口
function run(args: string[], options?: { env?: Record<string, string> }): string {
  return execFileSync("node", [cliEntry, ...args], {
    cwd: projectDir, encoding: "utf-8",
    env: buildTestEnv(options?.env),
    timeout: CLI_PROCESS_TIMEOUT_MS,
  });
}
```

**根因分析**：`cliEntry` 指向 `dist/index.js`（编译产物），它 import `@actalk/inkos-core`。在 pnpm workspace 中，`node_modules/@actalk/inkos-core` 是一个 symlink。vitest fork 池中的 `execFileSync("node", ...)` 子进程**不共享 vitest 的模块解析上下文**，导致 `ERR_MODULE_NOT_FOUND`。

三种不同失败表现均源于同一机制——只是运行条件不同：
- DS `pnpm -r test`：pretest build core dist → CLI 单元测试通过 → 集成测试子进程解析失败
- KM `pnpm --filter @actalk/inkos test`：可能触发了不同的 build 路径 → 仅 publish-package 失败
- GPT 冷启动 `pnpm test`：未 build core dist → 所有集成测试失败

**✅ 最终裁决：❌ CLI 集成测试子进程模块解析与 pnpm workspace 不兼容。三方描述的是同一问题的不同表现。P0 交付阻断。**

**严重度：P0（阻塞 CLI 独立分发包）。**
