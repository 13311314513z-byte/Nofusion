# InkOS LLM 写作参数调试报告（0608）

> 参照 SillyTavern 等主流创作前端可调的文本生成参数列表，对 InkOS 当前已暴露与未暴露的 LLM 控制节点进行全面 audit，为后续参数面板扩展提供依据。

---

## 一、参考标准：SillyTavern 参数谱系

SillyTavern 作为面向创意写作/角色扮演的开源前端，暴露了当前最完整的文本生成参数栈。其可调试参数可分为 **6 大类**：

### 1.1 基础采样参数（Basic Sampling）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `temperature` | float [0, ∞) | 控制随机性，值越高输出越发散 |
| `max_tokens` / `max_new_tokens` | int | 最大生成长度 |
| `top_p` | float [0, 1] | Nucleus sampling，按累积概率截断候选 tokens |
| `top_k` | int | 仅保留概率最高的 K 个 token |
| `min_p` | float [0, 1] | 按最大 token 概率的比例设置最低阈值 |
| `typical_p` | float [0, 1] | Typical sampling，按条件熵筛选典型 tokens |
| `tail_free_sampling` (TFS) | float [0, 1] | 截断尾部低概率 tokens 的另一种策略 |

### 1.2 重复控制参数（Repetition Control）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `repetition_penalty` | float [1, ∞) | 惩罚已出现 token 的概率 |
| `repetition_penalty_range` | int | 重复惩罚回溯的 token 范围 |
| `presence_penalty` | float | OpenAI 风格，对已经出现过的 token 施加固定惩罚 |
| `frequency_penalty` | float | OpenAI 风格，按出现频次递增惩罚 |
| `encoder_repetition_penalty` | float | 基于 embedding 相似度的重复惩罚 |
| `no_repeat_ngram_size` | int | 禁止连续 N-gram 重复出现 |

### 1.3 高级采样器（Advanced Samplers）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `mirostat_mode` | int {0,1,2} | 动态温度控制模式 |
| `mirostat_tau` | float | Mirostat 目标困惑度 |
| `mirostat_eta` | float | Mirostat 学习率 |
| `dynatemp_min` / `dynatemp_max` | float | 动态温度范围 |
| `dynatemp_exponent` | float | 动态温度曲线指数 |
| `smoothing_factor` | float | 概率平滑因子 |
| `smoothing_curve` | float | 平滑曲线形状 |
| `skew` | float | 偏度采样 |

### 1.4 新兴解码策略（Next-Gen Decoding）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `dry_multiplier` / `dry_base` | float | DRY (Do not Repeat Yourself) 惩罚强度 |
| `dry_allowed_length` | int | DRY 允许重复的长度 |
| `dry_penalty_last_n` | int | DRY 回溯范围 |
| `dry_sequence_breakers` | string[] | DRY 序列分隔符 |
| `xtc_threshold` | float | XTC (eXclusionary Top C) 阈值 |
| `xtc_probability` | float | XTC 应用概率 |

### 1.5 结构化控制（Structured Control）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `logit_bias` | map<token_id, float> | 提升/抑制特定 token 概率 |
| `stop` / `stop_sequences` | string[] | 遇到指定字符串时停止生成 |
| `seed` | int | 固定随机种子（可复现输出） |
| `guided_json` | object | 强制输出 JSON Schema |
| `guided_regex` | string | 强制输出匹配正则 |
| `guided_grammar` | string | 强制输出符合 BNFGrammar |

### 1.6 推理/思考模式（Reasoning）

| 参数 | 类型 | 作用简述 |
|------|------|---------|
| `thinking_budget` / `reasoning_effort` | int/enum | 扩展思考/推理预算 |
| `enable_thinking` | bool | 是否启用思考模式 |
| `preserve_thinking` | bool | 是否保留思考内容 |

## 二、InkOS 当前可控节点全景

### 2.1 配置层级

InkOS 的 LLM 参数配置位于三个层级：

```
project.inkos.json / inkos.json
├── llm: LLMConfig
│   ├── temperature          # 全局默认温度
│   ├── thinkingBudget       # 全局推理预算
│   ├── stream               # 是否流式
│   ├── apiFormat            # chat / responses
│   ├── extra: {}            # 透传到请求体的任意参数
│   ├── headers: {}          # 自定义 HTTP 头
│   └── services: []         # 多服务配置（每项可覆盖 temperature 等）
├── qualityGates
│   └── retryTemperatureStep # 失败重试时温度递增步长
└── agents
    └── <agent>: { model }   # 每个 agent 可独立指定模型
```

### 2.2 代码内已识别的控制节点

#### A. 项目配置 Schema（`packages/core/src/models/project.ts`）

| 字段 | 类型 | 默认值 | 前端可配置 |
|------|------|--------|-----------|
| `llm.temperature` | number [0, 2] | 0.7 | ✅ `ServiceDetailPage` |
| `llm.thinkingBudget` | int ≥ 0 | 0 | ⚠️  schema 中有，但 `ServiceDetailPage` 未暴露 UI |
| `llm.stream` | boolean | true | ✅ `ServiceDetailPage` |
| `llm.apiFormat` | "chat" \| "responses" | "chat" | ✅ `ServiceDetailPage` |
| `llm.extra` | Record<string, unknown> | {} | ❌ 无 UI，只能通过环境变量 `INKOS_LLM_EXTRA_*` 或手动改配置 |
| `llm.headers` | Record<string, string> | {} | ❌ 无 UI |
| `llm.services[].temperature` | number [0, 2] | 继承顶层 | ⚠️ 保存时会写入，但 UI 只展示单个 temperature |
| `qualityGates.retryTemperatureStep` | number [0, 0.5] | 0.1 | ❌ 无 UI |

#### B. Agent 级覆盖（`ModelOverridesPanel`）

| 字段 | 说明 |
|------|------|
| `provider` | agent 单独使用的服务商 |
| `model` | agent 单独使用的模型 |
| `temperature` | agent 单独温度 |
| `maxTokens` | agent 单独最大 token |
| `thinkingBudget` | agent 单独推理预算 |

> 注：`ModelOverridesPanel` 的 UI 已实现（360 行），但仅覆盖了 provider/model/temperature/maxTokens/thinkingBudget，缺少 SillyTavern 中的高级采样参数。

#### C. 代码内硬编码温度（`packages/core/src/agents/*.ts`）

| Agent | 硬编码温度 | 可调性 |
|------|-----------|--------|
| `writer.ts`（创意写作） | `creativeTemperature = input.temperatureOverride ?? 0.7` | ⚠️ 仅 pipeline input 可覆盖 |
| `writer.ts`（后处理） | 0.5 / 0.3 | ❌ 完全硬编码 |
| `architect.ts` | 0.8 / 0.5 / 0.7 | ❌ 完全硬编码 |
| `planner.ts` | 0.7 | ❌ 完全硬编码 |
| `reviser.ts` | 0.3 | ❌ 完全硬编码 |
| `polisher.ts` | `input.temperature ?? 0.4` | ⚠️ 部分可覆盖 |
| `continuity.ts` | 0.3 | ❌ 完全硬编码 |
| `chapter-analyzer.ts` | 0.3 | ❌ 完全硬编码 |
| `state-validator.ts` | 0.1 | ❌ 完全硬编码 |
| `style-rewriter.ts` | 0.3 | ❌ 完全硬编码 |
| `radar.ts` | 0.6 | ❌ 完全硬编码 |
| `short-fiction.ts` | 0.55 / 0.3 / 0.45 / 0.58 | ❌ 完全硬编码 |

#### D. Pipeline 调度器动态温度（`packages/core/src/pipeline/scheduler.ts`）

```typescript
// 失败重试时递增温度
temperature = Math.min(1.2, 0.7 + failures * this.gates.retryTemperatureStep);
```

这是一个很好的自适应机制，但 `retryTemperatureStep` 没有前端配置入口。

#### E. `extra` 字段的透传能力（`packages/core/src/llm/provider.ts`）

```typescript
const RESERVED_KEYS = new Set(["max_tokens", "temperature", "model", "messages", "stream"]);
function stripReservedKeys(extra: Record<string, unknown>) { ... }
```

**结论**：`llm.extra` 中的任意键值对（除保留字外）会直接透传到 LLM 请求体。这意味着——

- `top_p` 可以通过 `extra.top_p` 传入 ✅
- `presence_penalty` / `frequency_penalty` 可以传入 ✅
- `logit_bias`、`stop`、`seed` 都可以传入 ✅
- 但**前端没有为这些参数提供任何 UI**

测试用例 `effective-llm-config.test.ts` 明确验证了 `INKOS_LLM_EXTRA_top_p=0.9` 可以被正确解析并透传。

## 四、关键发现与分析

### 4.1 最大优势：`extra` 透传机制已就绪

InkOS 在架构层面已经为高级参数打开了通道：`llm.extra` 会原样透传到 LLM 请求体。这意味着：

- 技术上**不需要改动 provider 层**即可支持新参数
- 用户可以通过手动修改 `inkos.json` 或在 `.env` 中设置 `INKOS_LLM_EXTRA_*` 来实验

**示例（已验证可行）：**
```json
{
  "llm": {
    "temperature": 0.8,
    "extra": {
      "top_p": 0.92,
      "presence_penalty": 0.02,
      "frequency_penalty": 0.05,
      "seed": 42
    }
  }
}
```

### 4.2 最大短板：前端 UI 严重滞后

虽然 `extra` 提供了后门，但对于非技术用户：
- 不知道哪些参数可用
- 不知道如何取值
- 无法实时调试 A/B 对比

SillyTavern 的核心价值之一就是让这些参数**可视化、可滑动、可保存为预设**。

### 4.3 Agent 硬编码温度是隐性问题

当前大量 Agent 直接写死了温度：
- 审计/验证类 agent 统一 0.1~0.3（合理，需要确定性）
- 创意写作 0.7（较中庸）
- architect 0.8（较高）

**问题**：
1. 不同模型对温度的敏感度不同（Claude 的 0.7 与 DeepSeek 的 0.7 差异很大）
2. 用户无法根据体裁调整（诗歌需要更高温度，说明文需要更低）
3. 温度没有与文风分析结果联动（例如检测到 AI-Tells 严重时自动降低温度）

### 4.4 缺少针对创意写作的参数预设

SillyTavern 的一个核心功能是 **Parameter Presets**：用户可以为不同场景（对话、动作、NSFW、创意写作）保存不同的参数组合。InkOS 目前：
- 仅有 `service-presets.ts` 中的服务商级默认温度
- 没有用户级的"写作参数预设"概念
- 没有针对中文小说创作的推荐预设

### 4.5 `thinkingBudget` 已 Schema 化但未 UI 化

这是一个近在眼前的问题：`thinkingBudget` 已经在配置中，但 `ServiceDetailPage` 和 `ModelOverridesPanel` 没有为其提供输入框。用户即使知道这个功能存在也无法配置。

## 五、改进建议

### Phase 1：补全基础参数 UI（1-2 天）🔴

**目标**：把已经 schema/infra 支持的参数暴露到前端

1. **`ServiceDetailPage` 增加字段**：
   - `thinkingBudget` 数字输入框
   - `top_p` 滑块 [0, 1]（默认值 1.0，表示不启用）
   - `presence_penalty` 滑块 [-2, 2]
   - `frequency_penalty` 滑块 [-2, 2]
   - `seed` 数字输入框（可选，留空表示随机）

2. **`ModelOverridesPanel` 同步扩展**：
   - agent 级覆盖增加 `top_p`、`presence_penalty`、`frequency_penalty`

3. **参数说明 tooltip**：
   - 每个参数增加 `?` 图标说明
   - 参考 SillyTavern 的用户友好文案

### Phase 2：高级参数实验面板（2-3 天）🟠

**目标**：为进阶用户提供 SillyTavern 级别的自由度

1. **新增 `GenerationParametersPanel` 组件**：
   - 可折叠的"高级参数"区域
   - 支持 `repetition_penalty`、`min_p`、`top_k`、`mirostat` 等
   - 参数分组：Sampling / Repetition / Advanced / Structured

2. **Raw Extra JSON 编辑器**：
   - 为极客用户提供原始 JSON 编辑能力
   - 带 JSON Schema 校验（防止传入 `temperature` 等保留字）

3. **预设保存/加载**：
   - 用户可以保存多组参数预设
   - 默认内置几组推荐：
     - `creative`（高 temperature + 高 top_p）
     - `focused`（低 temperature + 低 top_p）
     - `longform`（高 max_tokens + 温和 repetition_penalty）
     - `deterministic`（temperature=0 + seed 固定）

### Phase 3：Agent 温度策略重构（3-5 天）🟡

**目标**：消除硬编码，让参数与创作场景智能联动

1. **引入 `AgentTemperatureProfile`**：
   ```typescript
   interface AgentTemperatureProfile {
     taskType: "creative" | "analytical" | "revision" | "validation";
     baseTemperature: number;
     topP?: number;
     presencePenalty?: number;
     frequencyPenalty?: number;
   }
   ```

2. **按任务类型而非 Agent 名称选择温度**：
   - `creative` → writer/planner/architect
   - `analytical` → chapter-analyzer/radar
   - `revision` → reviser/polisher
   - `validation` → continuity/state-validator/detector

3. **与文风分析联动**：
   - 当 `AI-Tells` 检测分数高时，自动降低 creative 任务的 temperature
   - 当检测到重复模式严重时，提升 frequency_penalty

### Phase 4：请求级参数与可观测性（5-7 天）🟢

**目标**：让每次 LLM 调用都可调、可追溯

1. **请求级参数覆盖**：
   - 在 `ChatPage` 中允许用户为单次生成临时调整 temperature/top_p
   - 在 `BookChaptersSection` 的"写下一章"时选择参数预设

2. **参数-效果关联分析**：
   - 记录每次生成的参数组合
   - 在 `Analytics` 中展示不同参数下的：
     - 平均输出 token 数
     - AI-Tells 分数
     - 用户接受/重写率
     - 成本

3. **A/B 测试支持**：
   - 同一提示用两组参数同时生成
   - 用户选择更满意的版本

## 六、市面其他 LLM 写作工具的参数实践（补充例证）

> 本节搜集 NovelAI、Sudowrite、KoboldAI、AI Dungeon、Jasper/Copy.ai 以及中文角色扮演产品的公开参数与风格稳定化方案，作为 InkOS 设计参数面板时的横向参考。

### 6.1 NovelAI：模块训练 + Lorebook + 完整采样面板

**NovelAI** 是面向小说创作最早的垂直化 LLM 产品之一，其风格稳定化并非单靠 temperature，而是靠**三层机制**：

| 层级 | 功能 | 用户可控点 |
|------|------|-----------|
| **模型层** | NovelAI-Sigurd（12B 文学语料微调）/ NovelAI-Kayra（RP & 对话优化） | 选择模型 |
| **模块层** | Training Modules（用户上传 2,000+ 字样本训练自定义风格模块） | 上传语料、调节模块强度 |
| **记忆层** | Memory + Lorebook（关键字触发的百科式记忆） | 权重、触发条件、注入位置 |
| **采样层** | Temperature / Top-P / Repetition Penalty / Presence Penalty / Length Penalty | 全参数 UI |

**NovelAI 的公开参数范围**（免费 tier 已可用）：
- `temperature`: 0.1–1.5
- `top_p`: 0.1–1.0
- `repetition_penalty`: 1.0–2.0
- `presence_penalty`: 有（OpenAI 风格）
- `length_penalty`: 有

**对 InkOS 的启示**：
1. 风格稳定化不能只靠 temperature，要靠**用户语料训练模块**（对应 InkOS 的 `analyzeStyleFingerprint` + 风格库）
2. Lorebook 的"关键字触发 + 权重注入"机制非常成熟（对应 InkOS 的 TruthFiles / BookTruthSection）
3. 采样参数应完整暴露，而非只给 temperature

### 6.2 Sudowrite：Story Bible + Creativity Slider + 场景化模板

**Sudowrite** 不讲采样参数术语，而是把复杂参数包装成创作者能懂的**场景化控制**：

| 功能 | 用户-facing 名称 | 背后的参数逻辑 |
|------|-----------------|---------------|
| **Write** | Auto / Guided / Tone Shift | temperature 中等偏高 |
| **Describe** | Sight / Sound / Smell / Touch / Taste | 低 temperature + 结构化输出 |
| **Rewrite** | Shorter / More descriptive / Show not tell / More intense | 温度微调 + system prompt 切换 |
| **First Draft** | 基于 Scenes 生成长段落 | 高 max_tokens + 中等 temperature |
| **Brainstorm** | 角色/地名/情节创意 | 高 temperature + 高 top_p |

**风格稳定化的核心机制**：
- **Story Bible**：集中存储世界观、角色、情节，系统自动引用（类似 InkOS RuntimeStateStore）
- **Style Examples**：用户上传样本文本，AI 学习语气、句法、POV（类似 InkOS 文风指纹）
- **Creativity Slider**：一个滑块控制"创意度"，底层映射到 temperature/top_p 组合

**对 InkOS 的启示**：
1. 高级参数应该被包装成创作者语言（如"创意度"、"描写强度"）而非原始采样术语
2. 每个写作功能（Write/Describe/Rewrite）应有自己的默认参数模板
3. Style Examples 与生成联动是减少风格漂移的关键

### 6.3 KoboldAI / KoboldCpp：最完整的本地采样器栈

**KoboldAI Lite** 是目前公开文档最详尽的本地 LLM 写作前端，其采样器栈与 SillyTavern 相当。

**推荐默认值**（来自官方文档）：
```
Top-P = 0.92
Repetition Penalty = 1.1
Temperature = 0.7
Sampler Order = [6, 0, 1, 3, 4, 2, 5]
```

**完整的可调试参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| Temperature | 0.7 | 随机性 |
| Top-P | 0.92 | Nucleus sampling |
| Top-K | 0（禁用） | 截断为 K 个候选 |
| Top-A | 0（禁用） | 按最大概率比例截断 |
| Typical | 1（禁用） | 按条件熵采样 |
| TFS | 1（禁用） | Tail Free Sampling |
| Min-P | 0（禁用） | 实验性替代 Top-P |
| Repetition Penalty | 1.1 | 重复惩罚 |
| Repetition Penalty Range | - | 回溯范围 |
| Repetition Penalty Slope | - | 惩罚曲线 |
| Mirostat | 0（禁用） | 替代采样策略 |
| DynaTemp / DynaTemp Range | 0 | 动态温度 |
| DRY | backend 决定 | 多 token 重复惩罚 |
| XTC | backend 决定 | Exclude Top Choices |
| Seed | 随机 | 可复现性 |
| Stop Sequences | 模式预配 | 自定义停止符 |
| Sampler Order | [6,0,1,3,4,2,5] | 采样器应用顺序 |

**KoboldAI 独特的工程细节**：
- **Sampler Order**：采样器执行顺序会显著影响输出质量，官方强烈建议不要改默认顺序
- **Context Size / Amount to Generate**：明确分开上下文窗口和生成长度
- **Memory / Author's Note / World Info**：三层记忆体系

**对 InkOS 的启示**：
1. 如果 InkOS 未来支持本地后端（llama.cpp / vLLM / Ollama），需要支持完整的采样器栈
2. `seed` 对创作者很重要——可以复现满意的生成结果
3. 停止序列（stop sequences）应该按写作模式预配置（chat/story/adventure/instruct）

### 6.4 AI Dungeon：模型选择优先于参数微调

**AI Dungeon** 走了一条不同的路：它把参数控制隐藏起来，只让用户选择模型和模式。

| 模型 | 参数量 | 特点 |
|------|--------|------|
| Griffin | 6B → 13B | 快速、成本低 |
| Dragon | 175B (GPT-3 Davinci) → 17B (Jurassic-2 Mid) | 连贯性更好 |

**用户可调的风格/控制项**：
- **Memory**：AI 必须记住的事实
- **Author's Note**：语气和风格指令
- **World Info**：按关键字触发的 lorebook
- **Randomness Slider**：底层映射 temperature（付费版）

**对 InkOS 的启示**：
1. 对于非技术用户，**模型选择**比参数调节更直观
2. 但隐藏所有参数会让高阶用户不满（AI Dungeon 因此常被批评缺乏控制）
3. Memory/Author's Note/World Info 的三层结构值得 InkOS 借鉴

### 6.5 Jasper / Copy.ai / Writesonic：营销写作的品牌声调和模板参数

这些产品面向营销文案，参数设计思路与小说写作不同：

| 工具 | 风格控制方式 | 用户可调参数 |
|------|-------------|-------------|
| **Jasper** | Brand Voice Training（上传品牌指南） | Tone（20+ 预设）、Length、Temperature（后台） |
| **Copy.ai** | Templates + Variations | Tone、Length、Format、Number of variations |
| **Writesonic** | SEO Score + Readability | Keywords、Tone、Length、Factual accuracy |
| **Rytr** | 20+ Tones × 30+ Languages | Tone、Use case、Creativity level |

**共性设计**：
- 不提供 raw sampling 参数，而是提供**创作场景 + 风格强度**的抽象控制
- 强调 **Variations（一次生成多个版本）** 来降低参数调优成本
- 品牌一致性通过**训练（Jasper Brand Voice）** 而非实时参数来保证

**对 InkOS 的启示**：
1. 可以为不同 Agent 设计场景模板（marketing/fiction/technical/blog）
2. 参数预设应该从"采样参数"升级为"创作意图"（如"描写丰富"、"对话紧凑"、"节奏舒缓"）
3. 一次生成多版本（variations）可以弥补单次生成参数不理想的遗憾

### 6.6 中文角色扮演/AI 陪伴产品：星野、猫箱、筑梦岛

国内主流 AI 聊天/陪伴产品走的是**重度产品化**路线：

| 产品 | 风格控制方式 | 用户可调参数 |
|------|-------------|-------------|
| **星野** | 角色卡 + 人设描述 + 音色 | 几乎无公开采样参数 |
| **猫箱** | 角色设定 + 剧情分支 | 几乎无公开采样参数 |
| **筑梦岛** | 小剧场 + 梦境收藏 | 几乎无公开采样参数 |
| **Character.AI** | 角色定义 + 对话示例 | 无参数控制 |
| **豆包/文心一言** | 提示词 + 插件 | temperature 等不暴露 |

**这些产品的共同特点**：
- 不暴露 LLM 采样参数给终端用户
- 通过**角色卡/人设/示例对话**来控制风格
- 风格稳定性主要依靠：
  1. 详细的 system prompt / 人设描述
  2. 对话示例（few-shot）
  3. 后过滤/重排序（选择最合适的回复）
  4. 模型本身的 instruction tuning

**对 InkOS 的启示**：
1. 对于纯 C 端用户，不暴露参数是合理的——他们更关心"角色是否保持一致"而非"temperature 是多少"
2. 但 InkOS 定位是**专业写作工具**，创作者需要比 C 端产品更细粒度的控制
3. 可以提供两套模式：
   - **简单模式**：只暴露"创意度"、"风格强度"等抽象滑块
   - **专家模式**：暴露全部采样参数

### 6.7 学术界/研究中的创意写作参数推荐

来自多篇论文和实验报告的基准配置：

| 来源 | 场景 | Temperature | Top-P | Top-K | Repetition Penalty | 其他 |
|------|------|-------------|-------|-------|-------------------|------|
| Probst 2024 | 通用文本生成 | 0.75 | 0.95 | 50 | 1.2 | max_tokens=512 |
| ACL 2024 SRW | 通用推理 | 0.7 | 0.7 | 50 | 1.0 | - |
| Creative Writing FT 2025 | 创意写作多样性 | 1.0 | 0.95 | 50 | 1.1 | max_length=2048 |
| Machine Learning Plus | 创意写作 | 0.7–1.0 | 0.9–0.95 | 40–100 | - | - |
| CSDN 教程 | 故事创作 | 0.85 | 0.95 | - | 1.2 | - |
| 学术论文 | 评估/分类 | 0.0–0.1 | 1.0 | - | 1.0 | 确定性优先 |

**共识性结论**：
- **创意写作**：temperature 0.7–1.0，top-p 0.9–0.95，repetition_penalty 1.1–1.2
- **事实/评估任务**：temperature 0.0–0.3，top-p ≤ 0.5
- **不要同时调节 temperature 和 top_p**（OpenAI 官方建议）
- **重复问题应通过 repetition_penalty 解决**，而非盲目降低 temperature

### 6.9 对 InkOS 参数设计的综合建议

综合以上横向对比，InkOS 在参数设计上应采取**分层策略**：

**Layer 1: 简单模式（面向普通创作者）**
- 只给一个 "Creativity" 滑块（低/中/高）
- 对应映射：
  - 低：`temperature=0.3, top_p=0.8`
  - 中：`temperature=0.7, top_p=0.9`
  - 高：`temperature=1.0, top_p=0.95`

**Layer 2: 场景预设（面向有明确需求的创作者）**
- `Narrative Focused`（叙事连贯）：低 temperature + 高 top_p
- `Descriptive Rich`（描写丰富）：中 temperature + 高 repetition_penalty_range
- `Dialogue Snappy`（对话紧凑）：中 temperature + 短 max_tokens
- `Experimental`（实验性）：高 temperature + min_p

**Layer 3: 专家模式（面向高级用户/本地后端）**
- 暴露全部 SillyTavern/KoboldAI 级别的参数
- Raw extra JSON 编辑器作为兜底

**Layer 4: 风格稳定化机制（超越参数本身）**
- 与 InkOS 已有的**文风指纹**结合：生成前先比对当前文本与目标风格的偏差
- 与 **TruthFiles / RuntimeState** 结合：自动注入 lorebook 式的记忆
- 与 **AI-Tells 检测**结合：实时调整参数避免 AI 痕迹过重

## 八、结论

InkOS 在**基础设施层面**已经为 LLM 写作参数的全面控制打下了良好基础：
- 统一的 `LLMConfig` schema
- `extra` 透传机制
- `ModelOverridesPanel` 的 agent 级覆盖
- `service-presets` 的模型卡约束

但在**用户可达性层面**，距离 SillyTavern 的差距明显：
- 60% 的参数完全没有前端入口
- 大量 Agent 温度硬编码
- 缺乏参数预设和 A/B 对比能力
- `extra` 的后门机制对普通用户不友好

**建议下一阶段的切入口**：先补全 `thinkingBudget`、`top_p`、`presence_penalty`、`frequency_penalty` 这四个参数的前端 UI（利用已有的 `extra` 透传能力，不需要改动 provider 层），随后做参数预设功能。这样可以用最小的成本显著提升创作控制力。
