# 新功能拓展可行性报告

> 报告日期：2026-06-07
> 评估对象：资料来源管理页、统一输入工作台、书籍健康总览
> 评估原则：核心稳定性优先、延展空间最大化、复杂度最小化、代码逻辑简洁清晰
> 参考依据：`NoFusion最终发展方向报告.md`、`0607项目实测与对照报告.md`

---

## 一、总述

本报告对三项涉及较大重构或独立开发的 Studio 前端功能进行可行性评估。三项功能的共同特点是：

- **不修改 Core Agent 主链路**（不改变写作/审计/修订流程）
- **优先复用 Core 已有契约与计算函数**，不在 Studio API 中复制业务口径
- **以 Studio 前端和 API 适配层为主**；涉及来源索引写入时，只增加 Core 边界服务，不让前端或 API 直接改写持久化文件

因此整体风险可控，但复杂度不只在前端组件设计，还包括 **API 契约一致性、来源索引并发写入和健康指标可信度**。

## 二、资料来源管理页

### 2.1 现状

Core 层已完整实现：

- `persistFoundationSourceBundle()` — 持久化资料包到 `story/sources/index.json` 和 `story/sources/{sourceId}.md`
- `FoundationSourceIndexEntry` — 结构化记录：`sourceId`、`sourceName`、`fileType`、`purpose`、`textHash`、`charCount`、`importedAt`、`mode`
- `FoundationSourcePurpose` — 8 种用途：`auto/world/character/era/plot/chapter/rule/style`
- `PipelineRunner.planFoundationImport()` / `commitFoundationImport()` — 架构导入预览与提交

**前端缺失**：
- 无独立的资料来源页面
- BookCreate 中文件上传时 `purpose` 固定为 `"auto"`，用户不可选
- ImportManager 中可以看到 foundation plan 中的资料列表，但无独立的"已导入资料"视图
- 无来源删除、更新、重新导入的 UI

### 2.2 核心稳定性影响

| 影响面 | 风险等级 | 说明 |
|--------|:--------:|------|
| Core Agent 管线 | 🟢 无影响 | 不触及 PipelineRunner 或 Agent |
| Core 文件写入 | 🟡 中风险 | 查询无写入风险；删除/归档必须由 Core 统一处理索引和来源文件 |
| API 层 | 🟡 中风险 | 新增查询端点风险低；写端点需校验书籍/来源 ID，并处理并发和部分失败 |
| 现有页面 | 🟢 无影响 | 新页面作为 BookWorkspace 的新 section，不修改 BookCreate/ImportManager 现有逻辑 |

### 2.3 实现方案（简洁版）

**不新增大端 API，复用来源文件已有的数据结构。**

#### Step 1：新增 API 端点

```typescript
// server.ts — 新增 2 个轻量端点

// 1) 获取书籍资料来源列表（只读）
app.get("/api/v1/books/:id/sources", async (c) => {
  // 读取 story/sources/index.json
  // 返回 FoundationSourceIndexEntry[]
  // 每个条目附带 sourceFileExists: boolean（对应 .md 文件是否存在）
});

// 2) 归档或删除单条来源
app.delete("/api/v1/books/:id/sources/:sourceId", async (c) => {
  // 校验书籍与 sourceId
  // 调用 Core 的 archive/removeFoundationSource()
  // Core 在同一写入边界内更新索引和来源文件
  // 明确：不修改书籍权威架构（outline/roles/hooks）
});
```

首版可先上线 GET 只读视图；DELETE 必须等 Core 受控写入函数与测试完成后开放。若需要保留审计追溯，优先采用“归档/停用”而不是物理删除。

**为什么不需要更多端点**：
- 重新导入 → 用户走 ImportManager（已有 plan/commit 流程）
- 修改用途 → 归档旧来源后重新导入（用途在生产时已决定，不直接改历史记录）
- 更新内容 → 归档旧来源后重新导入（hash 变化会产生新记录）

#### Step 2：新增前端页面 `SourceManager.tsx`

```typescript
// 极简设计：一张表格 + 一个删除确认弹窗
// 作为 BookWorkspace 的新 section（与 chapters / hooks 同级）
// 数据源：/sources API（约 20 行 fetch）

interface SourceEntry {
  sourceId: string;
  sourceName: string;
  fileType: string;
  purpose: string;
  charCount: number;
  importedAt: string;
  mode: "create" | "supplement" | "rebuild";
  sourceFileExists: boolean;
}
```

**页面交互**：
1. 加载书籍时检查是否有 `story/sources/index.json`
2. 有则显示来源表格，无则显示"暂无来源"占位
3. 每行显示：名称、用途(label)、字数、导入时间、模式
4. 归档/删除操作：确认弹窗 → 告知“不回滚权威架构且会影响来源追溯” → 调用受控 API
5. 不引入复杂状态管理，使用 `useApi` hook（与现有页面一致）

#### Step 3：整合入口

在 `BookWorkspace.tsx` 的 section 系统中注册新 section，而非修改旧版 `BookDetail.tsx`：

```typescript
// book-workspace-types.ts — 新增 section
export type BookSection =
  | "overview"
  | "chat"
  | "chapters"
  // ... 原有 section ...
  | "sources";  // 新增

export const BOOK_SECTIONS: BookSection[] = [
  // ... 原有 section ...
  "sources",
];
```

在 `BookWorkspace.tsx` 的 section 渲染区增加条件分支：

```typescript
{activeSection === "sources" && (
  <SourceManager bookId={bookId} t={t} />
)}
```

侧边栏保留“资料来源”入口；无来源时展示空状态和导入入口，避免用户无法发现该能力。

### 2.4 代码量估算

| 模块 | 新增代码 | 说明 |
|------|---------|------|
| Core 来源服务 | ~80-120 行 | 查询、归档/删除、索引校验、原子替换与测试 |
| API 端点 | ~40-60 行 | 2 个薄端点，只做参数校验和错误映射 |
| 前端组件 | ~120 行 | 表格 + 删除确认 + 空状态 |
| BookWorkspace 整合 | ~20 行 | section 注册 + 渲染分支 + 导航条件 |
| **总计** | **~260-320 行** | 不含测试和 i18n key（~12-15 条） |

### 2.5 延展空间

- 未来可增加"来源影响范围"提示（影响哪些 outline/roles/hooks）
- 未来可支持"重新生成预览"（复用现有 planFoundationImport）
- 未来可支持来源版本比对（hash 变化时标记"内容已更新"）
- **当前不做的**：来源编辑、来源合并、批量操作

### 2.6 结论：✅ 可行，建议实施

**理由**：查询视图改动较小，用户价值明确，并可通过 BookWorkspace section 系统嵌入。需要注意的是，删除不是纯前端功能；只有在 Core 受控写入、原子替换和异常恢复完成后，才能认定为低风险闭环。

## 三、统一输入工作台

### 3.1 现状

当前三个页面各自独立实现文件选择和文本处理：

| 页面 | 文件选择 | 文本提取 | 用途分类 | 去向 |
|------|:--------:|:--------:|:--------:|------|
| BookCreate | ✅ 拖拽/选择 | ❌ 无提取 | ❌ 固定 auto | 新书 foundation |
| ImportManager | ✅ 粘贴/上传 | ❌ 无提取 | ✅ 可选用途 | 存量书籍 foundation |
| StyleManager | ✅ 拖拽/粘贴/URL | ✅ 调用 extract-text | ❌ 不适用 | 风格分析/预处理 |

**关键发现**：三个页面并非"功能重叠"，而是"不同阶段的文件处理"。强行抽取共享组件会引入：

- 条件分支过多（三页去向不同）
- 接口设计膨胀（`SourceUploader` 需要支持所有三种模式的配置）
- 回归风险增大（每页独立测试过，重构图可能破坏边界情况）

### 3.2 核心稳定性影响

| 影响面 | 风险等级 | 说明 |
|--------|:--------:|------|
| Core 管线 | 🟢 无影响 | 重构纯前端组件 |
| API 层 | 🟢 无影响 | 三个页面已使用不同 API |
| 现有页面 | 🔴 **高风险** | 三页各自有完善的状态管理和边界处理，重构后须全部重新验证 |

### 3.3 复杂度分析

以 `SourceUploader` 组件抽取为例：

```
SourceUploader(props)
  ├── mode: "book-create" | "import" | "style"
  ├── onFileSelected: (text, sourceName, fileType) => void
  ├── onUrlImported: (text, sourceName) => void
  ├── enableExtraction?: boolean
  ├── enablePurpose?: boolean
  ├── enablePreprocess?: boolean
  ├── enableHealthCheck?: boolean
  └── purpose?: string  // 预选用途
```

**问题**：
- 6 个 props，其中 5 个条件开关，组合状态爆炸（2^5 = 32 种模式）
- BookCreate 需要拖拽 + 无提取 + 用途固定；ImportManager 需要粘贴 + 无提取 + 用途可选；StyleManager 需要拖拽 + 粘贴 + URL + 提取 + 体检 + 预处理
- 每个条件分支都会引入 if/else 逻辑，代码行数不会比重构前减少，反而因抽象层而增加
- 三页的文本处理后续发展方向不同（BookCreate 加创建前预览、ImportManager 加分片、StyleManager 加指纹），强行耦合后一个页面的改动会影响其他两个

### 3.4 替代方案（推荐）

**不抽取大而全的共享组件，改为：Core 契约 + 浏览器工具函数 + Studio API 适配器**

前端当前已经存在的重复代码主要是文件类型识别、浏览器文件读取和 API 调用。共享边界应拆为：

- Core：`DocumentFileType`、字符上限、提取结果和复合扩展名规则
- Studio shared：`File` / `FileReader` 等浏览器专用适配
- Studio API client：统一使用 `fetchJson()` 调用 `/style/extract-text`

```typescript
// src/shared/source-utils.ts — 新增 ~60 行

/** 从 File 对象读取文本内容 */
export function readFileAsText(file: File): Promise<string> { /* ... */ }

/** 按 Core 规则推断 API 可接受的文件类型，包含 .jsonl.md / .json.md */
export function inferFileType(fileName: string): ExtractApiFileType | null { /* ... */ }

/** 格式化文件大小 */
export function formatFileSize(bytes: number): string { /* ... */ }

/** 基础校验：文件大小、类型、空内容 */
export function validateSourceFile(file: File, maxBytes?: number): string | null { /* ... */ }
```

统一文档提取适配器（复用 `style-preprocess-adapter.ts` 中的白名单校验逻辑）：

```typescript
// src/shared/extract-adapter.ts — 新增 ~40 行

/** 调用 Studio API 进行文档提取（复用 use-api.ts 的 fetchJson，确保 base URL 一致） */
export async function extractDocument(
  text: string,
  options: ExtractOptions = {},
): Promise<ExtractedDoc> {
  // 通过 fetchJson 调用 /style/extract-text，而非裸 fetch
  // 确保在自定义 baseUrl 部署环境下仍能正常工作
}
```

**优点**：
- 改动范围可控
- 三页各自保留独立状态管理
- 重复代码从 ~30 行减到 ~5 行（每个页面调 `readFileAsText` 即可）
- 新的扩展可集中进入适配器，但只有完成各页面接入和回归测试后才会生效
- ImportManager 的粘贴文本导入可复用独立的 `validateSourceText()` 字符数校验

### 3.5 代码量估算

| 模块 | 新增代码 | 说明 |
|------|---------|------|
| 浏览器工具函数 | ~60 行 | 无 React 依赖，包含 `FileReader` 适配 |
| 文档提取适配器 | ~40 行 | 复用已有 API，通过 fetchJson 调用 |
| 三页调用适配 | ~-30 行 | 删除重复的提取逻辑 |
| **总计** | **~70 行净增** | 比重构方案少 200+ 行 |

### 3.6 延展空间

- CLI 只复用 Core 的类型、提取和限制契约；`File` / `FileReader` 工具仅供浏览器端使用
- 提取适配器可增加缓存层（避免重复解析大文件）
- **当前不做的**：SourceUploader 组件、拖拽区域统一样式、三页文件状态联动

### 3.7 结论：🟡 不建议重构为统一组件，建议分层共享契约与工具

**理由**：三页的去向和后续发展差异太大，强行抽取组件得不偿失。共享 Core 契约、浏览器工具和 API client 可以减少重复，同时避免把浏览器 API 错误扩散到 CLI。

## 四、书籍健康总览

### 4.1 现状

当前与书籍健康相关的数据和页面：

| 数据来源 | 所在页面 | 展示方式 |
|----------|----------|----------|
| 审计分数/趋势 | Analytics | 审计通过率 + 问题分类 + 问题最多章节 |
| Token 统计 | Analytics | 总量 + 均值 + 趋势图 |
| 章节目录与状态 | BookWorkspace / BookDetail | 章节列表 + 状态筛选 |
| 风格指纹 | StyleManager | 独立页面，不关联书籍 |
| 伏笔风险 | (无页面) | 仅在 `hooks.json` / `pending_hooks.md` |
| 最近导入/预处理回执 | (无页面) | 仅在 `sources/index.json` |
| Pipeline 失败阶段 | (无页面) | 仅在 daemon 日志 |

**关键问题**：不是数据不够，而是数据散落在 4 个页面中，且部分数据（伏笔、导入回执、管线失败）完全没有 UI。

### 4.2 核心稳定性影响

| 影响面 | 风险等级 | 说明 |
|--------|:--------:|------|
| Core 管线 | 🟢 无影响 | 只读数据聚合 |
| API 层 | 🟡 低风险 | 建议新增一个聚合端点而非多次请求 |
| Analytics 页面 | 🟢 可共存 | Analytics 保留深度分析 |
| BookWorkspace | 🟢 可整合 | 扩充现有 `BookOverviewSection`，不再增加平行首页 |

### 4.3 实现方案（简洁版）

**不新增独立页面或 `"health"` section，直接扩充现有 `BookOverviewSection`。**

#### Step 1：新增 API 聚合端点

```typescript
// server.ts — 新增 1 个聚合端点

app.get("/api/v1/books/:id/health", async (c) => {
  // 1. state.loadChapterIndexStrict(id) + Core computeAnalytics()
  // 2. Core summarizePendingHookHealth() 返回结构化伏笔风险
  // 3. Core 来源查询函数返回最近导入
  // 4. 暂无可靠来源的指标返回 unavailable，而不是 0
  // 5. 风格能力只返回 profile/score 是否可用，不伪造漂移分数
  //
  // 返回单个 JSON 对象，所有数据一次性返回
});
```

**为什么需要聚合端点**：
- 当前 Analytics 已有 `/books/:id/analytics` 端点
- 但健康总览需要更多数据维度
- 单个聚合端点避免 Overview 重复发起多组请求，并统一错误语义
- 聚合层在服务端做，前端只需 `fetchJson` 一次

**关键约束**：
- `styleDriftScore` 字段当前不存在。首版返回 `styleStatus: "available" | "unavailable"`；只有已有书籍风格档案且能计算最新章节评分时，才展示分值。
- Pipeline 近期错误当前没有可靠的持久化时间窗来源，应返回 `Metric.unavailable`，不能用 `0` 表示“无错误”。
- `pending_hooks.md` 解析和状态归一化必须留在 Core。建议新增并导出 `summarizePendingHookHealth()`，避免 API 直接拼接 `StoredHook` 与 `HookRecord` 两套类型。
- 当前 `state.loadChapterIndex()` 会将读取/解析失败降级为空数组。应增加严格读取方法并让 Analytics 与 Health 同步使用，否则损坏索引可能被误报为“0 章、100% 通过”。

#### Step 2：扩充现有 `BookOverviewSection`

```typescript
// BookOverviewSection.tsx 中加入健康卡片组

// 组件调用 /books/:id/health，并复用 Overview 的 StatCard：
// - auditPassRate / tokenStats 直接展示
// - Metric.available 展示 value
// - Metric.unavailable 展示 "--" 和“暂不可用”
// - 加载失败展示局部错误与重试
```

**设计原则**：
- 每张卡片只显示一个可信数字或明确的“暂不可用”
- 点击卡片跳转到对应详情页（Audit / Hooks / Sources）
- Analytics 页面保留不变；其 API 同步迁移到严格章节读取
- 保留现有 `"overview"` 默认入口，不新增 section 记忆逻辑
- 复用现有 `StatCard`、错误提示、重试和 SSE 最近事件区域

#### Step 3：伏笔风险数据准备

伏笔风险解析必须复用 Core 已有解析和生命周期工具，禁止 API 手写表格解析或强制类型断言：

```typescript
// Core 内部负责 parsePendingHooksMarkdown()、状态归一化和生命周期判断：
import { summarizePendingHookHealth } from "@actalk/inkos-core";

const hookRisk = summarizePendingHookHealth({
  markdown: hooksRaw,
  chapterNumber: currentChapter,
});

interface HookRiskSummary {
  readonly totalHooks: number;
  readonly staleCount: number;       // 超过 halfLifeChapters 未推进
  readonly criticalHooks: ReadonlyArray<{
    readonly hookId: string;
    readonly status: string;
    readonly risk: "stale";
  }>;
}
```

### 4.4 代码量估算

| 模块 | 新增代码 | 说明 |
|------|---------|------|
| Core 公共导出/查询 | ~40-80 行 | Hook 摘要、来源查询、章节索引严格读取与契约测试 |
| 聚合 API 端点 | ~70-100 行 | 数据组装、可用性状态和错误映射 |
| Overview 卡片扩充 | ~100-140 行 | 复用现有 StatCard、加载与重试结构 |
| **总计** | **~200-300 行** | 不含测试和 i18n key（~15-20 条） |

### 4.5 延展空间

- 每张卡片可单独扩展为详情弹窗（点击后展开趋势图）
- 未来可增加"健康评分"（综合审计/伏笔/风格/成本加权）
- 未来可增加"对比"（与上一卷/前 10 章对比）
- 待 Core 提供可复用的书籍风格漂移结果后，风格卡片可从“档案可用性”升级为数值趋势
- **当前不做的**：健康评分的自动化阈值、主动推送告警、跨书籍对比

### 4.6 结论：✅ 可行，建议实施（含前置约束）

**理由**：
- 数据层面基本可行，但不是所有指标都已就绪；首版只展示有可信来源的数据
- 聚合 API 端点使前端代码极简（一次请求，纯展示）
- 直接扩充现有 BookOverview，避免产生第二个书籍首页
- Analytics UI 保留深度入口；API 仅调整为严格读取并继续调用 `computeAnalytics()`

**前置约束**：
- 风格与 Pipeline 错误指标必须支持 `unavailable`，不能以 `0` 代替缺失数据
- Core 必须提供结构化 `summarizePendingHookHealth()`，API 不自行转换 Hook 类型
- 审计与 Token 数据必须通过严格章节读取后调用 `computeAnalytics()`，不接受“先复制、以后再统一”的实现

## 五、三项功能对比总结

| 评估维度 | 资料来源管理页 | 统一输入工作台 | 书籍健康总览 |
|----------|:------------:|:------------:|:------------:|
| 核心稳定性影响 | 🟡 查询低、写入中 | 🔴 重构风险高 | 🟡 口径一致性要求高 |

```typescript
// ============================
// 共享文本来源工具函数
// 浏览器适配层，无 React 依赖

import type { DocumentFileType } from "@actalk/inkos-core";

export type ExtractApiFileType = Exclude<DocumentFileType, "htm">;

/** 根据文件名推断文档类型。无法识别时返回 null。 */
export function inferFileType(fileName: string): ExtractApiFileType | null {
  const lower = fileName.toLowerCase();
  if (lower.endsWith(".jsonl.md") || lower.endsWith(".jsonl.markdown")) return "jsonl";
  if (lower.endsWith(".json.md") || lower.endsWith(".json.markdown")) return "json";
  if (lower.endsWith(".md") || lower.endsWith(".markdown")) return "md";
  if (lower.endsWith(".txt")) return "txt";
  if (lower.endsWith(".jsonl")) return "jsonl";
  if (lower.endsWith(".json")) return "json";
  if (lower.endsWith(".ts")) return "ts";
  if (lower.endsWith(".js")) return "js";
  if (lower.endsWith(".html") || lower.endsWith(".htm")) return "html";
  if (lower.endsWith(".css")) return "css";
  return null;
}

/** 从 File 对象读取文本内容 */
export function readFileAsText(file: File): Promise<string> {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = () => resolve(reader.result as string);
    reader.onerror = () => reject(new Error(`Failed to read file: ${file.name}`));
    reader.readAsText(file);
  });
}

/** 浏览器侧基础校验；服务端仍需执行 MAX_CHARS 等权威限制。 */
export function validateSourceFile(file: File, maxBytes = 50 * 1024 * 1024): string | null {
  if (!file.name || file.size === 0) return "File is empty";
  if (file.size > maxBytes) return `File exceeds ${Math.round(maxBytes / 1024 / 1024)}MB limit`;
  if (!inferFileType(file.name)) return `Unsupported file type: ${file.name}`;
  return null;
}

/** 格式化文件大小为人类可读 */
export function formatFileSize(bytes: number): string {
  if (bytes < 1024) return `${bytes} B`;
  if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)} KB`;
  return `${(bytes / 1024 / 1024).toFixed(1)} MB`;
}
```

**边界说明**：
- 此文件中的 `File` / `FileReader` 只能在 Studio 浏览器端使用，CLI 不复用。
- 文件类型契约来自 Core；复合扩展名规则必须有测试，防止 `.jsonl.md` 被误判为 Markdown。
- 50MB 是浏览器体验保护，不替代服务端 `MAX_CHARS`、分片和 `413` 校验。

```typescript
// ============================
// 文档提取适配器
// 封装 /style/extract-text API 调用
// 供完成接入的 Studio 页面共享同一调用路径
// ============================

import { fetchJson } from "../hooks/use-api";
import type { ExtractApiFileType } from "./source-utils";

export interface ExtractedDoc {
  readonly sourceName: string;
  readonly fileType: ExtractApiFileType;
  readonly text: string;
  readonly charCount: number;
  readonly textHash: string;
  readonly warnings: ReadonlyArray<string>;
  readonly truncated: boolean;
  readonly totalChunks: number;
  readonly chunkIndex: number;
  readonly originalLength?: number;
}

export interface ExtractOptions {
  readonly sourceName?: string;
  readonly fileType?: ExtractApiFileType;
  readonly maxChars?: number;
  readonly chunk?: number;
}

/** 调用 Studio API 进行文档提取 */
export async function extractDocument(
  text: string,
  options: ExtractOptions = {},
): Promise<ExtractedDoc> {
  const { sourceName, fileType, maxChars, chunk } = options;
  const body: Record<string, unknown> = { text, sourceName: sourceName || "sample" };
  if (fileType) body.fileType = fileType;
  if (typeof maxChars === "number") body.maxChars = maxChars;
  if (typeof chunk === "number") body.chunk = chunk;

  return fetchJson<ExtractedDoc>("/style/extract-text", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(body),
  });
}

/** 请求下一分片；调用方负责合并文本和更新 chunk 状态 */
export async function loadNextChunk(
  text: string,
  options: ExtractOptions & { currentChunk: number },
): Promise<ExtractedDoc> {
  return extractDocument(text, { ...options, chunk: options.currentChunk + 1 });
}
```

```typescript
    auditPassRate: analytics.auditPassRate,
    tokenStats: analytics.tokenStats ?? null,
    hookRisks,
    recentImports,
    pipelineErrors,
    styleStatus,
  } satisfies HealthOverviewData);
});
```

**Core 引用链**：
- `state.loadChapterIndexStrict()` + `computeAnalytics()`：审计和 Token 单一口径
- `summarizePendingHookHealth()`：伏笔解析、类型归一化和生命周期单一口径
- `listFoundationSources()`：来源索引校验和查询单一口径
- 不调用 Core Agent，不改变写作主链路

**数据口径一致性说明**：
- 不允许 Health 独立复制 Analytics 计算逻辑。
- 没有可靠来源的指标必须返回 `unavailable`；前端显示 `--` 和原因。
- `readOptionalText()` 只把 `ENOENT` 当作“无数据”，JSON 损坏和权限错误继续抛出。

#### 7.3.3 健康卡片组件骨架

```typescript
import { useApi } from "../../hooks/use-api";

type Metric<T> =
  | { status: "available"; value: T }
  | { status: "unavailable"; reason: string };

interface HealthData {
  auditPassRate: number;
  tokenStats: { totalTokens: number; avgTokensPerChapter: number } | null;
  hookRisks: Metric<{ total: number; stale: number; criticalIds: string[] }>;
  recentImports: Metric<number>;
  pipelineErrors: Metric<number>;
  styleStatus: Metric<"profile-ready">;
}

function metricValue<T>(metric: Metric<T>, format: (value: T) => string): string {
  return metric.status === "available" ? format(metric.value) : "--";
}

export function BookHealthCards({ bookId, nav, t }) {
  const { data, loading, error, refetch } =
    useApi<HealthData>(`/books/${encodeURIComponent(bookId)}/health`);

  if (loading) return <HealthCardsSkeleton />;
  if (error || !data) return <HealthCardsError error={error} onRetry={refetch} />;

  return (
    <div className="grid grid-cols-2 md:grid-cols-3 gap-3">
      <StatCard
        label={t("health.auditPassRate")}
        value={`${data.auditPassRate}%`}
        onClick={() => nav.toBookSection(bookId, "audit")}
      />
      <StatCard
        label={t("health.totalTokens")}
        value={data.tokenStats?.totalTokens.toLocaleString() ?? "--"}
      />
      <StatCard
        label={t("health.hookRisk")}
        value={metricValue(data.hookRisks, (v) => `${v.stale}/${v.total}`)}
        onClick={() => nav.toBookSection(bookId, "hooks")}
      />
      <StatCard
        label={t("health.recentImports")}
        value={metricValue(data.recentImports, String)}
        onClick={() => nav.toBookSection(bookId, "sources")}
      />
      <StatCard
        label={t("health.pipelineErrors")}
        value={metricValue(data.pipelineErrors, String)}
        sub={data.pipelineErrors.status === "unavailable" ? t("health.notAvailable") : undefined}
      />
      <StatCard
        label={t("health.styleStatus")}
        value={data.styleStatus.status === "available"
          ? t("health.profileReady")
          : "--"}
        sub={data.styleStatus.status === "unavailable" ? t("health.notAvailable") : undefined}
      />
    </div>
  );
}
```

`StatCard`、错误块和加载骨架应复用或提取自现有 `BookOverviewSection`，不要再建立第二套视觉与错误处理体系。

### 7.4 通用注意事项

#### 7.4.1 数据一致性保障

```
健康总览与 Analytics 共享数据口径的原则：
  1. Analytics 与 Health 都通过 state.loadChapterIndexStrict() 读取章节索引
  2. 两个端点都调用 Core computeAnalytics()
  3. Health 只从 AnalyticsData 中选取摘要字段
  4. Hook 与来源数据同样调用 Core 公共摘要/查询函数
  5. 禁止以 TODO 代替当前口径统一
```

#### 7.4.2 错误处理策略

```
所有新端点：
  - 书籍不存在 → 404
  - 可选文件不存在（如 pending_hooks.md）→ 返回空集合或 unavailable
  - 指标尚未采集 → { status: "unavailable", reason }
  - JSON/schema 解析失败 → 500 或结构化诊断，不伪装成空数据
  - 来源不存在 → 404
  - 书籍写锁冲突 → 409
  - 未知异常 → 500 + 稳定错误码，详细信息写服务端日志

前端所有 useApi 调用：
  - 已内置 loading / error 状态处理
  - 读取失败显示局部错误和重试，不阻塞其他 Overview 模块
  - 归档/删除失败保留确认上下文并显示错误，不提前移除表格行
  - unavailable 显示 "-- / 暂不可用"，不得显示为 0 或"良好"
```

#### 7.4.4 验收与回归测试

1. **资料来源**
   - 书籍不存在返回 404，非法 `sourceId` 返回 400。
   - 空来源仍可进入 Sources section，并显示导入入口。
   - 索引损坏返回诊断错误，不显示为“暂无来源”。
   - 归档与 Foundation 导入竞争时不会丢失索引记录；归档失败可恢复原文件和原索引。

2. **统一输入**
   - `.jsonl.md`、`.json.markdown`、`.htm` 等复合/别名扩展名与现有页面行为一致。
   - 三个页面接入后分别执行原有回归测试；未接入页面不宣称自动获得分片能力。
   - 超过服务端字符限制时保持 `413` 语义，不静默截断。

3. **健康总览**
   - Health 的 `auditPassRate`、`tokenStats` 与 `/analytics` 对同一本书完全一致。
   - 无持久化错误历史或无风格档案时显示“暂不可用”，不显示 `0` 或“正常”。
   - Hook 文件缺失返回空风险集合；Hook 文件损坏返回可见诊断。
   - 健康卡片加载失败不阻塞 Overview 的章节、审计、运行事件等原有模块。
