## 三、P1 级实现 Bug

### Bug-1：章节版本清理使用字典序排序，导致新版本被优先删除

**文件**：`packages/studio/src/api/server.ts`  
**位置**：`PUT /api/v1/books/:id/chapters/:num` 版本清理逻辑（约第 1950-1980 行）  
**严重度**：P1（数据损坏——版本历史顺序错乱）

**问题代码**：
```typescript
if (versionFiles.length >= MAX_CHAPTER_VERSIONS) {
  const sorted = [...versionFiles].sort();  // ← 字典序
  const toRemove = sorted.slice(0, sorted.length - MAX_CHAPTER_VERSIONS + 1);
  for (const stale of toRemove) {
    await unlink(join(versionsDir, stale)).catch(() => {});
  }
}
```

**根因分析**：
- 版本文件名形如 `0001_v1.md`、`0001_v10.md`、`0001_v2.md` … `0001_v50.md`
- JavaScript `Array.prototype.sort()` 默认按字典序（Unicode 码点）排序
- 字典序结果：`v1`, `v10`, `v11`, ..., `v19`, `v2`, `v20`, ...

**错误后果**：
- 当版本数 ≤ 50 时，本次写入后刚好 50 个，只会删除字典序第一个（恰好是 `v1`），结果正确
- 当版本数 > 50 时，会按字典序删除"最前面"的文件，导致 `v10-v19` 等较新版本被删除，而 `v2-v9` 等更旧版本保留
- 例如 60 个版本时，会删除 `v1, v10, v11, ..., v19, v2`，保留 `v3-v9, v20-v60`，版本历史严重错乱

**复现**：
1. 对同一章节保存 51 次以上
2. 检查 `chapters/versions/` 目录
3. 预期应保留 `v2-v51`（最新的 50 个），实际可能丢失 `v10-v11` 等

**修复建议**：
```typescript
const sorted = [...versionFiles].sort((a, b) => {
  const revA = parseInt(a.slice(versionPrefix.length, -3), 10);
  const revB = parseInt(b.slice(versionPrefix.length, -3), 10);
  return revA - revB;
});
```

## 四、P2 级潜在问题

### Issue-1：`BookCreate.tsx` 未同步新的书籍创建状态字符串

**文件**：`packages/studio/src/pages/BookCreate.tsx:480`  
**严重度**：P2（死代码中的潜伏 bug）

**问题代码**：
```typescript
const status = await fetchStatus(bookId);
lastKnownStatus = status.status;
if (status.status === "error") {  // ← 后端已改为 "failed"
  throw new Error(status.error ?? `Book "${bookId}" failed to create`);
}
// ...
if (lastKnownStatus === "creating") {  // ← 现在前面还有 "queued" 状态
  break;
}
```

**根因分析**：
- 6 月 12 日 `server.ts` 将 `bookCreateStatus` 的状态枚举从 `"creating" | "error" | "completed"` 改为 `"queued" | "creating" | "completed" | "failed"`
- `BookCreate.tsx` 是死代码（未被 `App.tsx` 引用），但 `waitForBookReady` 工具函数可能被测试或其他模块引用
- 当前代码仍检查 `"error"` 和 `"creating"`，无法识别 `"failed"` 和 `"queued"`

**后果**：
- 若未来复活 `BookCreate.tsx`，失败状态无法被识别，会轮询到最大次数后才抛错
- 看到 `"failed"` 时，会当作仍在创建中处理，用户体验差

**修复建议**：
```typescript
if (status.status === "failed") {
  throw new Error(status.error ?? `Book "${bookId}" failed to create`);
}
if (status.status === "queued" || status.status === "creating") {
  // continue polling
}
```

### Issue-2：`semantic-duplication.ts` "首语重复"正则存在误报与性能风险

**文件**：`packages/core/src/utils/semantic-duplication.ts:217`  
**严重度**：P2（误报 / 潜在性能问题）

**问题代码**：
```typescript
/^([\s\S]{2,4})(?:[\s\S]*?)\n\n[\s\S]{0,200}\n\1/gm,
```

**问题分析**：
1. **误报**：`m` 标志下 `^` 匹配任意行首，`[\s\S]{2,4}` 会捕获任何 2-4 个字符（包括空格、标点、数字）。例如两段都以 `"  "`（两个空格）或 `"第1"` 开头都会被判定为"首语重复"。
2. **性能**：`([\s\S]{2,4})` 有 4 种捕获长度，`*?` 非贪婪匹配配合 `\n\1` 回溯引用，在找不到匹配时可能产生较多回溯。虽然 200 字符窗口限制了规模，但在长文本上仍有 CPU 尖峰风险。
3. **语义变化**：原正则是 `^(.){2,4}`（逻辑错误，只能匹配连续相同字符），新正则虽然修复了原 bug，但改为捕获任意字符，误报率提高。

**修复建议**：
- 限制捕获组为 CJK 字符或字母：`([\u4e00-\u9fff]{2,4}|[a-zA-Z]{2,4})`
- 或增加断言，确保捕获的是"词"而非标点/空格
- 对复杂模式改用字符串分割算法替代正则

### Issue-3：`server.test.ts` 轮询等待时间过短，可能导致测试不稳定

**文件**：`packages/studio/src/api/server.test.ts`  
**严重度**：P2（测试可靠性）

**问题代码模式**（多处）：
```typescript
for (let i = 0; i < 20; i++) {
  const status = await app.request(".../create-status");
  if (status.status === 200) {
    lastStatus = await status.json() as { status: string; error?: string };
    if (lastStatus.status === "failed" || lastStatus.status === "completed") break;
  }
  await new Promise((r) => setTimeout(r, 50));
}
```

**问题分析**：
- 最大等待时间 = 20 × 50ms = **1 秒**
- 书籍创建涉及 LLM 配置、管道初始化、Agent 调用，在慢机器或高负载 CI 环境下可能超过 1 秒
- 若异步操作未完成，测试会超时失败或断言 `lastStatus` 非 null 失败

**修复建议**：
- 将轮询次数提高到 100 次（5 秒）或等待时间提高到 200ms
- 或引入可配置的测试等待上限

### Issue-4：`withChapterLock` 在上一把锁失败时会跳过当前操作

**文件**：`packages/studio/src/api/server.ts`（章节保存锁）  
**严重度**：P2（错误传播 / UX）

**问题代码**：
```typescript
const previous = chapterLocks.get(key) ?? Promise.resolve();
const next = previous.then(() => fn());  // ← previous reject 时 fn 不会执行
chapterLocks.set(key, next as Promise<void>);
next.finally(() => { ... });
return next;
```

**问题分析**：
- `previous.then(() => fn())` 没有传入 `onRejected` 回调。如果 `previous`  rejected，`fn()` 不会执行，`next` 直接 reject 并携带 `previous` 的错误
- 这意味着：如果某次保存因磁盘错误失败，后续保存请求会立即失败，并返回前一次的错误信息，而不是尝试当前保存
- 虽然 `finally` 会清理锁，不会永久阻塞，但用户会看到"上一次的失败"而不是"当前操作的结果"

**修复建议**：
```typescript
const next = previous
  .catch(() => { /* 忽略前一次错误，继续执行当前操作 */ })
  .then(() => fn());
```

或根据业务需求决定是否应排队而非跳过。
