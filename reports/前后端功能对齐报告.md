### 1.2 对齐判定标准

| 状态 | 含义 |
|------|------|
| **已对齐** | 后端端点存在且前端有明确调用方（页面/组件/hook） |
| **后端有余 / 前端未接入** | 后端完整实现，但在前端源码中未找到直接调用 |
| **骨架/后端有余** | 后端端点存在，但未真正调用 Core Agent / 未集成 Pipeline，仅做简单数据读写 |
| **前端有余 / 后端未实现** | 前端代码引用或 UI 入口存在，但后端无对应端点 |
| **重复/兼容/废弃** | 功能与另一端点重叠，或已被新接口替代 |
| **CLI 未命令化** | Core 已提供能力，但 CLI 尚未暴露为子命令 |

## 5. Core 导出 vs CLI 命令消费对齐

### 5.1 CLI 高层入口

CLI 主要通过 **7 个高层入口** 消费 Core：

1. `PipelineRunner`（写作/审计/修订管线）
2. `Scheduler`（调度/守护）
3. `StateManager`（项目状态）
4. `ConsolidatorAgent`（整合 Agent）
5. `runAgentLoop`（Agent 循环）
6. 各命令直接 import 的工具函数（如 `resolveWritingReviewRetries`）
7. 类型与 schema（`BookConfig`、`ChapterMeta`、`ProjectConfig`、`WritingConfig`、`AuthorChapterIntent` 等）

### 5.2 Core 导出被 CLI 直接引用情况

- **约 65 个 Core 导出**被 CLI 直接引用。
- 大量 Agent / 工具由 `PipelineRunner` 内部封装，CLI 不直接引用。

## 7. 主要风险与建议

### 7.1 P0 阻塞：Stage 5 数据证据不足（未改变）

- **现状**：代码与测试已闭环，但 Beta Reader 校准集仅 12/30 对，无人工标签，无校准报告。
- **建议**：继续第三批数据闭环，扩展校准集到 ≥30 对，统一胜率口径，输出 `Beta Reader 校准报告`。

### 7.2 已修复：Studio 测试不稳定

- **现象**：原 `pnpm --filter @actalk/inkos-studio test` 全量运行时 `server.test.ts` 出现大量失败。
- **根因**：模块状态污染。
- **修复**：`packages/core/vitest.config.ts` 改为 `pool: "forks"` + `isolate: true`，Core/Studio 全量测试已连续稳定通过。

### 7.3 功能缺口（按优先级）

| 优先级 | 缺口 | 建议 |
|--------|------|------|
| **P0** | **事件链 / 场景模板 / 声音画像 "有端点无能力"** | 这是 `2f1e452` 引入的最大风险：端点和模型存在，但未调用 Core Agent，也未集成 Pipeline/Writer。建议二选一：**(a)** 快速补完真正调用 `EventChainExtractor` / `VoiceProfileAnalyzer` / `SceneTemplate` 的 server 实现，并在 Planner/Writer prompt 中消费；**(b)** 若暂不投入使用，应在端点层面明确返回 `not implemented` 或移除，避免误导 |
| **P0** | **作家蒸馏页面未挂载** | `DistillationPage.tsx` 已完成只读展示，但 **未加入 `App.tsx` 路由** 与侧边栏/作家页入口。建议立即挂载，并补充"生成/更新/发布/应用到书籍"操作按钮 |
| **P1** | **计划变体未暴露** | Core Planner 已支持 `generateAlternatives`，但 `write-next` / `draft` / `write-preview` 端点未暴露参数，前端无选择 UI。建议在 `write-preview` 返回备选方案，或在 `write-next` 支持 `withAlternatives` 参数 |
| **P1** | **端点锁覆盖不完整** | 当前仅章节保存 `PUT /chapters/:num` 使用内存锁，`write-next/draft/rewrite` 等异步写操作未使用。建议将 `withChapterLock` 扩展到所有会写入同一章节文件的端点，或改用文件级锁 |
| **P1** | **作家诊断历史读取前端缺失** | 在 `StyleManager` 或新增 `StyleDiagnosticsHistoryPanel` 中调用 `GET .../diagnostics` 与单条读取 |

| **P2** | Beta Reader shadow / Prompt Manifest 可视化 | 在 `BookRuntimeSection` 或新增标签页展示 shadow 结果与 prompt 清单 |

- **Stage 5 准入条件未满足**：基线数据集、Beta Reader 校准数据、人工评测流程均未建立。
- **建议**：在启动 Stage 5 前，先完成：
  1. 至少 3–5 本不同题材书籍的完整写作/审计/修订基线数据。
  2. Beta Reader 评分与人工评分的配对校准（paired-preference）。
  3. 异构模型约束与评分一致性阈值定义。

## 8. 结论

1. **后端 API 覆盖完整**：120+ 端点覆盖了 InkOS 当前所有核心创作链路，异步任务统一通过 SSE 通知，架构清晰。
2. **前端调用覆盖率较高**：绝大多数后端能力均有对应页面/组件消费，核心创作链路（建书、写作、审计、修订、导出、风格、导入、角色、意图）已对齐。
3. **commit `2f1e452` 新增能力存在严重"空心化"风险**：
   - 事件链、场景模板、声音画像端点已注册，但 **未真正调用 Core Agent**，也未集成到 Pipeline/Writer，仅为骨架实现。
   - 作家蒸馏页面 `DistillationPage.tsx` 已存在，但 **未挂载到应用路由**，且无生成/更新/发布/应用操作。
   - 计划变体已在 Core Planner 落地，但 **Studio 端点未暴露参数**，前端无选择入口。
   - 端点锁仅在章节保存端点生效，**未覆盖异步写作操作**。
4. **CLI 与 Studio 能力差距扩大**：CLI 不仅未覆盖风格库、角色卡、章节目标/意图、雷达等旧能力，也未覆盖事件链、声音画像、场景模板、蒸馏等新能力。
5. **测试稳定性已修复**：Core/Studio 通过 `pool: "forks"` + `isolate: true` 解决模块污染，全量测试稳定通过。
6. **Stage 5 仍被数据证据阻塞**：代码测试已闭环，但 Beta Reader 校准数据不足，不能进入 Stage 5。
