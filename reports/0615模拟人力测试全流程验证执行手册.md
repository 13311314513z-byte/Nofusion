> 版本：v1.0
> 适用项目：`@actalk/inkos` v1.4.1
> 目标读者：质量工程师、算法工程师、测试负责人、产品经理
> 配套脚本：`scripts/extend-beta-calibration.mjs`、`scripts/beta-calibration-report.mjs`、`scripts/preference-eval.mjs`、`scripts/beta-reader-calibrate.mjs`
> 配套报告：`reports/BetaReader校准报告.md`、`reports/前后端功能对齐报告.md`

---

## 一、文档目的

本手册定义一套**可重复执行**的“模拟人力测试”流程，用于验证 InkOS 各阶段质量改进（模型、Prompt、Agent、补丁规则）对小说文本真实阅读体验的影响。

所谓“模拟人力测试”，是指：
- 用 **Beta Reader Agent**（LLM 驱动的模拟读者）替代部分人工初筛；
- 用 **成对偏好比较（paired preference）** 替代绝对评分，降低评分漂移；
- 用 **统计胜率 + 置信区间 + Reader 一致性** 替代主观“感觉更好”；
- 最终用**人工双盲抽检**校准模拟 Reader，确保数字可解释。

> **核心原则**：测试流程本身必须可复现、可审计、可回滚。任何一次“XX 改进了”的结论，都必须能追溯到具体 pair、模型配置、Git commit、Reader prompt 版本。

## 二、术语与口径

| 术语 | 定义 | 本项目对应 |
|------|------|-----------|
| **Pair（对子）** | 同一段文本的两种版本并排呈现给 Reader 做一次判断 | `calibration-results.csv` 中 `pairId` |
| **Version（版本/变体）** | 同一章节在某一组生成参数下的输出 | `low-temp` / `default` / `high-temp`，或模型 A / 模型 B |
| **Dimension（维度）** | 成对比较的评分角度 | `engagement`、`character`、`emotion`、`clarity`、`expectation` |
| **Answer（答案）** | Reader 在某一维度上的选择 | `A` = 选 A 侧；`B` = 选 B 侧；`tie` = 平局；`unable` = 无法判断 |
| **Confidence（置信度）** | Reader 对该判断的自信程度 | 1–4 分，4 分最高 |
| **版本胜率** | 某变体在所有含它的 pair 中被选中的次数 / （参赛次数 - 平局数） | 本手册统一口径 |
| **B 侧胜率** | 选择 B 的次数 / 非平局总数 | 兼容旧 `preference-eval.mjs` 口径 |
| **位置偏差** | A/B 放置位置导致的系统性偏好 | A 侧胜率应接近 50% |
| **Reader 一致性** | 多个 Reader 对同一 pair 判断的一致程度 | Fleiss' Kappa |

## 五、阶段 1：测试设计

### 5.1 明确要验证的假设

每次测试只验证**一个核心假设**。示例：

| 假设编号 | 假设内容 | 对应变量 | 成功标准 |
|:--------:|----------|----------|----------|
| H1 | 降低 temperature 可减少 AI 痕迹 | `temperature`（高 vs 低） | low-temp 在 `clarity` / `character` 维度胜率显著 > 50% |
| H2 | 加入意图注入可提升读者期待感 | `authorIntentBlock`（有 vs 无） | 有意图块在 `expectation` 维度胜率 > 55% |
| H3 | Issue Normalizer 减少跨章重复报错 | `issueNormalizer`（开 vs 关） | 开启后跨章 issue 计数更稳定（辅助审计日志验证） |
| H4 | 风格蒸馏规则可提升风格一致性 | `distillationRules`（有 vs 无） | 有规则时与目标作家 overallMatchScore 提升 |

> **原则**：一次测试只变一个变量，其他全部固定（模型、prompt、书籍、章节、温度）。否则无法归因。

### 5.2 选择测试类型

| 测试类型 | 适用场景 | 本项目工具 |
|----------|----------|-----------|
| 成对偏好 A/B | 两种生成版本的直接对比 | `scripts/preference-eval.mjs` |
| 单版本多维度评分 | 评估一个版本的综合质量 | `BetaReader` shadow |
| 修订前后对比 | 验证审计/修订是否真改善 | `PipelineRunner.reviseChapter` + A/B |
| 跨模型循环赛 | 3+ 模型/参数同时比较 | `scripts/beta-calibration-report.mjs` |

### 5.3 确定成功标准

在测试开始前写下：

```
成功标准（Go）:
1. 目标变体在目标维度胜率 ≥ 55%（Wilson 95% CI 下限 > 50%）
2. A/B 位置偏差在 45%–55% 之间
3. 至少 2 位独立 Reader 的 Fleiss' Kappa ≥ 0.4
4. 人工抽检一致性 ≥ 70%

失败标准（No-Go）:
1. 胜率 CI 跨越 50%
2. 位置偏差 > 60% / < 40%
3. Reader 一致性 Kappa < 0.2
```

## 六、阶段 2：样本准备

### 6.1 选择书籍与章节

| 维度 | 建议 |
|------|------|
| 数量 | 至少 1 本书，覆盖 opening / transition / conflict / climax / resolution 各 2 章 |
| 长度 | 每章 1,500–4,000 字，避免超长章节导致 Reader 注意力分散 |
| 题材 | 优先选择项目已有书籍，如 `药`、`通兰民间故事集`、`test-book-0609` |
| 去标识化 | 删除章节号、版本号、模型名，避免 Reader 被先验信息干扰 |

### 6.2 生成对比版本

以 H1（temperature）为例：

```bash
# 1. 使用默认参数生成版本 A
pnpm --filter @actalk/inkos inkos write <bookId> --chapter <num> --temperature 0.7
mv books/<bookId>/chapters/<num>.md books/<bookId>/tmp/versionA.md

# 2. 使用低温度生成版本 B
pnpm --filter @actalk/inkos inkos write <bookId> --chapter <num> --temperature 0.3
mv books/<bookId>/chapters/<num>.md books/<bookId>/tmp/versionB.md
```

> 实际 CLI 参数以 `inkos write --help` 为准；若 CLI 未暴露温度参数，可直接调用 Core `PipelineRunner` 脚本。

### 6.3 控制变量清单

| 变量 | 控制要求 | 记录位置 |
|------|----------|----------|
| 模型 ID | A/B 使用同一服务商/模型 | `calibration-results.csv` 的 `versionA`/`versionB` 备注 |
| Prompt 版本 | 固定 Git commit hash | 报告附录 |
| 书籍/章节 | 同章节对比 | `pairId` 元数据 |
| 长度 | A/B 字数差异 < 20% | 预检脚本 |
| 后处理 | 均不经过人工修改 | 测试协议 |

### 6.4 去标识化与随机化

- 将版本 A/B 的左右位置随机化（50% 概率交换）。
- 记录真实映射关系，但不对 Reader 暴露。
- 每对使用独立 `pairId`：`synth-XXX`。

## 七、阶段 3：模拟测试员配置

### 7.1 Beta Reader Agent 配置

`BetaReader` Agent 的配置项包括：

| 配置项 | 说明 | 建议 |
|--------|------|------|
| `readerModel` | Reader 使用的 LLM | 与 Writer 使用不同模型，避免“自评”偏差 |
| `temperature` | Reader 采样温度 | 0.1–0.3，保证判断稳定 |
| `promptVersion` | Reader system prompt 版本 | 固定并记录 hash |
| `dimensions` | 评测维度 | 默认 5 维：`engagement`、`character`、`emotion`、`clarity`、`expectation` |

### 7.2 多 Reader 异构策略

为降低单一 Reader 的模型偏差，建议配置 ≥2 个异构 Reader：

| Reader ID | 模型 | 角色定位 | 用途 |
|-----------|------|----------|------|
| `beta-reader-sim` | 轻量模型 | 快速初筛 | 大规模 pair 初评 |
| `beta-reader-crit` | 强模型 | 严格批评者 | 复检边界 pair |
| `beta-reader-audience` | 强模型 | 普通网文读者 | 模拟目标用户 |

> 若只有一个 Reader，Fleiss' Kappa 无法计算，报告应明确标注“单 Reader，一致性未验证”。

### 7.3 Reader Prompt 校准

在正式测试前，先用 5–10 对已知人工标签的 pair 校准 Reader：

1. 准备 10 对人工已标注的 A/B；
2. 用 Reader 跑一遍；
3. 计算 Reader-vs-人工 一致性；
4. 若一致性 < 70%，调整 Reader system prompt，重新校准。

## 八、阶段 4：执行测试

### 8.1 使用已有脚本批量执行

```bash
# 方式 1：使用 Core BetaReader shadow（推荐，与 Pipeline 集成）
node scripts/beta-reader-calibrate.mjs books/<bookId> --output reports/<test-name>-raw.md

# 方式 2：使用 preference-eval 处理已有 CSV
node scripts/preference-eval.mjs reports/baseline-data/calibration-results.csv --output reports/<test-name>-preference.md
```

### 8.2 自定义执行（新实验）

若要进行新实验，按以下格式生成 CSV：

```csv
pairId,readerId,questionId,questionText,answer,confidence,timestamp,versionA,versionB
synth-037,beta-reader-sim,engagement,哪个版本更让你想继续读下去？,A,3,2026-06-15T10:00:00Z,low-temp,default
synth-037,beta-reader-sim,character,哪个版本的角色更真实可信？,B,4,2026-06-15T10:00:00Z,low-temp,default
...
```

每对必须包含 5 个维度，answer 只能是 `A`、`B`、`tie`、`unable`。

### 8.3 异常处理

| 异常 | 处理 |
|------|------|
| Reader 返回无法解析 | 标记为 `unable`，不纳入胜率计算 |
| Reader 拒绝判断 | 标记为 `unable` |
| 网络/模型失败 | 重试 3 次，仍失败则记录失败日志，pair 作废 |
| A/B 字数差异 > 20% | 剔除该 pair，不纳入统计 |

## 九、阶段 5：数据收集与校验

### 9.1 数据汇合格式

统一 CSV 路径：`reports/baseline-data/calibration-results.csv`

字段约束：

| 字段 | 类型 | 约束 |
|------|------|------|
| `pairId` | string | 全局唯一，格式 `synth-XXX` 或实验名前缀 |
| `readerId` | string | 标识具体 Reader 配置 |
| `questionId` | enum | `engagement` / `character` / `emotion` / `clarity` / `expectation` |
| `questionText` | string | 固定问题文案 |
| `answer` | enum | `A` / `B` / `tie` / `unable` |
| `confidence` | int | 1–4 |
| `timestamp` | ISO string | UTC |
| `versionA` | string | 变体名，如 `low-temp`、`default`、`high-temp` |
| `versionB` | string | 同上 |

### 9.2 数据校验清单

```bash
node scripts/beta-calibration-report.mjs --csv=reports/baseline-data/calibration-results.csv --output=reports/BetaReader校准报告.md
```

校验项：

1. pair 总数 ≥ 30；
2. 每个 pair 有且仅有 5 条记录；
3. `answer` 取值合法；
4. `confidence` 在 1–4 之间；
5. `versionA` / `versionB` 不为空且互不相同；
6. 每个 Reader 对每个 pair 最多 5 条；
7. 无明显时间戳异常（如未来时间）。

### 9.3 无效数据处理

- `unable` 记录：保留但不参与胜率计算；
- 重复记录：按 `pairId + readerId + questionId` 去重，保留最新；
- 缺失维度：该 pair 整体剔除。

## 十、阶段 6：统计分析

### 10.1 执行统计脚本

```bash
# 生成统一口径报告
node scripts/beta-calibration-report.mjs \
  --csv=reports/baseline-data/calibration-results.csv \
  --output=reports/BetaReader校准报告.md
```

### 10.2 关键指标解读

| 指标 | 健康阈值 | 行动 |
|------|:--------:|------|
| 版本胜率 CI 下限 > 50% | ✅ | 该变体在对应维度有显著偏好 |
| 版本胜率 CI 跨越 50% | ⚠️ | 差异不显著，需增加样本或换变量 |
| A/B 位置偏差 45%–55% | ✅ | 无位置偏差 |
| A/B 位置偏差 < 40% 或 > 60% | ❌ | 存在位置偏差，需重新随机化 |
| 平局率 < 15% | ✅ | Reader 能区分 |
| 平局率 > 30% | ⚠️ | A/B 差异太小或 Reader 过于保守 |
| Bradley-Terry 最高分 > 0.8 | ✅ | 存在明确优势变体 |
| Reader 一致性 Kappa ≥ 0.4 | ✅ | 可接受 |
| Reader 一致性 Kappa < 0.2 | ❌ | Reader 不可靠，需校准或更换 |

### 10.3 与人工标签对比

若有人工标签，计算：

```
人机一致性 = 模拟 Reader 与人工判断一致的对数 / 人工参与判断的对数
```

目标 ≥ 70%。若低于 60%，说明模拟 Reader 不可信，必须调整 prompt 或模型。

## 十一、阶段 7：人工校准

### 11.1 双盲抽检

1. 从所有 pair 中随机抽取 20%（至少 10 对）；
2. 邀请 2 位人工标注员（目标读者或编辑）；
3. 标注员只能看到去标识化的 A/B 文本和 5 个问题；
4. 两位标注员彼此独立，不得讨论；
5. 记录人工答案到 `reports/baseline-data/human-labels.csv`：

```csv
pairId,questionId,humanReader1,humanReader2,confidence1,confidence2,notes
synth-001,engagement,A,B,4,3,
```

### 11.2 计算人工一致性

```bash
# 可使用 beta-calibration-report.mjs 的逻辑扩展，或临时脚本
node scripts/human-reader-agreement.mjs \
  --human=reports/baseline-data/human-labels.csv \
  --beta=reports/baseline-data/calibration-results.csv
```

输出：
- 人工间一致性（Kappa）
- 人工 vs Beta Reader 一致性
- 分歧案例分析

### 11.3 校准动作

| 情况 | 动作 |
|------|------|
| 人机一致性 ≥ 70% | 模拟 Reader 可信，可扩大测试规模 |
| 人机一致性 50%–70% | 调整 Reader prompt，重点校准分歧维度 |
| 人机一致性 < 50% | 暂停模拟测试，重新设计 Reader 角色或换模型 |
| 人工间一致性低 | 问题定义不清，需重新培训标注员或修改问题 |

### 附录 B：常见问题

**Q1: 为什么一次测试只能变一个变量？**
> 多变量同时变化会导致无法归因。若确实需要测试组合效果，应设计正交实验表（DOE），而非简单叠加。

**Q2: 模拟 Reader 能完全替代人工吗？**
> 不能。模拟 Reader 适合初筛和大规模对比，最终质量判断仍需人工双盲抽检校准。

**Q3: 胜率 55% 算显著吗？**
> 取决于样本量和置信区间。50 对时 55% 的 Wilson CI 可能跨越 50%；100+ 对时 55% 通常显著。本手册以 CI 下限 > 50% 为准。

**Q4: 如何处理模型输出不稳定？**
> 固定 seed、temperature、prompt 版本；每个 pair 只生成一次，避免多次采样污染对照。

**Q5: 校准集和测试集要分开吗？**
> 要。校准集用于调整 Reader prompt 和口径；测试集用于最终验证。建议 70% / 30% 划分，测试集在最终验证前不用于任何调参。
