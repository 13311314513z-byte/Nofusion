## 一、执行摘要

| 功能 | 当前完成度 | 实现难度 | 推荐路径 |
|------|-----------|----------|----------|
| **功能一：建书前资料导入** | **~60%**（框架已就绪，缺格式扩展与 UX 闭环） | ⭐⭐⭐☆☆ 中等 |  resurrect 死代码 + 扩展格式 + 增加 source 预览 |

## 二、功能一盘点：建书前资料导入

### 2.1 当前已实现的能力

#### A. 书籍创建时的 Foundation Source 导入（✅ 已实现）

**入口点：**

| 入口 | 文件 | 机制 |
|------|------|------|
| **Studio Chat** | `ChatPage.tsx` | 用户与 AI 对话，AI 调用 `create_book` 工具，生成 `creationDraft`，前端 POST `/api/v1/books/create` |
| **CLI** | `packages/cli/src/commands/book.ts` | `inkos book create --title <t> --genre <g> --brief <path> --source <paths...>` |
| **Studio 表单** | `BookCreate.tsx` | **已写好但未被路由引用（死代码）**，含文件上传组件 |

**API 契约**（`POST /api/v1/books/create`）：
```typescript
interface StudioCreateBookBody {
  title: string;
  genre: string;
  platform: "tomato" | "feilu" | "qidian" | "other";
  blurb?: string;                    // 创作简介/意图说明
  foundationSources?: FoundationSourceInput[];  // 前置资料数组
  language?: "zh" | "en";
  // ... 其他 BookConfig 字段
}

interface FoundationSourceInput {
  sourceName: string;
  fileType: "md" | "txt" | "jsonl" | "json" | "ts" | "js" | "html" | "htm" | "css";
  text: string;
  purpose?: "auto" | "world" | "character" | "era" | "plot" | "rule" | "chapter" | "style";
  normalized?: boolean;
}
```

**处理流程：**
1. 服务器将 `foundationSources` 组装为 `FoundationSourceBundle`（去重、校验、拼接 `contextBlock`）
2. `contextBlock` + `blurb` 合并为 `externalContext`，注入 `ArchitectAgent.generateFoundation()`
3. Architect 基于用户意图 + 资料生成世界观、角色、卷纲、伏笔等
4. 创建完成后，source bundle 持久化到 `story/sources/index.json` + `{sourceId}.md`

**限制：**
- 最大 20 个文件 / 总计 1,000,000 字符（处理后）
- `chapter` 和 `style` purpose 会被分流到章节导入和文风导入管线，不进入 foundation

#### B. 已存在书籍的 Foundation 增量导入（✅ 已实现）

**Plan → Commit 模式：**
- `POST /books/:id/import/foundation/plan` — 预览资料将如何修改现有 foundation
- `POST /books/:id/import/foundation/commit` — 应用变更（含乐观锁 `expectedRevision`、自动备份）

**功能：**
- 计算角色差异（新增/更新/删除）
- 生成 foundation 预览（story_frame、volume_map、roles 等）
- 支持 `supplement`（追加）和 `rebuild`（重建）两种模式

#### C. 文档读取与预处理（✅ 已实现）

`document-reader.ts` 支持：

| 格式 | 提取能力 | 验证 |
|------|----------|------|
| `.md` | 剥离 YAML frontmatter、Markdown 标记 | ✅ |
| `.txt` | 直接读取 | ✅ |
| `.jsonl` / `.json` | 智能提取 `text`/`content`/`message` 等字段，过滤 metadata | ✅ |
| `.ts` / `.js` | 提取含中文字符的字符串字面量和注释 | ✅ |
| `.html` / `.htm` | 剥离标签、解码实体 | ✅ |
| `.css` | 几乎无有效内容 | ⚠️ |

**预处理管线**（独立端点，未在导入时自动应用）：
- `POST /style/preprocess` — 去代码块、URL、邮箱、时间戳、ID、去重等
- `POST /style/relayout` — 对话格式化、段落合并、引号规范化

### 2.2 当前缺口与问题

| 缺口 | 严重程度 | 说明 |
|------|----------|------|
| **BookCreate.tsx 死代码** | P0 | 最完整的表单 UI 已写好但未被引用，用户只能通过 Chat 或 CLI 创建书籍 |
| **不支持 DOCX/PDF/EPUB** | P1 | 用户手稿/资料最常见格式无法导入（docx-indexer 可补此缺口，见融合报告） |
| **不支持图片/OCR** | P2 | 手写笔记、扫描件无法处理 |
| **导入时无自动预处理** | P2 | `document-reader` 提取后未自动过 `text-preprocessor`，噪声（URL、代码块、重复 prompt 标记）可能进入 Architect 上下文 |
| **Source 预览/管理体验弱** | P2 | 上传后无法预览提取内容、无法逐条编辑 purpose、无法查看字符数和警告 |
| **无"意图资料"独立概念** | P2 | 当前 `blurb` + `foundationSources` 混在一起，缺乏"这是用户的创作意图说明"与"这是参考资料"的语义区分 |
| **Purpose 自动识别不准** | P3 | `auto` 模式依赖简单启发式，用户常需手动指定 |

### 2.3 功能一可行性评估

**结论：高度可行。现有基础设施可直接复用，主要工作是 UX 增强和格式扩展。**

| 子功能 | 技术可行性 | 工作量 | 依赖 |
|--------|-----------|--------|------|
| 激活 BookCreate 表单 + 文件上传 | ✅ 高 | 1-2 天 | 无 |
| 扩展 DOCX/PDF 支持 | ⚠️ 中 | 1-2 周 | 可选：对接 docx-indexer 微服务 |
| 导入时自动预处理 | ✅ 高 | 2-3 天 | 复用 `text-preprocessor.ts` |
| Source 预览与 purpose 编辑 | ✅ 高 | 3-5 天 | 无 |
| 意图资料与参考资料分层 | ✅ 高 | 2-3 天 | 数据模型小改 |
| Purpose 智能分类（LLM） | ✅ 高 | 3-5 天 | 调用现有 LLM provider |

## 三、功能二盘点：写作前提示卡

### 3.1 当前已实现的能力

#### A. 角色卡（Role Card）— ✅ 高度成熟

**存储：** `story/roles/<tier>/<roleId>.md`

**层级：**
- `核心角色` / `core` — 主角团
- `主要角色` / `major` — 重要配角
- `次要角色` / `minor` — 功能性配角
- `功能角色` / `functional` — 龙套/工具人

**格式（YAML frontmatter + Markdown body）：**
```yaml
---
id: cheng-jinglin
name: 程靖霖
roleTier: core
aliases: [小程]
status: active
povEligible: true
firstAppearanceChapter: 1
lastSeenChapter: 55
voiceProfileId: v1
tags:
  - 势力:回春堂
  - 职业:药店伙计
  - attr:隐忍
relationshipIds: [cheng-shiyi]
arcStage: 成长期
secrets: [父亲留下的布包地址]
doNotWrite: [禁止让他主动谈论父亲过往]
---
```

**Body 章节（Architect 生成模板）：**
- `## 核心标签` — 3-5 个关键词 + 一句理由
- `## 反差细节` — 1-2 个打破标签的细节
- `## 人物小传` —  formative 背景故事
- `## 主角弧线（起点 → 终点 → 代价）` — 弧线三段论
- `## 当前现状` — 章节 0 时的状态
- `## 关系网络` — 动态关系
- `## 内在驱动` — 欲望、动机、支付意愿
- `## 成长弧光` — 全书内心位移

**次要角色简化版：** 仅保留核心标签、反差细节、当前现状、与主角关系

**Studio UI：** `BookCharactersSection.tsx` 提供角色列表、标签筛选、关系图入口

#### B. 场景/地点 — ⚠️ 仅聚合视图，无独立卡片

**当前实现：**
- `chapter.location` — 章节元数据中的字符串字段
- `BookScenesSection.tsx` — 按 location 分组聚合章节，只读
- 检测地点别名不一致（如"通兰县城"≈"通兰县"）
- 警告未分配地点的章节

**缺失：**
- 无地点/场景独立卡片
- 无场景描述、氛围、时间、关联角色、关联事件
- 无场景级 precondition

#### C. 先决条件（Preconditions）— ⚠️ 隐式存在，未显式化

**当前等效机制：**

| 机制 | 来源 | 作用域 | 显式程度 |
|------|------|--------|----------|
| `mustKeep` | `ChapterIntent` | 单章 | 结构化 ✅ |
| `该兑现的 / 暂不掀的` | `ChapterMemo` §3 | 单章 | 散文式 ⚠️ |
| `本章 hook 账`（advance/resolve） | `ChapterMemo` §7 | 单章 | 半结构化 ⚠️ |
| `book_rules` prohibitions | `story_frame.md` frontmatter | 全书 | 结构化 ✅ |
| `mustAvoid` | `ChapterIntent` | 单章 | 结构化 ✅ |
| `forbiddenMoves` | `ChapterGoalCard` | 单章 | 结构化 ✅（但 Planner 未读） |

**问题：**
- 用户无法在写作前**主动设定**先决条件，只能被动接受 Planner 的推导结果
- 无"角色 A 与角色 B 立场冲突"这类**关系型 precondition** 的显式表达
- `forbiddenMoves` 在 `chapter_goals.json` 中已定义但 Planner 未消费

#### D. 发展方向提示（Direction Hints）— ⚠️ 多层但分散

**当前等效机制：**

| 层级 | 来源 | 内容 | 可编辑性 |
|------|------|------|----------|
| 全书级 | `author_intent.md` | 长期创作方向 | ✅ 用户可编辑 |
| 近期级 | `current_focus.md` | 接下来 1-3 章的焦点 | ✅ 用户可编辑 |
| 卷纲级 | `volume_map.md` | 卷级路线图/节拍 | ⚠️ Architect 生成，用户可改 |
| 单章级 | `ChapterIntent.goal` | ≤50 字目标 | ❌ Planner 推导 |
| 单章级 | `ChapterIntent.arcContext` | 弧线上下文 | ❌ Planner 推导 |
| 单章级 | `externalContext` | 用户指令（最高优先级） | ✅ 单次有效 |
| 运行时 | `audit_drift.md` | 上一章审计漂移 | ❌ 自动携带 |

**问题：**
- 用户缺乏一个**结构化的"本章发展方向提示卡"**界面
- `externalContext` 是单次指令，无持久化、无模板、无复用
- 无"角色 A 与角色 B 在立场上发生激烈冲突，但角色 B 另有所图"这类**关系动态提示**的显式载体

#### E. Chapter Memo — ✅ 已非常接近"提示卡"

Planner 生成的 7 节结构：
1. `当前任务` — 具体动作
2. `读者此刻在等什么` — 期望管理
3. `该兑现的 / 暂不掀的` — 节奏控制
4. `日常/过渡承担什么任务` — 功能映射
5. `关键抉择过三连问` — 选择校验
6. `章尾必须发生的改变` — 硬性改变点
7. `本章 hook 账` — 伏笔管理
8. `不要做` — 硬性禁止

**这是目前最强的"提示"机制**，但它是**Planner LLM 的输出**，用户只能在生成后查看/微调，无法在写作前**主动设计**。

#### F. ChapterGoalCard — ⚠️ 数据模型存在，消费端未接通

```typescript
interface ChapterGoalCard {
  chapterNumber: number;
  title?: string;
  povCharacter?: string;
  location?: string;
  timeOfDay?: string;
  mainConflict?: string;        // ← 关系冲突可放这里
  requiredBeats?: string[];     // ← 先决条件可放这里
  forbiddenMoves?: string[];    // ← 禁止事项
  targetMood?: string;
  hookIdsToAdvance?: string[];
  targetChars?: number;
}
```

- 持久化于 `story/chapter_goals.json`
- 服务器暴露 CRUD 端点
- **但 Planner 未读取此文件** — 这是最大的已铺设但未接通的轨道

### 3.2 当前缺口与问题

| 缺口 | 严重程度 | 说明 |
|------|----------|------|
| **无场景卡实体** | P1 | 地点只是字符串，无氛围、时间、关联角色、关联事件 |
| **ChapterGoalCard 未接入 Planner** | P1 | 已有数据模型和端点，Planner 不读 = 无效 |
| **无"写作前提示卡"UI** | P1 | 用户无法在点击"Write Next"前查看/编辑本章的约束条件 |
| **无关系动态提示载体** | P2 | "角色 A 与 B 冲突"需要跨角色的关系型表达，当前只能在角色卡的关系网络中隐式体现 |
| **externalContext 无持久化** | P2 | 单次指令，无法积累为"本章创作笔记" |
| **无提示卡模板系统** | P2 | 无法保存"战斗场景模板"、"对峙场景模板"等复用 |
| **角色卡无法快速引用** | P2 | 写作界面无角色卡浮窗/侧边栏，需切换页面查看 |

### 3.3 功能二可行性评估

**结论：中高可行性。角色卡已成熟可复用；场景卡和提示卡需新增数据模型，但可建立在 ChapterGoalCard + ChapterMemo 的既有架构上。**

| 子功能 | 技术可行性 | 工作量 | 依赖 |
|--------|-----------|--------|------|
| 场景卡数据模型 + CRUD | ✅ 高 | 3-5 天 | 参考角色卡模式 |
| ChapterGoalCard 接入 Planner | ✅ 高 | 2-3 天 | 修改 `planner.ts` 上下文组装 |
| 写作前提示卡弹窗/面板 | ✅ 高 | 5-7 天 | React 组件，复用现有 UI 风格 |
| 角色卡侧边栏/浮窗（写作界面） | ✅ 高 | 3-5 天 | 无 |
| 关系动态提示（角色 A vs B） | ✅ 中 | 5-7 天 | 需新增 relationship 表达结构 |
| 提示卡模板系统 | ✅ 中 | 1 周 | 数据模型 + UI |
| externalContext 持久化（创作笔记） | ✅ 高 | 3-5 天 | 新增 `chapter_notes.md` 或类似 |

## 四、两功能联动设计

### 4.1 理想流程：从"意图资料"到"写作提示卡"

```
阶段 0：用户准备意图资料
  ├── 创作意图说明书（txt/md）→ blurb
  ├── 世界观草稿（md/txt/json）→ foundationSources[world]
  ├── 角色设定草图（md/json）→ foundationSources[character]
  ├── 已有章节手稿（docx/md/txt）→ foundationSources[chapter] 或 chapter import
  └── 参考资料（URL/文档）→ foundationSources[auto]
        ↓
阶段 1：创建书籍（功能一）
  ├── 上传资料 → 自动提取 + 预处理 + purpose 识别
  ├── 用户预览提取内容、确认/修正 purpose
  ├── Architect 基于资料生成 foundation（story_frame、volume_map、roles）
  └── 资料持久化到 story/sources/
        ↓
阶段 2：完善卡片体系（功能二）
  ├── 用户审阅 Architect 生成的角色卡 → 在 Studio 编辑完善
  ├── 用户新建/编辑场景卡（地点、时间、氛围、关联角色、关联事件）
  ├── 用户设定跨角色关系动态（A 与 B 的冲突、B 的隐藏动机）
  └── 用户保存场景卡模板、角色卡模板
        ↓
阶段 3：章节写作（功能二）
  ├── 用户点击"Write Next"或"Write Chapter N"
  ├── 弹出"写作前提示卡"面板：
  │     ├── 本章目标（可编辑，默认来自 volume_map）
  │     ├── 出场角色卡速览（POV + 关联角色）
  │     ├── 场景卡速览（地点、时间、氛围）
  │     ├── 先决条件（requiredBeats + mustKeep，用户可增删）
  │     ├── 发展方向提示（mainConflict + targetMood + 用户备注）
  │     ├── 禁止事项（forbiddenMoves + mustAvoid）
  │     └── Hook 账（本章节需要 advance/resolve 的伏笔）
  ├── 用户确认/编辑提示卡 → 提交
  ├── Planner 读取提示卡 + 现有 truth → 生成 ChapterIntent + ChapterMemo
  └── Writer 在提示卡约束下生成章节
```

### 4.2 数据模型扩展建议

#### A. 场景卡（SceneCard）

```typescript
// packages/core/src/models/scene-card.ts
interface SceneCard {

    auditory?: string;

    tactile?: string;
  };
  tags?: string[];
  firstAppearanceChapter?: number;
  notes?: string;                // 创作者备注
}

// 持久化：story/scenes/{sceneId}.md
```

## 五、具体执行流程建议

### 5.1 功能一：建书前资料导入 — 执行流程

#### Phase 1：Resurrect BookCreate 表单（1 周）

**目标**：让用户可以通过 Studio UI 直观地上传资料并创建书籍。

```
Step 1: 路由接入
  文件：App.tsx
  动作：在路由表中添加 BookCreate 页面引用，或从 ChatPage 的 book-create 模式
        跳转到 BookCreate 表单作为"高级模式"

Step 2: 表单功能补全
  文件：BookCreate.tsx
  动作：
    - 确保表单字段与 StudioCreateBookBody 对齐
    - 文件上传组件支持多选、拖拽
    - 上传后调用 document-reader 提取文本（前端或后端）
    - 显示提取预览（前 500 字符）
    - 允许用户为每个文件指定/修正 purpose
    - 显示字符数统计和警告（接近 100 万字符上限时提示）

Step 3: 创建流程串联
  文件：server.ts / book-create.ts
  动作：
    - 接收 foundationSources + blurb
    - 复用现有 PipelineRunner.initBook() 流程
    - 创建完成后跳转到新书工作台
```

#### Phase 2：自动预处理管线（3-5 天）

```
Step 1: 导入时自动调用 preprocessor
  文件：packages/core/src/import/foundation-source.ts
  动作：在 buildFoundationSourceBundle() 中，提取文本后自动过 text-preprocessor
        （可选开关：用户可在 UI 勾选"自动清理噪声"）

Step 2: 预处理报告
  文件：server.ts / BookCreate.tsx
  动作：返回预处理摘要（移除了 X 个 URL、Y 行代码块、Z 个重复段落）
```

#### Phase 3：格式扩展（2-3 周，可并行）

```
Option A: 轻量方案（仅 Node.js）
  - DOCX: 引入 mammoth.js 做基础文本提取（简单场景）
  - PDF: 引入 pdf-parse 或类似库
  - EPUB: 引入 epub 解压 + HTML 提取
  限制：复杂排版丢失，大文件性能一般

Option B: 微服务方案（推荐，见融合报告）
  - 复用 docx-indexer Python 服务做 DOCX 提取
  - 保持现有 Node.js 提取其他格式
  - 通过 API 网关统一接入
```

#### Phase 4：Purpose 智能识别（1 周，可选）

```
Step 1: LLM 分类器
  文件：新增 packages/core/src/import/source-classifier.ts
  动作：上传文件后，调用轻量 LLM（如 GPT-4o-mini）对提取文本分类
        Prompt: "请判断以下文本属于：世界观/人物设定/剧情梗概/时代背景/书籍规则/正文章节/文风样本"
  输出：purpose 建议 + 置信度

Step 2: 用户确认
  文件：BookCreate.tsx
  动作：显示建议 purpose，用户可一键接受或手动修改
```

### 5.2 功能二：写作前提示卡 — 执行流程

#### Phase 1：ChapterGoalCard 接通 Planner（3-5 天）

**目标**：让已有的 `chapter_goals.json` 真正发挥作用。

```
Step 1: Planner 读取 chapter_goals
  文件：packages/core/src/agents/planner.ts
  动作：在 planner 的上下文组装阶段，读取 story/chapter_goals.json
        找到当前 chapterNumber 对应的 ChapterGoalCard
        将以下字段注入 ChapterIntent / ChapterMemo：
          - mainConflict → memo "当前任务" 的冲突背景
          - requiredBeats → memo "该兑现的" 补充
          - forbiddenMoves → intent.mustAvoid 追加
          - targetMood → memo 氛围基调
          - targetChars → lengthSpec 覆盖

Step 2: 优先级处理
  动作：明确优先级层级：
        用户覆盖（externalContext / 提示卡编辑）> ChapterGoalCard > Planner 推导 > 卷纲默认值

Step 3: 向后兼容
  动作：若 chapter_goals.json 不存在或当前章节无对应 entry，保持现有 Planner 行为不变
```

#### Phase 2：场景卡系统（1 周）

```
Step 1: 数据模型 + 持久化
  文件：
    - packages/core/src/models/scene-card.ts（新增）
    - packages/studio/src/api/server.ts（新增 CRUD 端点）
  动作：
    - GET/PUT/POST/DELETE /books/:id/scenes
    - 持久化到 story/scenes/{id}.md（YAML frontmatter + body）

Step 2: Studio UI
  文件：新增 BookScenesSection.tsx（重写）或 BookSceneCardsSection.tsx
  动作：
    - 场景卡列表（卡片式布局）
    - 新建/编辑场景卡表单（地点、时间、氛围、描述、关联角色下拉选择）
    - 场景与章节关联：在章节元数据编辑中选择 sceneId

Step 3: 与 ChapterGoalCard 联动
  动作：chapter.location 可关联到 sceneCard.id，Planner 自动拉取场景氛围/感官细节
```

#### Phase 3：关系动态提示（1 周）

```
Step 1: 数据模型
  文件：packages/core/src/models/relationship-dynamic.ts（新增）
  动作：定义 RelationshipDynamic 接口
        持久化到 story/relationship_dynamics.json
        或在角色卡 body 中新增 ## 关系动态 章节

Step 2: Studio UI
  文件：新增 RelationshipDynamicsSection.tsx
  动作：
    - 关系动态列表（角色 A ↔ 角色 B）
    - 可视化：简单网络图或表格
    - 关联场景：选择此动态在哪些场景中体现

Step 3: Planner 接入
  文件：planner.ts
  动作：
    - 读取当前章节的 activeDynamics（通过关联角色或关联场景匹配）
    - 将 surfaceRelation + underlyingTension 注入 memo "当前任务" 和 "关键抉择"
    - currentPhase="peak" 时，提升冲突描写优先级
```

#### Phase 4：写作前提示卡面板（1-2 周）

```
Step 1: 面板组件
  文件：新增 packages/studio/src/components/WritingPromptCard.tsx
  动作：
    - 弹窗/抽屉形式，在点击"Write Next"或"Draft Only"时触发
    - 分区块显示：
      ├─ 本章目标（可编辑 textarea，默认来自 volume_map / ChapterGoalCard）
      ├─ 出场角色（自动从 volume_map 解析 + ChapterGoalCard.povCharacter，显示角色卡缩略）
      ├─ 场景设定（自动关联 sceneCard，显示氛围/感官细节）
      ├─ 关系动态（自动匹配 activeDynamics，显示冲突摘要）
      ├─ 先决条件（requiredBeats 列表，可增删 checkbox）
      ├─ 发展方向（mainConflict + targetMood + userNotes textarea）
      ├─ 禁止事项（forbiddenMoves + mustAvoid 列表）
      └─ Hook 账（自动从 pending_hooks 拉取本章节需要处理的 hook）

Step 2: 持久化用户编辑
  文件：packages/core/src/models/chapter-note.ts（新增）
  动作：
    - 用户编辑后的内容保存到 story/chapter_notes/{chapterNumber}.md
    - 作为 externalContext 的一部分传递给 Writer

Step 3: 一键应用 / 跳过
  动作：
    - "确认并生成" → 保存提示卡 → 调用 write-next/draft
    - "跳过提示卡" → 直接调用 write-next/draft（保持现有行为，向后兼容）
    - "保存为模板" → 将当前提示卡结构保存到 story/prompt_templates/{name}.json
```

#### Phase 5：角色卡快速引用（3-5 天，可选）

```
Step 1: 写作界面侧边栏
  文件：ChapterReader.tsx 或 BookWriteSection.tsx
  动作：
    - 右侧可折叠面板"角色速查"
    - 显示本章出场角色的 YAML frontmatter 核心字段
    - 点击角色名展开完整角色卡（read-only 或 inline 编辑）

Step 2: 角色卡模板
  动作：支持保存角色卡为模板（新建角色时可选择"从模板创建"）
```

## 七、风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| **BookCreate 表单死代码过期** | 中 | 复活前与当前 `StudioCreateBookBody` 类型做 diff 校验，补齐新增字段 |
| **ChapterGoalCard 改变 Planner 行为导致生成质量波动** | 高 | A/B 测试：对比开关 ChapterGoalCard 读取前后的生成质量；设置功能开关让用户选择 |
| **提示卡面板增加用户操作步骤，降低效率** | 中 | 提供"记住我的选择"、"跳过提示卡"、"使用上次设置"快捷选项 |
| **场景卡与现有 location 字符串冲突** | 低 | 场景卡 id 与 chapter.location 保持松耦合，允许 location 为自由文本，sceneCard 为可选增强 |
| **数据模型膨胀** | 中 | 场景卡/关系动态采用可选文件（不存在时不影响现有流程），避免强制迁移 |
| **LLM 上下文长度超限** | 中 | 提示卡内容需经过 context-filter 预算控制，与现有 truth 文件共享配额机制 |
| **多作者/并发编辑** | 低 | 复用现有 `StateManager.acquireBookLock()` 文件锁机制 |
