## 二、分阶段核验关键发现

### 发现 1：`tsconfig.server.json` 是「隐藏」typecheck 维度

合并报告（及 DS/GPT/KM 三源）均只验证了 `tsc --noEmit`（client typecheck）。但 `packages/studio/package.json` 中的 typecheck 命令为：

```json
"typecheck": "tsc --noEmit && tsc -p tsconfig.server.json --noEmit"
```

**第二阶段 `tsc -p tsconfig.server.json --noEmit` 存在 50+ 错误**，分类如下：

| 错误类型 | 数量 | 说明 |
|---------|:--:|------|
| TS7006 参数缺类型 | ~35 | callback 参数未标注类型 |
| TS2307 模块未找到 | ~20 | `@actalk/inkos-core` 动态 import subpath |
| TS2339 属性不存在 | ~5 | 类型推断问题 |

**影响**：整体 `pnpm typecheck` 在 server 编译阶段失败，但 `pnpm build` 和 `pnpm test` 不受影响（使用 esbuild/vitest 而非 tsc）。

**建议**：此为预存技术债务（非本轮引入），建议在 tsconfig.server.json 中设置 `"skipLibCheck": true` 和 `"noImplicitAny": false` 作为短期缓解，或逐项修复类型标注。

---

### 发现 2：新测试文件引入后再次引入 typecheck 回归

P1-3/P2-8 测试文件（`state-changelog-evidence.test.ts`、`p2-agent-smoke.test.ts`）在首次提交时含 5 个 TS 错误，导致 Core typecheck 从「通过」退化为「失败」。

**已修复**（commit `b367989`）：
- `state-changelog-evidence.test.ts`：移除无效 `it()` 调用、修复模块路径
- `p2-agent-smoke.test.ts`：移除类型不兼容的参数传递、简化为 import 检查

**教训**：任何新文件提交前必须运行 `pnpm typecheck` 验证。
