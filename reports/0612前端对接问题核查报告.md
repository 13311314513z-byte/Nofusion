## 三、具体对接问题

### 问题 1：SSE `tool:start` / `tool:end` 未注册，侧边栏操作状态失步 ⚠️ P1

**影响范围**：用户点击"AI 写作/审计/修订"后，侧边栏无法显示"正在处理中"指示器。

**证据链**：

1. **后端发出的事件**：`server.ts` 在 Agent 运行期间广播 `tool:start`、`tool:update`、`tool:end`。
2. **全局 SSE Hook 注册的事件**：`use-sse.ts` 通过 `STUDIO_SSE_EVENTS` 注册 55 个事件，但 **不包含 `tool:start` 和 `tool:end`**。
3. **侧边栏消费的事件**：`BookSidebar.tsx` 检查 `last.event === "tool:start"` / `"tool:end"` 来显示操作状态。

**根因**：`tool:*` 事件被设计成 Chat 流内部事件（由 `store/chat/slices/message/action.ts` 的独立 `EventSource` 消费），但侧边栏错误地依赖全局 `sse.messages` 监听这些事件。

**复现路径**：
1. 打开 BookSidebar
2. 触发一个会调用子 Agent 的操作（如 audit 或 revise）
3. 观察侧边栏：只有 `write:start`/`write:complete` 会触发状态变化，子 Agent 的 `tool:start` 不会

**修复建议**：
- **方案 A（推荐）**：在 `use-sse.ts` 的 `STUDIO_SSE_EVENTS` 中追加 `"tool:start"` 和 `"tool:end"`。
- **方案 B**：将 `BookSidebar.tsx` 的操作检测逻辑改为监听 Chat store 中的 `tool` 事件，或让它拥有自己的 `EventSource`。

### 问题 2：Foundation Import Plan 响应形状不一致，存在运行时 TypeError 风险 ⚠️ P1

**影响范围**：Foundation 导入预览时，如果 AI 未产生任何建议变更，前端可能访问 `undefined` 字段导致 UI 崩溃。

**证据链**：

1. **前端调用与类型**：`ImportManager.tsx`
   ```typescript
   postApi<FoundationPlan>(`/books/${id}/import/foundation/plan`, { ... })
   ```
   其中 `FoundationPlan` 要求字段：`planId`、`warnings`、`roleChanges`。

2. **后端实现**：`server.ts:6185`
   ```typescript
   // 仅当 result.proposed && result.roleChanges && result.foundationRevision 为真时返回完整形状
   if (result.proposed && result.roleChanges && result.foundationRevision) {
     return c.json({
       planId,
       bundle,
       proposed: result.proposed,
       warnings,
       roleChanges: result.roleChanges,
     });
   }
   // 否则返回：
   return c.json({ bundle, warnings, proposed: null });
   ```

3. **风险**：当 `proposed` 为 `null` 时，前端仍按 `FoundationPlan` 类型访问 `planId` 和 `roleChanges`，得到 `undefined`。

**修复建议**：
- **后端修复（推荐）**：始终返回统一结构，空值用 `[]` / `null` / `""` 填充：
  ```typescript
  return c.json({
    planId,
    bundle,
    proposed: result.proposed ?? null,
    warnings,
    roleChanges: result.roleChanges ?? { added: [], updated: [], removed: [] },
  });
  ```
- **前端修复**：将 `FoundationPlan` 的 `planId` 和 `roleChanges` 标记为可选，并在渲染前做空值保护。

### 问题 3：`useAutoSave` 死代码调用不存在端点 🧟 P2

**影响范围**：无实际功能影响（该 hook 未被引用），但误导性强，增加维护负担。

**证据链**：

1. **前端代码**：`hooks/use-auto-save.ts`
   ```typescript
   navigator.sendBeacon("/api/v1/session/auto-save", blob);
   ```

2. **后端端点**：`server.ts` 中 **不存在** `/api/v1/session/auto-save`。
   - 唯一相关的 session 端点是 `GET /api/v1/interaction/session`（用于 chat agent 运行时状态）。
   - 实际会话 CRUD 端点是 `/api/v1/sessions`（带 `?bookId=` 查询参数）。

3. **引用情况**：`useAutoSave` 在 `packages/studio/src/` 中 **零引用**。

**修复建议**：
- **方案 A（推荐）**：删除 `hooks/use-auto-save.ts` 文件。
- **方案 B**：如需自动保存草稿，实现真正的 `POST /api/v1/session/auto-save` 端点，并在编辑页面引用该 hook。

**影响范围**：后端维护了一套完整的蒸馏 API，但没有任何前端调用，造成代码冗余和测试负担。

| PATCH | `/style/authors/:authorId/distillations/current` | 更新当前蒸馏 |

**备注**：这与历史上下文中提到的"Writer蒸馏前端完全缺失（5 后端端点悬空）"一致，本次核查确认情况未变。

**修复建议**：

### 问题 5：Style 诊断保存后无读取 UI 🔌 P3

**影响范围**：用户可保存风格诊断，但无法回顾历史诊断，功能价值未完全释放。

**证据链**：

- **已使用**：`POST /style/authors/:authorId/diagnostics`（由 `StyleDiagnosticsPanel.tsx` 调用保存）
- **悬空**：
  - `GET /style/authors/:authorId/diagnostics`（列表）
  - `GET /style/authors/:authorId/diagnostics/:diagnosticsId`（单条详情）

**修复建议**：在作者画像页增加"诊断历史"标签页，调用上述两个 GET 端点。

### 问题 6：API 缓存失效路径不完整 ⚠️ P2

**影响范围**：章节保存、truth 编辑、角色删除等操作后，部分 UI 不会自动刷新，需手动 F5。

**证据链**：

- `use-api.ts` 中的 `deriveInvalidationPaths` 只覆盖少数路径的自动重取。
- 以下写操作**未自动失效**相关 GET 缓存：
  - `PUT /books/:id/chapters/:num` → 应失效 `GET /books/:id/chapters/:num` 和 `GET /books/:id`
  - `PUT /books/:id/truth/:file` → 应失效 `GET /books/:id/truth/:file` 和 `GET /books/:id/truth`
  - `DELETE /books/:id/roles/:roleId` → 应失效 `GET /books/:id/roles`
  - `DELETE /books/:id/sources/:sourceId` → 应失效 `GET /books/:id/sources`

**当前处理**：各调用点手动调用 `refetch()`，但覆盖不全。

**修复建议**：扩展 `deriveInvalidationPaths`，按写操作路径自动推导并失效相关 GET 路径。

## 四、已澄清的非问题

| 原记录问题 | 本次核查结论 |
|-----------|-------------|
| `GET /books/:id/chapters/:num/content` 前端期望但后端缺失 | **未复现**。前端实际调用 `GET /books/:id/chapters/:num`，后端返回 `{ chapterNumber, filename, content }`，`content` 字段存在。该问题可能已在某次修复中解决，或属于其他分支。 |
| 会话端点 `/books/:id/sessions` vs `/sessions?bookId=` | **前后端一致**。前端只调用 `/sessions?bookId=` 和 `/sessions/:id`，无旧端点引用。 |
| `POST /style/rhetoric/rewrite` 返回 `{ prompt }` 而非改写文本 | **接口契约一致**。前端也按 `{ prompt }` 处理并复制到剪贴板。虽然命名容易误解，但不算对接 bug。 |
