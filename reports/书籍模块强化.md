# NoFusion 书籍模块强化实践方案

> 编制日期：2026-06-06  
> 适用范围：NoFusion / InkOS 1.4.1 当前工作区  
> 目标：让新建书籍和现存书籍都能导入 `txt`、`md`、`jsonl`、`json` 等文本资料，并按世界观、主要人物、历史时代、故事内容等用途拆分到现有书籍架构中。  
> 约束：坚持最小改动，不重写 Pipeline，不新增独立 Agent，不引入数据库或插件系统，不改变现有章节生成与运行态文件的职责。

---

## 一、结论

本需求不需要新建一套“文档解析系统”或“书籍知识库”。当前项目已经具备三项关键底座：

1. `document-reader.ts` 已支持 `md`、`txt`、`jsonl`、`json` 等文本提取、噪声过滤、长度限制、文本哈希和警告。
2. `ArchitectAgent` 已能把输入材料拆分为故事框架、卷级规划、主要/次要人物、书籍规则和待处理伏笔。
3. 现有章节导入已经具备“预览计划 -> 用户确认 -> 提交执行”的交互范式。

因此，最小改动方案是：

> 在现有解析器和 Architect 之间增加一个轻量“书籍资料包”层；新建书籍把资料包作为初始化上下文，现存书籍把资料包作为架构修订上下文。资料导入与正文章节导入必须分流。

目标调用链：

```text
文件/粘贴文本
  -> document-reader 统一提取
  -> FoundationSourceBundle 资料包
  -> 分类与导入预览
  -> Architect 生成或修订
  -> 写入现有 story/outline、story/roles、pending_hooks
```

## 二、现状评估

### 2.1 已有能力

| 能力 | 当前入口 | 可复用方式 |
|---|---|---|
| 多格式文本提取 | `packages/core/src/utils/document-reader.ts` | 直接复用，不再写第二套解析器 |
| 新书架构生成 | `ArchitectAgent.generateFoundation()` | 把资料包作为 `externalContext` 输入 |
| 正文反向建档 | `ArchitectAgent.generateFoundationFromImport()` | 仅用于已明确为正文章节的材料 |
| 架构文件写入 | `ArchitectAgent.writeFoundationFiles()` | 保持现有权威文件布局 |
| 存量架构修订 | `PipelineRunner.reviseFoundation()` | 扩展为可携带资料包的补充修订 |
| 章节导入预览 | `chapter-import-planner.ts` | 复用 plan/commit 交互模式 |
| Studio 新建书籍 | `packages/studio/src/pages/BookCreate.tsx` | 在现有表单增加资料区，不新建页面 |
| Studio 导入管理 | `packages/studio/src/pages/ImportManager.tsx` | 增加“架构资料”分卡 |
| CLI 新建书籍 | `packages/cli/src/commands/book.ts` | 扩展已有 `--brief`，支持重复 `--source` |
| CLI 导入书籍 | `packages/cli/src/commands/import.ts` | 增加 `foundation` 子命令 |

### 2.2 当前缺口

1. Studio 新建书籍只有一段 `brief` 文本，不能同时携带多份资料。
2. CLI `book create --brief` 直接读取文件，只说明支持 `.md/.txt`，没有复用通用解析器。
3. ImportManager 的章节导入只接收粘贴文本，没有“架构资料导入”入口。
4. 存量架构修订主要接收用户反馈，没有标准化的多文件资料包。
5. `revise` 模式会重建角色目录，补充少量人物资料时可能误删未被本次生成覆盖的角色。
6. 正文章节导入会重置索引、快照及部分运行态；它不适合承担世界观、人物设定、时代资料导入。
7. 导入来源没有稳定的文件级记录，后续无法判断某份资料是否已导入、是否重复、用于何种分类。

## 三、最小改动边界

### 3.1 必须复用

- 复用 `extractDocument()` 和 `extractDocumentFromText()`。
- 复用 `ArchitectAgent`，不新增“资料分析 Agent”。
- 复用 `ArchitectOutput` 和现有架构文件。
- 复用章节导入的 plan/commit 产品交互。
- 复用当前文件系统存储，不扩展数据库。
- 复用 `outline-paths.ts` 对新旧书籍布局的兼容能力。

### 3.2 明确不做

- 不把每种格式做成独立导入器。
- 不新增向量数据库、RAG、知识图谱或长期索引服务。
- 不让用户直接指定任意目标文件路径。
- 不把世界观资料伪装成正文章节导入。
- 不在第一阶段支持 PDF、DOCX、图片 OCR。
- 不重构书籍创建路由、Studio 导航或整个 Pipeline。
- 不自动覆盖存量书籍架构，必须先预览再提交。

## 四、统一资料包设计

### 4.1 新增轻量核心类型

建议在 Core 新增：

```text
packages/core/src/import/foundation-source.ts
```

建议类型：

```ts
type FoundationSourcePurpose =
  | "auto"
  | "world"
  | "character"
  | "era"
  | "plot"
  | "chapter"
  | "rule"
  | "style";

interface FoundationSourceInput {
  sourceName: string;
  fileType: DocumentFileType;
  text: string;
  purpose?: FoundationSourcePurpose;
}

interface FoundationSource {
  sourceId: string;
  sourceName: string;
  fileType: DocumentFileType;
  purpose: FoundationSourcePurpose;
  text: string;
  charCount: number;
  textHash: string;
  warnings: readonly string[];
}

interface FoundationSourceBundle {
  sources: readonly FoundationSource[];
  totalChars: number;
  warnings: readonly string[];
}
```

这一层只负责：

1. 调用现有文档解析器。
2. 按 `textHash` 去重。
3. 校验文件数量和总字符数。
4. 保留用户指定的资料用途。
5. 将多份资料组装成 Architect 可识别的上下文。

它不负责生成架构，也不负责写文件。

### 4.2 支持格式

第一阶段 UI 明确开放：

- `.txt`
- `.md`
- `.markdown`
- `.jsonl`
- `.json`
- `.jsonl.md`
- `.json.markdown`

底层解析器现有的 `.ts/.js/.html/.css` 能力可以保留，但书籍资料入口暂不主动展示，避免产品范围过宽。

### 4.3 分类方式

分类采用“用户提示优先，自动判断兜底”，不新增分类 Agent。

| 用途 | Architect 目标位置 | 处理规则 |
|---|---|---|
| 世界观 `world` | `story/outline/story_frame.md` | 提取世界规则、地理、组织、力量体系和社会结构 |
| 主要人物 `character` | `story/roles/主要角色/*.md` | 提取身份、目标、关系、能力、秘密、初始状态和人物弧 |
| 历史时代 `era` | `story/outline/story_frame.md` | 形成时代锚点、技术边界、社会习俗和历史约束 |
| 故事内容 `plot` | `story/outline/volume_map.md`、`story/pending_hooks.md` | 提取主线、阶段冲突、关键事件、悬念和回收计划 |
| 正文章节 `chapter` | 现有章节导入流程 | 拆章、预览、回放并重建运行态 |
| 书籍规则 `rule` | `story_frame.md` YAML frontmatter | 提取禁区、主角约束、题材锁和叙事规则 |
| 文风样本 `style` | 现有 Style 导入能力 | 不混入世界观与剧情事实 |
| 自动 `auto` | 由 Architect 综合拆分 | 适合混合笔记和无法预分类的资料 |

### 4.4 资料包文本格式

将多份文本拼接为带边界的上下文，避免不同文件相互污染：

```text
<foundation-source
  name="世界观.txt"
  type="txt"
  purpose="world"
  hash="..."
>
规范化后的文本
</foundation-source>
```

Architect 提示中增加四条约束：

1. 资料是事实来源，不得把文件名、导入说明写进小说设定。
2. 同类冲突时，用户明确分类的资料优先于 `auto` 资料。
3. 资料与当前书籍冲突时，存量书籍默认保留当前设定并报告冲突。
4. `style` 资料只用于文风，不得被当作剧情事实。

## 五、新建书籍实践路径

### 5.1 Studio

在 `BookCreate.tsx` 现有“故事简介 / 核心设定”下方增加“资料文件”区域：

- 支持多选文件。
- 每份文件显示名称、格式、字符数、警告和用途下拉框。
- 允许删除单份文件。
- 保留原有 `brief`，它作为优先级最高的用户创作意图。
- 创建按钮提交 `brief + foundationSources`，不改变书籍基本字段。

建议创建请求在 `StudioCreateBookBody` 增加可选字段：

```ts
foundationSources?: readonly FoundationSourceInput[];
```

服务端流程：

```text
POST /api/v1/books
  -> 校验书籍基本信息
  -> 逐份 extractDocumentFromText()
  -> buildFoundationSourceBundle()
  -> brief + 资料包组成 externalContext
  -> 复用现有新书创建和 Architect 生成
  -> 保存资料来源索引
```

当资料被用户标记为 `chapter` 时，不在新书初始化阶段自动回放。先完成建书，再引导用户进入 ImportManager 的章节导入流程，避免一次请求同时承担建书、架构生成和运行态重建。

### 5.2 CLI

保留 `--brief` 兼容性，并增加可重复参数：

```powershell
inkos book create `
  --title "书名" `
  --genre "都市" `
  --brief ".\核心方向.md" `
  --source ".\世界观.txt" `
  --source ".\人物设定.md" `
  --source ".\时代资料.jsonl"
```

实践方式：

- `--brief` 和 `--source` 都改为调用 `extractDocument()`。
- `--source` 可重复出现。
- 第一阶段由文件名和内容自动判断用途；需要强控制时提供一份可选 manifest，而不是设计 Windows 路径不友好的 `purpose:path` 语法。
- 增加 `--dry-run`，只输出文件解析结果、分类、警告和总字符数，不创建书籍。

可选 manifest：

```json
{
  "sources": [
    { "path": "./世界观.txt", "purpose": "world" },
    { "path": "./人物设定.md", "purpose": "character" },
    { "path": "./旧稿.jsonl", "purpose": "plot" }
  ]
}
```

## 六、现存书籍实践路径

### 6.1 Studio 入口

在 `ImportManager.tsx` 增加“架构资料”分卡，不新建顶级页面。

操作流程：

1. 选择目标书籍。
2. 上传多份文件或粘贴文本。
3. 为每份资料选择用途，默认 `auto`。
4. 选择导入模式。
5. 生成预览。
6. 查看冲突、影响文件和角色变化。
7. 确认提交。

导入模式：

| 模式 | 默认性 | 行为 |
|---|---|---|
| 补充 `supplement` | 默认 | 当前架构为主，资料只补充缺失信息；保留未命中的现有角色 |
| 重构 `rebuild` | 高风险 | 当前架构与新资料共同重生成，允许删除或重组角色；必须二次确认 |
| 正文章节 `chapter` | 分流 | 转到现有章节导入，不走架构补充 |
| 文风 `style` | 分流 | 转到现有 Style 导入，不走 Architect |

### 6.2 Core 调用

不要直接调用 `importChapters()` 处理架构资料。建议扩展：

```ts
PipelineRunner.planFoundationImport(...)
PipelineRunner.commitFoundationImport(...)
```

`planFoundationImport()`：

- 读取当前权威架构文件。
- 构建并校验资料包。
- 调用 Architect 生成候选 `ArchitectOutput`，但不写文件。
- 返回候选结果、冲突摘要、角色新增/更新/可能删除列表和影响文件。

`commitFoundationImport()`：

- 再次校验目标书籍和预览数据。
- 复用 `reviseFoundation()` 的备份机制。
- 写入候选架构。
- 保存资料来源记录。
- 不修改章节正文、章节索引、快照、MemoryDB 和 runtime 文件。

### 6.3 角色安全合并

当前 `writeFoundationFiles(..., "revise")` 会删除并重建主要/次要角色目录。为满足“补充资料”语义，建议把写入模式扩展为：

```ts
mode: "init" | "revise-replace" | "revise-merge"
```

规则：

- `revise-merge`：按规范化角色名匹配并更新；新角色新增；未出现在候选结果中的旧角色保留。
- `revise-replace`：保持当前 `revise` 行为，清空后重建角色目录。
- 只有用户显式选择“重构”时使用 `revise-replace`。

这是存量书籍强化中唯一必须触及现有写入行为的核心改动，否则“最小改动”会以角色意外丢失为代价。

### 6.4 CLI

增加：

```powershell
inkos import foundation <book-id> `
  --from ".\补充世界观.md" `
  --from ".\新人物.jsonl" `
  --mode supplement `
  --dry-run
```

确认预览后执行：

```powershell
inkos import foundation <book-id> `
  --from ".\补充世界观.md" `
  --from ".\新人物.jsonl" `
  --mode supplement `
  --yes
```

CLI 与 Studio 必须调用同一 Core plan/commit 方法，避免两套导入规则。

## 七、资料来源持久化

不扩展 `BookConfig`，在书籍目录增加轻量来源记录：

```text
books/<book-id>/story/sources/
  index.json
  <source-id>.md
```

`index.json` 建议记录：

```json
{
  "sources": [
    {
      "sourceId": "时代资料-a1b2c3d4",
      "sourceName": "时代资料.jsonl",
      "fileType": "jsonl",
      "purpose": "era",
      "textHash": "a1b2c3d4e5f6g7h8",
      "charCount": 18342,
      "importedAt": "2026-06-06T00:00:00.000Z",
      "mode": "supplement"
    }
  ]
}
```

实践原则：

- 保存规范化文本，不强制复制原始二进制文件。
- 同一本书内以 `textHash` 去重。
- 同名但内容变化的文件作为新版本记录。
- 删除来源记录不自动回滚架构；回滚使用现有 `.backup-*` 目录。
- `story/sources` 是资料来源，不是权威设定；权威设定仍是 `story/outline` 和 `story/roles`。

## 八、API 路径

建议新增两组端点：

```text
POST /api/v1/books/:id/import/foundation/plan
POST /api/v1/books/:id/import/foundation/commit
```

`plan` 请求：

```json
{
  "mode": "supplement",
  "instruction": "补充港城旧案的历史背景，不改变主角身份",
  "sources": [
    {
      "sourceName": "港城历史.md",
      "fileType": "md",
      "purpose": "era",
      "text": "..."
    }
  ]
}
```

`plan` 返回：

- 解析后的来源摘要。
- 重复文件和质量警告。
- 候选架构。
- 世界观、人物、时代、故事内容的拆分结果。
- 新增、更新、保留、可能删除的角色。
- 将被修改的文件。
- 明确声明不会修改的章节和运行态文件。

`commit` 请求携带 plan 生成的候选架构及来源摘要。服务端必须再次执行 schema 校验，不能直接信任客户端传回内容。

新书不额外增加 plan 端点，第一阶段只在现有创建请求增加 `foundationSources`，控制改动范围。

## 十、容量与安全约束

沿用单文件最大 `500,000` 字符限制，并增加资料包级限制：

- 单次最多 20 个文件。
- 单次规范化文本总量建议最多 1,000,000 字符。
- 超限时拒绝提交，不静默截断整个资料包。
- 单文件截断必须在预览中明确显示。
- 空文本、乱码比例过高、有效内容过短时给出警告。
- 文件名只用于展示和来源记录，不直接拼接任意磁盘路径。
- 服务端只接收文本和声明格式，不接收客户端提供的本机绝对路径。
- 所有输出文件名继续使用现有角色名安全化规则。

对于超长小说正文，继续使用现有章节导入中的摘要、边缘章节和中段锚点策略，不把百万字全文一次性送给 Architect。

## 十一、实施顺序

### 阶段 A：Core 统一输入

1. 增加 `FoundationSourceBundle`。
2. 让 CLI `--brief` 改用现有解析器。

1. 扩展 `StudioCreateBookBody`。

1. 增加 foundation plan/commit。
2. ImportManager 增加“架构资料”分卡。

完成标志：已有章节的书籍能补充资料，且章节正文、索引、快照和运行态不变。

### 阶段 D：收尾

1. 完成中英文文案。
2. 补充来源管理和重复导入提示。
3. 增加回归测试和端到端验收。


覆盖：

- `txt/md/jsonl/json` 能统一提取。
- `.jsonl.md` 能按 JSONL 解析。
- 多文件按哈希去重。
- 用途标签在上下文中保留。
- 单文件和总量超限能报告。
- `style` 不进入剧情事实。
- `chapter` 会被分流而非直接修改架构。
- `supplement` 保留旧角色。
- `rebuild` 才允许移除旧角色。
- plan 阶段不写任何书籍文件。
- commit 前建立完整架构备份。
- commit 不修改章节索引、正文、快照和 runtime。

### 12.2 Studio/API 测试

- 新建书籍可提交多份资料。
- 无资料时完全保持原有创建行为。
- 预览能展示警告、分类和影响文件。
- 重构模式需要二次确认。
- 重复资料不能被无提示地再次导入。
- 非法文件类型和超限请求返回可读错误。

### 12.3 验收场景

1. 新建书籍同时导入 `世界观.txt`、`主要人物.md`、`历史时代.jsonl`，生成的世界框架、角色文件和卷纲归位正确。
2. 给已有 10 章的书补充人物资料，角色文件被更新，10 章正文和章节索引完全不变。
3. 给已有书补充时代资料，时代约束进入 `story_frame.md`，不写入 `current_state.md`。
4. 导入混合剧情笔记时，主线进入 `volume_map.md`，未回收线索进入 `pending_hooks.md`。
5. 同一文件重复导入时被哈希识别并阻止或明确跳过。
6. 用户误把正文标为架构资料时，预览提示可改走“正文章节导入”。
7. 用户取消预览时，书籍目录没有任何变化。
8. 生成失败时，原架构仍可从 `.backup-*` 恢复。

## 十三、主要风险与处理

| 风险 | 处理方式 |
|---|---|
| JSONL 对话被拆成重复片段 | 复用现有字段优先级和去噪逻辑，并以哈希去重 |
| 混合资料分类不准 | 用户用途优先；`auto` 只作为兜底 |
| 存量角色被误删 | 默认 `revise-merge`，只有重构模式允许替换 |
| 架构资料触发运行态重置 | 与 `importChapters()` 完全分流 |
| 一次输入过长导致模型失焦 | 文件级/资料包级限制，超长正文走章节摘要策略 |
| 新资料与旧设定冲突 | plan 返回冲突摘要，默认保留旧设定 |
| 新入口增加认知负担 | 只在 BookCreate 和 ImportManager 现有页面增加区域/分卡 |
| 来源文件成为第二份权威数据 | 明确 `story/sources` 只记录来源，架构文件仍是唯一权威 |

## 十四、最终推荐

推荐按以下最短路径实施：

1. 先完成 Core 资料包和 Architect 输入边界，不动章节 Pipeline。
2. 先接入新建书籍，验证多格式、多文件和分类结果。
3. 再接入现存书籍，必须同时完成 plan/commit 和角色安全合并。
4. 最后补 CLI 与来源记录，使 Studio 和 CLI 使用同一套 Core 逻辑。

最终产品形态应保持简单：

> 新建书籍时可以附加多份资料；已有书籍可以在导入管理中继续补充资料；系统负责把资料归入世界观、人物、时代、剧情和规则，但正文导入与文风导入仍走各自现有流程。

这条路径与当前 NoFusion “先复用、先文件、先呈现、少动核心”的发展原则一致，能够以有限改动补齐书籍模块最明显的输入能力缺口，同时避免破坏已经成熟的章节生成、真相文件和运行态治理体系。
