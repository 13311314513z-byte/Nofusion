# 书籍分卡修改

## 目标

将当前书籍页面中集中在右半边栏的信息模块，拆分为左侧二级菜单下的独立功能页面。最终效果是：进入一本书后，不再只有“会话”一个主入口，而是形成“会话 / 章节 / 角色 / 伏笔 / 设定 / 摘要 / 审计 / 导出”等书籍工作台分卡。

本次方案仅描述修改步骤，不直接改代码。

## 设计原则

1. 会话页只承担写作交互，不继续承载所有管理功能。
2. 右侧栏已有模块不废弃，而是页面化复用。
3. 先做只读迁移，再做编辑能力，避免一次性改动过大。
4. 左侧二级菜单负责导航，主内容区负责完整操作。
5. 右侧栏保留为“当前上下文详情”，不再是全部信息入口。
6. 移动端优先保证可访问，复杂详情可用抽屉或独立页面处理。

## 目标信息架构

进入某本书后，左侧二级菜单建议如下：

| 菜单 | 页面定位 | 数据来源 | 第一阶段能力 |
| --- | --- | --- | --- |
| 会话 | 写作/对话执行区 | 现有 chat store、SSE | 保持现状，减少右侧信息负担 |
| 章节 | 章节列表与章节状态管理 | chapters index、chapter content | 列表、筛选、查看、同步、重写入口 |
| 角色 | 角色卡管理 | roles API / role cards | 列表、详情、只读/基础编辑 |
| 伏笔 | 钩子/伏笔状态可视化 | hooks.json + pending_hooks.md | 看板、压力列表、依赖关系预览 |
| 设定 | Truth files 管理 | truth files API | 文件列表、查看、编辑非 shim 文件 |
| 摘要 | 章节摘要/卷摘要/长期记忆 | chapter_summaries.md、volume_summaries.md | 只读浏览、按章筛选 |
| 审计 | 审计历史和问题管理 | audit history、chapter meta | 问题列表、章节定位、手动审计入口 |
| 导出 | 成书导出 | export API | 格式选择、导出设置 |

### 2.4 伏笔页

目标：

建议文件：


```ts
    stale: number;

    core: number;
```

   - 种子
   - 开放
   - 推进中
   - 待回收
   - 阻塞
   - 已回收
   - 延后

3. 压力列表
   - 按 pressureScore 排序。
   - 显示“为什么它现在该处理”。

4. 依赖关系
   - 第一版用列表树。
   - 第二版再做节点图。

5. 详情抽屉
   - hookId
   - type
   - status
   - startChapter
   - lastAdvancedChapter

重要前置修复：

`state-bootstrap.ts` 从 `pending_hooks.md` 重建 `hooks.json` 时，应保留 Phase 7 字段：

验收标准：

## 阶段 4：API 调整建议

### 已有 API 可复用

1. `/api/v1/books/:id`
2. `/api/v1/books/:id/chapters`
3. `/api/v1/books/:id/chapters/:chapter`
4. `/api/v1/books/:id/truth`
5. `/api/v1/books/:id/truth/:file`
6. `/api/v1/books/:id/audit/:chapter`
7. 角色相关 API
8. 导出相关 API

### 建议新增 API

#### 4.1 伏笔聚合 API

```text
GET /api/v1/books/:id/hooks
```

用途：

为伏笔页提供结构化数据，不让前端解析 markdown。

后端逻辑：

1. 读取 `story/state/hooks.json`。
2. 读取 `story/pending_hooks.md` 作为 fallback 或补充 Phase 7 字段。
3. 调用 `describeHookLifecycle`。
4. 调用 `computeHookDiagnostics`。
5. 调用 `collectStaleHookDebt`。
6. 生成前端专用 view model。

#### 4.2 摘要聚合 API

```text
GET /api/v1/books/:id/summaries
```

用途：

聚合 chapter summaries 和 volume summaries。

#### 4.3 审计聚合 API

```text
GET /api/v1/books/:id/audit
```

用途：

返回全书审计问题，不需要前端遍历章节。

### 风险 2：BookDetail 继续膨胀

规避：

新页面逻辑放到 `book-workspace` 目录，逐步从 BookDetail 抽离。

### 风险 3：伏笔数据不准

规避：

优先修结构化状态字段保留；前端只读聚合 API，不直接解析 markdown。

### 风险 4：右侧栏和主页面功能重复

规避：

右侧栏只保留当前上下文，不作为主导航。

### 风险 5：移动端信息过载

规避：

移动端二级菜单改为顶部 tabs 或抽屉；详情用全屏页或底部抽屉。

## 第一阶段推荐交付范围

若希望最快看到 UI 变化，建议第一阶段只做：

1. 新增书籍 workspace layout。
2. 左侧二级菜单。
3. 会话页迁入 `chat`。
4. `TruthFiles` 迁入 `truth`。
5. 章节页只读列表。

不建议第一阶段就做：

1. 伏笔依赖图。
2. 审计全书聚合。
3. 复杂角色关系图。
4. 批量编辑。

## 最小可用版本验收

满足以下条件即可认为“书籍分卡”第一版完成：

1. 进入书籍后左侧出现二级菜单。
2. 默认分卡是会话。
3. 至少包含会话、章节、设定三个可用分卡。
4. 分卡切换不刷新整页、不丢当前书籍上下文。
5. 原有写作会话功能不回退。
6. 原有 truth files 查看/编辑功能不回退。
7. `pnpm typecheck`、`pnpm build`、Studio 测试通过。
