# 0609 NoFusion 完整测试计划

> 基于 `0609Nofusion测试计划.md` 与 `0609Nofusion测试计划_完善建议.md` 整合。本文档目标不是单纯罗列功能点，而是给测试人员一套可直接执行、可记录、可复测、可判定是否放行的完整流程。

---

## 1. 测试目标与范围

### 1.1 测试目标

- 验证 NoFusion / InkOS Studio 从新建书籍到写作、文风分析、审计、导出、服务商配置的主流程是否可用。
- 验证 0608 后续改动中的 P0/P1 问题是否已经修复，避免已知缺陷回归。
- 验证服务商 API Key、模型选择、代理连接、独立保存等配置行为是否稳定。
- 验证人工高频操作场景下的数据是否能够保存、刷新后是否恢复、异常情况下是否有明确提示。

### 1.2 覆盖范围

| 范围 | 内容 |
|---|---|
| 前端页面 | 首页、书籍工作区、Chat、文风、审计、模型配置、封面配置、日志、环境诊断、体裁管理 |
| 后端 API | 项目读取、书籍 CRUD、章节读写、服务商配置、连接测试、文风分析与调整、审计相关接口 |
| 本地数据 | `inkos.json`、`.inkos/secrets.json`、`books/`、`style-library/` |
| 第三方服务 | OpenAI、Moonshot/Kimi、DeepSeek 等配置与连接测试 |
| 异常场景 | 错误 Key、网络中断、超时、空输入、超长输入、刷新、双标签页并发 |

### 1.3 不纳入本轮验收的内容

以下内容只放入“待实现功能验证”，不作为当前版本失败项：

- `top_p`、`seed`、`presence_penalty`、`frequency_penalty`、`repetition_penalty` 等高级写作参数 UI。
- 模型参数预设一键应用。
- Agent 级 extra JSON 覆盖。

原因：这些功能依赖 `模型通用写作参数提升执行计划.md` 的后续实现，当前 UI 未完整暴露。

- 后一位测试人员接手前，必须阅读前一位提交的缺陷记录和阻断项。

## 3. 测试环境准备

### 3.1 环境检查

在项目根目录执行：

```powershell
node -v
.\pnpm.cmd -v
.\pnpm.cmd --filter @actalk/inkos-core typecheck
.\pnpm.cmd --filter @actalk/inkos-studio typecheck
```

预期：

- Node.js 版本不低于 20。
- pnpm 可用。
- Core 与 Studio typecheck 通过。

如 typecheck 失败，本轮人工测试标记为“环境阻断”，先修复代码问题再继续。

### 3.2 启动方式

推荐直接运行：

```powershell
.\start.bat
```

如需分开启动：

```powershell
cd packages/studio
pnpm run dev:server
pnpm run dev:client
```

预期端口：

- 前端：`http://localhost:4577`
- API：`http://localhost:4579`

### 3.3 测试数据准备

| 数据 | 用途 | 要求 |
|---|---|---|
| 中文叙事文本 | 文风分析、AI 痕迹检测、改写 | 500-2000 字 |
| 英文叙事文本 | 多语言、英文文风分析 | 500-2000 words |
| 超长文本 | 边界测试 | 20,000 字、50,000 字两档 |
| `.txt` 文件 | 本地导入 | UTF-8 编码 |
| `.md` 文件 | 章节导入、预处理 | 包含多级标题 |
| 错误 API Key | 负面测试 | `fake-key` |
| 有效 API Key | 服务连接 | OpenAI / Moonshot / DeepSeek 至少一个 |

### 3.4 测试前备份

测试前备份以下文件，避免污染正式数据：

```powershell
Copy-Item .\inkos.json .\inkos.json.bak-0609 -Force
Copy-Item .\.inkos\secrets.json .\.inkos\secrets.json.bak-0609 -Force -ErrorAction SilentlyContinue
```

如果本机没有 `.inkos/secrets.json`，不视为失败。

## 4. 证据与缺陷记录规范

### 4.1 必须保留证据的情况

| 情况 | 必需证据 |
|---|---|
| P0/P1 缺陷 | 截图、Console 报错、复现步骤、服务端日志 |
| 服务商连接失败 | 当前服务商、模型、协议、Base URL、返回错误文本 |
| 数据未保存 | 操作前后截图、本地 JSON 文件差异 |
| 页面白屏 | 浏览器 Console、Network 失败请求、当前 URL |
| LLM 输出异常 | 输入 prompt、服务商、模型、错误响应 |

### 4.2 文件命名

```text
缺陷_<模块>_<用例编号>_<简述>.png
录屏_<模块>_<用例编号>_<简述>.mp4
日志_<模块>_<用例编号>_<简述>.txt
```

示例：

```text
缺陷_服务商配置_5.1.3_OpenAI连接超时.png
日志_文风调整_R1_preview未配置LLM.txt
```

### 4.3 缺陷严重度

| 等级 | 判定标准 |
|---|---|
| P0 | 主流程不可用、数据丢失、服务无法启动、API Key 串写覆盖、页面白屏 |
| P1 | 核心功能失败但存在绕过方式，例如文风改写失败、连接测试异常、章节保存失败 |
| P2 | 非核心功能异常，例如文案错误、局部布局问题、提示不清晰 |
| P3 | 优化建议，例如交互效率、信息层级、视觉一致性 |

### 4.4 缺陷记录模板

| 字段 | 填写要求 |
|---|---|
| 缺陷编号 | `BUG-0609-001` 递增 |
| 模块 | 例如“服务商配置” |
| 用例编号 | 例如 `5.1.3` |
| 复现步骤 | 逐步写清点击和输入 |
| 实际结果 | 写具体错误，不写“有问题” |
| 预期结果 | 对照本文档预期 |
| 严重度 | P0/P1/P2/P3 |
| 证据 | 截图、录屏、日志文件名 |
| 当前状态 | 待修复 / 已修复待复测 / 复测通过 |

## 5. 放行与阻断标准

### 5.1 发布前放行标准

| 指标 | 放行要求 |
|---|---|
| P0 缺陷 | 0 |
| P1 缺陷 | 不超过 3 个，且必须有明确绕过方式 |
| 核心路径批次 A | 100% 通过 |
| 服务商配置 | 至少一个真实服务商连接成功 |
| 数据保存 | 新建书籍、章节生成、API Key 保存、作者档案保存均通过 |
| 控制台错误 | 不允许出现持续刷屏错误 |

### 5.2 必须停止测试并先修复的情况

- `start.bat` 无法启动，且前后端服务均不可访问。
- 首页白屏或路由无法进入核心页面。
- 新建书籍失败。
- API Key 保存出现跨服务商覆盖。
- 章节生成后本地文件丢失或乱码。
- 文风调整接受后原文被错误清空。

## 6. 测试执行总流程

### Step 1：自动化预检

执行：

```powershell
.\pnpm.cmd --filter @actalk/inkos-core typecheck
.\pnpm.cmd --filter @actalk/inkos-studio typecheck
.\pnpm.cmd --filter @actalk/inkos-core test
.\pnpm.cmd --filter @actalk/inkos-studio test
```

记录：

- 命令是否通过。
- 失败的测试文件与错误摘要。

判定：

- typecheck 失败时，不进入人工全量测试。
- 单元测试失败但不影响启动时，可进入人工测试，但必须标记“自动化预检失败”。

### Step 2：启动与冒烟

1. 运行 `.\start.bat`。
2. 打开 `http://localhost:4577`。
3. 打开浏览器 Console。
4. 点击侧边栏 3 个以上页面，确认无白屏。
5. 刷新当前页面，确认路由恢复。

通过标准：

- 页面可访问。
- API 可响应。
- Console 无持续报错。

### Step 3：创建本轮测试书籍

1. 首页点击“新建书籍”。
2. 标题填写 `0609人工测试书籍`。
3. 语言选择中文。
4. 体裁选择可用体裁。
5. 创建后进入书籍工作区。
6. 返回首页确认书籍在列表中。
7. 检查 `books/<id>/book.json` 是否存在。

通过标准：

- UI 显示新书。
- 本地文件存在。
- 刷新后仍可进入该书。

### Step 4：执行批次 A 核心路径

| # | 操作 | 预期 |
|---|---|---|
| A.1 | 打开普通 Chat，发送“你好” | 消息进入记录，无空白响应 |
| A.2 | 在书籍内 Chat 输入“请写第一章，主角在森林中醒来” | 生成章节或明确返回 LLM 错误 |
| A.3 | 打开章节列表 | 能看到生成章节、字数、状态 |
| A.4 | 打开章节阅读 | 正文完整显示 |
| A.5 | 点击章节文风诊断入口 | 跳转文风页面并预填文本 |
| A.6 | 对该章运行文风诊断 | 展示诊断结果或明确服务错误 |
| A.7 | 运行一次审计 | 审计页面不白屏，结果可查看 |
| A.8 | 刷新浏览器 | 当前书籍和页面状态可恢复 |

失败判定：

- A.1-A.5 任一失败，优先记为 P0/P1。
- LLM 因无有效 Key 失败时不直接判功能失败，但错误提示必须清楚。

### Step 5：执行批次 B 文风链路

| # | 操作 | 预期 |
|---|---|---|
| B.1 | 粘贴 500-2000 字中文文本并点击分析 | 展示基础统计和指纹 |

| B.4 | 追加样本 | 样本数增加，指纹更新 |
| B.5 | 选择参考作者并生成调整计划 | 建议按严重度或类别展示 |
| B.6 | 筛选“意图重复 / 过渡词 / AI 痕迹” | 列表随筛选变化 |
| B.7 | 应用单条补丁 | 文本局部替换，撤销可用 |
| B.8 | 撤销补丁 | 文本恢复 |
| B.9 | 勾选建议并预览改写 | 展示 diff 和风险对比 |

#### 6.2 API Key 独立保存专项

执行顺序必须固定：

1. 在 OpenAI 输入 `openai-test-key-0609` 并保存。
2. 在 Moonshot 输入 `moonshot-test-key-0609` 并保存。
3. 在 DeepSeek 输入 `deepseek-test-key-0609` 并保存。
4. 回到 OpenAI 页面，确认显示的是 OpenAI 掩码，不应显示后两个 Key。
5. 检查 `.inkos/secrets.json`，应按服务商分开保存。

预期结构示例：

```json
{
  "openai": "...",
  "moonshot": "...",
  "deepseek": "..."
}
```

失败判定：

- 任一服务商保存后覆盖其他服务商 Key，直接记 P0。

## 13. 复测流程

每个缺陷修复后按以下流程复测：

1. 拉取或确认最新代码。
2. 重新执行相关自动化测试。
3. 只执行失败用例的最小复现步骤。
4. 执行与该缺陷相邻的 2-3 个关联用例。
5. 更新缺陷状态为“复测通过”或“仍失败”。
6. 如果同一缺陷复测仍失败，保留原缺陷编号，不新开重复缺陷。

示例：

| 原缺陷 | 必测用例 | 邻近回归 |
|---|---|---|
| API Key 覆盖 | C.3、C.8、6.2 专项 | 封面配置保存、服务商连接测试 |
| OpenAI 超时 | C.4、C.X1 | Moonshot / DeepSeek 连接测试 |
| 文风预览过期 | B.9、B.10 | B.11、B.12 |
| 章节保存失败 | A.3、A.4、S.2 | 导出 TXT / Markdown |

## 15. 本轮重点关注项

本轮测试除原有主流程外，必须重点盯住以下问题：

- API Key 是否按服务商分开保存。
