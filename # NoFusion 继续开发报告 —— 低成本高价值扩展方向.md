## 1. 核心引擎（Core）

### 当前状态
- `PipelineRunner` 单文件 3400+ 行，耦合度高
- Agent 列表硬编码，无插件注册机制
- 状态快照基于文件系统，回滚能力完整但缺乏索引

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **Pipeline 阶段拆分** | 将 Runner 中的 `writeNextChapter` 按阶段拆分为独立函数（`planChapter`, `composeChapter`, `writeDraft`, `auditChapter`, `reviseChapter`, `persistChapter`），各阶段通过接口组合 | 2-3 天 |
| **Agent 注册表** | 将 `agents/index.ts` 中的 `ALL_AGENTS` 改为 Map，支持通过配置或文件扫描动态添加 Agent | 1 天 |
| **运行时产物查看器 API** | 为 `story/runtime/chapter-XXXX.*` 添加只读 REST API（直接暴露 JSON 即可，无需额外渲染） | 0.5 天 |
| **LLM 客户端池优化** | 将 `getClient()` 中的 `modelOverrides` 缓存改为 LRU + 自动失效（当前无过期机制） | 0.5 天 |

**预期收益**: 降低后续维护成本，为 Agent 插件系统铺路，提升运行时可观测性。

## 2. CLI / TUI

### 当前状态
- 25+ 子命令，覆盖全面
- TUI 仪表盘（Ink + React）功能完整
- 交互运行时（interaction）三端共享

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **命令参数自动补全** | 为 Commander 命令添加 `--completion` 选项，输出 bash/zsh 补全脚本 | 0.5 天 |
| **TUI 斜杠命令扩展** | 在 `slash-autocomplete.ts` 中添加常用命令别名（如 `/w` → `write next`, `/s` → `status`） | 0.3 天 |
| **JSON 输出标准化** | 所有命令的 `--json` 输出使用统一类型 `CliJsonResult<T>`，包含 `status`, `error`, `data`, `meta` 字段 | 1 天 |
| **守护进程日志轮转** | 为 `inkos.log` 添加简单轮转（按大小或天数），防止单文件无限增长 | 0.5 天 |

**预期收益**: 提升 CLI 用户体验，为外部 Agent 集成提供更稳定的接口。

## 3. Studio Web 工作台

### 当前状态
- React 19 + Vite 6 + Hono 后端
- SSE 实时推送，Zustand 状态管理
- 前后端共享类型契约（`contracts.ts`）

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **API 路由拆分** | 将 `server.ts`（3600+ 行）按功能拆分为 `routes/books.ts`, `routes/chat.ts`, `routes/services.ts`, `routes/analytics.ts` 等 | 1-2 天 |
| **审计问题可视化仪表板** | 新增 Studio 页面展示过往审计问题趋势（使用 Chart.js 或 Recharts），数据来自已经存在的 `analytics` API | 2 天 |
| **伏笔状态看板** | 读取 `pending_hooks.md` 并渲染为表格/看板，标记沉寂/阻塞/回收状态 | 1.5 天 |
| **SSE 事件类型枚举标准化** | 将 `api/lib/sse.ts` 中的 20+ 事件类型提取为 TypeScript 枚举，前后端共享 | 0.5 天 |
| **模型配置页增强** | 在服务配置页添加“模型能力标签”（maxOutput, thinking, 费用等），数据来自 `providers` 内置模型卡 | 1 天 |

**预期收益**: 提升用户对写作过程的可视化理解，降低新页面开发成本。

## 4. LLM 提供商与服务管理

- 28+ 提供商静态注册
- 服务发现与模型验证完整
- API Key 管理走 `.inkos/secrets.json`

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|

| **自定义服务端点模板** | 允许用户在 Studio 配置时填写“请求模板”（如添加自定义 headers），绕过代码修改 | 1.5 天 |

**预期收益**: 降低新增提供商的代码变更成本，提升用户对多模型路由的掌控力。

## 5. 审计系统

### 当前状态
- 37 维审计，含 LLM 审计 + 确定性规则
- 审计与修订循环（默认 1 轮）
- 番外模式额外 4 维（34-37）

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **审计维度配置化** | 在 `inkos.json` 中增加 `audit.dimensions` 数组，允许启用/禁用特定维度并调整严重级别 | 1 天 |
| **审计结果历史存储** | 每次审计结果追加到 `story/audit_history.jsonl`（已有类似日志，仅需统一格式） | 0.5 天 |
| **审计问题分组统计** | 在 `analytics` 命令中增加按维度分组的问题频率统计 | 0.5 天 |
| **确定性规则可配置** | 将 `PostWriteValidator` 的规则（如破折号禁令、Marker 词阈值）提取到配置文件中 | 0.5 天 |
| **修订模式智能推荐** | 根据审计问题类型自动推荐最佳修订模式（如“OOC” → `rework`，“词汇疲劳” → `polish`） | 1 天 |

**预期收益**: 提升审计系统的灵活性和可扩展性，减少误报误修。

## 6. 文风控制

### 当前状态
- 三层文风锁：方法论注入、风格指纹、反 AI 铁律
- `style_analyzer.ts` 纯统计分析
- `post-write-validator.ts` 确定性校验

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **风格指纹可视化** | 在 Studio 新增页面展示当前书籍的风格统计（句长分布、词汇多样性曲线、Marker 词趋势），数据来自已有的 `style_profile.json` | 1.5 天 |
| **章节级风格漂移评分** | 在 `LongSpanFatigue` 分析器中增加每章风格一致性评分（与 `style_profile` 对比），记录到章节元数据 | 1 天 |
| **反 AI 规则词表自定义** | 将 `ai-tells.ts` 中的 Marker 词、疲劳词提取为可编辑的外部 JSON 文件，允许用户按题材调整 | 0.5 天 |
| **对话声线初步提取** | 在 Writer 之后增加简单的说话人识别（按引号分割），统计每个角色的句长、用词偏好，输出 `voice_traits.json` | 1 天 |

**预期收益**: 让文风控制更加透明和可定制，为后续人物声线锁定打下基础。

## 7. 伏笔与状态管理

### 当前状态
- 7 个真相文件，伏笔生命周期完整（27 个工具文件）
- Hook debt、沉寂检测、提升规则
- SQLite 记忆数据库（Node 22+）

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **伏笔回收预测** | 在 Planner 生成 ChapterMemo 时，标记“本期建议回收的伏笔”并高亮沉寂超阈值项 | 1 天（已有机架，仅需增加提示词指令） |
| **伏笔关系图谱** | 从 `pending_hooks.md` 中解析 `depends_on` 字段，生成简单的依赖图（Mermaid 格式）供 Studio 展示 | 1.5 天 |
| **真相文件变更追踪** | 每次状态更新时记录 delta（谁变了、变什么），输出 `state_changelog.jsonl` | 1 天 |
| **内存数据库扩展** | 将 SQLite 记忆从 Node 22+ 独占改为向后兼容（polyfill or optional dep），降低运行时版本要求 | 1 天 |

**预期收益**: 增强伏笔治理的可视化和可预测性，降低长篇创作中的“伏笔丢失”风险。

## 8. 导入/导出

### 当前状态
- 导出 TXT / MD / EPUB
- 导入章节（支持分章模式）
- 导入风格（从参考文本）

### 低成本扩展方向

| **Markdown 目录索引** | 导出时自动生成带目录的 Markdown（已部分支持，增强跨平台兼容性） | 0.3 天 |
| **导入格式扩展** | 支持导入 Word (.docx) 和 HTML 格式章节，使用现有的 `cheerio` 或 `mammoth` 解析 | 1.5 天 |
| **断点续导校验** | 在 `import chapters` 中加入已导入章节号校验，避免重复导入 | 0.5 天 |

**预期收益**: 提升与其他写作工具的兼容性，方便作者从其他平台迁移。

## 9. 守护进程与通知

### 当前状态
- Cron 调度（雷达/写作）
- 通知渠道：Telegram、企业微信、飞书、Webhook
- 质量门控：暂停、温度递增、失败聚类

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **通知原生支持 Discord** | 实现 `NotifyChannel` 的 Discord Webhook 通道（类似 Webhook，仅改变 payload 格式） | 0.5 天 |
| **守护进程状态 API** | 为 Studio 添加守护进程状态查询 API（当前运行/暂停/错误），无需登录服务器查看 | 1 天 |
| **写作进度推送** | 在每章完成后推送字数、评分、审计通过的摘要到已配置的通知渠道 | 0.5 天 |
| **错误警报升级** | 连续失败 × N 次时，不仅暂停书籍，还发送“需要人工介入”的通知 | 0.3 天 |

**预期收益**: 增强远程监控能力，降低无人值守时的操作风险。

## 总结：NoFusion 继续开发优先级矩阵

| 优先级 | 模块 | 扩展方向 | 预估成本 | 用户可见价值 |
|--------|------|----------|----------|-------------|
| P0 | 审计 | 审计维度配置化 + 历史存储 | 1.5 天 | ⭐⭐⭐⭐⭐ |
| P0 | Studio | API 路由拆分 + 审计面板 | 3 天 | ⭐⭐⭐⭐⭐ |
| P1 | Core | Pipeline 阶段拆分 | 2-3 天 | ⭐⭐⭐⭐ |
| P1 | 文风 | 风格指纹可视化 + 章节漂移 | 2.5 天 | ⭐⭐⭐⭐ |
| P1 | 伏笔 | 伏笔关系图谱 + 回收预测 | 2.5 天 | ⭐⭐⭐⭐ |
| P2 | CLI | 命令自动补全 + JSON 标准化 | 1.5 天 | ⭐⭐⭐ |
| P2 | 导入/导出 | PDF 导出 + Word 导入 | 3.5 天 | ⭐⭐⭐ |
| P2 | UI | Trace 查看器 + 对比视图 | 3.5 天 | ⭐⭐⭐ |
| P3 | 通知 | Discord 支持 + 错误升级 | 1 天 | ⭐⭐ |
| P3 | LLM | 模型卡 JSON 外部化 | 1 天 | ⭐⭐ |

**建议第一阶段（两周）**: 优先完成 P0 和 P1 中的“审计配置化”、Studio 路由拆分、Pipeline 拆分。这些改动风险低、成本可控，且能显著提升后续扩展的迭代速度。

**第二阶段（一个月）**: 投入伏笔可视化、文风指纹面板和 Node.js 版本兼容性优化，让小说创作的“观测指标”更丰富，减少创作盲区。

**第三阶段（两个月）**: 补齐导入导出短板，增强守护进程通知能力，实现真正可组装的 NoFusion 创作平台。
