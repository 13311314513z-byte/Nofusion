## 1. 核心引擎（Core）

### 当前状态
- `PipelineRunner` 单文件 3400+ 行，耦合度高
- Agent 列表硬编码，无插件注册机制
- 状态快照基于文件系统，回滚能力完整但缺乏索引

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **Pipeline 阶段拆分** | 将 Runner 中的 `writeNextChapter` 按阶段拆分为独立函数（`planChapter`, `composeChapter`, `writeDraft`, `auditChapter`, `reviseChapter`, `persistChapter`），各阶段通过接口组合 | 2-3 天 |
| **Agent 注册表** | 将 `agents/index.ts` 中的 `ALL_AGENTS` 改为 Map，支持通过配置或文件扫描动态添加 Agent | 1 天 |
| **运行时产物查看器 API** | 为 `story/runtime/chapter-XXXX.*` 添加只读 REST API（直接暴露 JSON 即可，无需额外渲染） | 0.5 天 |
| **LLM 客户端池优化** | 将 `getClient()` 中的 `modelOverrides` 缓存改为 LRU + 自动失效（当前无过期机制） | 0.5 天 |

**预期收益**: 降低后续维护成本，为 Agent 插件系统铺路，提升运行时可观测性。

## 2. CLI / TUI

### 当前状态
- 25+ 子命令，覆盖全面
- TUI 仪表盘（Ink + React）功能完整
- 交互运行时（interaction）三端共享

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **命令参数自动补全** | 为 Commander 命令添加 `--completion` 选项，输出 bash/zsh 补全脚本 | 0.5 天 |
| **TUI 斜杠命令扩展** | 在 `slash-autocomplete.ts` 中添加常用命令别名（如 `/w` → `write next`, `/s` → `status`） | 0.3 天 |
| **JSON 输出标准化** | 所有命令的 `--json` 输出使用统一类型 `CliJsonResult<T>`，包含 `status`, `error`, `data`, `meta` 字段 | 1 天 |
| **守护进程日志轮转** | 为 `inkos.log` 添加简单轮转（按大小或天数），防止单文件无限增长 | 0.5 天 |

**预期收益**: 提升 CLI 用户体验，为外部 Agent 集成提供更稳定的接口。

## 3. Studio Web 工作台

### 当前状态
- React 19 + Vite 6 + Hono 后端
- SSE 实时推送，Zustand 状态管理
- 前后端共享类型契约（`contracts.ts`）

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **API 路由拆分** | 将 `server.ts`（3600+ 行）按功能拆分为 `routes/books.ts`, `routes/chat.ts`, `routes/services.ts`, `routes/analytics.ts` 等 | 1-2 天 |
| **审计问题可视化仪表板** | 新增 Studio 页面展示过往审计问题趋势（使用 Chart.js 或 Recharts），数据来自已经存在的 `analytics` API | 2 天 |
| **伏笔状态看板** | 读取 `pending_hooks.md` 并渲染为表格/看板，标记沉寂/阻塞/回收状态 | 1.5 天 |
| **SSE 事件类型枚举标准化** | 将 `api/lib/sse.ts` 中的 20+ 事件类型提取为 TypeScript 枚举，前后端共享 | 0.5 天 |
| **模型配置页增强** | 在服务配置页添加"模型能力标签"（maxOutput, thinking, 费用等），数据来自 `providers` 内置模型卡 | 1 天 |

**预期收益**: 提升用户对写作过程的可视化理解，降低新页面开发成本。

## 4. LLM 提供商与服务管理

### 当前状态
- 28+ 提供商静态注册
- 服务发现与模型验证完整
- API Key 管理走 `.inkos/secrets.json`

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **提供商模型卡 JSON 外部化** | 将 `providers/*.ts` 中的 `InkosModel[]` 硬编码数据提取为 JSON 文件，支持远程更新 | 1 天 |
| **自定义服务端点模板** | 允许用户在 Studio 配置时填写"请求模板"（如添加自定义 headers），绕过代码修改 | 1.5 天 |
| **服务连通性批量测试** | 新增 `inkos doctor --all-services` 一次性测试所有已配置服务的连通性 | 1 天 |
| **模型别名系统** | 允许用户为模型设置别名（如 `fast: deepseek-v4-flash`, `creative: claude-sonnet-4`），在 CLI 和 API 中使用别名 | 0.5 天 |

**预期收益**: 降低新增提供商的代码变更成本，提升用户对多模型路由的掌控力。

## 5. 审计系统

### 当前状态
- 37 维审计，含 LLM 审计 + 确定性规则
- 审计与修订循环（默认 1 轮）
- 番外模式额外 4 维（34-37）

### 低成本扩展方向

| 调整方向 | 具体做法 | 预估成本 |
|----------|----------|----------|
| **审计维度配置化** | 在 `inkos.json` 中增加 `audit.dimensions` 数组，允许启用/禁用特定维度并调整严重级别 | 1 天 |
| **审计结果历史存储** | 每次审计结果追加到 `story/audit_history.jsonl`（已有类似日志，仅需统一格式） | 0.5 天 |
| **审计问题分组统计** | 在 `analytics` 命令中增加按维度分组的问题频率统计 | 0.5 天 |
| **确定性规则可配置** | 将 `PostWriteValidator` 的规则（如破折号禁令、Marker 词阈值）提取到配置文件中 | 0.5 天 |
| **修订模式智能推荐** | 根据审计问题类型自动推荐最佳修订模式（如"OOC" → `rework`，"词汇疲劳" → `polish`） | 1 天 |

**预期收益**: 提升审计系统的灵活性和可扩展性，减少误报误修。
